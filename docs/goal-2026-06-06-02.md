## 1. 목적 (한 줄)

비밀값 보호)은 **그대로 상속**한다.

## 3. 완료 기준 (Definition of Done) — R1~R6 (G1~G9 위에 추가)

round-4의 **G1~G9가 모두 PASS임을 전제**로(적재본이 *맞고* *적재 가능*함이 이미 증명됨),
round-5는 그 적재본이 *실행 가능*하고 *재실행 안전*하며 *부족분이 정직하게 제안됨*을 추가로 증명한다.

| # | 게이트 | PASS 조건 (검증 가능 명제) |
|---|--------|---------------------------|
| **R1** | **멱등성** | 같은 적재 스크립트를 **2회** 적용해도 2회차에 행 변경 0 (`INSERT … ON CONFLICT DO NOTHING`, 또는 의도된 `DO UPDATE`만). DRY-RUN 2회 실행으로 증명. 재실행이 중복/PK 충돌을 일으키지 않는다. |
| **R2** | **트랜잭션 안전성** | 적재 전체가 단일 트랜잭션(`BEGIN … COMMIT`) 안에서 FK 위상정렬 순으로 실행되고, 임의 단계 실패 시 **전체 롤백**된다. 부분 적재(half-load) 불가능함이 스크립트 구조로 보장됨. |
| **R3** | **실행 가능성(문법·실행)** | 모든 `.sql`이 psql로 파싱·실행 가능(문법 오류 0). 로더 스크립트가 `.env.local` 자격증명으로 연결·`\copy`/parametrized INSERT를 수행. 손으로만 만든 비실행 SQL 0. |
| **R4** | **DDL 제안 정합** | 신규 엔티티 제안(CREATE TABLE/ALTER)이 (a) 라이브 `t_*` 네이밍·타입·코드 컨벤션과 정합, (b) 기존 스키마와 충돌·중복 0, (c) 정규화 위반 0(해당 GAP을 무손실 표현), (d) 마이그레이션 적용 순서와 영향(기존 행/FK) 명시, (e) **search-before-mint**(기존 테이블/컬럼으로 해결 가능한지 먼저 입증)를 통과. |
| **R5** | **라이브 DRY-RUN 통과** | lead 승인 하에 `BEGIN … (적재 시도) … ROLLBACK`을 라이브에서 실제 실행 시 제약 위반(타입/길이/NOT NULL/CHECK/FK/PK 중복) **0건**. 위반 시 위반 목록·원인 보고. (round-4 G6 보류분의 실집행.) |
| **R6** | **생성·검증 분리(독립성)** | SQL/DDL 산출(builder·proposer)과 게이트(validator)가 **분리된 에이전트**로 수행되고, 검증이 실행본/제안의 결함을 1건 이상 실제로 적발·정정한 이력이 있다(자기 승인 금지). round-4 G9의 round-5 연장. |

> **완료 선언 형식:** `03_validation/`의 round-5 최종 판정서(`load-execution-gate.md`)에 **G1~G9 carry-forward
> 확인 + R1~R6 각 PASS/FAIL**을 표로 명시한다. 하나라도 FAIL이면 미완료이며 반복 개선 루프로 되돌아간다.

## 4. 범위 (In / Out)

**In scope**
- 입력: round-4 GO 적재본(`09_load/_assembled/`, `_assembled_price/`) + 그 권위 게이트 판정서.
- 산출: ① **멱등 적재 SQL 스크립트**(트랜잭션 래핑·FK순·`ON CONFLICT` 가드) + **적재 로더 스크립트**
  (psql/Python, `.env.local` 자격증명), ② **신규 엔티티 DDL 제안서**(GAP·차단 해소용 CREATE/ALTER),
  ③ **라이브 롤백 DRY-RUN 검증 리포트**(멱등성·제약 무위반 실증).

**Out of scope (Non-goals)**

- **DDL의 실제 적용**(`CREATE`/`ALTER` 라이브 실행) — 제안서까지만. 적용은 인간 승인.
- 매핑 규칙 자체 재설계(round-2/3) · 수량구간 할인(round-1) · 위젯/프런트/주문 로직.
- round-4가 GO하지 않은 매핑의 적재 — GO 판정 없는 테이블은 round-5 진입 불가.

## 5. 타겟·DDL 제안 경계 [HARD]

**적재 대상은 round-4와 동일한 `t_*` 화이트리스트로만 한정한다**(goal-...-01 §5 상속).
DDL 제안 역시 **`t_*` 도메인 내 신규 엔티티/컬럼/제약**으로만 한정한다 — Django/비-`t_` 영역은 제안 대상이 아니다.

**DDL 제안이 정당한 경우 (GAP/차단을 무손실 표현하기 위해서만):**
- round-4가 GAP/BLOCKED로 분리한 항목 — 예: 박 2단 룩업(면적→분류→가격 중간키), goods-pouch 비치수
  size(47상품), sticker 형상 enum 축 귀속, 책등 param 슬롯 부재, addon template 부재.
- 코드행만으로 해소 불가하고, 기존 테이블/컬럼 재사용으로도 무손실 표현이 불가능함이 **입증된** 경우.

**DDL 제안이 금지인 경우:**
- 기존 테이블/컬럼/코드행으로 해결 가능한데 편의를 위해 새로 만드는 경우(search-before-mint 위반).
- "있으면 편할 것 같아서"식 추측 엔티티 — GAP에 실제로 묶이지 않은 제안.

## 6. HARD 제약 (round-4 상속 + round-5 추가)

**round-4에서 상속(불변):**
1. **`t_*` 외 테이블 매핑·적재 금지.**
2. **실제 `COMMIT` 금지.** 읽기전용 `SELECT` + 롤백전용 DRY-RUN(`BEGIN…ROLLBACK`)만. `COMMIT`·파괴적 SQL 라이브 실행 금지.
3. **추측·발명·dodge·침묵폴백 금지.** 코드값·존재·등록 여부는 라이브 실재만 권위.
4. **비밀값 노출 금지.** `RAILWAY_DB_*`는 `.env.local`(chmod 600·gitignore)에만. `_workspace`·stdout 금지.
5. **라이브 반복 조회 최소화.** 등록/NULL/존재 판정은 라이브 권위(추출본 stale 주의).

**round-5 추가:**
6. **DDL 직접 적용 금지.** 신규 엔티티는 **제안서**(CREATE/ALTER + 근거 + 적용순서)까지만. 라이브 `CREATE`/`ALTER` 실행 금지.
7. **멱등성 필수.** 적재 SQL은 재실행 안전해야 한다. `INSERT`는 `ON CONFLICT` 가드 없이 산출하지 않는다(자연키/PK 명시 필수).
8. **트랜잭션 원자성 필수.** 적재 스크립트는 단일 트랜잭션·전부 아니면 전무. 부분 커밋 경로 금지.
9. **search-before-mint.** DDL/코드행을 제안하기 전, 기존 라이브 구조로 해결 가능한지 먼저 탐색·입증한다(round-4 `SIZ_000506` 중복 mint 사건의 교훈).

## 8. 작업 방법 (명령 우선)

**적재 실행본 빌드 (dbm-load-builder, round-5 확장):**
1. round-4 GO 적재본(`09_load/_assembled*/load/*.csv` + manifest)을 **유일 입력**으로 삼는다(재매핑 금지).
2. FK 위상정렬 순서(manifest 권위)대로 테이블별 **멱등 `INSERT … ON CONFLICT`** 문을 생성한다.
   자연키/PK는 라이브 제약에서 읽어 `ON CONFLICT (cols)` 타겟으로 명시한다.
3. 전체를 **단일 트랜잭션**(`BEGIN; … ; COMMIT;`)으로 래핑하고, 코드행 선적재·신규 DDL 적용을 step 00 이전 순서로 배치한다.
4. **적재 로더**(psql `\copy`+SQL 또는 Python)를 산출 — `.env.local`에서 자격증명을 읽고, 기본은 DRY-RUN(롤백) 모드, `--commit`은 인간 승인 플래그로만.
5. 산출은 재현 가능 스크립트로(손편집 금지), per-row provenance 유지.

**신규 엔티티 DDL 제안 (dbm-ddl-proposer):**
1. round-4 `blocked-and-gaps.md`의 GAP/BLOCKED를 입력으로, 각 항목이 **기존 t_*로 무손실 표현 불가**임을 먼저 입증(search-before-mint).
2. 무손실 표현에 필요한 **최소** 신규 엔티티(테이블/컬럼/제약/코드)를 라이브 `t_*` 컨벤션 정합으로 설계한다(정규화 위반·중복 0).
3. CREATE/ALTER DDL + 적용 순서 + 영향분석(기존 행·FK·백필) + 해당 GAP이 어떻게 닫히는지를 제안서로 산출.
4. 직접 적용하지 않는다 — 제안서는 인간 승인 게이트로 간다.

**라이브 DRY-RUN 멱등성 증명 (dbm-validator, round-5 게이트):**
- 단일 트랜잭션에서 적재본 적용을 시도하고 **반드시 ROLLBACK** — 제약 위반 0 확인(R5).
- 같은 트랜잭션 패턴을 2회 반복(또는 1회 안에서 2회 INSERT)해 멱등성 확인 — 2회차 행 변경 0(R1).
- 위반·비멱등 발견 시 원인별 분류해 builder/proposer로 라우팅, 개선 루프.

**산출물 컨벤션:**
- round-5 산출은 `_workspace/huni-dbmap/09_load/_exec/`(상품마스터)·`_exec_price/`(가격) + `11_ddl_proposals/`(DDL 제안서).
- 파일명: `{NN}_{table}.sql`(적재), `apply.sh`/`load.py`(로더), `ddl-proposal-{gap}.sql`+`.md`(제안). 최종 판정은 `03_validation/load-execution-gate.md`.

## 9. 진행상태 보존·재현성·되돌리기 — round-4 상속

- **HANDOFF 갱신:** 세션 종료 시 round-5 HANDOFF에 *다음 시작점·미해결 블로커·이번 세션 결정·건드리지 말 것* 기록.
- **세션 시작 루틴:** (a) 본 GOAL → (b) goal-...-01 → (c) 최신 게이트 판정서 순.
- **멱등성:** 스크립트는 같은 입력에 같은 출력. 손작업 결과를 권위로 삼지 않는다.
- **되돌리기:** 산출물은 git 커밋 추적(`.env.local` IGNORED 확인 후).

## 10. 인간 승인 체크포인트 (자율 진행 금지)

1. **라이브 DRY-RUN 실행 직전** — 롤백전용이라도 쓰기 트랜잭션이므로 lead가 승인 후 1회.
2. **실제 `COMMIT`(영구 적재) 직전** — 본 GOAL 종착점 너머. R1~R6 + G1~G9 PASS 산출물 제시 후 승인 요청.
3. **신규 엔티티 DDL 적용 직전** — 제안서 검토 후 후니가 라이브에 적용. 제안과 적용은 분리.
4. **코드행 선적재 적용** — 라이브에 없는 코드값을 후니가 등록(placeholder 해소).

## 11. 실패 모드 처리 — round-4 상속 + 추가

- **조기 종료 방지:** G1~G9 carry-forward + R1~R6 전부 PASS 전까지 완료 선언 금지. 컨텍스트 압박은 HANDOFF로 대응.
- **멱등성 회귀:** DRY-RUN 2회차에 행 변경이 잡히면 비멱등 — `ON CONFLICT` 누락/잘못된 충돌키. 즉시 차단·수정.
- **DDL 제안 남발 방지:** 같은 GAP에 2개 이상 제안이 나오면 최소안으로 수렴. search-before-mint 재확인.
- **차단의 정직한 표기:** 적재 불가·제안 보류는 "처리 완료"로 포장 금지. 사유·해소 조건 명시.

## 12. 비목표 재확인 (오해 방지)

- 본 트랙은 **적재 실행 산출물 + 적재 가능성 실증**이지, 라이브 적재 실행기가 아니다(COMMIT은 인간 승인).
- DDL 제안은 **GAP을 정직하게 닫기 위한 수단**이지, 스키마를 마음대로 바꾸는 권한이 아니다(제안·승인·적용 분리).
- 이 GOAL은 **하네스 확장의 입력**이다. round-5 에이전트/스킬 설계는 본 문서 확정 후 진행한다.
