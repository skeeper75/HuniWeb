임베드 = 웹컴포넌트(Custom Element) + Shadow DOM

요구사항이 ① 여러 사이트에서 사용 ② 이질감 없이 녹아듦 ③ (그러면서) 안 깨짐 이 셋을 동시에 만족해야 하므로, 이 조합이 사실상 유일한 답입니다. 아래는 그 이유와 "어떻게 녹아드는가"를 중심으로 정리한 것입니다.

왜왜 이 방식이 답인가 — 세 방식 비교

핵심 딜레마: "녹아듦"과 "안 깨짐"은 보통 서로 상충합니다. 남의 디자인을 다 받아들이면 자연스럽지만 사이트마다 깨지고, 다 차단하면 안 깨지지만 동떨어져 보입니다.

한 줄 요약 Shadow DOM은 "방음·단열된 방"입니다. 남의 집(사이트) 소음(CSS)이 안 들어오고 내 소리도 안 새나갑니다. 대신 창문(CSS 변수·상속)은 열 수 있어서, 바깥 햇빛(그 사이트의 글꼴·색)은 골라서 받아들입니다. 그래서 여러 사이트에서 안 깨지면서도 그 사이트처럼 보일 수 있습니다.

★핵심어떻게 이질감 없이 녹아드는가 — 4단 전략

"자동으로 어디서나 native하게 보이는 마법"은 없습니다. 대신 위에서 아래로 4단계를 쌓으면, 대부분의 사이트에서 설정 없이도 자연스럽고, 부족하면 관리자가 마지막에 손봅니다.

0토대
Shadow DOM = 깨지지 않게 하는 보호막
이게 깔려 있어야 1~4단이 의미가 있습니다. 보호막이 없으면(=Light DOM) 어떤 사이트는 멀쩡하고 어떤 사이트는 그 사이트의 * { box-sizing }·CSS 리셋·button{} 규칙 때문에 위젯이 깨집니다. "여러 사이트에서 일관되게"가 불가능해지죠. Shadow DOM이 그 충돌을 원천 차단합니다. (대신 위젯은 자기만의 작은 리셋 CSS를 들고 들어갑니다.)
1공짜
글꼴·글자색·줄간격을 그냥 물려받기
:host { font: inherit; color: inherit; } 한 줄. 텍스트가 즉시 그 사이트의 글처럼 보입니다. 비용 0, 효과 최대 — 사람 눈은 글꼴·글자색이 같으면 "같은 집안"으로 인식합니다.
2자동
그 자리에 적용될 스타일을 "읽어서" 자동 채택 웹컴포넌트의 결정적 강점
웹컴포넌트는 그 페이지 DOM 안에 실제로 박혀 있습니다. 그래서 getComputedStyle(요소)로 "이 위치에 페이지가 적용했을 글꼴·글자크기·기본 텍스트색·링크색"을 읽어낼 수 있습니다. 이 값을 위젯 테마의 기본값으로 삼으면, 사이트가 아무 설정을 안 해도 그 사이트 톤을 자동으로 입습니다. (강조색은 근처 버튼·링크 색을 샘플링) — iframe은 분리돼 있어 이게 불가능합니다. 이게 웹컴포넌트를 택하는 결정적 이유입니다.
3약속
사이트가 디자인 토큰을 주면 그걸 우선
잘 만든 사이트는 --brand-primary 같은 CSS 변수를 갖고 있습니다. 있으면 그걸 씁니다: color: var(--huni-primary, var(--brand-primary, 기본값)). CSS 변수는 Shadow 경계를 통과하므로 가능합니다.
4수동
빌더에서 관리자가 직접 지정/고정
자동(1~3)으로 부족하거나, 특정 사이트에 딱 맞추고 싶으면 위젯 빌더에서 색·폰트·모서리·버튼 스타일을 직접 지정하거나 "페이지 따라가기"를 선택합니다. 사이트별 위젯이면 사이트별로 테마를 다르게 줄 수 있습니다.
동작 순서(우선순위) 4(관리자 고정) ＞ 3(사이트 토큰) ＞ 2(자동 감지) ＞ 1(공짜 상속) ＞ 위젯 기본값. 위에서 값이 있으면 그걸 쓰고, 없으면 아래로 내려가며 채웁니다. 대부분 1·2단에서 이미 "그 사이트 같다"가 됩니다.

    const autoColor = probe.color;

    root.innerHTML = `

`;

정리하면: 0이 안전을 보장하고, 1·2가 대부분의 "녹아듦"을 공짜로 해결하고, 3·4가 마무리를 합니다.

+ 레이아웃도 자연스럽게 흐릅니다 (이미지 오른쪽 배치)

웹컴포넌트는 그냥 일반 블록 요소입니다. 그래서 상품 메인이미지 오른쪽 컨테이너에 넣으면 그 칸 너비에 맞춰 자연 배치되고, 내용이 길어지면 페이지가 알아서 늘어납니다.
iframe과의 차이iframe이면 높이를 미리 알 수 없어 스크롤바가 생기거나 잘리고, postMessage로 높이를 계속 주고받아 맞춰야 합니다. 웹컴포넌트는 이 고생이 없습니다 — "녹아듦"이 디자인뿐 아니라 레이아웃에서도 성립.

설치여러 사이트에 까는 법 — 스니펫 한 줄

각 사이트는 아래 두 줄만 붙이면 됩니다. 어떤 위치에든(상품 이미지 오른쪽 칸) 둘 수 있습니다.

<script src="https://cdn.huni.../widget.js" async></script>
<huni-widget data-widget-id="WGT-123"></huni-widget>
  • 프레임워크 무관: customElements.define는 브라우저 표준이라 React·Vue·순수HTML·카페24·쇼피파이 어디서든 동일하게 동작.
  • 한 페이지에 여러 개도 가능(각 인스턴스가 자기 Shadow 방을 가짐).
  • 버전·자동업데이트: 스크립트 URL을 한 곳에서 관리 → 전 사이트 일괄 갱신.
  • CORS: 위젯 JS가 우리 API를 부르므로 허용 도메인 화이트리스트 필요(아래 운영 이슈).

정직이 방식의 한계 (알고 가야 할 것)

유일한 방법인가? 문자 그대로 유일하진 않습니다(Light DOM도 기술적으로 가능). 하지만 "여러 미지의 사이트 + 자연스러움 + 안 깨짐"을 동시에 만족하는 건 이 조합뿐이라, 사실상 유일한 현실적 선택입니다.
  • 자기 리셋 CSS 지참: 사이트의 전역 리셋이 안 들어오므로, 위젯이 자기만의 최소 리셋·기본 스타일을 들고 들어가야 함(=오히려 일관성에 유리).
  • 일부 서드파티 위젯(전역 CSS가 필요한 라이브러리)은 Shadow 안에서 까다로울 수 있음 → 대표 사례인 Edicus 에디터는 어차피 자체 iframe으로 띄우므로 영향 없음.
  • 브라우저 지원: 모든 최신 브라우저가 Shadow DOM·Custom Element 지원(IE만 불가 — 무시 가능).
  • 완전 자동 100%는 환상: 1~3단으로 대부분 해결되지만, 디자인이 까다로운 사이트는 4단(빌더 테마 지정)으로 마무리 필요. 그래서 빌더에 테마 입력을 꼭 둡니다.

나머지나머지 기술 이슈 (이 기준 위에서)

가격 계산 + 【보안】 견적 위변조 방지

가격 엔진(evaluate_price())은 수십 개 테이블을 읽는 서버 파이썬 → JS로 못 옮김. 선택이 바뀔 때마다 공개 가격 API로 서버 왕복.

디바운스빠르게 클릭 시 매번 쏘지 말고 ~250ms 모아 1회. 로딩 표시 필요.
가격 위변조 (중요)화면 숫자를 페이지가 받아 장바구니에 넣으면 사용자가 가격 조작 가능.
추천가격 API가 서명된 견적 토큰({상품,선택,가격,만료}를 서버 비밀키로 HMAC)을 함께 반환 → 실제 주문 시스템이 그 토큰을 재검증. 화면 가격이 아니라 토큰이 진실.

파일 업로드 + Edicus 에디터 — "artwork 참조"로 추상화

상품에 file_upload_yn/editor_yn 플래그만 있고 구현은 없음(models.py:515-516).

  • 업로드: Railway는 파일 휘발성 → 오브젝트 스토리지(S3/R2) + presigned URL로 브라우저→S3 직업로드(큰 PDF가 서버 우회). PDF·용량·규격 검증 이슈.
  • Edicus: 자체 iframe 에디터로 띄움(Shadow 영향 없음). 별도 미니 프로젝트.
핵심 추상화업로드든 에디터든 결국 "artwork 참조 1개"(파일키/디자인ID)를 만들어 출력 JSON에 넣는 것. 교체 가능한 모드로 설계 → 1차는 업로드만, Edicus는 나중에.

운영 이슈

  • 드래프트/퍼블리시 + 버전: 저장 즉시 라이브 X → 초안/발행 분리, 발행 시 스펙 캐시 무효화.
  • 캐시/성능: 스펙 JSON은 CDN 캐시(발행 때만 무효화). 위젯 JS는 가볍게(프레임워크 의존 X).
  • CORS 화이트리스트: 위젯이 실릴 사이트 도메인만 API 허용 + 가격 API 레이트리밋.
  • 상품당 위젯 1 vs N: 같은 상품을 다른 사이트/페이지에 다른 구성·테마로? → ID 체계 결정.
  • 접근성/키보드: 기존 드롭다운 키보드 수준을 버튼·체크박스에도 유지.
