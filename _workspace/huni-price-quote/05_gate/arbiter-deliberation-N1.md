## 0. 한 줄 결론 (TL;DR)

N-1은 검증자가 지목한 "단가값 불일치 + 임포지션 변환 부재" **둘 다 맞지만, 둘 다 더 깊은 단일 근본원인의 증상**이다.
근본원인 = **권위 `디지털인쇄비` 매트릭스의 2개 축이 라이브 적재에서 통째로 소실/오해석됐다**:
1. **도수축 소실** — 권위 14열(흑백·칼라CMYK·별색5종 × 단/양면) 중 라이브 DIGITAL comp엔 **흑백 2열만** 적재. 칼라CMYK 행은 **어느 comp에도 부재**.
2. **수량행 의미 오해석** — 권위 단가행 라벨 = **출력매수(판수)**인데, 라이브 엔진은 그 행을 **주문수량(qty)** 으로 직접 룩업(임포지션 변환 0).

→ 검증자의 "단가 470 vs 800"은 **판형(300x625 vs 국4절)·도수(흑백 vs 칼라) 둘 다 어긋난 우발적 비교**였다. 진짜 골든 매칭 단가(국4절 흑백단면 25행=350)는 권위와 byte-identical로 일치한다. **단가값 자체는 정확히 적재됐다 — 틀린 건 "어느 열(도수)·어느 행 의미(판수 vs 수량)"이다.**

## 1. 근본원인 규명 (라이브 실측 + 권위 역추적)

### 1.1 검증자의 470 vs 800 — 정밀 해부

검증자(recompute-log §2)는 `PRD_000017 / plt_siz_cd=SIZ_000077 / POPT_000001 / PROC_000004 / qty=25` → 인쇄비 11,750(=470×25)을 얻고, 골든 800(`디지털인쇄비:D16`)과 불일치 판정했다. 라이브 직접 실측으로 470의 정체를 역추적:

| 실측 사실 | 라이브 값 | 권위 대조 |
|-----------|----------|-----------|
| `SIZ_000077` 정체 | **"300x625"** (판형) | 권위 국4절=**316x467**(=`판걸이수`:G열 출력판형). 300x625는 국4절 **아님** |
| `POPT_000001` 정체 | **"단면"** (t_prt_print_options) | 면 축만 구분 — **도수(흑백/칼라/별색) 안 풀림** |
| `PROC_000004` 정체 | **"디지털"** (인쇄방식, clr_cd NULL) | 도수 아님 |
| DIGITAL_S1 `clr_cd` | **전 행 NULL** | 도수 차원 미충전 |
| DIGITAL_S1 print_opt distinct | **POPT_000001/2 (단/양면)만** | 권위 14열 중 면 2분기만 |

→ 검증자가 쓴 SIZ_000077(300x625)은 골든 케이스1의 사이즈(100x150 → 국4절 출력)와 **무관한 판형**이다. 그리고 그 판형의 흑백단면 25행=470이 골든 칼라단면(국4절)과 비교돼 11,750이 나온 것. **판형도 도수도 어긋난 비교.**

### 1.2 진짜 골든 매칭 — 단가값은 정확하다

라이브 `SIZ_000499`(="**316x467**"=국4절) 단면 시퀀스를 권위 `디지털인쇄비` 흑백단면(B열)과 전수 대조:

```
            라이브 SIZ_000499 단면   권위 디지털인쇄비 흑백단면(B열)
출력매수 1 :       3000              3000  [B4]   ✅
출력매수 2 :       2000              2000  [B5]   ✅
출력매수 3 :       1600              1600  [B6]   ✅
출력매수 5 :       1200              1200  [B8]   ✅
출력매수 7 :        800               800  [B10]  ✅
출력매수 25:        350               350  [B16]  ✅
...전 53행 byte-identical
```

→ **라이브 단가 = 권위 흑백단면 verbatim. 값 결함 0.** (round-21 "단가행 값=가격표 verbatim·값결함0" 패턴 재확인.)
검증자의 "라이브 단가표가 골든 출처와 체계적 불일치"(recompute-log §2-1)는 **부정확** — 체계적 불일치가 아니라, 골든이 **칼라단면(D열)** 을 인용했는데 라이브엔 **흑백단면(B열)만** 적재돼 비교 대상 자체가 없었던 것이다.

### 1.3 도수축 소실 — 칼라CMYK 행이 어디에도 없다

권위 `디지털인쇄비` 매트릭스 = **7도수 × 2면 = 14열**:
흑백·칼라(CMYK)·별색화이트·별색클리어·별색핑크·별색금색·별색은색 × 단/양면.

라이브 comp 분포(실측):
| 권위 도수 | 라이브 comp | 상태 |
|-----------|------------|:--:|
| 흑백(1도) | `COMP_PRINT_DIGITAL_S1/S2` (clr_cd NULL·POPT 단/양) | ✅ 적재(흑백값) |
| **칼라(CMYK)** | **— (어느 comp에도 부재)** | 🔴 **소실** |
| 별색 5종 | `COMP_PRINT_SPOT_{WHITE,CLEAR,PINK,GOLD,SILVER}_S1/S2` | ✅ 분리 적재 |

→ **DIGITAL comp는 흑백값만 담고 있다.** 골든 케이스1(칼라CMYK 단면)은 라이브에서 **표현 자체가 불가능**하다. 도수 선택이 가격축인데(authority-golden §1.1), DIGITAL_S1의 use_dims·단가행에 도수 차원이 없다(clr_cd NULL·print_opt는 면만). 이는 dimension-mapping-matrix §1이 DIGITAL_S1을 "✅ 3원 일치"로 본 것이 **불완전 판정**임을 뜻한다 — 면·판형·수량은 맞지만 **도수축 전체 누락을 못 봤다**(검증자도 N-1에서 단가값 차원으로만 접근, 도수 소실은 미포착).

> **dodge 방지 자기검토:** "DIGITAL이 흑백만"이라는 판정의 반증 가능성 — 칼라가 별도 PROC나 별도 comp로 있을 수 있다. 실측으로 COMP_PRINT% 전수 확인 → DIGITAL(흑백)·SPOT 5종(별색)뿐, 칼라 comp 0. PROC_000004=디지털 단일(도수 아님). **반증 기각 — 칼라 부재 확정.**

### 1.4 수량행 의미 오해석 — "출력매수" 라벨을 "주문수량"으로 룩업

권위 산정식(authority-golden §1.3·[공식집:행5~6]):
```
출력매수(판수) = ceil(주문수량 / 판걸이수)      (판걸이수=사이즈별 고정, 판걸이수 시트)
인쇄비 = [출력매수 행단가] × 출력매수
```
권위 `디지털인쇄비` **수량(국4절) 열의 행 라벨(1,2,3…25…)= 출력매수(판수)**이지 주문수량이 아니다. (100x150 엽서 판걸이=8 → 200매 주문 = 판수 25 → 25행 룩업 × 25 = 인쇄비.)

라이브 엔진 실측(engine-contract §3.2·price_views.py:107 `_tier_order_val`):
- `dim=="min_qty"` → **주문 qty를 그대로** 티어값으로 사용. 임포지션 변환 grep **0건**(pricing.py·price_views.py 둘 다).
- `t_siz_sizes`에 **판걸이수 저장 컬럼 부재**(work/cut/margin/impos_yn만 — 판수 분모 없음).

→ 라이브 단가행 격자(1…25…200…1000000)는 권위 **출력매수** 격자를 그대로 복사했는데, 엔진은 그 행을 **주문수량**으로 룩업한다. 200매 주문 → 엔진은 min_qty=200행(라이브 180)을 룩업 → 인쇄비 36,000(recompute-log §2 qty=200 케이스가 정확히 이 발화). 권위 정답(판수25 × 25 = 칼라면 20,000·흑백면 8,750)과 **구조적으로 다른 행**을 본다.

### 1.5 근본원인 종합 — 3겹

| # | 결함 | 증상 | 권위 근거 | 심각도 |
|---|------|------|-----------|:--:|
| **N-1a** | **도수축 소실** (칼라CMYK comp 부재·DIGITAL=흑백만) | 칼라 선택 표현 불가·골든(칼라) 재현 불가 | `디지털인쇄비` 14열 중 흑백 2열만 적재 | 🔴 High (돈크리티컬·전 엽서·상품권·슬로건) |
| **N-1b** | **수량행 의미 오해석** (출력매수 격자를 qty로 룩업·임포지션 변환 부재) | 모든 합산형 인쇄비/코팅비가 판수 아닌 주문수량으로 룩업 → 전 수량대역 오산 | [공식집:행5~6]·라이브 판걸이수 컬럼 부재·엔진 grep 0 | 🔴 High (돈크리티컬·전 합산형) |

> **단가값 자체는 결함 아님**(흑백값 verbatim). N-1의 본질은 **차원 모델 결함**(도수축 부재 + 수량축 의미 오정렬)이다. 이건 [[dbmap-schema-design-intent-first]]의 "같은 값을 잘못된 t_*/차원 의미에 매핑" 위험의 정확한 사례다.

## 2. 판걸이수→출력매수 변환의 귀속 심의 (트레이드오프)

핵심 질문: 변환(출력매수=ceil(qty/판걸이수))을 **누가** 책임지는가. 세 후보를 펼친다.

### 후보 ⓐ — 엔진/앱 런타임 계산 ([[dbmap-compute-in-app-db-stores-lookup]] 정합)

판수=임포지션/네스팅 런타임 계산·DB는 룩업만(off-grid ceiling과 동일 철학). 엔진이 selections의 사이즈→판걸이수 룩업→ceil(qty/판걸이)→그 값을 인쇄비/코팅비 comp의 min_qty 티어로 주입.

- **장점:** 권위 도메인 모델([[...compute-in-app...]])과 정합. 판걸이수는 사이즈별 고정 소수 데이터(7행)라 DB 룩업 1회면 충분. 단가행 격자(=출력매수)를 **그대로 두고** 의미를 회복(재적재 0).
- **단점:** **엔진(pricing.py) 코드 수정 필요** → webadmin은 개발자 GitHub 배포·read-only oracle([[dbmap-axis-staged-load-round22]] HARD). 우리가 못 고침 → 개발자 백로그. 또한 `t_siz_sizes`에 판걸이수 컬럼 **신설(DDL)** 필요(현재 부재) → 사이즈별 판걸이수 적재.
- **영향 범위:** 합산형 전체(엽서·상품권·슬로건 10상품군) 인쇄비·코팅비 동시 정상화. 면적매트릭스(실사·아크릴)는 무관(수량축 없음).

### 후보 ⓑ — 단가행을 "주문수량 격자"로 재적재 (변환을 데이터로 흡수)

권위 출력매수 격자를 버리고, 각 사이즈별로 "주문수량 → 인쇄비" 를 미리 전개해 min_qty=주문수량 격자로 단가행 적재(엔진은 현행 qty 직접 룩업 유지).

- **장점:** 엔진 수정 0(개발자 의존 없음). 우리 트랙(dbm-load-execution)으로 닫힘.
- **단점 (치명):** ① 판걸이수가 **사이즈마다 다름**(73x98=18·100x150=8·148x210=4) → 같은 도수·면이라도 **사이즈별로 단가행 격자를 따로 전개**해야 함(조합 폭발). ② 주문수량은 연속(1~1000000)인데 출력매수 격자는 이산 → 주문수량격자로 변환 시 **비선형 구간 경계가 사이즈마다 달라짐**(예 판걸이8: 주문1~8→판수1, 9~16→판수2…). 격자 수천 행. ③ **권위 단가표 구조를 파괴** — 가격표가 출력매수 기준인데 주문수량 기준으로 미리 곱해 박으면 가격표↔DB 추적성 상실([[dbmap-print-domain-recipe-philosophy]] "실무진이 가격표로 정리한 것 존중" 위배). ④ 가격표 개정 시 재전개 지옥.
- **판정:** **권고 안 함.** 돈크리티컬 영역에서 권위 구조 파괴 + 유지보수 부채. 변환을 데이터로 흡수하는 건 안티패턴.

### 후보 ⓒ — 호출측(위젯/price_simulate)이 변환 후 보정 qty 주입

위젯/뷰가 사이즈→판걸이수 룩업→출력매수 계산→엔진엔 `qty=출력매수` 주입, 최종 ×는 호출측이.

- **장점:** pricing.py 본체 미수정(price_views.py 또는 위젯 어댑터만).
- **단점:** ① 인쇄비/코팅비만 출력매수 기반이고 용지비(=출력매수+5장)·후가공비(=주문수량)는 분모가 달라 **단일 qty 주입으로 못 풂**(comp마다 다른 수량 의미). ② price_simulate도 webadmin(개발자 영역). ③ 가격 책임이 엔진 밖으로 새어 위젯/뷰/주문 재검증 3곳에 변환 중복 → 불일치 위험(돈크리티컬). 
- **판정:** **권고 안 함**(부분적 보조는 가능하나 단독 해법 불가).

### 권고 — **ⓐ (엔진/앱 런타임 계산) + DDL(판걸이수 컬럼) — 개발자 백로그**

근거: ① 권위 도메인 모델([[...compute-in-app...]]) 정합 — 판수는 계산값이지 저장 단가가 아니다. ② 단가행 격자(출력매수)를 **그대로 보존**(재적재 0·byte-identical 권위 유지). ③ 합산형 전체 1곳 수정으로 정상화(comp별 분모 차이도 엔진이 comp 유형별로 처리 가능: 인쇄/코팅=출력매수·용지=출력매수+5·후가공=주문수량). ④ ⓑ의 격자 폭발·권위 파괴를 회피.

**단, ⓐ는 webadmin 코드(개발자 GitHub·read-only) + DDL이 걸려 우리가 직접 적용 불가** → 개발자 백로그로 인계. 우리가 즉시 할 수 있는 건 **N-1a(도수축) 데이터 재적재**(아래 §3) — 이건 엔진 무관·우리 트랙으로 닫힘.

## 3. 정립 방안 (트랙별·트레이드오프·컨펌)

N-1을 두 갈래로 분리한다: **데이터 결함(N-1a 도수축)** = 우리 트랙 가능 / **구조·엔진 결함(N-1b 수량축)** = 개발자 백로그.

### 방안 R-1 — 칼라CMYK 인쇄비 단가행 적재 (N-1a 해소)

| 항목 | 내용 |
|------|------|
| **무엇** | 권위 `디지털인쇄비` 칼라(CMYK) 단/양면 열(D/E열) → 라이브 칼라 인쇄비 comp 단가행 |
| **어느 t_\*** | `t_prc_component_prices` (단가행) + 도수 차원 표현 결정(아래 옵션) |
| **트랙** | dbm-load-execution(멱등 UPSERT·R1~R6) ← dbm-price-arbiter 심의 GO 후 인간 승인 |
| **선결 컨펌** | **도수를 어느 차원으로 표현?** (Q-DGP-COLORAXIS — 아래) |

**도수 표현 3옵션 (컨펌 필요):**
- ⒜ **별도 comp** `COMP_PRINT_DIGITAL_COLOR_S1/S2` 신설 (현 SPOT 5종과 동일 패턴) — 흑백/칼라/별색이 comp 단위로 분리. 옵션이 comp 택1로 풀림. **권고**(현 별색 적재 패턴과 동형·search-before-mint로 SPOT 패턴 재사용).
- ⒝ DIGITAL_S1에 `clr_cd` 차원 추가 + 칼라 행 append (흑백 clr_cd=흑백코드·칼라 clr_cd=CMYK코드) — 한 comp에 도수축. use_dims에 clr_cd 추가 필요(엔진 NON_QTY_DIMS에 clr_cd 있는지 확인 필요 — engine-contract §3 NON_QTY_DIMS에 clr_cd **없음** → 엔진이 clr_cd 매칭 안 함 → **불가**·엔진 수정 동반).
- ⒞ print_opt_cd로 흑백단/칼라단/흑백양/칼라양 4분기 — 의미 오염(print_opt=면인데 도수 섞음·[[dbmap-schema-design-intent-first]] 위배). **비권고.**

→ **권고 = ⒜** (엔진 무관·SPOT 패턴 동형·우리 트랙으로 닫힘). ⒝는 엔진이 clr_cd를 비수량 차원으로 안 봐서 개발자 의존 발생.

> **트레이드오프:** ⒜는 옵션 레이어(L2)에서 "도수" 옵션이 comp 택1(DIGITAL흑백 vs DIGITAL칼라 vs SPOT5)로 풀려야 함 → CPQ option_items 재배선 동반(round-6 트랙). 단가행 적재(R-1)와 옵션 배선(별건)을 분리 인계.

### 방안 R-2 — 임포지션 변환 + 판걸이수 컬럼 (N-1b 해소)

| 항목 | 내용 |
|------|------|
| **무엇** | ① `t_siz_sizes`에 판걸이수 컬럼 신설(국4절·3절 2개·`판걸이수` 시트 F/G/H열) ② 사이즈별 판걸이수 적재 ③ pricing.py에 출력매수=ceil(qty/판걸이수) 변환 + comp 유형별 분모(인쇄/코팅=판수·용지=판수+5·후가공=주문수량) |
| **어느 t_\*** | `t_siz_sizes`(DDL+적재) · `pricing.py`(코드·개발자) |
| **트랙** | ①② = dbm-ddl-proposer(컬럼 제안) + dbm-load-execution(적재) ③ = **webadmin 개발자 백로그**(read-only oracle·우리 수정 금지) |
| **선결 컨펌** | Q-IMPOSITION(변환 위치 — §2 권고 ⓐ 비준) · 판걸이수 컬럼 2개(국4절/3절) 데이터 모델 확정 |

> **트레이드오프:** R-2 ③은 우리가 못 고침(개발자 영역). 그러나 ①②(판걸이수 컬럼+데이터)는 우리가 준비 가능 → 개발자가 ③만 붙이면 됨. **분리 인계로 병목 최소화.** 단, ③ 미반영 동안 **합산형 인쇄비 전부 오산**(주문수량≈판수 우연 일치 구간만 맞음) → 위젯 노출 시 오청구 위험(§4).

### 방안 R-3 (보강) — 유광코팅·도수 누락의 동일 패턴 점검

N-1a(도수 소실)와 D-4(유광코팅 0행)는 **같은 결함 유형 = "권위 매트릭스 열이 라이브에 부분 적재"**. R-1 적재 시 **권위 14열 × 코팅 4열 전수 커버리지 체크리스트**를 함께 산출(누락 열 재발 방지). 트랙=dbm-price-import-prep(가격표 그릇 전수 분해) → dbm-load-execution.

## 4. 위젯 계약 영향 (widget-price-contract 함의)

| 함의 | 내용 |
|------|------|
| **도수 선택 → 가격 미도달 (N-1a)** | 위젯이 "흑백/칼라/별색" 도수 옵션을 노출해도, 라이브엔 칼라 단가행 부재 → 칼라 선택이 흑백값으로 침묵 계산되거나 0원. **위젯 옵션→comp 매핑 계약에 도수축이 빠져 있다.** R-1(⒜) 적재 + option_items 도수 택1 배선 선결. |
| **수량 입력 → 인쇄비 오산 (N-1b)** | 위젯이 주문수량을 엔진 qty로 그대로 보내면, 엔진이 그걸 판수로 룩업 → 200매가 200행(판수200) 단가로 계산. **위젯/어댑터가 출력매수 변환을 하거나(권고 ⓐ는 엔진이), 변환 책임을 계약에 명시**해야. 현 계약은 "qty=주문수량" 단일 의미 가정 → 합산형에선 깨짐. |
| **단일키 가드 (D-5 연계)** | DIGITAL S1(판형)/S2(3절) 양키 동시 전송 시 이중합산(recompute §3 D-5). 위젯은 출력판형 단일키만 보내야. 도수 comp 추가(R-1 ⒜) 시 흑백/칼라/별색 comp가 **택1**임을 위젯이 보장(동시 선택=이중합산). |
| **strict 모드 노출** | 실서비스(주문 재검증)는 strict. 칼라 단가행 부재 시 strict에서 매칭0건→ok=False(견적 거부) 또는 0원. **위젯 출시 전 R-1·R-2 선결 = 합산형 상품군 출시 게이트.** |

> **종합:** 합산형(엽서류) 상품군은 **R-1(도수 적재) + R-2(임포지션) 둘 다 닫히기 전엔 위젯 가격 계약이 성립 불가**. 면적매트릭스(실사·아크릴)는 골든 완전 재현(P2 PASS)이라 위젯 선출시 가능 — gate-verdict의 "상품군별 분기" 판정과 정합.

## 6. 라우팅 요약 (생성자≠검증자 — 본 심의는 GO 판정 안 함)

| 결함 | 방안 | 트랙 | 우리가 닫을 수 있나 |
|------|------|------|:--:|
| N-1a 도수축 소실 | R-1(칼라 comp 적재 ⒜) | dbm-price-import-prep → dbm-load-execution | ✅ (엔진 무관·Q-DGP-COLORAXIS 컨펌 후) |
| N-1b 수량축 ①② 판걸이수 컬럼+데이터 | R-2 ①② | dbm-ddl-proposer + dbm-load-execution | ⚠️ 준비 가능(DDL 인간 승인) |
| N-1b ③ 임포지션 변환 | R-2 ③ | **webadmin 개발자 백로그** | ❌ (read-only oracle) |
| 매트릭스 열 누락 재발방지 | R-3 | dbm-price-import-prep 전수 커버리지 | ✅ |

> **독립 게이트:** 본 정립안의 GO/NO-GO·권위 인용 실재성은 **dbm-validator**가 독립 검증(생성자≠검증자). 본 arbiter는 심의·제안까지. 실 COMMIT·DDL·코드는 인간 승인 + 개발자 협업.

## 부록 A. 권위 셀 출처 (재현 가능)

- `디지털인쇄비` 흑백단면 B열: B4=3000·B16=350(출력매수25). 칼라단면 D열: D4=4000·**D16=800**(출력매수25). 양면=C/E열. (직접 파싱·data_only.)
- `판걸이수`: 100x150 엽서 판걸이=8(F4)·국4절 출력=316x467(G2). 73x98=18(F2)·148x210=4(F8).
- 라이브 `t_prc_component_prices` COMP_PRINT_DIGITAL_S1: SIZ_000499(316x467 국4절) 단면 = 권위 흑백단면 byte-identical(53행). clr_cd 전 행 NULL.
- 라이브 `t_prt_print_options`: POPT_000001=단면·POPT_000002=양면(면 축만).
- 라이브 `t_proc_processes`: PROC_000004=디지털(인쇄방식·clr_cd NULL).
- `pricing.py`·`price_views.py`: 임포지션/판걸이/출력매수 grep 0건. `_tier_order_val`(price_views.py:107) min_qty=qty 직접.
- `t_siz_sizes`: 판걸이수 컬럼 부재(work/cut/margin/impos_yn만).
