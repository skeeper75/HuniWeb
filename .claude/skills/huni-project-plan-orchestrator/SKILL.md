# 후니프린팅 프로젝트 일정관리 오케스트레이터

후니프린팅 리뉴얼(Classic ASP → 신규)의 **프로젝트 일정관리 통합 IA 문서(엑셀)**를 산출·갱신한다. 레드프린팅·와우프레스 같은 인쇄회사로, 인쇄 자동견적 위젯 + 쇼핑몰이 결합된 주문제조 구조다. Shopby는 직접 종속하지 않고 표준 기능 체크리스트·외부연동 후보로만 참조한다.

## 실행 모드
**하이브리드** — Phase 1(콘텐츠 수집)은 서브에이전트 병렬 팬아웃(읽기전용 리서치), Phase 2~4(종합·빌드·QA)는 순차 파이프라인. 모든 Agent 호출은 `model: "opus"`.

## Phase 0: 컨텍스트 확인
- `_workspace/huni-project-plan/` 존재 + 부분 수정 요청 → **부분 재실행**(해당 에이전트만)
- 존재 + 새 입력(새 IA 버전 등) → **새 실행**(기존을 `_workspace_prev/`로 이동)
- 미존재 → **초기 실행**
- 입력 IA 엑셀(`docs/huni/후니프린팅_통합IA_*.xlsx`)을 먼저 파싱해 현황 파악.

## Phase 1: 근거 콘텐츠 수집 (서브에이전트 병렬, 읽기전용)
한 메시지에서 3 에이전트를 병렬 호출(`_workspace/huni-project-plan/01_research/`에 산출):
- **hw-reverse-engineer** → 위젯·에디쿠스 기능목록(쉬운말 A부 + 개발자상세 B부). 원천 `docs/reversing/*`·`_workspace/huni-widget/*`.
- **dbm-schema-analyst** → 주문→생산(MES) 사이트내부/외부 흐름. 원천 주문프로세스·공정관리 PDF·`table-spec`·라이브 DB 읽기전용 실측.
- **pq-researcher** → Shopby 표준 갭 + 레드 1차 런칭 벤치마크 + 외부계약 체크리스트. 원천 Shopby OpenAPI·레드 공개구조(읽기전용).

> 안전: 라이브 DB 쓰기 금지(읽기전용 SELECT만)·상대 서비스 주문/결제/폼 금지·비밀값 stdout 금지.

## Phase 2: 종합 (hpp-ia-curator)
입력 IA(권위) + 3 리서치 → 보강 IA 마스터(담당자·주차·구분)·페이즈 일정(병렬 레인·병목 대응)·진행현황 재집계. 가상 기능 창작 금지 — 실 IA에 실측 근거로 누락분만 보강.

## Phase 3: 빌드 (hpp-xlsx-builder)
종합 결과 → 가독성 엑셀 11시트. 결정적 빌드 스크립트(openpyxl)·색코딩·헤더고정·자동필터·집계 재계산.

## Phase 4: QA (hpp-plan-qa, 독립)
사용자 11요건 커버리지·근거 정합·집계 정합·가독성·안전 게이트(Q1~Q5)로 GO/NO-GO. NO-GO면 Phase 2/3 보정 루프.

## 데이터 전달
- 파일 기반(`_workspace/huni-project-plan/{01_research,02_synthesis,03_build,04_qa}/`).
- 최종 산출만 `docs/huni/`에 출력. 중간 파일 보존(감사 추적).
- 미해결 결정(정책·계약·도메인)은 BLOCKED로 분리·오케스트레이터가 사용자에 AskUserQuestion(서브는 사용자 질문 금지).

## 에러 핸들링
1회 재시도 후 재실패 시 해당 결과 없이 진행(리포트에 누락 명시). 상충 데이터는 삭제하지 않고 출처 병기. 라이브 실측과 문서가 충돌하면 라이브 실측이 권위(stale 문서 주의).

## 핵심 규칙 (HARD)
- 실제 기능목록 권위 = 후니 논의 IA. 가상 신규 창작 금지·실측 근거로 누락분만 보강.
- 전문용어 배제(쉬운 말) + 용어집/범례 동반. 개발자 상세는 별도 시트.
- 담당자(신우진 PM+인쇄·서희항 인쇄·김동학 쇼핑) + 고객(최숙진 실장 디자인·계약) 한눈에.
- 주차=상대일정(W1 착수 기준)·절대 날짜 날조 금지·병렬 레인+의존순서. 1·2·3차 단계.
- 쇼핑개발 병목 → AI 에이전트 병행 + Shopby 표준 흡수 가속 명시.
- 자격증명은 `.env.local`만·`_workspace`/stdout/엑셀 비노출.

## 테스트 시나리오
- **정상**: "일정관리 문서 업데이트" → Phase 0 부분 재실행 판별 → 변경 영역 에이전트만 → 빌드 → QA GO.
- **에러**: 라이브 DB 접속 실패 → dbm-schema-analyst 1회 재시도 → 실패 시 직전 실측본 사용·리포트에 'DB 재실측 누락' 명시·중단 없이 진행.
