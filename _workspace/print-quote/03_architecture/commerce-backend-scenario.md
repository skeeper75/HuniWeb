## 0. 전제 — 무엇이 이미 확정인가

| 영역 | 결정 | 출처 |
|---|---|---|
| 회원/인증 | 자체 (NextAuth v5, CI/DI 불필요) | ADR-004b §2 Bucket1 |
| 카트/주문 | 자체 (7-상태 + 파일 부속 상태머신, 인쇄 fit 강제) | ADR-004b §2 Bucket3 |

→ B와 C의 **유일한 차이는 결제(PG) 처리**다. B = 결제 위임(Shopby PG channel 또는 외부 통합), C = 결제 자체구축(이니시스 직연동 + 간편결제 직계약). 교차검증은 이 한 축에 집중한다.

## 1. B를 C 관점에서 공격 — "결제 위임이 만드는 종속·비용·장애점은 C로 가면 실제 사라지는가?"

| C의 공격 | 실제 검증 | 판정 |
|---|---|---|
| B는 Shopby Enterprise 월정액에 묶인다 | PG channel만 쓰려고 Enterprise 유지 시 비용 비효율 (OQ-5). 단 **이니시스 직연동(B-2)은 Shopby 없이도 B 성립** → 월정액 종속은 B-1에만 해당 | **부분 인정** — B-2로 회피 가능 |
| B는 Shopby 5xx 장애 시 결제 정지 (외부 단일장애점) | B-1은 사실. 그러나 C도 이니시스 단일 PG 장애 시 동일 정지 — 다중 PG fallback 자체구현 전까지는 C도 단일장애점 | **상쇄** — C가 우월하지 않음 |
| B는 동적 견적가를 PG channel이 막을 수 있다 (OQ-2) | Shopby PG channel이 어드민 SKU 가격 재계산을 강제하면 견적가 결제 불가 = deal-breaker | **인정** — 단 B-2(이니시스 직연동)는 가맹점 지정 금액이라 동적가 자연 지원 |
| B는 게스트 결제 시 Shopby memberNo 강제 (OQ-1) | 검증 전. 강제 시 그림자 회원 hack 필요 | **미검증 리스크** — B-1만 해당 |
| B는 데이터 동기화 코드(members 캐시·webhook polling) 부담 | B는 회원이 자체이므로 회원 동기화는 없음. 결제 결과 webhook 1방향만 | **반증** — 결제만 위임 시 동기화 최소 |

**결론:** C가 제거한다고 주장한 종속의 대부분은 **B-1(Shopby PG channel)에만** 존재하고, **B-2(이니시스 직연동)는 이미 그것들을 제거한 상태**다. 즉 "결제 위임=종속"은 위임처가 Shopby일 때만 참. 이니시스 직연동은 위임이지만 As-Is 종속 그대로(신규 종속 0).

## 2. C를 B 관점에서 공격 — "결제 자체구축의 비코드 차단요소가 V1 런칭을 실제로 막는가?"

| B의 공격 | 실제 검증 | 판정 |
|---|---|---|
| C는 이니시스 가맹 계약·심사 리드타임에 종속 (코드 아님) | As-Is에서 이니시스 운영 중 → **계약 승계 가능 시 신규 심사 회피**(OQ-3). 승계 불가면 신규 심사 = 외부 일정 차단 | **조건부** — OQ-3가 가른다 |
| C는 에스크로/구매안전 의무(전자상거래법 §13) | 선결제 통신판매 법적 의무. As-Is 에스크로 운영 중 → 승계 검토. 신규 구축 시 에스크로 사업자 계약 | **조건부** — 승계 시 해소 |
| C는 간편결제(네이버/카카오/토스) **직계약 N건 심사** | 각 페이사 가맹 심사 = 외부 일정 ~수주. **Agentic Coding 해결 불가**. V1 런칭 최대 병목 | **인정** — 직계약 시 V1 차단 |
| C는 전자금융거래법 책임·정산대사·24-7 장애대응 자체 부담 | PG 직연동 = 가맹점이 결제 분쟁·정산 불일치·환불 책임 직접. As-Is에 운영 경험 있으나 신규 시스템 무중단 운영 부담 | **인정** — 운영 리스크 |
| C는 컷오버와 결제 단일장애점을 동시 부담 | Big-Bang 컷오버 직후 + 결제 신규 자체구축 = 리스크 중첩 | **인정** — manager-strategy 핵심 경고 |

**결론:** C의 비코드 차단요소는 **카드/가상계좌(이니시스 승계 시)는 통과 가능**하나, **간편결제 직계약 N건의 심사 일정**이 V1을 직접 차단한다. 또 컷오버+결제 단일장애점 동시 부담은 정성 리스크가 크다.

## 3. 수렴 — 두 공격을 견딘 단일 권고 시나리오

두 공격의 교집합:
- **C의 우위(종속 제거)는 B-2 이니시스 직연동이 이미 대부분 달성** → "결제 자체구축해야 종속이 사라진다"는 거짓. 이니시스 직연동도 위임의 일종이며 As-Is 종속 승계.
- **C의 약점(간편결제 직계약 심사 + 컷오버 동시부담)은 V1에서 실재** → V1에 간편결제 다중 직계약을 신규로 묶으면 런칭 차단.
- 따라서 **V1은 검증된 결제 경로로 안전 런칭하고, 완전 독립은 검증 후 단계 전환**이 합리적.

### 통합 시나리오 (단계별 결제 처리)

| 단계 | 회원/카트/주문/CS | 결제(PG) | 게이트(통과 조건) |
|---|---|---|---|
| **V1 (컷오버)** | 자체 (확정) | **검증된 최소 경로**: 이니시스 카드/가상계좌/에스크로 **계약 승계**(OQ-3 OK 전제). 간편결제는 이니시스 통합형(중계)으로 1차. **Shopby PG channel은 백업 옵션**(OQ-1/OQ-2 통과 시에만 채택) | 컷오버 안정화 + 결제 무중단 검증 |
| **V1.5 (검증·PoC)** | 자체 운영 안정화 | V-001(게스트결제 회원강제) + OQ-2(동적 견적가) 검증. **PG 직결 PoC**(간편결제 직계약 사전 심사 착수 — 코드 아닌 리드타임을 미리 소진) | PoC 성공 + 직계약 심사 통과 |
| **V2 (조건충족 시)** | 자체 | **C-extreme 전환**: 간편결제 직계약 N건 + 다중 PG fallback 자체구현. Shopby 완전 해지(B-1 잔존 시) | 운영팀 24-7 대응 체계 + 다중 PG 안정 |

→ **핵심: V1은 "PG 제외 전 영역 자체 + 결제는 검증된 승계 경로"(=B의 실질, 단 위임처는 이니시스 직연동 우선). 100% 자체(C)는 비코드 차단요소를 V1.5에서 선소진한 뒤 V2에 전환.** V1 역순(완전자체 즉시)은 manager-strategy 경고대로 컷오버+결제 단일장애점 동시부담으로 비추천.

## 4. B/C 분기 결정 트리거 표

V1 시점에 결제 경로를 가르는 결정 변수:

| 결정 변수 | 값 | V1 분기 |
|---|---|---|
| **OQ-3: 이니시스 가맹 계약 승계** | 승계 가능 | → **B-2 (이니시스 직연동)** 우선. 신규 심사 0, 동적가 자연 지원, As-Is 종속 유지 |
| | 승계 불가 (신규 심사) | → B-1(Shopby PG channel) 임시 + 이니시스 신규 심사 병행 → 심사 완료 시 B-2 |
| **V-001: 게스트결제 회원강제 (Shopby)** | 강제 안 함 + OQ-2 통과 | → B-1(Shopby PG channel) 채택 가능 (다중 간편결제 1계약 흡수) |
| | 강제함 / OQ-2 실패 | → B-1 탈락, **B-2(이니시스 직연동) 강제** |
| **OQ-2: 동적 견적가 결제 (Shopby)** | 지원 | → B-1 후보 유지 |
| | 미지원 (어드민 SKU 강제) | → **B-1 deal-breaker**, B-2로 |
| **팀 결제도메인 경험** | 충분 (이니시스 운영 경험 있음) | → B-2/C 자체구축 자신감 ↑, V2 전환 앞당김 |
| | 부족 | → V1 B-1(위임)로 안전, V2 보류 |

**판독 규칙:** 위 변수 중 하나라도 Shopby 불리(V-001 강제 / OQ-2 미지원)면 **B-1 제외 → B-2(이니시스 직연동)로 수렴**. OQ-3 승계 가능하면 V1부터 B-2가 최적해. 즉 **현실적 기본 경로 = B-2 (이니시스 직연동 + 회원/카트/주문/CS 자체)**, Shopby PG channel은 OQ 3종 모두 통과 시에만 비교 대상.

### 가장 큰 미해결 (BLOCKER)

**OQ-3 (이니시스 계약 승계)** — 승계 가능하면 V1부터 B-2(이니시스 직연동)가 신규 심사 없이 즉시 가능해져 Shopby PG channel 비교 필요성 자체가 약해진다. 승계 불가(신규 심사)면 심사 리드타임이 V1 일정을 직접 좌우하므로, 그 기간 Shopby PG channel(B-1)을 임시 가교로 쓸지 결정해야 한다. **이 답이 통합 시나리오 V1의 결제 경로를 확정한다.**

## 전략 독립검증 (manager-strategy)

검증일: 2026-05-28 | 검증자: manager-strategy (독립·냉정) | 대상: 본 문서 §1~§5

### 종합 판정: **큰 틀 동의, 1개 핵심 보정 + 2개 누락 리스크**

본 시나리오의 단계 전략(V1→V1.5→V2)과 B-2 기본경로 수렴은 전략적으로 건전하다. 이전 manager-strategy 평가(strategy-shopby-assessment.md)와 일관되며, "PG 제외 전 영역 자체 + 결제는 검증경로"라는 V1 상한선을 정확히 반영했다. 그러나 핵심 논거 1개에 용어 혼동이 있고, BLOCKER 식별에 2개 누락이 있다.

### Q1 — "B-2가 C의 종속제거를 이미 달성" 논거 검증: **타당하나 용어 보정 필요**

타당한 부분: B-2(이니시스 직연동)는 As-Is 종속을 *승계*할 뿐 **신규 종속 0**이고, 동적 견적가를 가맹점 지정금액으로 자연 지원한다(§1, §4 정확). "결제 자체구축해야 독립"이 거짓이라는 결론은 옳다.

**허점 1 (용어 혼동):** §1 결론은 B-2를 "위임의 일종"이라 부르는데, 이는 부정확하다. 이니시스 직연동은 PG *직접 연동*이지 제3자(Shopby) *위임*이 아니다. 전자금융 책임·정산대사·환불분쟁이 **가맹점(후니)에 직접 귀속**된다 — 이건 §2에서 C의 약점으로 인정한 바로 그 부담이다. 즉 **B-2는 "코드 종속은 없으나 운영·법적 책임은 C와 동일하게 후니가 짊어지는" 구조**다. "B-2=위임이라 부담이 가볍다"는 뉘앙스는 위험하다. B-2의 진짜 이점은 "위임이라서"가 아니라 "As-Is에서 이미 운영 중이라 학습·계약 비용이 0에 가까워서"다. 이 구분이 흐려지면 V2 운영체계(OQ-Ops) 준비를 과소평가하게 된다.

**허점 2 (반증 과대):** §1 마지막 행 "결제만 위임 시 동기화 최소(webhook 1방향)"은 B-1(Shopby) 기준으로만 참이다. B-2도 결제결과 webhook·정산 대사·부분환불 역전표(ecount 연동)는 동일하게 필요하다. "동기화 최소"는 회원 동기화 부재를 말하는 것이지 결제 운영 부담 부재가 아니다.

### Q2 — 단계 전략(V1→V1.5→V2) 순서: **옳다. 더 나은 순서 없음**

리스크 최소화 관점에서 정순(검증경로 런칭 → 비코드 리드타임 선소진 → 조건부 전환)이 정답이다. 역순(V1 완전자체)은 컷오버+결제 단일장애점 동시부담으로 비추천 — 이전 평가와 일치. **단 1개 보강:** V1.5의 "간편결제 직계약 사전심사 착수"는 더 일찍, **V1 개발과 병행 착수**해야 한다. 직계약 심사는 외부 리드타임이라 V1.5까지 미루면 V2가 그만큼 밀린다. 심사 착수는 V1 시점, 활성화만 V2가 맞다.

### Q3 — OQ-3를 V1 단일 BLOCKER로 보는 것: **부분 동의 — 단일 아님, 2개 누락**

OQ-3(이니시스 승계)가 최대 변수인 것은 맞다. 그러나 "단일 BLOCKER"는 틀렸다. 본문 §5가 이미 OQ-EsCrow를 BLOCKER=YES로 표기해놓고 §5 마지막에서 OQ-3만 "가장 큰 미해결"로 부른 것은 자체 모순이다. 누락·과소평가된 V1 차단 리스크:

- **OQ-EsCrow (에스크로 승계) — OQ-3와 동급 BLOCKER.** 선결제 통신판매 법적 의무(전자상거래법 §13)다. 이니시스 카드 승계가 돼도 에스크로 사업자 계약이 별도면 V1 결제 흐름이 법적으로 미완성. OQ-3와 **반드시 동시 확인**해야 하며 하나라도 실패 시 V1 결제 경로 재설계.
- **누락 R1 — 컷오버 시점 결제 데이터 마이그레이션/대사 공백.** Big-Bang 컷오버 순간 As-Is 진행중 주문의 결제·환불·정산 상태를 신규 시스템이 이어받는 단절 구간. 이건 OQ-3(계약)와 별개의 **데이터·운영 BLOCKER**이며 본 문서에 없다.
- **누락 R2 — 이니시스 승계의 "기술 버전" 함정.** 계약 승계가 돼도 As-Is 연동이 구버전 모듈(예: 표준결제창 구판)이면 신규 Next.js 스택에서 재연동(신규 키·API 버전·인증방식)이 필요. "승계=즉시 가능"이 아닐 수 있음. OQ-3에 "계약 승계 가능 여부"뿐 아니라 "현 연동 모듈 버전·신규 스택 호환성"을 하위 질문으로 추가 권고.

### Q4 — 최종 1줄 권고: **방향 동의, 책임 명시로 수정**

원안의 "B-2 기본경로 + 단계전환"에 동의하나, B-2를 "위임"으로 칭하는 프레임은 운영책임 과소평가를 부르므로 책임 소재를 명시하도록 1줄을 수정 권고(아래).
