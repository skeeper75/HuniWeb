## 2. 완료 기준 (Definition of Done)

아래 게이트가 **모두 PASS**일 때만 작업을 완료로 선언한다. 각 항목은 "완료됐을 때 무엇이
참인가"로 검증 가능해야 하며, "그럴듯하다"는 근거가 될 수 없다.

| # | 게이트 | PASS 조건 (검증 가능 명제) |
|---|--------|---------------------------|
| G1 | **타겟 화이트리스트 준수** | 모든 적재 대상 테이블이 §5의 `t_*` 화이트리스트에 속한다. Django/비-`t_` 테이블 적재본 0건. |
| G2 | **무손실 추출** | 엑셀 원본 셀 → L1 추출본 역대조 100%. 블록 통째 누락 0, 발명(invented) 0, dodge(placeholder로 회피) 0. |
| G3 | **매핑 무결성** | 엑셀 컬럼 ↔ `t_*` 컬럼 매핑이 1:1로 추적 가능. 자연키 중복 0, 과소적재 0, 미보유 침묵 폴백 0. |
| G4 | **스키마 적합성** | 모든 값이 타입·길이·NOT NULL·CHECK 제약을 만족. `comp_cd` 등 길이 초과 0건. |
| G5 | **FK 무결성 + 적재 순서** | 모든 FK 참조 대상이 (a) 라이브에 존재하거나 (b) 코드행 선적재 목록에 있다. 적재 CSV에 FK 의존 위상정렬 순서가 명시됨. |
| G6 | **DRY-RUN 통과** | 트랜잭션 `BEGIN … ROLLBACK` 안에서 적재 시도 시 제약 위반(타입/길이/NOT NULL/CHECK/FK/PK 중복) 0건. 위반 시 위반 목록과 원인이 보고됨. |
| G7 | **차단/에스컬레이션 명시** | 적재 불가 행(placeholder=후니 미등록 코드 대기, GAP=모델링 미정)이 별도 목록으로 분리되고, 차단 사유·해소 조건이 기록됨. 차단을 "원천 부재"로 침묵 처리하지 않는다. |
| G8 | **재현성** | 추출→변환→검증이 스크립트로 재실행 가능(`profile → extract → transform → validate`). 손으로만 만든 산출물 0. |
| G9 | **교차 검증 독립성** | 매핑 산출(designer)과 검증(validator)이 분리된 에이전트로 수행되고, 검증이 설계 결함을 1건 이상 실제로 적발·정정한 이력이 있다(자기 승인 금지). |

> **완료 선언 형식:** `03_validation/` 의 최종 판정 문서에 G1–G9 각 항목 PASS/FAIL과 근거를
> 표로 명시한다. 하나라도 FAIL이면 미완료이며, 반복 개선 루프로 되돌아간다.

## 5. 타겟 스키마 — `t_*` 엔티티 화이트리스트 [HARD]

**적재 대상은 아래 `t_*` 테이블로만 한정한다.** 이 목록에 없는 테이블(특히 Django `auth_*`,
`django_*` 등 비-`t_` 테이블)에는 어떤 행도 매핑·적재하지 않는다.

**상품마스터 적재 타겟 (`t_prd_*` 중심)**
- `t_prd_products` (상품 마스터, `prd_cd` PK)
- `t_prd_product_categories` · `t_prd_product_sizes` · `t_prd_product_plate_sizes`
- `t_prd_product_materials` · `t_prd_product_processes` · `t_prd_product_print_options`
- `t_prd_product_sets` · `t_prd_product_bundle_qtys` · `t_prd_product_page_rules` · `t_prd_product_addons`
- CPQ: `t_prd_product_option_groups` · `t_prd_product_options` · `t_prd_product_option_items`
  · `t_prd_templates` · `t_prd_template_selections` · `t_prd_product_constraints`

**가격표 적재 타겟 (`t_prc_*` + 상품-가격 연결)**
- `t_prc_component_prices` · `t_prc_price_components` · `t_prc_price_formulas` · `t_prc_formula_components`
- `t_prd_product_price_formulas` · `t_prd_product_prices`

**마스터 참조 (FK 대상 — 원칙적으로 read; 부족 시 코드행 INSERT *제안*만)**
- `t_cod_base_codes` · `t_cat_categories` · `t_clr_color_counts`
- `t_mat_materials` · `t_siz_sizes` · `t_proc_processes`

> 화이트리스트 밖이지만 알아둘 것: `t_dsc_*`(할인)는 round-1 별도 트랙, `t_cus_*`(고객)는 무관.

## 6. HARD 제약 (금지 규칙 — "하지 마라")

1. **`t_*` 외 테이블 매핑 금지.** Django/비-`t_` 테이블은 타겟이 될 수 없다.
2. **실제 DB 쓰기 금지.** 읽기 전용 `SELECT` + 롤백 전용 DRY-RUN(`BEGIN…ROLLBACK`)만 허용.
   파괴적 SQL(`INSERT`/`UPDATE`/`DELETE`/`TRUNCATE`/`ALTER`/`DROP`) 실행 금지.
3. **DDL 변경 금지.** 스키마 부족분은 코드행 INSERT *제안*으로 우회, 직접 변경하지 않는다.
4. **추측 금지.** 코드값·존재 여부·등록 여부는 **라이브 실재**만 권위. note·라이브에 없는 값을
   추론으로 채우지 않는다(번호 연속 추론도 추측이다).
5. **placeholder 발명 금지.** 코드값을 placeholder로 만들기 전 **반드시 라이브 실코드를 먼저
   탐색**한다. 못 찾으면 발명하지 말고 §7 차단 목록으로 분리한다(dodge 금지).
6. **침묵 폴백 금지.** 미보유 데이터를 조용히 기본값/빈값으로 대체하지 않는다. 누락은 드러낸다.
7. **비밀값 노출 금지.** `RAILWAY_DB_*` 자격증명은 `.env.local`(chmod 600·gitignore)에만.
   `_workspace`(git 추적)·stdout에 비밀값 금지.
8. **라이브 DB 반복 조회 최소화.** 추출본 기반으로 작업하되, **등록/NULL/존재 판정은 라이브
   권위**(추출본 stale 주의 — 권위 반전 지점).

## 7. 권위 우선순위 (충돌 해소 규칙)

값이 충돌할 때 아래 순서로 권위를 적용한다. 상충 데이터는 삭제하지 말고 **출처를 병기**한다.

1. **라이브 DB 실재** — 코드값·테이블 구조·등록 여부·제약의 최종 권위.
2. **엑셀 명시값** — 입력 데이터의 권위(외부 표준으로 재고치지 않는다). 자가 확보는 미지·공란만.
3. **실무진 셀 텍스트 = 가격 의미 [HARD]** — 포함가("출력+코팅+가공")·세트("2개1세트"=`bdl_qty`)
   ·추가옵션을 note·차원에 **절대 간과 없이** 반영한다.
4. **계산공식집 초안** — 상품↔공식 바인딩의 권위(`06_extract/calc-formula-draft-l1.csv`).
   가격표=단가, 계산공식=상품마스터 몫이라는 경계를 지킨다.
5. **기존 검증 산출물** — `03_validation/*-final.md` 의 확정 판정(건드리지 말 것).

## 8. 작업 방법 (명령 우선)

**ETL 3단 — 재현 가능 스크립트로 수행** (설계≠검증 분리가 dodge·과소적재 적발의 핵심):
1. **Profile** → 시트 구조 카탈로그화 (블록·밴드헤더·수량축·차원).
2. **Extract (L1)** → 전 컬럼 행구조 보존·무손실. 9게이트 self-check로 누락 0 증명.
3. **Transform (L2)** → `t_*` 차원으로 평면화/정규화 (유형별 변환 로직).
4. **Validate** → 적대 교차검증(역대조·자연키 중복·날조·dodge·과소적재·발명) + DRY-RUN.

**DRY-RUN 적재 가능성 증명 절차:**
- FK 의존 위상정렬로 적재 순서를 확정한다(부모 → 자식).
- 단일 트랜잭션 안에서 적재본을 시도하고 **반드시 `ROLLBACK`** 한다.
- 제약 위반(타입/길이/NOT NULL/CHECK/FK/PK 중복)을 수집해 리포트로 산출한다.
- 위반 0이면 G6 PASS. 위반이 있으면 원인별로 분류해 개선 루프로 되돌린다.

**산출물 컨벤션:**
- 중간 산출물은 `_workspace/huni-dbmap/` 기존 폴더 체계(06_extract·02_mapping·03_validation 등)에 둔다.
- 파일명: `{phase}_{agent}_{artifact}.{ext}`. 최종 판정은 `03_validation/`에 표로.

## 9. 진행상태 보존·재현성·되돌리기 (롱러닝 하네스 원칙)

- **HANDOFF 갱신:** 세션 종료 시 `_workspace/huni-dbmap/HANDOFF.md`(또는 트랙별 HANDOFF)에
  *다음 시작점·미해결 블로커·이번 세션 결정·건드리지 말 것*을 기록한다.
- **세션 시작 루틴:** 새 세션은 (a) 이 GOAL → (b) HANDOFF → (c) 최신 검증 판정 순으로 읽고 재개한다.
- **멱등성:** 추출/변환/검증 스크립트는 같은 입력에 같은 출력을 낸다. 손작업 결과를 권위로 삼지 않는다.
- **되돌리기:** 산출물은 git 커밋으로 추적(`.env.local` IGNORED 확인 후). 결함 산출은 되돌릴 수 있어야 한다.
- **부분 재실행:** 특정 시트/상품군만 재작업 시 해당 단계만 재실행하고 나머지 확정본은 보존한다.

## 10. 인간 승인 체크포인트

다음 지점에서 멈추고 사용자 승인을 받는다(자율 진행 금지):
1. **실제 DB 적재(INSERT) 직전** — 본 GOAL의 종착점 너머. DRY-RUN PASS 산출물 제시 후 승인 요청.
2. **코드행 선적재 제안** — 라이브에 없는 코드값을 후니가 등록해야 하는 경우(placeholder 해소).
3. **GAP 모델링 결정** — 라이브 스키마로 무손실 표현 불가한 구조(예: 박 2단 룩업, gp size 비치수).
   억지 평면화하지 말고 에스컬레이션한다.

## 11. 실패 모드 처리 (실패에서 성장)

- **조기 종료 방지:** G1–G9 전부 PASS 전까지 완료 선언하지 않는다. 컨텍스트 압박은 HANDOFF
  저장으로 대응하고 중단하지 않는다.
- **반복 실수 회고:** 같은 유형 결함(dodge·과소적재·발명)이 2회 이상 나오면 원인을 일반화해
  검증 게이트/스킬에 규칙으로 반영한다(특정 예시용 좁은 수정 금지).
- **차단의 정직한 표기:** 적재 불가는 "처리 완료"로 포장하지 않는다. 차단 사유와 해소 조건을 §7
  목록에 남긴다. 침묵 truncation은 "전부 커버됨"으로 오독된다.
- **재시도 한도:** 동일 작업 1회 재시도 후 재실패 시 해당 결과 없이 진행하되 산출물에 누락을 명시한다.
