>
> **인과 체인(사용자 명시):** 인쇄 도메인 지식 → 엑셀 해석 → DB 엔티티(인쇄 자동견적 위젯 구축).

> DB read-only(라이브 조회 불요 — 기수확 07_domain 사용). 비밀값 무관.

> **확정도 범례:** ✅PDF/DB 권위 · 🟡PDF부분+표준 교차 · 🔴PDF미수록(가설, 실무 컨펌).

## 0. 큰 그림 — "레시피" 모델

사용자 비유: **인쇄방식 = 요리 레시피**. 1상품=1인쇄방식(Decision 14 v3, PDF p3·p8 독립 확증)이고,
그 방식이 **가능한 공정 부분집합·파일포맷·생산팀·기초데이터**를 결정한다.

```
인쇄방식 (5종, 최상위 축)
   └─ 가능 공정 부분집합 (방식별 레시피)
        └─ 공정 전후 의존 (오시→접지, 출력→코팅→재단→후가공→포장)
             └─ 공정별 필요 기초데이터 (param·자재·반제품)
```

본 문서 구성: §1 인쇄방식별 레시피 → §2 공정 순서·전후 의존(PDF 17 Case 권위) →
§3 제본 종류별 차이·필요 기초데이터(레이플랫 vs PUR 결론) → §4 셋트/반제품 vs 단일/완제품 생산방식(11 상품군 매핑).

## 1. 인쇄방식별 공정 레시피 (5종 — 옵셋/디지털/실사/UV/실크)

> 권위: PDF 주문프로세스 p3 §03 인쇄타입 분류 + p8 §03-1 인쇄타입별 파일/공정 + DB `PROC_000001` 자식 5종.
> 인쇄방식은 `t_proc_processes` 트리(PROC_000002~6)로만 표현(코드그룹 아님, KB §3 주의).

| 인쇄방식 (proc_cd) | 후니 정의·용도 (PDF) | 접수 파일포맷 (p8) | 담당팀 (p1·p10) | 가능 후가공 레시피 | 확정도 |
|---|---|---|---|---|:---:|
| **디지털**(PROC_000004, "토너 후니 현재 운영") | 디지털출력 = 토너 인쇄. **후니 현행 주력**. 낱장·책자·스티커·문구 대부분 | PDF / PDF+커팅AI / PDF+박AI | 디지털인쇄팀 | 코팅→재단→(오시·접지·타공·귀돌이·미싱·완칼·반칼·제본)→포장 | ✅ |
| **실사**(PROC_000006, "b.9 실사 default") | 실사출력 = 대형 잉크젯. 포스터·패브릭·배너·현수막·족자·보드/액자 | JPG / PDF+커팅AI | 특수인쇄가공팀(실사출력) | (쿨코팅)→(봉제/타공/족자/부착/완칼)→1차+박스포장 | ✅ |
| **UV**(PROC_000002, 변형 enum 5종) | UV평판출력 = UV경화 평판 인쇄. **아크릴·투명/리지드 소재**(§별색·표준 §UV) | PDF / PDF+레이저커팅AI | 쿠마샵(외주)·UV인쇄(단품/아크릴) | →레이저커팅→(아크릴가공/조립)→포장 | ✅ |
| **옵셋**(PROC_000003) | 옵셋 인쇄. **PDF 인쇄타입 리스트 미명시** — DB 마스터에만 존재 | (PDF 미수록) | (PDF 미수록) | 옵셋→코팅→재단→후가공→제본 (표준) | 🔴 |
| **실크**(PROC_000005) | 실크스크린. **PDF 인쇄타입 리스트 미명시** — DB 마스터에만 존재 | (PDF 미수록) | (PDF 미수록) | (표준: 굿즈·소재 직인쇄) | 🔴 |

**핵심 결론(레시피 축):**
- **디지털·실사·UV = 후니 현행 운영 3대 방식**(PDF 자체제작 인쇄출력으로 명시 확증). 옵셋·실크는 DB 마스터에만 있고 PDF 인쇄타입 리스트 미수록 → **미운영 가설**(실무 컨펌). 🔴
- **방식이 후가공 레시피를 가른다:** 디지털=코팅·제본 중심 / 실사=봉제·족자·타공(대형 후가공) / UV=레이저커팅·아크릴가공(리지드 가공). PDF 17 Case가 이 분기를 직접 보여준다(§2).
- **외주 인쇄출력**(화이트인쇄/패브릭/전사, 코드005 쿠마샵): PDF상 외주이나 DB `PROC_000001` 자식엔 별도 코드 없음 🔴. 화이트인쇄=별색(PROC_000008)으로 별도 표현(§3-별색·entity-semantic-model 참조).

### 1-1. UV 변형 enum (PROC_000002 `변형`, prcs_dtl_opt) — 표준 교차

DB KB §2-4: UV 변형 = **일반/배면양면/풀빼다/투명테두리/단면** 5종. 표준(§UV 표준)과 교차하면:

| UV 변형 (DB enum) | 도메인 의미 (표준 교차) | 용도 | 확정도 |
|---|---|---|:---:|
| 일반 | 표면 직인쇄 | 불투명 소재 기본 | 🟡 |
| 배면양면 | 투명 소재 **뒷면 인쇄**(양면) | 투명아크릴 양면 시안 — 표준 "selective transparency, CMYK+화이트+CMYK 다층" 정합 | 🟡 |
| 풀빼다 | 화이트 베이스 **전면 채움**(full white underbase) | 투명소재 위 색 발색용 화이트 전면 | 🟡 |
| 투명테두리(★) | 테두리만 무인쇄(투명 유지) | 투명 가장자리 노출 | 🟡 |
| 단면 | 한 면만 인쇄 | 기본 단면 | ✅ |

> ⚠️ **현 DB 오적재(KB acrylic G-AC-5):** 이 4변형이 `t_prd_product_print_options.print_side`(인쇄"면")에 욱여넣어짐.
> 도메인상 print_side(단/양면 도수)가 아니라 **UV 인쇄방식 옵션**이다 → entity-semantic-model §UV 참조. 표준이 후니 변형명을 직접 검증하진 못하나(후니 고유어), "투명소재 화이트 베이스/다층 투명제어" 표준 원리와 정합. 🟡

## 2. 공정 순서·전후 의존 (PDF 17 Case 권위)

### 2-1. 공통 골격 (전 방식 공통 백본)

```
출력(인쇄방식) → (코팅, 점선=조건부) → 재단/커팅 → 후가공/제본 → 포장
```

- **코팅은 항상 출력 후·재단 전**(점선=조건부, 전 Case 동일 위치). ✅
- **`입고` 동반 공정 = 별도 팀/외주 후 합류:** 제본·아크릴가공·봉제미싱·박작업·링제본. → 이들은 선행 공정 완료가 전제(공정 순서 강제). ✅
- **공정 순서 강제(PDF p1 To Do):** "앞공정 안 찍으면 다음 공정 못 찍히게" = **선행 공정 완료 후 다음 공정** 도메인 규칙. ✅

| **하드커버 트윈링**(PROC_000024?) | 하드보드 표지+트윈링 결착 | 완전 평탄 | 표지 반제품, 면지, 링컬러, 타공 | 하드커버 링책자(082) | ✅ |
| **레이플랫**(PROC_000025, "포토북 전용") | **펼침제본** — 보드/특수힌지로 **gutter 없이 완전 평탄** | **완전 평탄(파노라마 무손실)** | 내지(보드/두꺼운지), 표지, 면지, 책등, **펼침 힌지 구조** | **(DB 적재 0건 — §3-3 충돌)** | ✅(정의)/🔴(적재) |

### 3-2. 제본별 "필요 기초데이터" 의존 — 도메인 규칙

제본방식이 **어떤 반제품/자재/param을 요구하는가**가 갈린다(사용자 핵심 질문):

1. **트윈링 → 링 정보 필수:** 링컬러(자재 USAGE.07)·D링 mm(바인더)·방향(좌철/상철)·타공. 표준 교차(§제본): Wire-O는 쌍고리 와이어+사각타공, 링 직경/색상이 제품 식별자. ✅ → DB booklet G-BK 트윈링(071) mat=링3+투명커버2 정합.
2. **하드커버 → 표지/면지 반제품 필수:** 하드보드 표지(싸바리/레더)+면지가 독립 자재 슬롯(USAGE.02/.03). PDF Case5b "레더+싸바리(입고)"가 이 반제품 제작 공정. ✅ → booklet 072·077·082는 표지·면지 parent 적재(내지종이는 미적재 G-BK-1).
3. **무선/PUR → 책등(mm) 필수:** 책등 두께가 페이지수·종이두께에 의존. photobook 책등 10/12/14/16(하드)·4/6/8/10/12/14(소프트) = 페이지수 종속 선택지. ✅
4. **떡제본 → 묶음수(권), 페이지 무의미:** 떡메모지(097) page_rule 3/3/3은 **잡음**(G-BK-5) — 떡제본은 낱장 묶음수(50장1권) 도메인이지 페이지 도메인 아님. ✅
5. **중철 → 페이지 4의 배수:** 중철은 접지 후 가운데 철. 페이지=4배수 제약(표준). 🟡

### 3-3. 레이플랫 vs PUR — 도메인 충돌 결론 (G-PB-1)

> **사용자 핵심 질문: 포토북 제본 DB는 PUR 적재인데 도메인상 레이플랫 의심.**

**사실 확정(라이브·엑셀·표준 3중):**
- **라이브:** 포토북(PRD_000100) process = **PROC_000020 PUR** 단독. 레이플랫(PROC_000025)은 **전 상품 적재 0건**.
- **엑셀:** photobook 전 행 `제본사양_제본=PUR`.
- **표준(✅ 강한 근거):** 레이플랫 = 보드/특수힌지로 **gutter 없이 완전 평탄**, 파노라마 spread 무손실. PUR = 폴리우레탄 접착 perfect binding으로 **gutter에 이미지 손실·중앙 휨** 발생(반복사용 시 다소 이완). 둘은 **물리적으로 다른 제본**.

**결론(단정 금지 — 도메인 충돌로 분류):**
- 엑셀·DB는 **일관되게 PUR**로 표기. DB note의 "PROC_000025 레이플랫=포토북 전용"은 **마스터 정의일 뿐 미운영**(적재 0).
- 표준상 포토북은 레이플랫이 이상적이나, **후니 권위(PDF/DB/엑셀)가 PUR이면 PUR이 사실**(HARD: 표준 충돌 시 후니 권위).
- **가설①(유력):** 후니 포토북은 실제 PUR 제본 운영, 레이플랫은 마스터에만 정의된 미운영 옵션.
- **가설②:** 레이플랫이어야 하나 PUR로 오적재(엑셀 원천부터 PUR이므로 가능성 낮음).
- **→ 실무 컨펌 1순위(C-10 해결의 핵심):** "후니 포토북 제본은 PUR인가 레이플랫인가? 레이플랫(025)은 미운영 마스터인가?" 표준은 레이플랫 우위를 알려주나 **후니 운영 사실이 PUR임을 뒤집지 못함**.

## 4. 셋트/반제품 vs 단일/완제품 생산방식 (11 상품군 매핑)

> 권위: 라이브 booklet/photobook 규명(08_remediation) + DB `PRD_TYPE`(.01 완제품/.02 반제품/.03 기성/.04 디자인) + `t_prd_product_sets` + `usage_cd`.
> 사용자 핵심 질문: 어느 상품이 내지/표지/면지로 분해되고, 어느 게 통합 단일인가, 왜 그렇게 생산되나.

### 4-1. 후니 적재 모델의 핵심 발견 (권위 반전, booklet.md ②③)

라이브는 **3가지 생산구조**로 갈린다 — B-rules 가설("자재는 sub_prd에 붙음")은 **라이브에서 반전**됨:

| 구조유형 | 정의 | 자재 적재 위치 | sets | 적용 |
|---|---|---|---|---|
| **A. 통합상품**(단일행 캐스케이드) | 내지/표지가 **반제품 분해 안 됨**. parent 한 상품에 usage_cd로 슬롯 구분 직접 적재 | parent + usage_cd(.01내지/.02표지) | **0행(정상)** | 중철·무선·PUR·트윈링책자(068~071) |
| **B. 셋트상품**(반제품 분해) | 표지/면지가 **독립 반제품 sub_prd**(PRD_TYPE.02)로 분해. sub_prd는 **빈 껍데기**, sets로만 연결. 자재는 **여전히 parent에 usage_cd 적재** | parent + usage_cd | **N행(parent→sub_prd)** | 하드커버류(072·077·082·088·094·100) |
| **C. 완제품/단일**(분해 없음) | 낱장·굿즈 — 내지/표지 개념 자체 없음 | parent 단일 | 0행 | digital-print·sticker·acrylic·silsa·goods-pouch·calendar |

> **HARD 교정(booklet 머리말):** 모든 sub_prd(반제품 073~107)는 **9속성 전부 0행(빈 껍데기)**. sets만 parent→sub_prd를 가리킨다.
> 자재 권위 위치 = **parent + usage_cd**(sub_prd 아님). "자재를 sub_prd에서 찾으면 전량 MISSING 오판" 위험.

### 4-2. "왜 그렇게 생산되나" — 도메인 인과

- **통합상품(A)이 되는 이유:** 중철·무선·PUR·트윈링은 **내지+표지를 한 번에 출력·재단·제본**한다(PDF Case4·5). 표지가 내지와 같은 디지털 출력물(종이)이라 별도 반제품 제작이 불필요. → 엑셀에서 표지/내지가 단일 행 캐스케이드.
- **셋트상품(B)이 되는 이유:** 하드커버는 **표지가 별도 공정**(하드보드+싸바리/레더, PDF Case5b "싸바리 입고")으로 만들어지는 **반제품**이다. 면지도 별도 자재 슬롯. 표지·면지가 내지와 다른 생산 라인을 타므로 sub_prd로 분해해 sets로 묶는다. 포토북은 표지타입 variant(하드/레더하드/소프트) 5종이 표지 sub_prd. ✅
- **완제품/단일(C)이 되는 이유:** 낱장·스티커·아크릴·굿즈는 **내지/표지 개념이 없는 단일 인쇄물**. 분해 불요.

## 5. 큰 그림 적용 (도메인 → 엑셀 → DB → 위젯)

- **도메인→엑셀:** 인쇄방식·제본종류가 엑셀 시트축(`제본사양`·`인쇄사양`·`소재`)으로 1:1 인코딩.
- **엑셀→DB:** 제본 8종 → PROC_000017 family, 반제품 분해 → sets+usage_cd. 통합상품 sets=0이 정상(booklet 권위 반전).
- **DB→위젯:** 인쇄방식이 가능 공정 부분집합을 게이팅 → 위젯 옵션 캐스케이드(제본 선택→링컬러/책등 종속, 오시→접지 종속).
- **선결 결함:** stationery 제본 0건(G-ST-1)·calendar 택일 멤버 끊김(G-CL-1)·booklet 내지종이(G-BK-1)가 잔존하면 위젯에서 제본/택일/종이 옵션이 통째로 사라짐.
