# 전략 평가 — Shopby 위임 vs 자체 구축 (Agentic Coding 시대 관점)

생성일: 2026-05-28
작성자: manager-strategy (독립·냉정 평가)
질문 출처: 오너 — "냉정하게 레거시 시스템을 확인할 때 Shopby를 이용해야 하는지, Agentic Coding으로 직접 구축하는 게 나은지 판단해달라"
선행 결정: D-004(옵션 C 채택), D-005/ADR-004(C-minimal 잠정 1순위, V-001만 미해소)

> 본 문서는 오너가 듣고 싶은 답이 아니라 사실 기반 판단을 목표로 한다. 불확실한 지점은 "가정"으로 명시한다.

---

## 0. 질문의 재정의 (냉정한 출발점)

오너의 질문은 "Shopby를 쓸 것인가 말 것인가"의 이분법처럼 들리지만, **이미 프로젝트는 그 이분법을 지났다.** D-004에서 Aurora(프론트)는 폐기됐고, D-005/ADR-004에서 커머스 백엔드 위임 범위는 이미 **C-minimal(PG 채널만 Shopby, 회원·정산은 자체+ecount)**로 좁혀졌다. 즉 Shopby의 잔존 가치는 **회원·결제·주문·CS 전체 위임**이 아니라 **"한국 다중 PG 통합 한 조각"으로 이미 축소**되어 있다.

따라서 오너의 진짜 질문은 이렇게 번역된다:

> **"이미 한 조각(PG)만 남은 Shopby 의존마저 Agentic Coding으로 걷어내고 C-extreme(완전 자체)로 갈 것인가?"**

이 프레임으로 평가한다. 핵심 변수는 결제 통합 한 조각의 가치 대 부담이지, 커머스 전체가 아니다.

## 1. 6개 전략 차원 평가

| # | 차원 | 핵심 판단 (1줄) |
|---|------|----------------|
| 1 | **TCO (총소유비용)** | Shopby Enterprise 월정액은 PG 채널만 쓸 때 정당화 어려움 — PG 직결(토스페이먼츠 등)은 수수료만 발생하므로 장기 TCO는 C-extreme가 우위, 단 초기 다중계약 공력이 선납 비용. |
| 2 | **Lock-in / 종속성** | C-minimal조차 IP 화이트리스트·NHN 정책변동·게스트결제 강제(V-001 미해소)에 묶임 — ADR-004 §3에서 외부의존 차원 C-extreme(8) > C-minimal(6)로 이미 정량화됨. |
| 3 | **규제·컴플라이언스 주체** | **Agentic Coding으로 해결 안 되는 영역.** PG 심사·전자금융 정산 의무·에스크로·소비자보호법 책임은 코드가 아니라 "사업자가 누구와 계약했는가"가 결정 — C-extreme는 이 부담을 후니가 직접 짊어진다. |
| 4 | **Agentic Coding 레버리지 한계** | 코드 생산성은 실제로 오르지만(옵션·가격 엔진·관리자 self-build 타당), **운영 장애대응·PG 계약·규제 계약은 코드 외 영역** — 과대평가 시 V1 런칭 직후 운영 리스크로 회귀. |
| 5 | **인쇄 도메인 특수성** | 수량구간가·작업파일 RENAME·교정 게이트·공정 라우트는 표준 커머스(Shopby) 모델과 본질 충돌 — **자체 구축이 명백히 우월** (D-004 deal-breaker가 이미 증명). |
| 6 | **개발팀 역량 전제** | "경험 있는 개발자 중심"이 성립하면 self-build 리스크 급감, 불성립 시 C-extreme의 PG/정산/장애대응 자체부담이 단일 실패점 — **이 전제의 진위가 권고를 가르는 최대 변수**. |

### 차원별 보충 (냉정한 디테일)

**1. TCO.** 정확한 Shopby Enterprise 단가는 본 자료에 없다(가정: 월정액+거래수수료 구조, ADR-004 §2 명시). 냉정한 비교는 "월정액을 PG 한 조각 때문에 계속 낼 가치가 있는가"이다. ecount(월 4만원)가 정산·세금계산서를 이미 흡수했으므로(V-002 Closed), Shopby가 남긴 유일한 실효 가치는 **다중 PG 통합 편의 1개**뿐이다. 이 한 조각을 위해 Enterprise 월정액을 내는 것은 장기적으로 비효율이다. 다만 PG 직결은 사업자 심사·통합 어댑터·webhook 정규화라는 **선납 공력**이 들고, 이건 Agentic Coding으로 코드는 빨리 짜도 **PG사 심사 기간 자체는 단축 불가**다.

**2. Lock-in.** ADR-001 D3, ADR-004 §3에서 이미 식별된 종속(IP 화이트리스트, NHN 업스트림, private registry)은 C-minimal에서 PG 한 채널로 줄었다. 그러나 V-001(게스트 결제에 Shopby memberNo 강제 여부)이 **아직 미검증**이다. 만약 강제된다면 게스트 주문마다 그림자 회원을 만드는 hack이 필요(ADR-004 §5 V-001)하고, 이는 인쇄 B2B 게스트 견적 흐름에 독성이 강하다. **이 한 가지 미검증 항목이 C-minimal의 실효성을 좌우한다.**

**3. 규제·컴플라이언스 — 가장 냉정해야 할 지점.** 한국 전자상거래는 (a)전자금융거래법상 PG/정산 (b)정보통신망법상 개인정보·휴면 (c)소비자보호법상 청약철회·표시광고로 나뉜다. 오너 입력(ADR-004 §1)으로 **본인인증 CI/DI 불필요·휴대폰 인증 충분**이 확인되어 (b)의 무거운 부분은 이미 가벼워졌다. 그러나 **(a) PG/정산은 위임을 걷어낼수록 후니가 직접 PG사와 계약·심사·정산의무를 진다.** Agentic Coding은 결제 연동 코드는 생성하지만, **PG 가맹 심사·전자금융 분쟁 책임·정산 대사(reconciliation) 운영 책임은 이전하지 못한다.** 이것이 self-build 확대의 진짜 비용이다.

**4. Agentic Coding 레버리지.** 명확히 분리해야 한다(아래 §2). 요지: 코드량이 많고 도메인이 명확한 영역(옵션 폼·가격 엔진·관리자·공정 워크플로)은 self-build가 합리적이고 Agentic Coding이 가장 빛나는 곳이다. 반대로 코드량은 적지만 **계약·심사·운영 책임이 본질인 영역(PG, 본인인증, 정산 규제)**은 Agentic Coding이 거의 도움이 안 된다. 오너가 "다 직접 만들 수 있다"고 느끼는 자신감의 90%는 전자에서 오고, 위험의 90%는 후자에 있다.

**5. 인쇄 도메인 특수성.** 이미 D-004에서 Aurora의 가격 무결성 충돌·옵션 모델 강성·공정 부재가 deal-breaker로 정량 증명(13/30점). 표준 커머스 위임이 **오히려 방해**가 되는 유일한 영역이고, self-build가 100% 정답인 구간이다. 여기엔 이견 없음.

**6. 팀 역량 전제.** "경험 있는 개발자 중심"이라는 전제는 양날이다. 성립 시: 옵션·가격·관리자 self-build는 충분히 통제 가능하고 C-extreme의 PG 통합도 감당된다. 불성립 시(1인 또는 인쇄 도메인 경험만 있고 결제/규제 경험 없는 팀): C-extreme의 다중 PG·정산·장애대응이 **단일 인력에 집중되는 운영 단일 실패점**이 된다. 본 자료에 팀 규모·결제도메인 경험은 명시되지 않음(가정 필요).

## 2. Agentic Coding이 바꾸는 것 vs 바꾸지 못하는 것

### 해결하는 것 (self-build 확대 정당화)

1. **도메인 로직 생산성** — 옵션 폼·수량구간 가격 엔진·관리자·공정 워크플로처럼 "명세가 명확하고 코드량이 많은" 영역의 구현 속도·반복 비용을 크게 낮춘다. (edicus.man 12K LOC 흡수 + 23K 신규 LOC가 현실 일정이 되는 근거)
2. **표준 통합 어댑터 작성** — PG webhook 정규화, ecount Open API 연동, 데이터 마이그레이션 스크립트 같은 "정형 통합 코드"를 빠르게 생성·테스트.
3. **유지보수·리팩터링 민첩성** — 단일 코드베이스의 변경·테스트 커버리지 확보 비용을 낮춰, 자체 구축의 장기 부담을 과거보다 가볍게 만든다.

### 해결하지 못하는 것 (위임을 걷어낼 때 그대로 후니에 남음)

1. **계약·심사 리드타임** — PG 가맹 심사, KCB/NICE 본인인증 계약, 사업자 등록 검수는 코드 속도와 무관한 외부 기관 일정. Agentic Coding이 0으로 만들 수 없다.
2. **규제·법적 책임의 주체** — 전자금융 분쟁, 정산 의무, 개인정보 사고 책임은 코드가 아니라 계약·사업자에게 귀속. self-build는 책임을 후니로 이전시킨다(반대 방향 X).
3. **24/7 운영·장애대응** — 결제 정지·정산 오류·PG 장애 fallback은 코드 품질이 아니라 운영 체계(온콜·모니터링·대사 프로세스)의 문제. 위임 시 Shopby가 흡수하던 운영 부담이 self-build 시 후니의 상시 부담이 된다.

> **냉정한 요약:** Agentic Coding은 "만드는 비용"을 낮추지만 "책임지고 운영·규제 대응하는 비용"은 낮추지 못한다. self-build 확대는 전자를 절감하는 대신 후자를 떠안는 거래다.

## 3. 최종 전략 권고

### 자체구축 확대 가능 범위 (냉정한 경계선)

| 영역 | 권고 | 근거 |
|------|------|------|
| 옵션 폼·가격 엔진·견적·카탈로그 | **100% 자체 (확정)** | 인쇄 도메인 본질, Shopby 부적합 증명됨 |
| 관리자·공정 워크플로·검수 게이트 | **100% 자체 (확정)** | 도메인 명확, Agentic Coding 최적 영역 |
| 회원·인증·휴면·약관 | **자체 (C-minimal대로)** | CI/DI 불필요로 위임 가치 소멸 |
| 정산·세금계산서 | **자체 + ecount (확정)** | V-002로 Shopby 정산 가치 소멸 |
| **결제 PG** | **단계적 — 즉시 C-extreme 금지** | V-001 미검증 + PG 심사 리드타임이 코드와 무관 |

**결론: 자체구축 확대는 "PG를 제외한 전 영역"까지가 안전한 상한선.** PG까지 자체(직결)로 가는 C-extreme는 *기술적으로는* Agentic Coding으로 가능하지만, *전략적으로는* V-001 검증과 PG 심사 일정·운영체계 준비가 선행되어야 한다.

### 단계 전략 (권고)

- **V1 (런칭): C-minimal 유지** — 결제만 Shopby PG 채널로 두고 나머지 전부 자체. 이유: PG 다중 직결의 심사·통합 리드타임을 Big-Bang 컷오버의 critical path에 올리지 않는다(컷오버 자체에 이미 20건 미결정 D-PM이 쌓여 있음 — decisions.md).
- **V1.5 (런칭 후): V-001 검증 + PG 직결 PoC** — 게스트 결제 가능 여부 확정하고, 토스페이먼츠 등 1개 PG 직결을 운영 부하 없는 상태에서 검증.
- **V2: 조건부 C-extreme 전환** — V-001 결과·PG 직결 PoC·운영체계(온콜/대사) 준비가 모두 충족되면 Shopby Enterprise 해지하고 PG 직결로 완전 독립. 충족 안 되면 C-minimal 영구 유지(부담/가치 비율상 합리적).

> **즉, "V1은 Shopby(PG)에 최소 위임, V2에서 조건 충족 시 자체 전환"이 정답.** 반대(V1 완전 자체 → 안정화)는 컷오버 리스크와 결제 단일 실패점을 동시에 떠안아 비추천.

## 4. 오너가 오해하기 쉬운 지점 (냉정한 경고)

1. **"Agentic Coding이 있으니 PG도 직접 만들면 된다"는 착시.** 결제 *연동 코드*는 만들 수 있어도 *PG 가맹 심사·전자금융 책임·정산 대사 운영*은 코드로 해결되지 않는다. self-build 확대의 진짜 비용은 코드가 아니라 **이전 불가능한 운영·규제 책임**이다. 코드 생산성 향상을 운영·계약 리스크 감소로 착각하면 V1 런칭 직후 결제·정산 사고로 회귀한다.

2. **"이미 C-minimal로 거의 자체인데 한 발만 더"라는 과소평가.** PG 한 조각이 남은 이유는 그게 "어려워서"가 아니라 **다중 PG 통합 편의 + 심사 리드타임 회피**라는 전략적 선택이기 때문이다. 게다가 **V-001(게스트 결제 시 Shopby 회원 강제 여부)이 아직 미검증**이라, 이 검증 없이 C-minimal의 실효성조차 확정되지 않았다. 마지막 한 조각을 걷어내기 전에 V-001부터 닫는 것이 순서다.
