### base 출처 신뢰도 badge (R-4 채택 2026-06-12)

base 사실은 보편 지식이라 huni badge(✅🟡🔴⚪)를 쓰지 않으나, **출처 신뢰도**는 사실마다 다르다. base 사실 각 항목에 다음 표기를 붙인다(huni badge와 별개 축 — 충돌 없음).

| 표기 | 의미 |
|---|---|
| **[검증]** | 2+ 독립 외부 출처가 일치(`_research/base-verification.md` CONFIRMED). |
| **[단일출처]** | 단일 출처/관용어 등 — 사실이나 교차검증 1건. |
| **[추정]** | 외부 근거 부재·도메인 통념. 사실로 단정 금지. |

> 권위 = `_research/base-verification.md` verdict(CONFIRMED→[검증]·SINGLE→[단일출처]·UNVERIFIABLE→[추정]). 나무위키 등 약한 출처는 base 권위로 쓰지 않고 검증된 표준(EPA·Wikipedia·CIP4 등)으로 격상한다.

## 3. 페이지 컨벤션

각 항목 = 조회 가능한 원자 블록:

```
### [항목ID] 제목  {상태badge}
- 내용: <한 줄 사실/규칙>
- 출처: <file:§ | 인터뷰ID | DB>
- 연결: [[다른-페이지#항목]] (교차참조)
- answers_cq: CQ-XXX
- tags: #도메인 #키워드
```

**상태 코드 (HARD — 매핑 권위 구분):**
| badge | 의미 | 매핑 시 취급 |
|---|---|---|
| ✅결정 | 확정 결정/운영 사실 | 권위로 사용 |
| 🟡권장 | 권장안 존재, 미확정 | **사실로 쓰지 말 것**("후보" 표기) |
| 🔴미결정 | 결정 없음(오픈) | To-Be 결정 필요, 매핑 보류 |
| ⚪명세 | 기능 명세만(결정 불요) | 참고 |

### 안정 식별자(stable @id) 운용 (R-7 채택 2026-06-12)

- 원자 블록 항목ID(`[FAMILY-SECTION-NNN]` 또는 base/축 ID)는 **페이지 이동·이름 변경 시에도 불변**하는 안정 식별자다. [[페이지#항목ID]] 교차참조가 ID 기반이므로, ID는 한 번 부여되면 재사용·중복하지 않는다.
- **분할 lint 규칙:** 페이지가 500줄 근접해 분할(`<family>-pricing.md` 등)할 때, 항목ID는 따라가되 **링크의 페이지 부분만** 갱신한다(ID 부분 불변). 분할 후 [[옛페이지#ID]]를 [[새페이지#ID]]로 일괄 교정하는 것을 lint에 포함.

### 관계 그래프 원칙 (사용자 directive 2026-06-12 — 위키의 본질)

> "인쇄자동견적 서비스의 구성요소를 **가장 작은 단위까지 분리해서 서로서로 연계**한다." 원본 엑셀을 json/md로만 두면 관계성 관리가 어렵다 → 요소를 원자 항목(안정 @id)으로 등재하고 관계를 **명시 링크**로 표현한다.

**1. 원자 항목 등재 (요소 분리, HARD)**
- 구성요소(자재·공정·사이즈체계·가격공식·CPQ옵션·위젯계약 필드·적재경로 단계)는 **축 페이지(`huni/<axis>.md`)에 원자 항목(`[항목ID]` 안정 @id, R-7) 단위로 1회만 등재**한다.
- 항목 = **자기완결 블록**: 정의 한 줄 · 라이브 t_* 앵커 · 출처(file:§) · badge(✅🟡🔴⚪) · 관계 링크.
- **과분할 금지(R-9):** 1항목 = 1 의미 단위이지 1 DB행이 아니다. 예: "자재 1행 ≠ 1항목"; "자재 유형/사용축(MAT_TYPE·usage_cd)" 같은 **개념 단위**로 묶는다. 행 단위 데이터는 라이브 t_*/CSV 권위로 두고 항목은 그 구조·규칙을 기술한다.

**2. 관계 유형 (명시 링크, 최소 셋 + 확장)**
관계는 `[[axis-x#항목ID]]` 링크로 표현하고, 링크 옆에 **관계 동사**를 명시한다(축 팩 도메인에 정합):

| 동사 | 의미 | 방향 예 |
|---|---|---|
| **uses** | 요소가 다른 요소를 구성요소로 사용 | 레시피 → 자재/공정 항목 |
| **requires** | 필수 선행/동반(없으면 불가) | 박 공정 → 박색 옵션풀 · 옵션 → 차원행 선적재 |
| **excludes** | 상호배제(캐스케이드 제약) | 옵션 A ↛ 옵션 B (JSONLogic constraints) |
| **priced-by** | 가격이 특정 공식/사슬로 결정 | 상품 → PRF_<X> 공식 항목 |
| **loaded-via** | 적재 경로/순서로 등록 | 요소 → load-path 단계 항목 |
| **mapped-to** | 위젯 계약 필드 ↔ DB 컬럼/componentType | 정규화 계약 → t_* / componentType |

(확장 동사는 팩 의미에 맞게 추가하되 위 6종을 1급으로 둔다.)

**3. 역링크 슬롯 규약**
- 각 축 원자 항목은 끝에 **`- 사용처:`(역링크 슬롯)** 을 둔다(초기 빈칸 또는 기존 known만). 레시피 페이지 집필 시 그 family가 항목을 `uses/priced-by` 하면 이 슬롯에 `[[recipes/<family>#항목ID]]`를 추가한다 → 요소↔상품군 양방향 추적.
- 보편 지식 연계는 `- 연결:`에 `[[../base/...]]`로 둔다(요소 → base 토대).

**4. 레시피 = 조립 뷰 (복붙 금지, HARD)**
- 레시피 페이지(`recipes/<family>.md`)는 **조립 뷰**다: 축 원자 항목을 `[[링크]] + 관계 동사 + 한 줄 요약`으로 **참조만** 한다. 횡단 사실 본문 복붙 금지(갱신 시 모순의 뿌리 — 단일 사실 원칙 §8/§9).
- 레시피 고유 사실(이 family만의 prd_cd 목록·교정대기 행)만 레시피 본문에 원자 블록으로 둔다.

## 4. 워크플로 (Karpathy)

**Ingest(원천 추가):** 원천 읽기 → 핵심 정리 → `sources/`에 요약 페이지 → `index.md` 갱신 →
관련 엔티티/개념 페이지 10~15개 수정·교차참조 → `log.md` 기록.

**Query(질문):** `index.md` 먼저 → 관련 페이지 읽기 → 인용과 함께 합성 →
좋은 답은 새 페이지로 환원(file back) → `log.md` 기록.

**Lint(주기 점검):** 모순·stale 주장·고아 페이지·누락 교차참조·데이터 공백 점검 → `log.md` 기록.

## 8. index.md = llms.txt 형식 (R-1·R-2 채택 2026-06-12)

- `index.md`는 **llms.txt 사실상 표준 구조**로 운용한다: H1 위키명 → blockquote 한 줄 목적 → H2 섹션별 불릿 링크 리스트(`- [페이지](경로): 한 줄 설명 — 상태`). 표(table)보다 불릿 링크가 llms.txt 파서·LLM 컨텍스트 효율에 정합.
- **1급 섹션(HARD):** `RECIPES`(상품군 11)·`AXES`(횡단 축 6: materials·processes·price-engine·cpq-options·widget-contract·load-path)를 BASE·HUNI와 함께 index의 1급 H2 섹션으로 둔다. 미래 LLM이 "상품 조립" 질의 시 가장 먼저 닿는 곳이 레시피여야 한다.
- 모든 페이지는 ingest마다 index에 등재(Karpathy 워크플로). `_`-prefix 작업 폴더는 카탈로그에 넣지 않는다.

## 9. 레시피 페이지 표준 템플릿 (DB-anchored 8절, HARD)

`recipes/<family>.md` 섹션 순서 고정. 각 섹션은 라이브 t_* 실재 구조에 앵커한다. 앵커 불가 항목은 **"앱 계산"**(판수·박 등급 등) 또는 **🔴 GAP**(미모델링)으로만 존재 — 스키마 발명 금지.

```markdown
# <family> 레시피  {전체상태: ✅|🟡}

## CQ 헤더 (이 페이지가 답하는 질문)  ← R-5
- 이 상품군은 무엇인가 / 어떤 차원·옵션이 있는가
- 가격은 어떻게 계산되는가 / DB에 어떻게 등록하는가
- 현재 라이브 적재 상태·교정 대기는 무엇인가

## 0. 정체 (identity)        앵커: t_prd_products · t_cat_categories
## 1. 차원 (dimensions)      앵커: t_prd_product_sizes/plate_sizes/bundle_qtys/sets · t_siz_sizes
## 2. 자재·공정 BOM           앵커: t_prd_product_materials/processes · t_mat_materials · t_proc_processes
## 3. 가격 사슬 (price chain) 앵커: t_prc_* 4단 + t_dsc_*
## 4. CPQ 옵션 레이어         앵커: t_prd_product_option_groups/options/option_items · constraints · templates
## 5. 위젯 계약              앵커: 정규화 계약(huni-widget 04_build adapters) — DB 외 앵커임을 명시
## 6. 적재 레시피 (load path) 앵커: raw/webadmin sql/tools · round-8 admin-ui-spec
## 7. 현황·결함 (state)       round-7 매트릭스 셀 + round-13 교정대기 양면표기(아래)
## Sources                   페이지 전체 인용 원천 목록(큐레이션 팩 ID 포함)
```

### 섹션 자기완결성 (R-6 채택)

- 각 H2 섹션 머리에 **상품군명 명시**(예 "## 3. 가격 사슬 — 디지털인쇄"). markdown H2 = 자연 청크 경계이므로 단독 retrieval 시에도 의미가 통하게 한다.
- 앵커 t_* 명을 섹션 머리에 노출(위 템플릿대로). 횡단 링크는 "한 줄 요약 + [[링크]]" 패턴(링크 대상 미로드 시에도 최소 의미 전달).

### 7절 라이브 오적재 양면 표기 (R-8 채택, 필수 슬롯, HARD)

round-13이 라이브를 다수 오적재로 확정했으므로, 7절에 **"라이브 현재값 ↔ 정답"** 표를 필수 슬롯으로 둔다. 라이브 값을 사실로 단정 금지.

```markdown
| 항목 | 라이브 현재값 | 정답 | 상태 | 출처 |
|---|---|---|---|---|
| 레더 자재 | MAT_000186 mat_typ=.08 | .06(가죽) | 🔴 교정대기 | 17_correctness/photobook/correction-manifest.md |
```

- 양면 표기 형식(인라인): `라이브 현재값 X → 정답 Y {🔴 교정대기}`.
- correction-manifest 미대조 라이브값 인용 금지(G-1·F-PB-1 교훈: 인용은 의미 일치까지).
