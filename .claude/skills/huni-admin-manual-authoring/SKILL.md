# Huni Admin 운영자 매뉴얼 집필

비개발자 운영자가 화면을 보며 따라 할 수 있는 **task-oriented 사용 매뉴얼**을 집필하는 방법론. 기술 레퍼런스가 아니라 업무 수행 가이드를 만든다.

## 왜 이렇게 쓰는가 (원칙)

좋은 사용 매뉴얼은 "이 화면에 무슨 필드가 있다"가 아니라 "내가 하려는 일을 어떻게 끝내는가"에 답한다. 운영자는 기능을 공부하러 오지 않고 **일을 끝내러** 온다. 그래서 절차(task)가 1차 구조이고, 화면/필드 설명은 절차를 뒷받침하는 참조다.

### 적용하는 베스트프랙티스

- **Diátaxis** — 문서를 4유형(튜토리얼/howto 절차서/레퍼런스/설명)으로 구분. 이 매뉴얼은 **howto 절차서 + 레퍼런스**의 혼합이 중심이고, "시작하기"는 튜토리얼, "개념"은 최소한의 설명으로 둔다. 한 섹션에 유형을 섞지 않는다(절차서에 긴 개념 설명을 끼우지 않음).
- **task-oriented** — 모든 핵심 챕터는 동사로 시작하는 작업 단위("상품 등록하기", "가격 설정하기", "옵션 구성하기").
- **위키독스/기술문서 스타일** — 짧은 문장, 한 단계 한 동작, 번호 목록, 일관된 용어. 화면 명칭·버튼 명칭은 실제 라벨 그대로 따옴표로(예: "저장", "추가").
- **Anthropic 명료성 원칙** — 모호함 제거, 한 번에 한 가지, 구체적 예시, 독자가 이미 아는 것에서 출발. 전문용어는 처음 등장 시 한 줄로 풀이.
- **접근성** — 모든 스크린샷에 대체 텍스트(캡션), 색에만 의존하지 않는 안내(번호·텍스트 병기), 명확한 제목 위계.

## 문서 구조 (표준 목차)

```
00 시작하기
   - 로그인 (URL·계정·첫 화면=상품 뷰어)
   - 화면 둘러보기 (좌측 도메인 메뉴·홈·상단 바)
   - 모두에게 공통인 동작 (목록 보기·검색·필터·추가·수정·저장·논리삭제·자동채번·표시순서)
   - 안전 수칙 (라이브 운영 DB — 신중히)
[업무별 절차 — howto, 우선순위 순]
   - 상품 등록하기 / 수정하기
   - 상품 하위정보 다루기 (인라인: 사이즈·자재·공정·인쇄옵션·판형·묶음수·추가상품·페이지룰)
   - 가격 설정하기
   - 옵션 구성하기 (상품뷰어 옵션 드릴다운 3계층)
   - 구성 템플릿(SKU) 만들기 (2계층)
   - 제약 설정하기 (폼빌더 + 검증 미리보기)
   - 수량구간 할인 설정하기
   - 기초코드·카테고리·자재·공정 마스터 관리하기
   - 사용처(impact)·SKU 카탈로그 보기
[레퍼런스]
   - 화면 레퍼런스 (전 모델/뷰: 목록 컬럼·필드·필터·동작) — 표 형식
   - 입력 항목 사전 (코드값 도메인·필수·의미)
[부록]
   - 용어집 (운영자 언어 ↔ 시스템 용어)
   - 자주 겪는 문제 / 저장이 안 될 때 (제약 위반 → 해결)
   - 작업 → 화면 역인덱스 ("이걸 하려면 어디로")
```

분량이 크면 챕터별 파일(`manual/00_index.md`, `manual/01_getting-started.md` …)로 분할하고 `00_index.md`에 목차+링크.

## 절차(howto) 서식 표준

각 작업은 동일 골격으로 쓴다:

```markdown
## 상품 등록하기

**언제** 새 상품을 카탈로그에 추가할 때.
**미리** 상품이 속할 카테고리가 카테고리 마스터에 등록돼 있어야 합니다.

1. 좌측 메뉴에서 **"상품"** 을 클릭합니다.
   ![상품 목록 화면](../captures/tprdproducts__changelist.png)
   *목록 화면. 우측 상단 ① "상품 추가" 버튼.*

2. **"상품 추가"** 를 클릭합니다.
   ![상품 추가 폼](../captures/tprdproducts__changeform.png)
   *① 상품명(필수) ② 카테고리(드롭다운) ③ 상품코드(비우면 자동 생성)*

3. **상품명**(`prd_nm`)을 입력합니다. *필수.*
4. **카테고리**(`cat_cd`)를 드롭다운에서 고릅니다. 들여쓰기로 상·하위가 구분됩니다.
5. **상품코드**(`prd_cd`)는 비워 둡니다 — 저장 시 `PRD_000000` 형식으로 자동 생성됩니다.
6. **"저장"** 을 클릭합니다.

> ⚠️ 저장이 안 되면: 빨간 오류 메시지의 필드가 필수 누락이거나, 참조하는 마스터가 없어서입니다. → "저장이 안 될 때" 참조.
> 💡 팁: 표시순서를 비우면 같은 그룹에서 맨 뒤로 자동 배치됩니다.
```

규칙:
- 한 단계 = 한 동작. 클릭/입력/선택을 섞지 않는다.
- 화면 라벨은 실제 한글 그대로 따옴표. 시스템 컬럼명은 괄호 병기(운영자가 개발자에게 문의할 때 다리).
- 스크린샷은 단계 옆에 배치하고 캡션으로 항목 번호를 짚는다.
- 콜아웃: `> ⚠️ 주의`(데이터 위험·실패), `> 💡 팁`(편의), `> ℹ️ 참고`(배경).

## 운영자 언어 번역 규칙

시스템 개념을 운영자가 겪는 현상으로 번역한다(근거: DB 도메인표·소스):

| 시스템 | 운영자 매뉴얼 표현 |
|--------|-------------------|
| NOT NULL | "반드시 입력해야 합니다(비우면 저장 안 됨)" |
| FK 참조 | "먼저 OOO 마스터에 등록돼 있어야 선택할 수 있습니다" |
| CHECK 제약 | "OOO~OOO 범위만 허용됩니다" |
| UNIQUE | "이미 있는 값과 중복될 수 없습니다" |
| 자동채번 PK | "코드는 비워 두세요 — 저장 시 자동 생성됩니다" |
| 논리삭제(del_yn) | "삭제해도 기록은 남고 목록에서만 사라집니다" |
| use_yn='N' | "끄면 더 이상 사용처에서 선택되지 않습니다" |
| 복합PK 인라인 | "이 항목은 상위 화면(상품 등) 안에서 함께 관리합니다" |

## 입력 항목 설명 표준

각 입력 필드는 다음을 갖춘다(추측 금지 — 소스·DB·캡처 근거만):
- 한글 라벨 (+ 시스템 컬럼명)
- 필수/선택
- 입력 가능한 값 (DB 도메인표의 실제 코드값; 드롭다운이면 선택지 목록)
- 의미 (무엇을 뜻하는지)
- 비우면/잘못 넣으면 (기본값·자동채번·저장 실패)

## 품질 체크 (작가 자가검토)

- 모든 절차가 "언제·미리·번호 단계·저장·문제 시"를 갖췄는가
- 모든 화면/모델이 레퍼런스에 1:1 존재하는가(전수 커버리지 — QA가 재검증)
- 스크린샷 경로가 실제 `captures/` 파일과 일치하는가
- 코드값·필수 표기가 DB 도메인표와 일치하는가
- 추측 문장 0 (근거 없는 "아마 ~일 것" 금지)
- 용어집에 처음 등장 전문용어가 모두 등재됐는가
