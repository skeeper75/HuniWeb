# 가격 클러스터: 중복 vs 상보 판정 (쌍별 경계·겹침)

감사일: 2026-06-18 · 근거: 정의 파일(에이전트 .md·오케스트레이터 SKILL.md) + STALE 메모리 표시 + 라이브 사실 실측(pricing.py 실재).

판정 기준: 두 하네스가 **같은 산출을 같은 입력으로** 내면 중복. **입력 단위·렌즈·시점이 의도적으로 다르면** 상보적 레이어(사용자가 이번 세션 포함 의도적으로 만든 구조 — push-back 신중).

## 0. 사용자 설계 의도 (정의 파일이 명시한 레이어 서사)

각 오케스트레이터 SKILL.md의 "정체성" 섹션이 서로를 명시 참조하며 경계를 선언한다 — 즉 **레이어는 우연이 아니라 설계**다.

```
        [§14 huni-price-engine-diag]  ──선행 이해·진단──▶  [§13 huni-price-quote]  ──검증 게이트──▶  결론
         5장치 역할 원리 정의                              대표 상품군 파일럿 P1~P7
         코드↔DB 속성 진단                                생성≠검증
         아는것/모르는것 분리                                    │
                                                              ├──▶ [§15 huni-quote-verify]
                                                              │    단일 상품 온디맨드(Claude+Codex)
                                                              │    §13·§14 산출을 입력 재사용
                                                              ▼
                                              [§7 dbmap 가격 서브셋]  ◀── 실 교정/적재 위임 대상
                                              분석·논의·적재(arbiter·formula·import·option)
```

- §14 SKILL.md: "huni-price-quote(검증·게이트)와 분리된 **선행 트랙** — 검증·결론이 아니라 이해·정의·진단까지"
- §13 SKILL.md: "기존 dbm-price-* 6종은 분석·논의·제안·적재 위주… 본 하네스는 그 산출을 **입력·도구로 재사용**하되 정체성은 다르다(냉철한 검증 관문)"
- §15 SKILL.md: "§13·§14와 달리 **단일 상품 온디맨드** 트랙… 그 둘의 산출을 입력 재사용"
- 세 검증 하네스 모두 "실 COMMIT/교정/DDL은 인간 승인 후 **dbmap 트랙(§7)** 위임" — §7을 적재 백엔드로 명시.

판정: **서사는 일관·의도적**. 단 서사와 실제 정의 파일 사이에 (a) STALE 누수, (b) 검사 항목 실질 중첩이 존재 → 아래 쌍별 분석.

## 1. 쌍별 경계·겹침 판정

### 쌍 A — §14 hped ↔ §13 hpq  (이해 → 검증)
- **경계(명확)**: §14 = "장치가 무엇을 위한 것인가 + 코드가 속성대로 구현됐나"(이해·진단·결론 금지). §13 = "그 장치 권위로 라이브가 권위 엑셀과 맞나"(게이트 GO/NO-GO).
- **겹침**: C1(엔진 계약) vs C10(장치 역할) — §13 engine-cartographer의 `engine-contract`와 §14 mechanism-researcher의 `device-roles`가 **둘 다 evaluate_price 우선순위·차원매칭·단가/합가를 서술**. §14 SKILL.md가 §13 engine-contract를 "인용(권위 아님)"한다고 명시해 중복을 일부 흡수하나, 산출 문서 레벨에서 30~50% 내용 중첩 추정.
- **판정: 상보(레이어 정당) — 단 C1/C10 산출 경계 명문화 필요.** §14는 "왜·원리·미지", §13은 "계약 추출(검증 자)"로 역할을 못박으면 중첩 해소. 근거: §14는 §13 검증자조차 도수축을 오해한 사건(메모리 F-2)에서 태어남 → 선행 이해 레이어의 존재 이유가 실증됨.

### 쌍 B — §13 hpq ↔ §15 hqv  (대표 파일럿 ↔ 단일 상품)
- **경계(의도)**: §13 = 가격산정 4구조(MATRIX·합산형·면적매트릭스·2단룩업) 대표 2~3 상품군 종단(동형 확대 전제). §15 = 사용자가 지목한 단일 상품 온디맨드(+ Codex 병행 + 개선 심의).
- **겹침(상당)**: C3·C5·C8·C13이 거의 동일. §13 hpq-quote-gate-validator의 P1~P7과 §15 hqv-quote-verifier의 3축+골든 재계산은 **검사 항목이 본질적으로 같다**(SOT 일치·공식↔구성요소 매핑·차원 매칭·evaluate_price 실호출 골든 대조). 실제로 §15 메모리(프린트엽서 검증)의 결함 F-1(도수축 붕괴)·F-2(인쇄비 이중배선)는 §13 메모리(엽서 NO-GO N-1a/N-1b)와 **같은 결함을 다시 발견**.
- **판정: 상보적이나 경계 가장 흐림(겹침 최대).** 차별점은 **실재**: (i) §15만 Codex 독립 교차(C20)·명령 해독(C19), (ii) §13만 사이즈 중복(C6)·동형 확대 프레임. 그러나 "단일 상품 검증"은 §13 파일럿의 부분집합으로 보일 위험. **권고: §13 = 구조(동형 클래스) 커버리지 게이트 / §15 = 임의 상품 온디맨드 진입점 + Codex 보강**으로 역할 분리 명문화. 통합은 비권장(Codex 병행·온디맨드 UX가 §15 고유).

### 쌍 C — §7 engine-verifier ↔ §13 gate-validator / §15 quote-verifier  (재구현 ↔ 실호출)
- **경계(설계상 의도)**: §7 = 전 상품군 가격사슬 완전성 + **명세 기반 재계산기 재구현**. §13·§15 = evaluate_price **실호출**.
- **겹침(진짜 중복 + STALE)**: §7 dbm-price-engine-verifier의 **존재 전제가 노후**. 에이전트 .md / 스킬 SKILL.md 둘 다 "`evaluate_price` **미구현**·`pricing.py` **부재**·ROADMAP Phase 11 미완 → 호출할 엔진이 없으므로 **재구현이 핵심**"을 [HARD] 전제로 박음. **그러나 실측: `raw/webadmin/webadmin/catalog/pricing.py` 실재(26KB·2026-06-18)·§13/§15가 라이브 evaluate_price 직접 호출 성공**(메모리: "evaluate_price 미구현 완전 STALE·P1 동치입증 불요"). 즉 §7의 "재구현" 경로는 §13·§15의 "실호출"로 대체됨.
- **판정: D-2(독립 재계산)는 진짜 중복 + §7 경로 STALE.** 재구현 검증기는 실호출이 가능해진 지금 가치 하락. **권고: dbm-price-engine-verify 스킬/에이전트를 STALE 정정(deprecate-or-repurpose).** 상세는 consolidation-proposal.

### 쌍 D — §7 dbm-option-mapper ↔ §13 hpq-option-constraint-mapper  (설계 ↔ 검사)
- **경계(명확)**: §7 dbm-option-mapper = CPQ 옵션 레이어 **설계가**(차원행→option_groups/items 재구성·적재 설계). §13 hpq-option-constraint-mapper = 그 적재 결과가 **의미대로 됐나 검사**(option_items·templates·constraints·dim_vals 정합 보드).
- **겹침**: 둘 다 polymorphic ref_dim_cd·fn_chk_opt_item_ref·BUNDLE을 다룸. 그러나 한쪽은 **만들고**(설계 CSV) 한쪽은 **검증**(결함 보드).
- **판정: 상보(생성≠검증 정석).** 통합 금지 — 같은 에이전트가 설계·검증하면 자기승인. 건전.

### 쌍 E — §7 dbm-price-arbiter ↔ §13 / §15  (심의 단일 SOT)
- §13 SKILL.md "기존 자산 재사용: dbm-price-arbiter" + §15 Phase4 "dbm-price-arbiter로 심의". 즉 두 검증 하네스가 **§7 arbiter를 공유 도구로 호출**.
- **판정: 중복 아님 — 의도적 단일 심의 SOT.** 정립 심의(C18)를 한 에이전트에 모은 건 모범. 유지.

### 쌍 F — §7 dbm-price-formula-auditor / import-builder / formula  ↔  검증 하네스 전체
- §7의 정적 정리(C15)·그릇 분해(C16)·매핑 설계(C17)는 검증 하네스에 대응물 없음.
- **판정: 완전 상보.** 검증 하네스의 입력(골든 그릇·공식 인벤토리)을 만드는 적재·정리 도구.

## 2. 종합 판정

| 쌍 | 관계 | 겹침도 | 조치 |
|----|------|--------|------|
| A §14↔§13 | 상보 레이어 | 중(C1/C10 산출 중첩) | 경계 명문화(원리 vs 계약) |
| B §13↔§15 | 상보(경계 흐림) | **고**(C3/C5/C8/C13) | 역할 분리 명문화(구조 커버리지 vs 온디맨드+Codex)·통합 비권장 |
| C §7 verifier↔§13/§15 | **진짜 중복 + STALE** | **고**(C8 재구현 vs 실호출) | **dbm-price-engine-verify deprecate/repurpose** |
| D §7 option-mapper↔§13 | 상보(생성≠검증) | 저 | 유지 |
| E §7 arbiter↔§13/§15 | 단일 SOT 공유 | — | 유지(모범) |
| F §7 정리·적재↔검증 | 완전 상보 | 무 | 유지 |

**결론: 클러스터는 대체로 의도적 상보 레이어다.** 4개 하네스를 "이해(§14)→게이트(§13)→온디맨드(§15)→적재(§7)"로 분화한 설계는 일관되고, 각 하네스에 고유 가치(C6·C10·C11·C12·C19·C20·C14~C17)가 실재한다.

**단 두 지점이 진짜 문제다:**
1. **§7 dbm-price-engine-verifier/verify = STALE 전제(엔진 부재) 위에 선 중복**(쌍 C). evaluate_price 실재로 "재구현 검증" 명분 소멸 → §13·§15의 실호출과 겹침.
2. **§13↔§15 검사 항목 실질 중첩(쌍 B)** — 통합은 비권장이나 역할 경계를 정의 파일에 명문화하지 않으면 "같은 검증 두 번" 위험.

그 외 STALE 누수(prcx01·clr_cd·frm_typ)는 통합 문제가 아니라 노후 정정 문제 → consolidation-proposal §STALE.
