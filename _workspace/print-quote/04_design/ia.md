## 1. 사용자 멘탈 모델 — 페르소나 × 주요 태스크

화면 그룹화의 기준은 페르소나의 의사결정 흐름이다. (user-journeys.md와 동기화)

### 1.1 P1 — 게스트 비회원 (단순·1회성)
- 상황: 명함 100장만 빠르게 견적·결제. 핵심 질문: "얼마야? 언제 받아? 회원가입 안 해도 되나?"
- 태스크: 메인(SM-M-01) → 리스팅(SM-M-03) → 상품페이지(SM-Q-01) → 옵션·수량 → 가격 확인 → 파일 업로드(SM-Q-10) → 비회원 결제(SM-Q-12) → 비회원 조회(SM-CS-07).
- REQ: REQ-PQ-013, 021, 063~064.

### 1.2 P2 — 반복 주문 회원
- 상황: 매월 동일 사양 스티커 재주문. 핵심 질문: "지난번 옵션 그대로?"
- 태스크: 로그인(SM-AC-01) → 마이페이지 옵션보관함(SM-MY-02) → 견적 재사용 → 장바구니(SM-Q-10.1) → 결제(SM-Q-12).
- REQ: REQ-PQ-039, 094~096.

### 1.3 P3 — 사용자 정의 옵션·복합 견적
- 상황: 박명함·아크릴 등 비규격·종속옵션 견적. 핵심 질문: "이 후가공 가능? 동판비 별도? 매트릭스 밖 사이즈?"
- 태스크: 리스팅 → 상품페이지(SM-Q-01) → 옵션 8축 → 비규격 입력 → 후가공 안내 → LineItem 명세 → 옵션보관함 저장(SM-MY-02) / 견적서 요청(SM-CS-04 대량견적).
- REQ: REQ-PQ-010, 024, 025, 028, 038.

### 1.4 P4 — 디자인 의뢰·풀세트 (V1=에디터 fallback)
- 상황: 디자인 시안부터. V1은 디자인 편집기(에디터 슬롯)로 직접 제작, 디자인 의뢰(SM-Q-14)는 V2.
- 태스크: 상품페이지 → 옵션 → 파일/에디터 분기(SM-Q-10) → 에디터 슬롯 → 미리보기(SM-MY-01.1).
- REQ: REQ-PQ-076, 077.

### 1.5 P5 — 관리자/CS 운영자 (V1 = 단일 관리자)
- **중요:** huni-ia-master는 단일 관리자다. v1.0의 7조직 분할 페르소나(P5a~e)는 V2로 이연. V1 P5는 통합 단일 관리자.
- 핵심 질문: "오늘 신규 결제 주문? 파일 확인할 주문? 상태 변경?"
- 태스크: 관리자 로그인 → 주문관리(SM-A-60) → 파일확인(SM-A-61) → 재업로드 요청(SM-A-62) → 주문서 출력(SM-A-63) → 상태변경+문자(SM-A-64).
- REQ: REQ-PQ-065~067, 074, 048, 049.

```

                                                                                       ├ 디자인상담 (SM-CS-06, V1) — 에디터 연계
                                                                                       └ 가이드 (11종)
```

- 카테고리 메뉴: 출력상품 4종(디지털/책자/실사/패키지) + 포장재 + 굿즈. **수작 브랜드 진입(SM-M-07)은 V2**(D-PM-30 갱신 2026-05-30 — V1 제외·V2 재평가, 브랜드 자산 보존). V1 GNB 카테고리에서 수작은 미노출.
- 통합검색: 상품명 + 카테고리 + 종이/소재명.
- 장바구니: 비회원도 표시. 인쇄옵션+파일이 함께 담긴 모듈(IA-28↳).
- 로그인/마이페이지 토글: 게스트=로그인, 회원=마이페이지.

## 3. 화면 그룹화 원칙

### 3.1 기준

| 기준 | 동일 그룹화 사유 |
|---|---|
| 태스크 응집도 | 동일 의도 연속 방문 화면(견적: 상품→옵션→수량/가격→파일→결제)은 같은 흐름. |
| 데이터 응집도 | 동일 도메인 엔티티(Order/Quote/Product)를 다루는 화면은 LNB 인접. |
| 권한 응집도 | V1은 단일 관리자라 권한 분리 없음. 게시판류·증빙류 등 업무 단위로만 묶음. |
| 상태 일관성 | 주문상태 7-superstate 표시(SM-MY-01.1, SM-A-60) 컴포넌트 공유. |

### 3.2 그룹화 결정 사례

- **견적 흐름(SM-Q-01~04 → SM-Q-10~13):** 상품페이지 옵션 → 파일/편집(IA-28) → 배송(IA-29) → 결제(IA-30) → 완료(IA-31). 단일 상태머신 흐름으로 묶어 실시간 가격 갱신(REQ-PQ-031) + 0.5초 응답(REQ-PQ-021) 동시 만족. URL은 `/quote/{productId}/...`.
- **파일/디자인 분기(SM-Q-10):** 파일 업로드 / 디자인 편집기(에디터 슬롯) / 디자인 의뢰(V2)의 3분기. 사용자에게는 "디자인 입력" 1단계로 보임. 에디터는 외부 편집기 iframe.
- **가격관리 8 팝업(SM-A-14):** 동일 부모 등록 화면에서 진입하는 8 단가표(DP02/04/06/GD01/02/PK01/PR01/02). 좌측 트리 내비.
- **상품Q&A 중복(IA-12 마이페이지 vs IA-22 고객센터):** 동일 게시판의 진입점 2개로 처리(SM-MY-05 = 내 글, SM-CS-03 = 전체). 관리자 답변은 SM-A-35 단일.

## 5. 폼·CTA·에러·확인·빈 상태 패턴

### 5.1 폼 패턴
| 패턴 | 사용처 |
|---|---|
| 단일 페이지 폼 | 회원정보수정·배송지·문의 작성 |
| 마법사(multi-step) | 견적 흐름 / 회원가입 3단계 |
| 인라인 편집 | 관리자 단가표 셀 |
| 모달 폼 | 비규격 사이즈·도서산간 안내·재업로드 요청·쿠폰등록 |

### 5.2 CTA 패턴
| Level | 사용 |
|---|---|
| Primary | "견적 받기", "결제하기", "주문 등록" |
| Secondary | "장바구니 담기", "옵션 저장", "취소" |
| Tertiary | "더보기", "옵션 변경" |
| Destructive | "삭제", "주문 취소", "환불 처리" |

### 5.3 에러 메시지 패턴
| 상황 | 처리 |
|---|---|
| 폼 필드 오류 | 필드 하단 적색 텍스트 + 아이콘 |
| 결제 실패 | 모달 — 사유·재시도·다른 수단 |
| 파일 포맷 오류(REQ-PQ-064) | 업로드 영역 inline + 허용 포맷 |
| 최소주문 미만(REQ-PQ-033) | 수량 입력 옆 가이드 |
| 가격 산출 거부 | 견적 결과 영역 placeholder + 사유 |

### 5.4 확인 다이얼로그 (돌이킬 수 없는 행위)

## 6. 응답성·접근성·모바일 IA 원칙

- 반응형 3 브레이크포인트: 모바일(360~768)/태블릿(768~1024)/데스크톱(1024+).
- 모바일 GNB: 햄버거 + 검색 + 장바구니. 카테고리는 풀스크린 슬라이드.
- 견적 모바일: 단계 indicator 상단 sticky + 가격 미리보기 하단 sticky bar.
- 관리자: 데스크톱 우선. 모바일은 읽기 전용·간단 액션(상태 변경)만.
- 접근성: 모든 폼 label-for, 상태는 색상+아이콘+텍스트 병기, 키보드 내비. (상세 accessibility.md)
