# round-18+ 가격계산 파이프라인 게이트형 심의 — arbiter review

> 작성 2026-06-14 · `dbm-price-arbiter` 심의자. 입력 = 오케스트레이터 SKILL "Pipeline (round-18+)" · verifier/arbiter 스킬 · 엽서 파일럿(`26_price-engine-verify/postcard/`) · `_class-map.md`. 비파괴(분석·심의·제안). 실 교정·파이프라인 개정은 인간 승인.
>
> **사용자 의도(목표 흐름):** ① 구성요소 매핑이 엑셀대로 맞는지 **먼저 검증** → ② **통과해야(GATE)** 가격계산 → ③ 경쟁사 대조 → ④ 가격차로 공식 검증. **핵심: 틀린 매핑 위에서 계산 금지 — 매핑 검증이 계산의 선행 하드 게이트.**

---

## 결론 먼저 (3줄)

1. **현행은 사용자 의도를 달성하지 못한다.** 파이프라인이 "계산(Phase 2) → 정합 정립(Phase 4)" 순서라 **매핑 검증이 계산보다 뒤에** 있다. 게다가 verifier가 자기 코드(`recompute.py`)에 옵션→구성요소 매핑을 **하드코딩 우회**해 틀린 매핑 위에서 "계산 성공"을 내고 있다 — 사용자가 막으려던 바로 그 일.
2. **권고: arbiter의 `mapping-integrity`(엑셀 대조 의미 정합)를 계산 前 하드 게이트(GATE-A)로 승격**하고, verifier의 `chain-completeness`(배선 연결 확인)를 그 앞 구조 게이트(GATE-S)로 분리해 **2단 게이트(구조→의미)**를 만든 뒤, 통과분만 계산(Phase 3)으로 넘긴다. 실 적재 DB 검증은 round-13(라이브=교정대상)을 GATE-A의 라이브 실측 차원으로 재사용한다.
3. **신규 에이전트 불필요.** 기존 5에이전트(verifier·arbiter·benchmark·validator + round-13 correctness-auditor)의 **호출 순서·게이트 지위 재배치**만으로 폐루프가 닫힌다. arbiter 스킬에 "계산 前 게이트" 모드와 "재검증" 종착 게이트만 추가하면 된다.

## ① 현행 게이트 진단 (달성 / 미달)

현행 Phase 흐름(오케스트레이터 SKILL line 351~365):

```
Phase 1 클래스 분류
Phase 2 클래스별 재계산 (verifier)          ← 계산이 먼저
Phase 3 경쟁사 벤치마크 (benchmark)
Phase 4 정합 정립 심의 (arbiter)            ← 매핑 의미 정합이 계산보다 뒤
Phase 5 게이트 (validator: PE/B/A)
Phase 6 보고
```

| 사용자 의도 요건 | 현행 | 판정 | 근거 |
|---|---|:--:|---|
| ① 구성요소 매핑이 엑셀대로 맞는지 **먼저** 검증 | arbiter `mapping-integrity`가 Phase 4(계산 後) | ❌ **미달** | mapping-integrity는 "각 클래스의 가격=f(자재·공정·사이즈·도수·옵션)가 엑셀대로 적재됐나"를 판정하는 **바로 그 단계**인데, Phase 2 계산 뒤에 놓여 선행 게이트가 아님 |
| ② 검증 통과해야(GATE) 계산 | 계산(Phase 2)에 진입 게이트 없음 | ❌ **미달** | verifier Phase 0은 "입력 파일 존재"만 점검(스킬 §3). 매핑 정합 GO 여부가 계산 진입 조건이 아님 |
| 틀린 매핑 위에서 계산 금지 | **위반 실증됨** | 🔴 **역행** | 아래 "결정적 증거" 참조 — verifier가 `compute_corrected()`로 우회 계산 |
| 구조 배선 확인 vs 의미 정합 2단 분리 | `chain-completeness`(배선)와 `mapping-integrity`(의미)가 **다른 라운드·다른 게이트**로 흩어짐 | 🟡 **부분** | 둘 다 존재하나 "구조 통과→의미 검사→통과→계산"의 순차 게이트로 엮여 있지 않음 |
| ③ 경쟁사 대조로 가격차 확인 | benchmark Phase 3 | ✅ 달성 | absurd-gap = 우리측 결함 신호 directive 반영 |
| ④ 가격차로 공식 검증 | arbiter root-cause가 🔴 받아 원인 규명 | 🟡 **부분** | 원인 규명은 하나, "공식 검증 결과 → 재검증" 폐루프 미연결(아래 ④) |
| 개선/수정 시 요건 충분 반영(폐루프) | 없음 | ❌ **미달** | arbiter가 remediation-plan(제안)까지만. 교정 후 **재검증 게이트**가 파이프라인에 없음 |
| 실 적재 DB가 엑셀 권위와 맞는지 검증 단계 | 명시 부재 | ❌ **미달** | mapping-integrity가 "라이브 적재 반영"을 보긴 하나, round-13(라이브=교정대상 전수 diff) 트랙과 연결 안 됨 |

### 🔴 결정적 증거 — "틀린 매핑 위 계산"이 이미 일어남

엽서 파일럿(`postcard/recompute.py` line 86~104 `compute_corrected`)에서:

- **명세 순수 차원매칭(naive)** = 27개 comp 동시매칭 = **22,849,330원** (라이브 데이터를 11-CONTEXT 명세대로 그대로 돌린 값).
- **보정매칭(corrected)** = **44,330원** (verifier가 `side/coating/spot/postproc → comp_cd`를 **파이썬 코드에 하드코딩**해 옳은 구성요소만 골라낸 값). 516배 차이.
- `defects.md` D-2가 정직하게 적시: *"고객 선택(면·코팅·별색·후가공)을 명시적으로 구성요소 집합에 매핑해야 함(recompute.py 보정매칭). 이 매핑 권위 = CPQ 옵션 레이어일 가능성 — 엔진이 옵션 선택→comp를 해석해야."*

**의미:** 라이브 적재에는 "옵션 선택 → 어느 comp를 합산할지"를 가르는 차원/배선이 **없다**(D-2). verifier는 그 빠진 매핑을 **자기 코드로 임시 보충**해 계산을 통과시켰다. 즉 현행 파이프라인은 "매핑이 불완전한데도 계산이 GO를 낸다" — 사용자가 정확히 막으려던 상황. **계산값(44,330)은 verifier의 외부 지식이 만든 것이지 라이브 매핑이 만든 것이 아니다.** 이건 PE5(결함 정직성) 게이트는 통과하지만(우회를 숨기지 않음), 사용자 의도(매핑 GATE 선행)는 위반한다.

> 이 한 건이 진단 전체를 압축한다: **매핑 정합을 계산 前 하드 게이트로 두지 않으면, 계산기는 매핑 결함을 코드로 메워 거짓 GO를 낸다.**

| 재검증 | 없음 | **Phase 9 ⟳ GATE 재실행** | 사용자 의도 — 폐루프(요건 충분 반영) |

### 2단 게이트 역할 분리 (심의질문 1 직접 답)

- **GATE-S (구조 / verifier chain-completeness):** "사슬이 *연결*됐나?" — 공식 바인딩 존재, formula_components 배선 ≥1, comp마다 component_prices 단가행 ≥1, t_dsc 연결. **연결 자체만** 본다(값이 옳은지는 안 봄). 라이브 실측. 엽서 파일럿은 여기 **통과**(사슬 데이터 완결).
- **GATE-A (의미 / arbiter mapping-integrity):** "*엑셀대로* 맞나?" — 엑셀이 규정한 구성요소가 빠짐없이 배선됐나, 차원 키가 엑셀 가격축과 같나(같은 값이 엉뚱한 t_*/차원에 갔나), prc_typ 단가/합가 정확한가, 옵션 선택→comp 해석이 데이터에 존재하나. 엽서 파일럿은 여기서 **🔴 불일치**(D-2: 옵션→comp 매핑 부재). **즉 구조는 통과하지만 의미는 실패** — 이 분리가 정확히 사용자가 원하는 2단 게이트다. 현행은 이 의미 게이트를 계산 뒤에 둬서 무력화됐다.

> 엽서가 GATE-S 통과·GATE-A 실패라는 사실이 2단 분리의 정당성을 실증한다. "단가행이 다 있다(구조 OK)"와 "그 단가행들이 옵션 선택에 따라 옳게 골라진다(의미 OK)"는 다른 실패면이고, 후자가 계산 정확도를 좌우한다.

## ③ 실 적재 DB 검증 단계 정립안 (심의질문 2)

**진단:** 현행 파이프라인에 "라이브에 *적재된* 가격·구성요소 데이터 자체가 엑셀 권위와 맞나"를 전수 검증하는 단계가 **명시 부재**. mapping-integrity가 "라이브 적재 반영"을 보긴 하나 클래스 대표 1건 수준이고, round-13(라이브=교정대상·전수 diff·webadmin load_master oracle)과 **연결 안 됨**.

**권고: 신규 단계 신설 대신 round-13을 GATE-A의 라이브 실측 차원으로 재사용.** GATE-A는 두 축으로 구성:

| GATE-A 축 | 출처 | 무엇을 검증 |
|---|---|---|
| A-1 엑셀↔공식정의 정합 | 권위 엑셀(가격표·상품마스터) ↔ 라이브 `t_prc_*` | 엑셀이 정한 구성요소·차원·prc_typ가 공식 배선에 맞게 반영됐나 (arbiter 본연) |
| A-2 적재값 정확성(전수) | round-13 `17_correctness/<family>/` correction-manifest | 라이브에 적재된 size·자재·공정·도수·옵션이 엑셀대로인가 (= 가격 차원 키가 가리키는 마스터 행 자체가 옳은가) |

**왜 재사용이 맞나:** round-13은 이미 라이브를 "피고", 엑셀+webadmin oracle+스키마의도를 "판관"으로 전수 diff하는 트랙이다(11/11 시트 중 6 GO). 가격 차원 키(siz_cd·mat_cd·proc_cd·clr_cd)는 **그 마스터 행을 참조**하므로, 마스터 행이 round-13에서 MIS-LOADED면 가격도 틀린 행을 조회한다. 별도 가격용 전수검증을 새로 만들면 round-13과 중복·드리프트. **GATE-A는 A-1(가격 구성 정합·arbiter) + A-2(참조 마스터 정확성·round-13 인용)을 합쳐 판정**한다.

- A-2가 해당 클래스의 참조 마스터에 round-13 미GO(미감사 또는 MIS-LOADED 잔존)가 있으면 → GATE-A는 그 클래스를 **"마스터 미정합 — 계산 차단"**으로 분리(추측 통과 금지). 이때 라우팅 = round-13 해당 시트 선행.
- A-2 round-13 GO + A-1 정합이면 → GATE-A ✅, 계산 진입.

이로써 "실 적재 DB가 엑셀과 맞나"가 파이프라인에 명시되고, round-13과 1:1 연결된다(재유도 0).

## ④ 폐루프(요건 충분 반영) 보장 방안 (심의질문 3)

**진단:** arbiter는 remediation-plan(제안)까지가 종착이고, 그 제안이 round-5/13/ddl-proposer로 적용된 뒤 "요건이 충분히 반영됐는지"를 다시 확인하는 게이트가 없다. → 열린 루프.

**권고: Phase 9 재검증 게이트(GATE 재실행) 신설로 폐루프 강제.** 메커니즘:

```
remediation-plan(GO 제안) → [인간 승인] → round-5(적재)/round-13(교정)/ddl-proposer(신규 엔티티) 실 적용
   → ⟳ 해당 클래스 GATE-S + GATE-A 재실행(arbiter→validator)
       ├ 재통과 = 그 클래스 "정립 완료" 마킹(remediation-plan에 RESOLVED 기록 + 라이브 실측 증거)
       └ 미통과 = 잔존 결함 finding으로 다시 remediation-plan(루프 계속)
```

**충분 반영 보장의 3요소:**

1. **요건 명세화:** remediation-plan의 각 정립 항목은 "**재검증 기준**(어떤 GATE 축이 어떤 라이브 실측값이면 통과)"을 함께 적는다. 예: D-2 정립 = "option_items에 comp 연결 후 → GATE-A A-1에서 단면 선택 시 활성 comp 집합이 라이브 매핑만으로 {S1·용지·선택코팅·선택별색}로 한정됨을 재계산이 보정 하드코딩 없이 재현". 기준 없는 정립은 닫을 수 없다(검증 불가).
2. **생성자≠검증자 유지:** Phase 9 재검증도 arbiter(재판정)·validator(A1~A6 독립 게이트) 분리. 자기 정립안을 자기가 GO 안 함.
3. **돈-크리티컬 보수성:** 재검증 미통과·불확실은 "정립 미완·컨펌 필요"로 분리(추측 RESOLVED 금지). 가격은 청구 금액이라 거짓 완료가 곧 과·오청구.

이로써 "구성요소 매핑 검증 GATE → 계산 → 경쟁사 → 공식 검증 → 정립 → **재검증**" 폐루프가 닫힌다.

## ⑤ 과설계 경계 — 신규 에이전트 필요 여부 (심의질문 4)

**권고: 신규 에이전트 0개.** 게이트형 흐름은 **기존 에이전트의 호출 순서·게이트 지위 재배치 + 스킬 모드 추가**로 전부 달성된다.

| 필요 역할 | 담당 (기존) | 추가 작업 |
|---|---|---|
| 클래스 분류 | verifier / 인라인 | 없음(`_class-map.md` 기존) |
| GATE-S 구조 배선 | **verifier** (chain-completeness) | 게이트 지위 명문화(통과/단절→계산 진입 차단) |
| GATE-A 의미 정합(계산 前) | **arbiter** (mapping-integrity) | 스킬에 "계산 前 게이트 모드" + A-2(round-13 인용) 추가 |
| 적재값 전수 정확성 | **round-13 correctness-auditor** | 신규 아님 — GATE-A A-2로 인용 재사용 |

| 근본원인·정립 | **arbiter** (root-cause·remediation) | 재검증 기준 명세 의무 추가 |

**과설계 경계선 (하지 말 것):**
- ❌ "가격 적재 전수검증 에이전트" 신설 — round-13과 중복. A-2 인용으로 충분.
- ❌ "게이트 오케스트레이터 에이전트" 신설 — 게이트 순서는 오케스트레이터 SKILL Pipeline 섹션의 Phase 선언으로 표현(에이전트 아님).
- ❌ recompute에 정교한 옵션→comp 추론 엔진 추가 — 그건 보정 하드코딩의 연장(거짓 GO 재발). 옵션→comp는 **데이터(option_items/차원)로 정립**할 사안이지 계산기가 추론할 사안 아님(D-2 정립 권고와 일치).

**최소 변경 적용 목록 (인간 승인 대상):**
1. 오케스트레이터 SKILL "Pipeline (round-18+)" 섹션: Phase 2~4를 위 ②의 GATE-S→GATE-A→계산 순서로 재배치 + Phase 9 재검증 신설.
2. `dbm-price-arbiter` SKILL: §1 mapping-integrity에 "**계산 前 하드 게이트(GATE-A)**" 지위 + A-2(round-13 인용 차원) 명문화 + remediation-plan에 "재검증 기준" 필수 필드 추가.
