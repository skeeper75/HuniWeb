# 개발 수정 요청서 — `fn_calc_pansu` 판걸이수 산출 함수

작성 2026-07-01 · 대상=개발팀(HuniProductPrice2 레포 + Railway 라이브 DB) · 라이브 읽기전용 실측 재현. 실 함수 교체=개발팀 배포·인간 승인.

## 0. 한눈 요약 (비전문가)

디지털인쇄는 큰 종이(전지) 1장에 작은 제품을 여러 개 앉혀 한 번에 찍습니다. "한 장에 몇 개 앉느냐"가 **판걸이수**이고, 이 숫자로 필요한 전지 매수가 정해져 **인쇄비·용지비**가 매겨집니다. 지금 가격엔진의 `fn_calc_pansu`는 **종이 면적만 보고 기하학적으로 꽉 채워** 계산해서, 실제 인쇄의 칼선 간격·물림변·자재별 여백을 빼먹습니다. 그 결과 실무진이 엑셀에 손으로 적은 판걸이수와 어긋나 **가격이 양방향(저/과청구)으로 틀립니다(±7~80% 실측)**. 일부 큰 제품(썬캡)은 아예 0이 나와 **견적 자체가 불가**합니다. 실무진 판걸이수가 정답이므로, 엔진이 그 값을 먼저 쓰도록 고쳐 주세요.

---

## 1. 현재 함수 (고칠 대상)

`raw/webadmin/sql/32_fn_calc_pansu.sql` (라이브 동일):

```sql
-- 판 실제영역 = 판형 작업사이즈(work) − 여백
pw := plate.work_width  - margin_lft - margin_rgt;
ph := plate.work_height - margin_top - margin_bot;
-- 아이템 = 작업사이즈(여분포함)
iw := item.work_width;  ih := item.work_height;
IF (치수 NULL/0) THEN RETURN NULL;   -- 계산불가
n_normal := floor(pw/iw) * floor(ph/ih);   -- 정상 배치
n_rotate := floor(pw/ih) * floor(ph/iw);   -- 90° 회전
RETURN GREATEST(n_normal, n_rotate);       -- 더 많이 앉는 쪽
```
- 입력: `(판형 siz_cd, 아이템 siz_cd)` 2개. 시그니처에 **상품·자재 정보 없음**.
- 호출부: `pricing.py:287 _calc_pansu()`, `:215 plate_qty()`(판수=⌈주문수량÷판걸이수⌉), `price_views.py:1495`.

## 2. 한계 (왜 틀리나)

| # | 한계 | 결과 |
|---|---|---|
| L1 | **칼선 간격(gutter) 미반영** — 제품 사이 여백 0 가정 | 한 행/열 더 앉는 것으로 착각 → 판걸이수 과다 → 저청구 |
| L2 | **물림변(gripper) 미반영** — 기계가 무는 변 여백 미차감 | 동상 |
| L3 | **자재별 임포지션 차이 미반영** — 투명PET 등은 더 넓게 앉힘 | 투명엽서 전 사이즈 +43~50% 저청구 |
| L4 | **출력용지규격(전지) 매칭 미반영** — 큰 제품에 맞는 전지 선택 로직 없음 | 썬캡(작업320×404)은 등록 전지에 기하학적으로 0개 → **fn_calc_pansu=0 → 견적불가** |

핵심: **순수 기하 패킹(면적÷면적)** 이라 실무 인쇄의 물리 제약을 못 담습니다. 방향이 양쪽(과다=저청구·과소=과청구)으로 갈립니다.

## 3. 권위 (정답)

- **상품마스터 디지털인쇄 시트 "판수" 컬럼**(27 상품×사이즈, 실무진 기록) — 1차 권위.
- **인쇄상품가격표 "판걸이수" 시트**(94행, 카테고리×사이즈 + 출력용지규격) — 보강·미출시 상품 포함.
- 두 권위 일치(엽서 100×150=8 등). 실무 엑셀 = 이전 후니프린팅(ASP) 산식일 가능성(교차검증 키).

## 5. 극단 사례 — 썬캡 (PRD_000051·견적 0)

- 권위(판걸이수 시트): 썬캡 출력용지규격 **3절 330×540**, **판걸이수 1**, 재단 320×404.
- 현재: 판형이 국4절(SIZ_000499 316×467)로 바인딩 → 작업 320×404가 국4절 재단영역(306×457)보다 커서 `fn_calc_pansu=0` → 인쇄·용지·커팅 전부 합산 제외 → **견적 0(불가)**.

## 6. 제안 수정

### 6-1. lookup 우선 (핵심)
권위 판걸이수 저장소를 먼저 조회, 없을 때만 기존 기하 폴백:
```sql
SELECT pansu INTO v FROM t_siz_pansu WHERE ...키...;
IF v IS NOT NULL THEN RETURN v; END IF;
-- (폴백) 기존 기하 계산 — 권위 미등록 사이즈 회귀 방지
... RETURN GREATEST(n_normal, n_rotate);
```
폴백 유지 → 권위 미등록분은 기존 동작 보존(회귀 격리).

### 6-2. ★키 설계 — 자재/상품 종속 주의 (설계 결정 필요)
투명엽서(98×98)=판수 **8** vs 일반엽서(98×98)=**12** — **같은 (판형,사이즈)인데 자재(상품)별로 판수가 다름**. 따라서 `(plate, item)` 키만으로는 투명을 표현 못 함.
- **(권장) `(prd_cd, item_siz_cd) → pansu`** — 상품 단위. 호출부 `pricing.py`가 prd_cd 보유 → `fn_calc_pansu`에 prd_cd 인자 추가(`fn_calc_pansu(plate, item, prd_cd DEFAULT NULL)`) 또는 pricing.py에서 lookup 후 함수 폴백.
- (대안) `(plate, item, mat_typ)` — 자재 임포지션 군으로.
- 시그니처 변경 시 호출부(`pricing.py:287`·`price_views.py:1495,1888`)도 prd_cd 전달하도록 소폭 수정.
