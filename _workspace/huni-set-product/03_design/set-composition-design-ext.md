## 3. ★택1 면지/표지 함정 판정 (directive 3 [HARD] · §21 always-add 가드)

### 3.1 라이브 실측 = 평면 다중행(현황)

하드커버/링책자/포토북의 동일 역할(면지·표지) 다중 구성원이 **별개 sub_prd_cd 평면 행**으로 등록(disp_seq 전부 1). 이는 권위 엑셀의 `제본(옵션)_면지(옵션)` = **색상 택1 옵션값**(화이트/블랙/그레이/인쇄)을 라이브가 행으로 평면화한 것(set-authority-spec §2.4 CONFIRM-2).

### 3.2 판정 = **(b) 라이브 현황 유지** (행 삭제/슬롯1개化 안 함)

근거 — **현 가격 모델에서 평면 4행은 과대청구가 아니다**:
- `evaluate_set_price`(pricing.py:718)의 `members`는 **호출자가 런타임에 전달하는 배열**이다(`for mb in members`). t_prd_product_sets의 행은 **선택 가능한 구성원 카탈로그**이지 자동 SUM 대상이 아니다. 손님이 면지 1색을 고르면 호출자가 그 1개만 `members`에 넣는다 → 택1이 호출단에서 해소.
- 게다가 **면지 구성원 전부 가격공식 0·가격 차원 0**(라이브 실측: 074/075/076/087/104 등 member에 사이즈·자재·공정 0). 따라서 설령 4행을 동시에 넣어도 각 contribution=0 → **합산 오염 0**. always-add silent 가산(§21·RC-2 교훈)의 전제(차원행 존재)가 성립 안 함.
- 행 삭제/슬롯1개化는 **권위 택1 옵션값 4종을 손실**시킨다(상품뷰어 면지색 선택지 = 이 4행). 구조적 손실이므로 금지.

### 3.3 평면 4행 합산 금지 가드(미래 위험 차단)

현재 무해하나, **면지/표지 구성원에 향후 가격공식·차원이 붙으면 평면 합산이 즉시 과대청구로 전환**된다(RC-2 always-add 함정과 동형). 따라서:
- 택1 그룹 의미를 **note + disp_seq**로 명시(이번 보정 §2).
- 가격공식 신설 시(§18/dbmap) **면지 색상은 셋트 member 합산이 아니라 옵션축(상품뷰어 자재추가/옵션 택1)으로 모델링**하도록 BLOCKED 보드에 가드 명시(아래 §6 GUARD-1). evaluate_set_price 호출단이 택1 1개만 members에 전달하는 계약을 가격설계가 보증해야 함.

→ **이번 적재본 = 라이브 현황 유지(행 보존)·disp_seq/note 보정만**. 택1 구조 오염 정정 불요(오염 아님). 단 가격 신설 시 GUARD-1 준수.

## 4. 개수규칙(min/max/incr) 충전 판정 (directive 2)

엽서북은 내지(095)가 실 member이고 페이지 20/30/10이 member-level 가변이라 충전 가능했다. **6셋트는 전부 충전 불가** — 사유별:

| 072 하드커버책자 | 24/300/2 (booklet-l1 row38) | **없음**(내지=`*별도설정`=MES) | ✗ BLOCKED | 페이지축=부모상품 옵션·내지 member 미등록 |

| 088 레더 링바인더 | 권위 페이지 공란(row55) | 없음 | ✗ BLOCKED | 권위 침묵(내지 미정의·바인더형) |

| 100 포토북 | **미특정**(CONFIRM-3) | 내지101 있음 | ✗ BLOCKED | 권위행 미특정(모호) |

핵심 — **페이지 가변은 셋트-member 범위(min_cnt/max_cnt/cnt_incr)가 아니다**:
- 하드커버/링책자: 내지가 MES `*별도설정`이라 **sub_prd_cd 자체가 없다** → 충전할 행이 없음. 페이지(24~300)는 부모상품의 페이지축/옵션으로 별도 모델링 대상(셋트 행 아님).
- 떡메: 권위 페이지 공란 → 충전 금지(directive). 제작수량 3/1000/3은 **부수(copies)**이지 member-qty가 아님(evaluate_set_price copies 인자). 묶음수 50/100은 `t_prd_product_bundle_qtys`(별 테이블·라이브 실재) 소관.
- 포토북: CONFIRM-3 권위 미특정 → 모호로 BLOCKED.

→ **6셋트 전부 min/max/incr NULL 유지**. directive "권위 침묵분은 충전하지 말고 blocked" 충실. 페이지축은 §6 RM-4(부모 페이지옵션·내지 member)로 라우팅.

## 5. 가격 사슬 점검 (directive 5 · evaluate_set_price)

### 5.1 라이브 바인딩 실측 (2026-06-24)

| 상품 | 가격공식 | 결과 |
|---|---|---|
| 6셋트 부모(072/077/082/088/097/100) | **전부 없음** | set_eval base=0 |
| 전 구성원(073~107, 엽서북094 외) | **전부 없음** | member contribution=0 |

→ `evaluate_set_price` = Σ(구성원 0) + 셋트공식(0) + 할인 = **base_total 0 → final_price 0 → PRICE=0 견적 불가**.

### 5.2 엽서북 교훈 적용 — "부재" vs "미바인딩/오청구" 구분

엽서북 게이트가 30P를 "부재"가 아닌 "comp 존재·미바인딩→오청구"로 정정한 교훈(set-verdict R-1)을 적용해 6셋트를 라이브로 구분:
- 6셋트는 **부모 공식 자체가 0**(PRF_* 바인딩 행 없음)이고, **구성원 공식·차원도 0**이다. 엽서북처럼 "comp는 있는데 미바인딩"이 아니라 **셋트공식·구성요소가 라이브에 아예 없음**(진성 부재).
- 따라서 6셋트의 가격 결함 = **진성 BLOCKED-PRICE**(셋트공식 신설 필요). 30P류 "미바인딩 오청구"와는 등급이 다름(과소청구가 아니라 견적 자체 불가).

→ **6셋트 전부 가격 BLOCKED**. set-designer는 셋트 행만 보정·가격공식 신설 안 함(§18 재사용·없으면 BLOCKED). §18/dbmap에 셋트 제본/조립 공식(PRF_BIND_* 류) 신설 라우팅.

### 5.3 이중합산 점검

- 구성원가 0·셋트공식 0 → 합산 자체가 0 → 이중합산 위험 0(현 상태). 단 §3.3 GUARD-1대로, 가격 신설 시 면지 택1 평면합산 + 제본비 셋트공식 이중 계상 방지가 설계 계약.
