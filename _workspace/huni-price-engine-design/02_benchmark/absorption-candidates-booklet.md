## 1. 제본방식별 부품 합산 분기 — 경쟁사가 어떻게 나누나 ★

### 경쟁사 관측

| 제본방식 | RedPrinting `[red:book·TP]` | WowPress `[wow:booklet]` | 부품 합성 차이 |
|----------|-----------------------------|--------------------------|----------------|
| **중철** | `STA_CLD`(중철·쫄대)·INN_PAGE 소(효도달력 1면) | 중철책자 variant | 표지=내지 동일 용지 多 → 부품 1~2개·제본 단순 |
| **무선/PUR** | 무선 24~300면·제본방향 | 무선책자·특가 variant | 표지(별 용지) + 내지(페이지수 비례) + 무선/PUR 제본 = 3부품 |
| **트윈링/스프링** | `book2025` 트윈링·`RIN_DFT`(링)·제본방향(좌철/상철) | 스프링·윤전책자 | 표지 + 내지 + 링자재(별도) + 제본공정 |
| **하드커버(싸바리)** | 하드커버 24~300·`COV_MIN_WGT`·PVC추가커버 | (해당 캡처 외) | **표지(하드보드+싸개) + 면지 + 내지 + 제본** = 최다 부품 |

★**제본방식별 부품 수가 다르다**: 중철(부품 적음·합산 단순) → 무선/PUR(표지+내지+제본 3항) → 트윈링(+링자재) → 하드커버(표지=하드보드+싸개+면지 = 최다·면지가 별 prd_cd로 분리될 만큼 부품성). RedPrinting `book2025_price`가 제본방식에 따라 합산 부품 집합을 분기. WowPress는 `jobqty0`(작업량) 환산 시 제본방식별 연(매수)·도무송수 가중치가 다름.

### 후니 매핑 — 흡수 불요(그릇 동형)·배선이 본 과업

| 항목 | 후니 그릇 `[huni:live·prior]` | 흡수 판정 |
|------|------------------------------|----------|
| 제본방식 분기 | 제본 comp **11종**(`COMP_BIND_JUNGCHEOL/MUSEON/PUR/TWINRING/SSABARI/HC_MUSEON/HC_TWINRING/CAL_*`)·`comp_typ=.04 공정비` | ★라이브 실재 — RedPrinting `book2025` 제본 variant와 1:1 동형 |
| 부품 합산 | 합산형 `addtn_yn='Y'` Σ + 세트 부모 prd(표지/면지 별 prd_cd) | ★그릇 보유·**(A) 중철~트윈링은 제본비 단일항만 배선**(`PRF_BIND_SUM`)·**(B) 하드커버는 미바인딩** |
| 제본방식별 부품 집합 | 상품마스터 BOM이 부품 집합 정의(표지/면지/내지) | 권위 엑셀이 분기의 최종 권위 |

★**흡수 실질 = 0건(메커니즘)**. 후니는 제본방식 분기(comp 11종)·부품 분리(별 prd_cd 세트)를 경쟁사보다 명시적으로 보유. **흡수가 아니라 배선-gap**: (A) 중철~트윈링이 "제본비 단일항"만 배선 / (B) 하드커버류가 미바인딩. → designer가 **제본방식별 부품 집합을 합산 공식에 배선**(표지비 + 내지비[페이지수] + 제본비 Σ·권위 가격표가 부품 단가 주면 부품 합산이 정답·D-BIND-SCOPE 인간 결정).

### 가드
- **제본방식별로 부품 수가 다르다** → 한 공식 강제 금지. 중철=제본비 단일항이 정당할 수 있음(부품 분리 부재). 하드커버=표지+면지+내지+제본 다항.
- **(B) 표지/면지 별 prd_cd = 세트 "구성"** vs **합산 공식 = 세트 "가격"** 분리(P-1b 원칙·디지털 종단 set-product-design §0 실증).

## 2. 이중수량(부수×페이지) + 페이지 파생 — 앱계산 vs DB저장 ★

> 후니 "중간계산=앱·DB=룩업" 원칙(`dbmap-compute-in-app-db-stores-lookup`)과의 정합 확인이 본 절 핵심.

### 경쟁사 관측 — 어디서 계산하나

| 계산 | RedPrinting `[red:book·TP]` | WowPress `[wow:booklet]` | 계산 위치 |
|------|-----------------------------|--------------------------|----------|
| **페이지수(INN_PAGE)** | `MIN_INN_PAGE=2`·`MAX=130`(트윈링)·캘린더 2~200·STEP1 = **입력 차원** | `jobqty0` 입력 인자(페이지수) | ★**입력**(DB는 min/max/step 제약만 저장) |
| **이중수량(부수×디자인수)** | `ORD_CNT`="디자인 수(건수)" × `PRN_CNT`="수량" (`usePDFordCnt`·`quantityGroup`) = 2축 입력 | `jobqty0`(작업량) ← 부수·페이지·제본 환산 | ★**입력 2축**(RedPrinting) / **서버 환산**(WowPress jobqty) |
| **내지 단가 = 1면 × 페이지수** | 내지 단가 페이지수 비례(곱) | `jobqty0`→`jobcost0` 2단 API에서 곱 | ★**런타임 계산**(곱)·DB는 1면 단가만 |
| **책등(seneca)** | `seneca=0.64`·`max 1000` = 페이지수×내지두께 **파생값**(입력 아님·무선/하드커버 표지 영향) | (해당 없음) | ★**파생 계산**(앱·DB 미저장) |

★**RedPrinting = 페이지수·이중수량 입력 + 곱/파생은 런타임**. WowPress = `jobqty0`(작업량) **서버 사전 환산** → `jobcost0` 2단(페이지·제본·부수를 작업량 단일 스칼라로 압축 후 단가 룩업).

### 후니 매핑 — 후니 원칙과 정합·흡수 불요

| 후니 처리 `[huni:live·prior]` | 정합 판정 |
|------------------------------|----------|
| 페이지수 = `t_prd_product_page_rules`(11행·page_min/max/incr) **입력 차원** | ★RedPrinting INN_PAGE 1:1 동형·라이브 실재 |
| 이중수량 = `bdl_qty`(묶음수) + 수량 차원 / 디자인수(ORD_CNT)는 옵션 입력 | 후니 그릇 보유(P-2)·디자인수는 견적 입력축 |
| 내지단가×페이지수 곱 = **앱 런타임**·DB는 1면(또는 1대) 단가 룩업 | ★메모리 `compute-in-app-db-stores-lookup` **정확 정합**(off-grid ceiling·판수·박등급과 동일 철학) |
| 책등 = 페이지수×내지두께 **앱 파생**·DB 미저장 | ★동일 철학(seneca 신규 축/단가행 불요) |

★**흡수 판정 = RedPrinting 정합(앱계산/DB룩업 분리), WowPress 2단 엔진 부결**:
- **RedPrinting 방식 = 후니와 같은 철학** — 페이지수/이중수량/디자인수는 입력, 곱·책등은 런타임. 후니가 그대로(흡수 불요).
- **WowPress `jobqty0→jobcost0` 작업량 2단 엔진 = 흡수 부결**(C-ST6 재확인). 후니 단일 `evaluate_price` + 앱 계산(작업량 환산을 앱에서)/단가 룩업 분리로 이미 표현. 작업량을 DB 사전 환산하는 2단계는 **과분화**(후니 원칙 위배·DB가 파생값 저장). ★다만 WowPress가 보여주는 **"제본방식·페이지·부수 → 단일 작업량 스칼라 환산"**의 환산 규칙은 designer가 후니 **앱 계산 로직**(내지 출력매수·판수)으로 흡수할 만한 도메인 단서(가격축 신규 아님·앱 함수).

### 가드 [HARD]
- **페이지수 = 입력 차원·SKU 폭발 금지**(2~300면을 siz_cd SKU로 베이크 금지·접지 P-3와 구분·디지털 종단 set-product-design §3 확정).
- **곱·책등 = 앱 런타임·DB 미저장**(WowPress처럼 작업량을 DB에 사전 저장하면 후니 원칙 위배).
- **이중수량(부수×디자인수) 라우팅** — 디자인수(ORD_CNT)는 가격에 곱해지는 입력이지 단가 차원 아님. 부수=수량 차원(볼륨할인), 디자인수=주문 건수 곱. designer가 두 축 구분(혼합 금지·돈크리티컬).

## 3. 제본비/표지/내지 단가의 prc_typ 성격 — 경쟁사 단서 + 라이브 결판 ★★

> designer의 돈크리티컬 결판(부수당 단가 vs 묶음총액) 보강. 디지털 종단의 .01→.02 교정 클래스와 같은 사안 — **책자는 라이브 실측이 결판을 줌**.

### 3.1 ★라이브 제본비 단가행 = 부수당 단가(볼륨할인)·묶음총액 아님 `[huni:live]`

라이브 `COMP_BIND_*` 단가행 실측(2026-06-20·읽기전용):

```
COMP_BIND_CAL_DESK130:  min_qty 1→5000 · 4→4000 · 10→3000 · 50→2500 · 100→2300 · 1000→2000
COMP_BIND_CAL_DESKMINI: min_qty 1→4500 · 4→3500 · 10→2500 · 50→2000 · 100→1800 · 1000→1600
prc_typ = PRICE_TYPE.01 (단가형) · comp_typ = .04 (공정비) · use_dims = [proc_cd, min_qty, proc_grp:PROC_000017]
```

★**결정적 단서**: 제본비 단가가 **수량(min_qty)이 오를수록 하향**(5000→4000→3000→…2000). 이는 **부수당 단가 + 볼륨디스카운트**의 명백한 패턴이지, "묶음 총액"이 아니다(묶음 총액이면 수량 구간마다 총액이 커져야 함). → **`.01 단가형 = 부수당 단가가 옳다**(min_qty=수량 구간 하한·해당 구간 부수당 단가).

★**디지털 종단과의 결정적 차이**: 디지털 명함/엽서북은 `.01`인데 단가가 **묶음/구간 총액**(100매=3500원 = 묶음총액)이라 `×qty` 과청구 → `.02 합가형 교정` 필요. **책자 제본비는 `.01`이고 단가가 진짜 부수당 단가**(볼륨할인 구간) → **교정 불요·`.01` 정당**. ★**같은 `.01`이라도 단가 의미가 정반대**(디지털=묶음총액 오적재 / 책자제본=부수당 정상) — designer가 `min_qty` 단가 추이로 판별(하향=부수당 / 일정 또는 상향 의심=묶음총액).

### 3.2 경쟁사 단서 — 부수당 단가 지지

| 단서 | RedPrinting `[red:book]` | WowPress `[wow:booklet]` |
|------|--------------------------|--------------------------|
| 제본비 | 제본방식별 단가·부수 tier(100~500) → 부수당 단가 하향 | `jobcost0` = 작업비(작업량×단가)·작업량에 부수 반영 |
| 표지/내지 | 표지 자재가·내지 자재가 = 부수·페이지 종속 단가 | `paperno` 용지가 = 부수·연 종속 |

★**경쟁사도 부수당 단가 + 볼륨 tier** — 라이브 후니 `.01` 부수당 단가 해석을 독립 지지. WowPress `jobcost0`도 작업량(부수 포함)에 단가를 곱하는 부수 종속 단가.

### 3.3 표지/내지 단가 — 미배선이라 라이브 결판 불가(권위 가격표 필요)

- 하드커버 표지(073)·면지(074~076)는 **미바인딩**(단가행 없음) → 라이브로 prc_typ 결판 불가.
- ★designer 입력: **권위 가격표(260527)가 표지/내지 단가를 "1매당/1면당"으로 주는지 "묶음 총액"으로 주는지** 확인 후 prc_typ 결정. 경쟁사 단서(부수당 단가)는 보강(권위 아님). 제본비 `.01` 부수당 패턴을 표지/내지에도 동형 적용 가설(검증 대상·designer 골든으로 확정).

### 가드 [HARD]
- **`.01` 단가형 ≠ 자동 묶음총액**(디지털 결함을 책자에 무비판 전이 금지). 책자 제본비는 `min_qty` 하향 추이로 **부수당 단가 정당** 입증됨. prc_typ 결판은 **단가행 값 추이 + 권위 가격표 구조**로 판별(comp_typ 코드만으로 결정 금지).
- **표지/내지 prc_typ 미확정**(미배선) → 권위 가격표 대조 후 결정·경쟁사는 부수당 단가 지지 보강만.

### 4.2 한 줄 결론

**책자(반제품 세트) 경쟁사 가격 골격(표지비 + 내지비[페이지수 비례·런타임 곱] + 제본비[부수당 단가] + Σ후가공·부품 합성)은 후니가 세트 그릇·페이지 규칙·제본 comp 11종·합산형 공식으로 1:1 동형(부품 분리는 후니가 별 prd_cd로 우월)하게 담는다.**

- **신규 가격축 = 0** — 문구 종단 결론(배선-gap·신규 0)과 **일관**. rpmeta TP/책자 분석도 distinct #18 부결·기존 축 무왜곡 흡수(`huni-rpmeta-harness` TP 그룹 C/D)와 정합.
- **흡수 실질 = 0건(메커니즘)** — 후니 권위 충분(그릇 보유). RedPrinting `book2025`/WowPress `jobqty`는 흡수 대상이 아니라 **후니가 이미 동등하거나 우월**.
- **실질 과업 = 배선-gap 닫기**: (A) 중철~트윈링 `PRF_BIND_SUM` 제본비 단일항 → 표지+내지+인쇄 comp 합산 / (B) 하드커버류(072~087·173/174) 미바인딩 → 세트 부모 + 부품 합산 공식 배선. **vessel-gap 아닌 data/배선-gap**(designer·dbmap 트랙).
- **돈크리티컬 결판은 책자가 라이브로 줌** — 제본비 `.01` = 부수당 단가(min_qty 하향 입증). 디지털 종단의 `.01→.02` 교정과 **다른 결론**(전이 금지). 표지/내지 prc_typ만 권위 가격표 대조 미결.

### 4.3 naming 유입 가드 [HARD]

`book2025_price`·`MTRL_CD`(RXART250 등)·`INN_PAGE`·`paperno3/4/5`·`jobqty0`/`jobcost0`·`seneca`·`ORD_CNT`/`PRN_CNT` 후니 유입 금지. 후니 `frm_cd`(`PRF_BIND_SUM` 등)·`comp_cd`(`COMP_BIND_*`)·`page_rules`·`bdl_qty` 컨벤션으로 번역(`dbmap-naming-standardization` 권위순서: 후니 레거시 → 라이브 DB → rpmeta 흡수 → 인쇄표준).

### 4.4 designer 인계 (후속 에이전트가 판정·설계)

기준점만 제공(흡수 후보 정리). 설계/판정은 hpe-engine-designer·검증은 hpe-validator:
- **D-BIND-SCOPE**(제본비 단일항 vs 표지+내지+제본 부품 합산) = 권위 가격표가 부품별 단가 주면 부품 합산(경쟁사 입증)·인간 결정.
- **표지/내지 prc_typ** = 권위 가격표 "1매/1면당 vs 묶음총액" 확인 후 결정(제본비 `.01` 부수당 패턴 동형 가설).
- **세트 "구성"(세트 부모+표지/면지 prd_cd) vs 세트 "가격"(합산 공식) 분리** = P-1b 원칙(디지털 종단 set-product-design §0 실증·이중계상 가드).
