# 예전사이트 골든 추출 + PR(가격재현) 채점 — 폼 구동법·파일럿 결과·한계

> 2026-06-29 신설. 배치 채점기에 **"가격이 맞나"(정답 대조)** 를 붙이는 토대.
> 기존 채점(CALC/PR-pansu/R1/R3/OC)은 "가격이 나오나"만 봤다. 여기에 예전사이트
> (huniprinting.com goods.asp)를 **독립 가격 오라클**로 써서 엔진 simulate 금액과
> 대조하는 **PR(가격 재현)** 점수를 추가했다.
>
> ★위상[HARD]: 예전사이트 = **구 시스템**(리뉴얼 대상). **권위 엑셀이 절대 기준.**
> PR mismatch 는 **조사 신호**일 뿐 자동 교정 트리거가 아니다(예전사이트에 맞춰
> 우리 값을 바꾸지 않는다).

## 산출물
| 파일 | 역할 |
|------|------|
| `build_pcode_map.py` | 라이브 prd_cd ↔ 예전사이트 pcode 매핑표 생성기 |
| `prd-pcode-map.csv` | 매핑표(276행·high 31·mid 30·미상 215). PR 채점이 어느 폼을 쓸지 |
| `golden_fetch.py` | 한 상품의 대표 구성으로 예전사이트 폼을 네이티브로 채우고 공급가 읽기 |
| `pr_score.py` | 엔진 simulate 금액 ↔ 골든 공급가 대조(match/mismatch/no-golden) |
| `score_batch.py` | `--pr` 플래그로 PR 채점 통합(기존 흐름 보존·브라우저 독점) |

## ★폼 구동법 (실측 2026-06-29 — 재사용 가능한 사실)
예전사이트 가격은 **onchange→`setActionCheck()` 의 서버 AJAX** 로 계산된다.

1. **반드시 gstack browse 의 네이티브 `select`/`fill`** 로 옵션을 바꾼다.
   - raw JS `el.value=...` 는 AJAX 재계산이 안 걸린다(0원/고정 함정·사용자 실측 확인).
2. **사이즈가 sub-select 를 활성화한다.** `tmp_p01`(사이즈) 선택 →
   - 용지 sub-select `tmp_p07_H/X/M`(사이즈군별) 중 **하나가 보이게(visible) 활성**
   - 수량 select `qty_<내부코드>`(사이즈별) 가 활성. 폼에 따라 단순 `qty` 도 있음.
   - → **보이는(offsetParent!==null) select 를 동적으로 찾아** 채운다(이름 하드코딩 X).
3. **`setActionCheck()` 호출 + 대기**(서버 왕복 ~2초)로 재계산을 돌린다.
4. **공급가 읽기**(부가세 전):
   - `input[name=price_01]` 에는 **`id` 가 없다** → `getElementById('price_01')` 는
     null(함정). `document.getElementsByName('price_01')[0].value` 또는 본문
     `"공급가 : N원"` 파싱으로 읽는다.
   - 공급가 0원 = 미계산(필수옵션 누락) 또는 off-grid(견적문의) = **미실측 신호**.

폼 필드 사전: `tmp_p01`(사이즈)·`tmp_p07_*`(용지·사이즈군별)·`tmp_p08`(출력방식)·
`tmp_p09`(단/양면)·`qty`/`qty_<N>`(수량)·`tmp_p14/05/06/11/13`(귀돌이·넘버링·난수·
미싱·오시)·`tmp_p15`(봉투)·`tmp_page`(페이지·책자).

## ★엔진 라벨 → 예전사이트 라벨 브리지 (`pr_score.py`)
| 축 | 엔진 라벨 | 예전사이트 라벨 | 변환 |
|----|-----------|-----------------|------|
| 사이즈 | `73x98` / `100x150` | `73mm x 98mm` | `NxM` → `Nmm x Mmm` |
| 자재 | `백색모조지 220g (MAT_000074)` | `백색모조지 220g` | `(CODE)` 제거 |
| 인쇄 | `단면 (POPT_000001)` | `단면` | `(CODE)` 제거·단/양면 추출 |
| 수량 | `100` | `100 장` | golden_fetch 가 `"N 장"`/`"N장"` 매칭 |

매칭은 `_opt_match`(공백 무시·부분 일치)라 `mm` 유무·미세 표기차를 흡수.

## 어느 상품군이 자동되고 어느 군이 어려운가
| 자동화 수준 | 상품군 | 이유 |
|-------------|--------|------|
| ✅ **자동** | 디지털 엽서·명함·카드(siz_cd×수량 모델) | 폼 구조 단순(사이즈→용지→인쇄→수량) · 라벨 브리지 직결 |
| ◐ **부분** | 스티커·접지카드 | pcode 폼 1개가 여러 prd 커버(mid confidence) · 폼 내 구성 좁힘 필요 |
| ◐ **부분** | 아크릴·굿즈(면적/자유치수) | 엔진=면적(w×h)·예전사이트=치수 입력(`tmp_width/height`) → 케이스 빌더 보강 필요 |
| △ **수동 보강** | 명함(고정 사이즈) | sim-meta 에 siz_cd 미노출(UNSCORED) → 명함 전용 케이스 빌더 필요 |
| ✗ **불가(설계상)** | 기성상품(.03)·반제품(.02 내지) | 예전사이트 **단독 폼 없음** = 정답가 오라클 부재(매핑 미상-비단품) |
| ✗ **세트(.01 부모)** | 책자·엽서북·떡메 | 폼이 페이지·표지·내지 다축 → `golden_fetch` extra_selects + 세트 케이스 빌더 확장 필요 |

## 한계·주의
- **다대일 매핑**: 예전사이트 pcode 는 **카테고리 단위 폼**(예 pcode 33 엽서 폼이
  프리미엄/코팅/스탠다드엽서 모두 커버). prd 별 정답은 폼 안 용지·옵션으로 좁힌다.
  같은 pcode 의 여러 prd 가 같은 골든을 받을 수 있으니 **용지/옵션 정합이 핵심**.
- **브라우저 독점**: golden_fetch = gstack browse 단일 세션. PR 채점은 **다른 작업과
  동시에 브라우저를 쓰면 안 된다.** 1상품 1케이스(첫 priced)만 대조해 호출 최소화.
- **off-grid/필수옵션 누락 = 0원** → `no-golden`(미실측·누락 은폐 X).
- **mismatch ≠ 결함 확정**: 차이는 ⓐ 우리 적재 오류 ⓑ 의도된 리뉴얼 가격변경
  ⓒ 예전사이트 stale 중 무엇인지 **사람이 조사**(자동 교정 금지·권위=엑셀).
- **실 배치 전면 적용은 인간 승인** — 전 상품 `--pr` sweep 은 라이브 접속이 많아
  명시적으로만(브라우저 점유·예전사이트 부하 고려).

## 사용
```bash
# 1) 매핑표 (재)생성
python3 build_pcode_map.py

# 2) 골든 단독 추출(점검)
python3 golden_fetch.py 33 --size "100mm x 150mm" --paper "백색모조지 220g" --print 단면 --qty 100
python3 golden_fetch.py --selftest          # pcode 33 엽서 재현 확인

# 3) PR 단독 채점(한 상품)
python3 pr_score.py PRD_000016

# 4) 배치 + PR 통합 (--pr 플래그 · 끝에)
python3 score_batch.py digital-print PRD_000016 --pr   # 단일
python3 score_batch.py digital-print --pr              # 그룹
# 플래그 없으면 PR OFF = 기존 동작 100% 보존
```
