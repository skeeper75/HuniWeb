## 0. 심의자 한 줄 결론

프린트엽서의 NO-GO는 **per-product 값 오류가 아니라 4개 구조축이 동시에 무너진 시스템 결함**이다. 그리고 **이 결함은 엽서만의 것이 아니다** — 라이브 재실측으로 인쇄비 comp(`COMP_PRINT_DIGITAL_S1/S2`)가 **PRF_DGP_A~F 6개 공식 전부**에 배선됨을 확인(아래 §1-A). 따라서 정립안은 "엽서 단가행 몇 개 고치기"가 아니라 **디지털인쇄비 데이터 모델 + 엔진 산정 규칙 두 층의 합작 교정**으로 설계해야 한다.

**핵심 분리 [HARD]:**
- **우리 트랙(데이터 정립)으로 닫히는 것**: F-1 칼라 단가행 복원(도수축), F-2 배선 use_yn=N(택1), F-5 유광 단가행 — 단, 셋 다 **선결 컨펌(Q-COLOR-INTENT·Q-S1S2-EXCL·Q-유광선택)** 후에만 안전.
- **개발자 백로그(엔진)로만 닫히는 것**: F-3 출력매수 변환(`×ceil(qty/판걸이수)`), F-4 plate↔work 환산(임포지션), 도수 차원을 엔진 매칭축(NON_QTY_DIMS)에 추가하는 부분.
- **★결정적 트레이드오프**: F-1을 "데이터로만" 닫으려면 도수가 엔진 매칭에 통하는 통로가 있어야 한다. 현재 `use_dims`에 도수 차원이 없어(라이브 실측 확정) **데이터를 넣어도 엔진이 못 읽는다.** → F-1은 데이터+엔진 **양쪽이 동시에** 닫혀야 골든이 재현된다(아래 §3-B에서 정량 입증).

**정리 후 PC-2 골든(20,000) 재현 가능성: 우리 트랙 단독 = NO. 우리 트랙 + 개발자 백로그 동시 적용 = YES (정량 입증 §6).**

## 1. 라이브 재실측 확정 (정립안 근거 — 본 세션 읽기전용 SELECT)

### §1-A 광역 영향 (★엽서 단독 아님)
```
COMP_PRINT_DIGITAL_S1 배선 공식 = PRF_DGP_A,B,C,D,E,F (6개)
COMP_PRINT_DIGITAL_S2 배선 공식 = PRF_DGP_A,B,C,D,E,F (6개)
PRF_DGP_A 바인딩 상품 = 9개
```
→ F-1(도수)·F-2(이중)·F-3(출력매수)·F-4(plate환산)는 **디지털인쇄 전 상품군 동형 결함**. 엽서 1상품 교정이 아니라 인쇄비 comp 데이터 모델 자체를 고쳐야 6공식 전부가 동시에 정상화된다(또는 동시에 회귀 위험).

### §1-B F-1 도수축 — 정확한 오적재 정체 (work-spec·verdict보다 한 단계 더 규명)
라이브 `COMP_PRINT_DIGITAL_S1 @ SIZ_000499`(국4절) 실측:
| print_opt | min_qty=15 | 20 | 25 | 200 | = 권위 어느 열? |
|:--:|:--:|:--:|:--:|:--:|:--:|
| POPT_000001(단면) | 450 | 400 | 350 | 130 | = **흑백단면 B열** verbatim |
| POPT_000002(양면) | 900 | 850 | 800 | 310 | = **칼라단면 D열** verbatim |

라이브 `COMP_PRINT_DIGITAL_S2 @ SIZ_000499`:
| print_opt | min_qty=200 | = 권위 |
|:--:|:--:|:--:|
| POPT_000001(단면) | 260 | = **흑백양면 C열** |
| POPT_000002(양면) | 620 | = **칼라양면 E열** |

`use_dims` 라이브: `S1=[proc_cd, plt_siz_cd, print_opt_cd, min_qty, proc_grp]` · `S2=[proc_cd, siz_cd, print_opt_cd, min_qty, proc_grp]` → **도수(색) 차원 둘 다 0.**

**★확정 진단**: 권위 가격표는 **8도수(흑백1도/칼라CMYK/별색 화이트·클리어·핑크·금·은) × 2면(단/양) = 16열**. 라이브는 그중 **흑백·칼라 두 도수의 단면값만** 가져와 면축(POPT 단/양)에 **흑백단면=단면, 칼라단면=양면**으로 1:1 오매핑했고, 흑백양면·칼라양면은 S2에 비슷하게 밀어넣었다. 즉 **2D 매트릭스(도수×면)를 1축(면)으로 평면화하며 도수와 면을 뒤섞었다.**
- 결과: 손님이 "칼라 단면" 선택 → POPT_000001 매칭 → **흑백단면 단가(350@25)** 산출. 정답 칼라단면(800@25)은 라이브에 존재하나 **POPT_000002(양면)에 잘못 묶여** 도달 불가.
- PRD_000016 print_opt 정의: POPT_000001=단면(front CLR_000005 CMYK4도, back 인쇄안함) / POPT_000002=양면(양면 CMYK). 즉 **PRD_000016은 칼라전용**(흑백 옵션 없음) — 그런데 단면에 흑백단가가 실린 것은 명백 오적재(Q-COLOR-INTENT로 의도 확인 필요하나 강한 결함 신호).

### §1-C F-2 인쇄비 이중 — "동일값 이중"이 아니라 "서로 다른 도수열 동시 매칭"
S1·S2 둘 다 PRF_DGP_A seq 1·2에 배선(addtn_yn='Y'). S1=plt_siz_cd 키, S2=siz_cd 키. 동일 SIZ_000499 200매 단면:
- S1 = 130 (흑백단면) · S2 = 260 (흑백양면) → 호출이 `plt_siz_cd=SIZ_000499` **그리고** `siz_cd=SIZ_000499` 동시 공급 시 둘 다 included → 130+260 합산. `_combo_key`가 plt vs siz로 달라 ERR_AMBIGUOUS도 안 걸림(엔진 lenient).
- **정정**: reconcile-synthesis는 "동일값 2배"로 봤으나 실측은 **단면값+양면값 합산**(130+260=390/매)이다. 더 위험 — 면 선택과 무관하게 단·양면이 합쳐진다.

### §1-D F-4 plate↔work — 가격 comp는 plate에만 키
- `COMP_PAPER` siz_cd = SIZ_000499만 / `COMP_COAT_MATTE`·`S2` siz_cd = SIZ_000077,SIZ_000499만 / `S1` plt_siz_cd = SIZ_000077,SIZ_000499만. (전부 impos_yn=Y = 출력판형)
- PRD_000016 등록 사이즈 = SIZ_000001~007 (impos_yn=N = work size). **가격 단가행에 work size 0행.** 손님이 work size로 견적 → 전 comp no-match → 0원(recompute PC-2-naive=0 입증).

### §1-E F-5 유광 / F-6 투명
- `COMP_COAT_GLOSSY` 단가행 = **0** (GROUP BY에서 아예 미출현). 무광 92행만. → 유광 선택 시 0원 침묵.
- PRD_000019(투명엽서) `t_prd_product_price_formulas` = **0행** (바인딩 결손).

### §1-F CX-3 (별색 중복) — 라이브 실재 신호 (컨펌큐)
`COMP_PRINT_SPOT_WHITE_S1 @ SIZ_000499` 실측: POPT_000001/min_qty=15 에 **1500/1350/1350/1500/1500 등 5개 단가행 중복**. use_dims에 색 변별자 없이(`[plt_siz_cd, proc_cd, print_opt_cd, min_qty, proc_grp:PROC_000007]`) 여러 별색값이 한 comp에 다중 적재됨 → 호출 시 동일 키에 복수행 매칭 위험. **Codex CX-3가 라이브 신호로 확인됨**(단 엽서 칼라 결함과 별개 트랙·도수축 정립과 함께 재검토).

## 3. 결함별 정립안 (어느 t_*·무엇·어떻게·대안·트레이드오프·트랙)

### ★ F-1 도수축 붕괴 (1순위·돈크리티컬·광역 6공식)

**근본원인**: 권위 2D 매트릭스(도수×면)를 적재 시 1축(면)으로 평면화 — 도수를 차원으로 두지 않고 흑백단면=POPT단면·칼라단면=POPT양면으로 면축에 욱여넣음. `use_dims`에 도수 차원 부재 → 엔진이 도수를 읽을 통로 자체가 없음.

**정립 옵션 (대안 비교):**

| 옵션 | 무엇 | t_* | 트레이드오프 |
|------|------|------|------|
| **ⓐ print_opt에 도수×면 4종 재정의** | POPT를 흑백단/흑백양/칼라단/칼라양 4개로 늘려 단가행 재배치 | `t_prd_product_print_options`+단가행 | 데이터만으로 닫힘(엔진 무변경)이 장점이나 ★print_opt의 의미가 "면"인데 "도수+면"을 인코딩 = 의미축 오염(스키마 설계의도 위반·F-1을 똑같은 방식으로 재발). **반대.** |
| **ⓑ 도수 차원 신설 + use_dims 추가** (권장) | 인쇄비 comp use_dims에 도수 차원(front_colrcnt_cd 또는 신규 print_clr 축) 추가 + 권위 16열(또는 엽서 사용 도수만)을 (도수×면×수량) 단가행으로 재적재 | `t_prc_price_components.use_dims` + `t_prc_component_prices`(차원 컬럼) + **엔진 NON_QTY_DIMS 등록** | 권위 모델과 1:1·재발 방지. **단점=엔진 백로그 동반 필수**(use_dims에 선언해도 엔진이 그 컬럼을 매칭축으로 안 읽으면 무효). 차원 컬럼이 라이브에 없으면 DDL(ddl-proposer). |
| ⓒ PRD_000016 칼라전용 가정 → 단면=칼라단가로 교정 | print_opt 그대로, 단면에 칼라단면(D), 양면에 칼라양면(E) 재적재 | 단가행만 | 엔진 무변경·즉효. **단점=상품별 도수옵션이 다르면(흑백 가능 상품) 깨짐·근본 미해결(도수축 여전히 없음). 임시책.** PRD_000016이 정말 칼라전용이면 단기 정당하나 6공식 광역엔 부적합. |

**권고 = ⓑ (근거)**: ① 권위가 도수×면 2D이므로 도수는 차원이어야 한다(price-engine-diag SOT2 "같은 가격테이블 같은 차원=결합"과 정합 — 인쇄비 테이블에 도수·면 둘 다 있으니 둘 다 use_dims 차원). ② F-2가 "단면값+양면값 합산"인 진짜 이유도 도수×면을 S1/S2 두 comp로 쪼개 면축에 뭉갰기 때문 → ⓑ가 F-1·F-2를 **한 번에** 닫는다(도수·면이 한 comp의 두 차원이면 S1/S2 분리 자체가 불요). ③ price-engine-diag F-2가 "도수=clr_cd 폐기→print_opt_cd"로 결론냈으나, 그건 **면 한 축만 있을 때** 얘기 — 도수×면 2D 상품에선 print_opt(면)와 별개로 도수 축이 필요함을 본 검증이 드러냄. **이 충돌은 §7 컨펌(U-1 재검토)로 올림.**

**우리 트랙 분담**: 단가행 재적재(권위 16열→long-format, 도수·면·수량 차원) = dbmap(load-execution/axis-staged). use_dims 차원 컬럼 선언 = ddl-proposer(라이브에 도수 차원 컬럼 부재 시).
**개발자 백로그 분담**: 엔진 NON_QTY_DIMS에 도수 차원 등록(없으면 단가행을 넣어도 매칭 안 됨).

### F-2 인쇄비 이중배선 (S1+S2 단·양면값 합산)

**근본원인**: 도수×면 2D를 S1(plt 키)·S2(siz 키) 두 comp로 분리 적재 → 둘 다 PRF_DGP_A에 배선(addtn_yn=Y) → 호출이 plt·siz 동시 공급 시 합산. F-4(plate/work)와 F-1(도수/면 평면화)의 합작 증상.

**정립 옵션:**
| 옵션 | 무엇 | 트레이드오프 |
|------|------|------|
| **ⓐ S1/S2 중 택1, 나머지 use_yn=N** (단기 권장) | Q-S1S2-EXCL 확정 후 정답 경로 1개만 배선 유지·다른 하나 단가행/배선 use_yn=N | 즉효·가역. **단 [HARD] S1=흑백단면+칼라단면, S2=흑백양면+칼라양면이라 둘 중 하나만 끄면 양면(또는 단면) 도수가 통째로 사라짐** → ⓐ는 F-1 ⓑ 재적재(도수×면 통합)와 **반드시 함께** 해야 안전. 단독 적용 금지. |
| **ⓑ F-1 ⓑ로 통합 → S1/S2 분리 소멸** (근본) | 도수·면을 한 comp의 두 차원으로 통합하면 S1/S2 2-comp 구조 자체가 불요 → 단일 인쇄비 comp(plate 키) | 근본 해결·F-1과 동시에 닫힘. 작업량 큼·6공식 회귀 테스트 필요. |

**권고 = ⓑ(근본)이되, 단계적이면 ⓐ를 F-1 재적재와 묶어 동시 실행.** S1/S2를 "plate 경로 vs work 경로 택1"로 오해하면 안 됨 — 실측상 둘은 **서로 다른 면(단/양)의 도수값**을 들고 있다(택1 대상이 아니라 한 매트릭스의 두 조각). **이것이 F-2 정립의 핵심 트레이드오프이자 함정.**

**우리 트랙**: 배선 use_yn=N(use_yn 토글) + 단가행 통합 재적재 = dbmap. **개발자 백로그**: 없음(데이터로 닫힘, 단 §F-1 엔진 도수축이 선결).

### F-3 출력매수 미적용 (×qty vs ×ceil(qty/판걸이수)) — ★개발자 백로그 (엔진 전속)

**근본원인**: 권위 인쇄/코팅/용지는 단가 × **출력매수**(=ceil(주문수량/판걸이수)), 수량행 룩업도 출력매수 기준. 라이브 comp는 전부 prc_typ .01(단가형) = 단가 × **주문수량 qty**, min_qty 티어도 qty로 선택. 출력매수 환원 차원이 가격사슬에 없음.

**왜 우리 트랙으로 못 닫는가 (트레이드오프 명시):**
- 출력매수 = ceil(qty/판걸이수)는 **런타임 계산**이다. 판걸이수는 사이즈별 룩업값(임포지션/네스팅 결과)이고 DB는 룩업만 저장(메모리 [[중간계산=앱·DB=룩업]]·price-engine-diag F-3). 단가행에 "출력매수별 단가"를 박을 수는 있어도(이미 권위 수량행=출력매수행), **곱셈 인자(× 출력매수)와 행룩업 입력(출력매수)을 qty에서 출력매수로 바꾸는 것은 엔진 로직**이다.
- 데이터로 흉내내려면 "주문수량→출력매수 매핑 테이블"을 사이즈마다 깔아야 하는데(판걸이수 8이면 200매→25, 18이면 200매→12…), 이는 **차원 폭발 + 판걸이수 변경 시 전면 재적재** = 안티패턴. price-engine-diag가 명시적으로 "박 면적등급·off-grid ceiling과 동일 — DB정착 아님"으로 닫음.

**정립 = 개발자 백로그 (엔진):**
- 엔진이 인쇄비/코팅비 comp 평가 시 곱셈 인자를 `qty` 대신 `출력매수 = ceil(qty / 판걸이수[siz])`로, 수량행 룩업 입력도 출력매수로 전환.
- 용지비는 `× (출력매수 + 5)` (손지율 +5장).
- 판걸이수 룩업 = 사이즈별 테이블(가격표 판걸이수 시트 → 어딘가 적재 또는 앱 임포지션 계산). **CX-2(판걸이수 앱↔가격표 비동기)와 직결** — 판걸이수 소스 단일화가 선결.

**우리 트랙 분담(데이터)**: 판걸이수 룩업값을 엔진이 읽을 위치에 적재(현재 가격표 시트값이 라이브 어디에 있는지 컨펌 필요 — §7 Q-PANSU-SRC). **개발자 백로그**: 출력매수 환산 + 곱셈/행룩업 입력 전환(F-3 핵심).

### F-4 plate↔work 사이즈 단절 — ★개발자 백로그 (엔진/앱 환산) + 우리 트랙(판형 테이블)

**근본원인**: 가격 단가행은 plate size(국4절/3절)에만 키. 손님 선택은 work size. work→plate 환산(국4절/3절 판별 = 임포지션) 장치가 엔진·데이터 어디에도 없음.

**왜 우리 트랙으로 못 닫는가**: work size(100x150)에 plate 단가를 복사하면 = 같은 단가를 모든 work에 중복 → plate별 차이(국4절 vs 3절) 소멸·임포지션 무시. work→plate는 **계산**(작업사이즈가 어느 전지에 몇 장 앉는지)이지 룩업이 아님. price-engine-diag·[[중간계산=앱·DB=룩업]] 정합.

**정립 = 개발자 백로그 (엔진/앱):**
- 호출측(위젯/시뮬레이터)이 work size 선택 → 임포지션으로 plate(국4절 SIZ_000499 / 3절 SIZ_000077) 판별 + 판걸이수 산출 → 엔진에 `plt_siz_cd`(plate) 공급. F-3 출력매수와 동일 환산 레이어에서 함께 처리(판걸이수가 곧 임포지션 결과).
- **우리 트랙(데이터)**: work size ↔ plate size ↔ 판걸이수 매핑 테이블이 라이브에 있는지 확인·적재(가격표 판걸이수 시트가 곧 이 매핑). 단 환산 "로직"은 엔진.

**트레이드오프**: 환산을 앱에 두면(위젯) 후니 자체 주문/시뮬레이터와 위젯이 같은 로직을 중복 구현할 위험 → **엔진(evaluate_price 진입 전 normalize 레이어)에 두는 것을 권고** (단일 알고리즘 원칙·price-engine-diag 5장치 정합). 최종 위치는 개발자 결정(Q-PANSU-ENGINE).

## 6. 정리 후 기대 가격 — PC-2 골든(20,000) 재현 정량 입증

**시나리오 A — 우리 트랙만 (F-1 단가행 재적재, 엔진 무변경):**
```
손님: 100x150 칼라단면 200매
- F-4 미해결: work size(SIZ_000003) → plate 환산 없음 → no-match → 0원 (또는)
- 호출이 plate(SIZ_000499) 직접 공급해도:
- F-3 미해결: ×qty(200) → 도수축 데이터 넣어도 엔진이 도수 차원 못 읽으면(use_dims 미반영) 여전히 흑백 매칭
→ 골든 20,000 재현 = NO
```

**시나리오 B — 우리 트랙 + 개발자 백로그 동시:**
```
1. 앱/엔진 환산: work 100x150 → plate 국4절(SIZ_000499), 판걸이수 8 → 출력매수 = ceil(200/8) = 25
2. 도수축(엔진 NON_QTY_DIMS 등록) + 칼라단면 단가행(우리 트랙 재적재):
   인쇄비 단가 = 칼라단면(D) 출력매수25행 = 800
3. 인쇄비 = 800 × 출력매수(25) = 20,000       ← 골든 일치 ✅
   (F-2 해소: S1/S2 통합 → 단일 인쇄비 comp → 합산 없음)
4. 용지비 = 70.64 × (25+5) = 2,119.2
→ final ≈ 22,119.2 (반올림 정책 Q-ROUND 별개)
```
**결론: PC-2 골든(인쇄비 20,000) 재현 = 우리 트랙 단독 NO · 우리 트랙 + 개발자 백로그 동시 YES.** 네 결함(F-1 도수 데이터+엔진·F-2 통합·F-3 출력매수 엔진·F-4 plate환산 엔진)이 **모두** 닫혀야 골든이 나온다. 어느 하나라도 빠지면 0원 또는 흑백값 또는 ×200 괴리.

## 9. 미해소 (컨펌 전 정립 불가 — 정직 기록)

- **F-1의 데이터 적재 구조는 엔진 도수축 계약(U-1) 확정 전엔 못 짠다** — 엔진이 도수를 어느 컬럼으로 매칭할지 모르면 어느 컬럼에 단가를 넣을지 모름. 따라서 **개발자 백로그(엔진)가 데이터 트랙보다 선행**. 이것이 본 결함의 가장 큰 정립 제약.
- F-3·F-4는 본 하네스(데이터 정립)로는 닫히지 않음 — 엔진 작업 없이는 골든 재현 불가(§6 시나리오 A 입증).
- 광역 6공식(PRF_DGP_A~F) 동형이므로 엽서 단독 교정 금지 — 디지털인쇄비 데이터 모델 전체를 한 번에(회귀 위험 관리).
