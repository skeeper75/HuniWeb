## 1. goods-pouch 비치수 size 77상품 (H1 BLOCKER) — **[사용자결정]** (권고: master siz_cd 신설)

### 도메인 해석 (근거 인용)
- **엔티티 의미축(entity-semantic-model §1·#1):** size축 = **재단치수(고객 선택 완성품 치수)**. "비치수(`사용자입력`) = `nonspec_*_min/max` 범위"로 인코딩하는 것이 의미축 정의.
- **L1 원천 직접 재판독(waveB1 §3-A + 본 패스 재집계):** BLOCKED 96 size행 / 기적재 29행. L1 `파일사양_재단사이즈` 컬럼은 **실존하나 비치수 상품 전부 공란**(303행 전수 공란 확인). WxH는 `작업사이즈`(plate)에만 존재 → plate 복제·siz_cd 발명 거부 정당(검증 UPHELD).
- **결정적 추가 발견 — "비치수 77"은 단일 유형이 아니라 4종 혼재**(본 패스 L1 sub-분류):

  | sub-유형 | 건수 | 예시 | 진짜 의미축 |
  |---|:---:|---|---|
  | **(a) 형상+치수** (치수 내포) | 14 | `원형 90mm`·`사각 110mm`·`무광 75mm`·`미니 50mm` | **size축**(지름/한변이 곧 재단치수) + `prcs_dtl_opt.모양` 짝(benchmark §2) |
  | **(b) 용량** | 21 | `11온스`·`350ml`·`500ml` | size축 **아님**(머그/보틀 물리용량). 색상×용량 variant → material/옵션 |
  | **(c) 단/양면** | 12 | `단면`·`양면`·`양면 가로형` | **print_option축**(인쇄면 도수) — size 오적재(mis-axis) |
  | **(d) 순수 라벨** | 49 | `1구`·`2구`·`CD커버세트`·`2개1팩`·`A5용` | bundle/config축 또는 옵션 — size **아님** |

- **벤치마킹(benchmark §2):** RP는 자유형 치수를 `MIN/MAX_CUT` 슬롯, WP는 `non_standard=1 → req_width/req_height`로 둠 — **경쟁사 둘 다 비규격을 별도 min/max 슬롯**으로 처리. 후니 `nonspec_*_min/max`가 이에 대응.

### 권고 (후니 엔티티 + 관리 용이성)
**의미축별로 분리 처리 — "size 77 일괄"이 오류였다.** 처리안:
1. **(a) 형상+치수 14건 → master `t_siz_sizes` siz_cd 신설**(사각손거울 짝 적재 모델). 지름/한변이 명확한 재단치수라 siz_cd 등록이 정공법. `prcs_dtl_opt.모양`(원형/사각)과 캐스케이드 짝. **← 이것이 사용자 결정 핵심**(master 쓰기).
2. **(b) 용량 21건 → size 적재 안 함.** 머그 용량은 `material`(색상 variant) 또는 단순 옵션. nonspec 불요(인쇄치수 아님).
3. **(c) 단/양면 12건 → `t_prd_product_print_options`로 이축**(size 아님, mis-axis 정정). 발명 0, 의미축 정정.
4. **(d) 순수 라벨 49건 → size 적재 안 함.** `1구/2구`=조각수/구성(bundle_qty 후보), `세트/2개1팩`=묶음(bundle), `A5용`=plate. size축 부적합.
5. **자유형(사용자입력) 상품 → `nonspec_yn=Y` + `nonspec_*_min/max`** (현 전상품 NULL = 글로벌 갭, benchmark §2 보강대상).

→ **권고 결과: master siz_cd 신설은 14건뿐**(96이 아님). 나머지 82는 다른 축이거나 비-size. **"77상품 size 누락"이라는 BLOCKER 프레임 자체가 과대**였다.

### 분류: `[사용자결정]` (master siz_cd 14건 신설 = master 쓰기 인가 필요)
- 우리측 추정 발명 불가는 (a) 14건의 **siz_cd 코드 신설**뿐. (b)(c)(d)는 권고로 의미축 정정 확정 가능(`[도메인해소]` 성격).

### 위젯 영향
- 미해소 시: goods-pouch 14상품(거울·코스터·핀버튼·키링류)에서 **사이즈 선택 옵션이 비어 위젯이 단일 SKU로 고정**(원형 90mm vs 사각 90mm 선택 불가). 용량/단양면/구수 상품은 size축이 빈 게 정상이라 영향 없음.

## 2. sticker 형상 enum (066 원형 11종 + 058~062) — **[사용자결정]** (권고: master siz_cd 신설 + 모양=공정 유지)

### 도메인 해석 (근거 인용)
- **벤치마킹 권고(benchmark §2.2):** "**모양 = 공정 `prcs_dtl_opt.모양`(기존 유지)** + **치수 = size축 `t_prd_product_sizes`에 적재하되 `prcs_dtl_opt.모양`과 짝지음**. 후니는 모양=공정, 치수=사이즈로 자연 분리되므로 RP(PCS 2중표현)/WP(non_standard 플래그)보다 단순." → **신규 테이블·process param 재배치 불요**.
- **라이브 사실(sticker load-spec §3-R2):** 066 합판도무송 = 모양공정 PROC_000055(스완) 기적재 + size 26행(정사각12+직사각14) 기적재. **원형 11종(10~60mm)만 size 드롭**(`siz_nm like '원형%'`=0건). master `t_siz_sizes`에 합판도무송 원형 siz_cd 부재 → link 적재 전 **master mint 선행**.
- **058~062 형상 enum:** `커팅` 컬럼이 형상 본체(원형 25~90mm 등) + `★사이즈선택:A4/A5` 분기. 현 DB는 size=판형(A4/A5)만 → 형상 enum 축 자체 부재. 단 `★사이즈선택`이 **"도형별 판당 EA = 재단치수인가 임포지션(plate) 메타인가" 모호**(load-spec D-3).

### 권고 (후니 엔티티 + 관리 용이성)
1. **모양 = 공정 유지**(PROC_000053완칼/054반칼/055스완 + `prcs_dtl_opt.모양`). 재배치 0. benchmark §2 권위.
2. **066 원형 11종 → master `t_siz_sizes` 원형 siz_cd 11개 신설 후 `t_prd_product_sizes` link 적재.** 치수(10~60mm)는 L1 실값 추출(blocked CSV에 siz_cd=`(MINT_NEEDED)`로 대기). **← 사용자 결정(master 쓰기)**.
3. **058~062 형상 enum → 치수는 size축, 단 `★사이즈선택` 분기 의미 확정 선행.** 권고: 도형별 치수(원형 25mm 등)는 **재단치수=size축**, `A4/A5`는 **plate 임포지션 메타**(이미 size에 A4/A5 적재된 게 plate 성격). 즉 "A4 모지에 원형 25mm 다수 임포지션" → size축엔 원형 치수, A4/A5는 plate. **master 원형/띠지/하트 siz_cd 부재라 066과 동일 mint 선행**.
4. **자유형(052~057·064) → `nonspec_*_min/max` 보강**(현 NULL).

### 분류: `[사용자결정]` (master siz_cd 신설 인가 — 066 원형11 + 058~062 형상치수)
- 모양=공정 유지·축 귀속(치수=size/A4A5=plate)은 `[도메인해소]`로 권고 확정. **siz_cd 코드 신설만 master 쓰기 인가 필요**.

### 위젯 영향
- 미해소 시: 합판도무송에서 **원형 도형 선택지 11종이 통째 사라짐**(정사각/직사각만 노출). 058~062 반칼스티커는 도형 옵션 축 부재로 **사이즈만 A4/A5 선택**되고 모양 선택 불가.

## 3. calendar 택일 미연결 6멤버 + 장수 — **[도메인해소]** (권고로 확정 가능)

### 도메인 해석 (근거 인용)
- **미연결 6멤버 분류(waveB1 §3-C + calendar load-spec §R1):** 가공없음(110·111)·제본없음(112)·우드거치대(110)·삼각대(108·109).
- **"없음" 선택지의 도메인 의미:** SEL_TYPE.01 택일그룹은 "반드시 하나 선택"(`mand_yn=Y` 헤더). **"가공없음/제본없음"은 택일 UI의 "선택 안 함" 빈 선택지** — 신규 proc_cd 부여 대상이 **아니다**(공정이 없는 게 곧 그 선택지). RP `VIEW_YN`/WP `type=radio`의 "기본 미선택" 옵션과 동형(benchmark §5). → **별도 master proc_cd 신설 불요**.
- **거치대류(우드거치대·삼각대):** C-6(부속 맥락별) + entity-semantic #9 "추가상품(완제 부속: 거치대) = addon축, 부착공정과 축 분리". 우드거치대는 이미 addon(PRD_000012) 적재됨(G-CL-3 이중분류 정정 대상). 삼각대도 **거치대 = addon축**이 도메인 정합 — process 택일 멤버 아님.
- **장수(4(8P)/8(16P)):** entity-semantic §2 variant표·benchmark §3.2. RP=시트 FixedUnit, WP=pages 배열. 후니 "장수=수량(매)" vs "장수=page_rule" 택1. **캘린더 장수는 내지 페이지수가 아니라 "달력 면 수"** = 가격 비선형이면 page_rule, 단순 매수면 qty.

### 권고 (후니 엔티티 + 관리 용이성)
1. **"가공없음/제본없음" → 신규 proc_cd 신설 안 함.** 택일그룹 UI의 "선택 안 함"(빈 멤버)으로 처리. 110·112는 실멤버(타공/링) 1개 + "없음" 1개 = **택일 ≥2 성립**(현 PARTIAL 해소). 위젯에서 "가공 안 함 / 1구타공+끈" 2지선다 노출.
2. **우드거치대·삼각대 → addon축**(거치대 = 완제 부속). 108·109는 삼각대 addon만 있고 가공 택일그룹 불요 → **excl_group 헤더 신설 안 함**(현 0행이 정상). 우드거치대는 G-CL-3대로 addon(PRD_000012) 유지 + process 택일에서 제거.
3. **장수 → page_rule 아님, "선택옵션(매)" 권고.** 캘린더 장수는 내지 묶음(page_rule = 책자 내지 min/max)과 도메인이 달라 page_rule 부적합(잡음 위험, 떡제본 page 잡음과 동형). 장수별 가격차는 round-2 가격엔진의 옵션 차원으로. **현 page_rule 0행 = 정상**.

### 분류: `[도메인해소]` (권고로 확정, 사용자 승인만)
- 미연결 6멤버는 전부 "없음=빈선택지" 또는 "거치대=addon"으로 도메인 확정 가능. 신규 master 코드 0. 장수=옵션(매) 권고. **단 "삼각대를 addon으로 적재"는 addon_prd_cd 매칭 확인 필요**(거치대 기성상품 prd 실재 여부 — 라이브 SELECT 1회).

### 위젯 영향
- 미해소 시: 110·112 택일 UI에 **단일 옵션만 노출**(택일 기능 미성립, waveB1 H5 PARTIAL). 권고 적용 시 "없음 포함 2+지선다" 정상 노출.

## 4. acrylic 완칼 코드 + over-reach 3 — **[사용자결정]** (권고: 레이저커팅 proc_cd 신설 + 161/168/169 보류 유지)

### 도메인 해석 (근거 인용)
- **도메인 공정 = 레이저커팅(process-recipe-tree §1·§2-3·entity-semantic §3):** "acrylic 전체 = **UV→레이저커팅→아크릴가공**"(PDF Case11/12). 마스터 83공정 중 컷계열 = 완칼053·반칼054·스완055 = **전부 종이/스티커용**. 아크릴 모양컷의 정확 도메인 공정인 **레이저커팅 proc_cd가 마스터에 부재** → 설계자가 053(종이완칼) 차용.
- **C-7(binding):** "완칼도 공정 … 완칼(PROC_000053/054) = 공정으로 적재" → RULE 레벨에서 "아크릴 절단=완칼-family 공정 적재"는 사용자 권위로 grounded(발명 아님). 단 **코드 의미는 레이저커팅과 불일치**.
- **over-reach 3(waveB2 §3-A):** 161 판아크릴=recipe Case10 "UV단품→포장"(컷 단계 부재) / 168 입체코롯토=L1 소재·인쇄·조각수·가공 **전부 공백**(옵션 신호 0) / 169 입체블럭=라미(10T) 적층(시트컷 아닌 조립). **product-level 근거 없는 C-7 RULE blanket 확대**(grounded 14 / over-reach 3).

### 권고 (후니 엔티티 + 관리 용이성)
1. **마스터에 `레이저커팅` proc_cd 신설 권고**(예 PROC_0000XX `레이저커팅`{모양 param}). 도메인 정확성·관리용이성 둘 다 유리 — 아크릴 컷을 종이완칼(053)로 영구 차용하면 위젯/가격/검색에서 "종이 완칼"과 의미 혼동. **← 사용자 결정(master proc_cd 신설)**. 대안(차용 유지): 053 유지 + provenance에 "레이저커팅 차용" 명시 → 관리 단순하나 의미 부정확.
2. **161/168/169 완칼 행 = 보류 유지(active 미적재).** 168은 L1 신호 전무라 가장 ungrounded. 적재 직전 라이브 SELECT로 161/169 실제 컷공정 보유 시에만 승격. **grounded 14는 적재 유지**.
3. **151 PROC_000081(맥세이프 부착) BLOCKER** = 라이브 기적재 중복PK → conditional CSV로 분리됨(해소). 적재 직전 라이브 재확인.

## 5. photobook 책등 param — **[도메인해소]** (권고: master prcs_dtl_opt 상속, 상품별 슬롯 불요)

### 도메인 해석 (근거 인용)
- **마스터 보유 사실(photobook load-spec §G-PB-4):** PROC_000017(제본) `prcs_dtl_opt`에 **`책등(number, mm)` param 이미 존재**. 책등은 마스터 레벨에서 param 형상 보유.
- **구조 갭(HARD):** `t_prd_product_processes` 스키마 = `prd_cd,proc_cd,excl_grp_cd,mand_proc_yn,disp_seq,reg_dt,upd_dt` — **상품별 param 값 슬롯 없음**. 책등 상품별 선택지(하드 10/12/14/16 vs 소프트 4/6/8/10/12/14, 표지타입 종속)를 적재할 슬롯이 현 연결테이블에 부재.
- **벤치마킹(benchmark §3.2):** "책등은 후니가 이미 제본 param(`책등 mm`)으로 보유 — 경쟁사보다 우위. 유지." RP/WP 모두 책등 명시 신호 약함.
- **표준(process-recipe §3-2):** 책등 두께는 **페이지수·종이두께에 종속**(파생값). 즉 상품별 고정 enum이라기보다 페이지수→책등 함수.

### 권고 (후니 엔티티 + 관리 용이성)
**책등은 마스터 `prcs_dtl_opt.책등` param에서 상속 — 상품별 슬롯 신설(DDL) 불요가 1순위 권고.** 근거: 책등 mm는 페이지수 종속 파생값이라 위젯이 "페이지수 선택 → 책등 계산/표시"로 런타임 처리하면 됨(가격엔진 round-2 영역). 상품별 책등 enum을 DB에 고정 적재하는 것은 과적재(페이지수마다 달라짐).
- **대안 비교(관리용이성):**
  - ① master prcs_dtl_opt 상속(권고): DDL 0, 위젯 런타임 계산. **관리 최단순.**
  - ② size variant(책등별 작업사이즈): L1 row15 `*책등에따라다름` 단서 있으나 plate 메타라 size축 부적합.
  - ③ constraint_json(jsonb 실존): 표지타입별 책등 범위를 JSON으로 둘 수 있으나 책등이 캐스케이드 제약이 아니라 파생값이라 부적합.
- → **DDL 확장(상품별 param 슬롯 신설) 불요.** 책등은 마스터 param + 위젯 런타임. 레이플랫(PROC_000025)은 미운영 마스터(적재 0) 정합 — 별도 확인 1건만 사용자에게(아래).

### 분류: `[도메인해소]` (권고로 확정 — master param 상속, DDL 불요)
- 단 **포토북 제본=PUR vs 레이플랫**(D-PB-LP)은 운영 사실 확인 1건 필요(아래 §8). 책등 param 위치 자체는 도메인 해소.

### 위젯 영향
