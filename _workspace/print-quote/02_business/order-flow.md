## 1. 주문 상태 머신 (p.1 `01. 주문프로세스`)

### 1.1 상태 노드 (7개 정식 상태)

| 코드 | 한글 | 의미 | 다음 상태 |
|------|------|------|------|
| `unpaid` | 미입금 | 가상계좌 입금 대기 | paid / cancelled |
| `paid` | 결제완료 | 결제 승인 (카드/포인트/에스크로/가상계좌/신용결제) | preparing |
| `preparing` | 제작대기 | MES로 주문 이관, 의뢰서 출력 대기 | producing |
| `producing` | 제작중 | 의뢰서 출력 완료, 생산 라인 진입 (process-flow Case 1~17) | done |
| `done` | 제작완료 | 1차포장 완료 (process-flow `1차포장:제작완료` 마커) | shipped |
| `shipped` | 출고완료 | 송장출력 또는 직접방문 수령 | (종결) |
| `cancelled` | 주문취소 | 환불·이체환불 처리 | (종결) |

### 1.2 결제 케이스 (p.1)

미입금 → 결제완료 전이는 5가지 케이스로 분기:

| Case | 결제수단 | 트리거 |
|------|--------|------|
| Case 1 | 카드결제 | 이니시스 승인 즉시 → 결제완료 |
| Case 2 | 포인트결제 | 시스템 자동 차감 → 결제완료 |
| Case 3 | 에스크로 | 이니시스 승인 → 결제완료 |
| Case 4 | 가상계좌 | 입금 확인 시 → 결제완료 |
| Case 5 | 신용결제 | MES에서 생성된 주문 → 제작대기에서 시작 (미입금 우회) |

### 1.3 파일 흐름 부속 상태 머신 (p.1 `파일번호`)

주문번호와 별도로 **파일번호**(고유 생성번호) 단위로 부속 상태 머신이 동작.

#### Case 1 — 배송상품 (재고)

`상품준비중 → 포장완료`

#### Case 2 — 파일업로드 상품

```
확인전 ─→ 파일확인 ─→ 다운로드완료 ─→ 포장완료
   │
   └─→ 재업로드요청 ─→ 재업로드완료 (고객 액션 후 확인전으로 복귀)
```

#### Case 3 — 편집기 상품 (Edicus)

```
랜더링전 ─→ 랜더링완료 ─→ 다운로드완료 ─→ 포장완료
              │
              └─→ 수정요청 ─→ 수정완료 (고객 액션 후 랜더링완료로 복귀)
```

**관찰:**
- 주문 상태(7개)와 파일 상태(상품 타입별)는 **이중 상태 머신**. 주문 상태가 `producing` 일 때 파일 상태가 `포장완료` 가 되어야 주문 상태가 `done` 으로 전이.
- 한 주문에 여러 상품(파일 N개)이 있는 경우 **모든 파일이 포장완료** 일 때 주문이 `done` (p.1 명기: "모든상품이 포장완료이면 제작완료").
- 색상 범례 (p.1): **빨강 화살표 = 시스템 자동처리**, **녹색 화살표 = 관리자 수동처리**, **보라 화살표 = 고객 액션**.

### 1.4 상태 머신 다이어그램 (요약)

```
                  ┌─── Case1~4 결제 ───┐
                  ▼                    │
  [unpaid] ─── 입금/승인 ──→ [paid] ──→ [preparing] ──→ [producing] ──→ [done] ──→ [shipped]
      │                                    ▲
      │                                    │ Case5 신용결제 (MES 발생, paid 우회)
      ├─── 자동취소(1일전 미입금 알림 후) ──→ [cancelled]
      └─── 고객 직접 취소 ─→ [cancelled]
                                                            ┌── 택배출고 ──→ shipped
                                                            ├── 퀵 출고 ──→ shipped
                                                            └── 직접방문 ──→ shipped
```

## 2. 상태 전이 트리거 (p.1 + p.2)

### 2.1 전이 트리거 표

| 전이 | 트리거 | 처리 | 출처 |
|------|------|------|------|
| `unpaid → paid` | 이니시스 승인 / 가상계좌 입금 / 포인트 결제 / 에스크로 승인 | 시스템 자동 | p.1 |
| `unpaid → cancelled` (자동) | 가상계좌 미입금 시한 도래 (정책 미정) | 시스템 자동 + 1일전 알림 | p.2 011 |
| `unpaid → cancelled` (수동) | 고객 직접 취소 버튼 | 사용자 액션 | p.1 |
| `paid → preparing` | 결제완료 즉시 (또는 MES 주문 등록 시) | 시스템 자동 | p.1 |
| `preparing → producing` | 의뢰서 출력 + 주문상태 변경 | 관리자 수동 (발주팀) | p.1 + p.9 |
| `producing → done` | 모든 상품 포장완료 (= 1차포장 바코드 입력) | 시스템 자동 (집계) | p.1 + 공정관리 PDF |
| `done → shipped` (택배) | 송장출력 (바코드 입력) | 관리자 수동 (출고팀) | p.2 008 |
| `done → shipped` (퀵) | 납품명세서 출력 | 관리자 수동 | p.2 007 |
| `done → shipped` (직접방문) | 출고명세서 출력 + 고객 방문 수령 | 관리자 수동 | p.2 006 |

### 2.2 파일 상태 전이 트리거

| 파일 상태 전이 | 트리거 | 액터 |
|--------|------|------|
| `확인전 → 파일확인` | 발주팀이 파일 확인/수정 후 등록 | 관리자 수동 |
| `확인전 → 재업로드요청` | 발주팀이 파일 결함 발견 → SMS/메일 발송 | 관리자 수동 |
| `재업로드요청 → 재업로드완료` | 고객이 신규 파일 업로드 | 고객 |
| `랜더링전 → 랜더링완료` | Edicus Prepress 자동 렌더링 | 시스템 자동 |
| `랜더링완료 → 수정요청` | 고객 검토 후 수정 의뢰 | 고객 |
| `수정요청 → 수정완료` | 고객이 수정 사항 확정 | 고객 |
| `파일확인/랜더링완료 → 다운로드완료` | 생산팀이 파일 다운로드 (제작중 진입) | 관리자 수동 (생산) |
| `다운로드완료 → 포장완료` | 1차포장 바코드 입력 | 관리자 수동 (포장) |

## 6. Pain Point (p.5 `05. Pain point`)

### 6.1 발주 단계 Pain (4채널 OEM 비교)

후니프린팅은 **자사사이트(현 후니프린팅)** 외에 3개 OEM 채널 운영:

| 채널 | 거래처코드 | 주요 Pain |
|------|--------|------|
| **비즈하우스** | BIZ | 웹페이지 로그인 후 다운로드 → 해외배송(주소 검색) → 색깔종이에 출력 |
| **후지필름** | FUJI | 생산관리 프로그램에서 생산대기 선택 후 SCV 다운로드 → EXCEL 변환 후 등록 |
| **컨티뉴** | (별도) | 웹하드에서 다운로드 → EXCEL 확인 (수동주문이라고 생각하면 됨) |
| **자사사이트** | HUNI | 대부분 AI/JPG 업로드 → 조판, 파일변환 등 100% 파일명 수정 |

### 6.6 사용자 단계 Pain (해결책 명기)

- 출력시간 분배가 어려움

**해결:** "MES의 상품과 사용자의 상품을 매칭하여 파일네임을 일관화해야 합니다."

### 6.8 출고 단계 Pain (해결책)

- 합배송의 리스트를 알 수 없음
- 송장추가시 배송정보를 입력해야함

**해결:** "하나의 주문에 여러개의 송장 부여가 가능하도록 합니다."

## 11. 발주 업무 프로세스 (p.10 `09. 발주업무 프로세스`)

### 11.1 발주 흐름

```
[제작대기] ──→ 접수파일 등록 ──server→ 썸네일생성 ──→ 의뢰서 출력 + 주문상태변경 ──→ [제작중]
                                                              │
                                                              ├─→ 외주발주 (외주처별 발주서 생성)
                                                              └─→ 수동조판 (틴거울, 아크릴, 자석북마크)
```

### 11.2 두 가지 접수파일 등록 케이스

| Case | 내용 |
|------|------|
| **Case 1** | 원본파일 그대로 접수파일로 사용 — 업체에서 완성된 파일을 받을 경우 (`디자인파일여부` Y/N로 자동처리) 또는 Edicus Prepress를 통해 랜더링한 경우 |
| **Case 2** | 원본파일을 다운 받아 확인하고 가공 후 접수파일 등록 — 고객 파일을 다운 받아 접수파일로 변환 (후니프린팅 파일업로드 주문). 하나의 파일을 2개 파일로 분리 (예: 자유형 스티커). 접수파일이 AI인 경우 썸네일 생성을 위한 JPG 파일을 함께 업로드 (예: 아크릴스티커, 박) |

### 11.3 외주 발주

- 접수파일을 모아 외주처별 발주서 생성 → 발주서 출력
- 책자 상품의 경우 수량·크기에 따라 **선택적 외주 발주**

### 11.4 수동 조판

여러 주문을 수동 조판: 예시 — 틴거울, 아크릴상품, 자석북마크. **수동조판한 파일은 별도 전달**.

### 11.5 생산 업무 프로세스

"각 팀별 권한"별로 MES를 통해:
- 제작중인 상태의 주문건 AND "품목관리에서 생산다운 여부 Y" 상품만 다운로드
- 해당 문서 03-1과 상품마스터 품목관리 참고

```
[접수완료] ─→ [제작대기] ─→ 의뢰서출력하면 빨간점 표시 ─→ [제작중] ─→ [출고준비중] ─→ [출고완료]
                                                              │                       │
                                                              제작중으로 상태변경됨    송장부여
```

## 13. 파일 FLOW (p.12 `06-1. 파일 FLOW`)

### 13.1 인프라 구조

```
                ┌── AMAZON (클라우드)
                │      │
                │      ▼
[오프라인주문] ──→ MES ─────→ [사용자사이트] (코드엠샵 호스팅서버 buysangsang.co.kr)
                │       
                │       ┌── 아이락 오아시스 (Edicus 서버)
                │       └── 후지필름 생산컨트롤타워 / 모션원 프리프레스 커스텀
                │                  ↑
                │                  └─ "프리프레스 로 변경" (통신해야함)
                ▼
       [파일서버 + NAS] ⭐ 신규구입 — 접수파일 취합 / 조판파일 생성
                │
                │ 접수파일 이동
                ▼
       ┌────────┴────────┐
       │ 발주팀 ── 비즈하우스 다운로드 + 후지필름 랜더파일
       ▼
   년도/날짜(발주일자) 단위 디렉토리 구조 — 9개 인쇄타입별
       │
       ├── 디지털인쇄 ──→ 출력팀 (DOWN, 파일명 RENAME)
       ├── 실사출력 / 레이저커팅 / 시트커팅 ──→ 실사출력팀 (DOWN, 파일명 RENAME)
       ├── 커팅 ──→ 커팅파트
       └── UV인쇄 / 패브릭인쇄 / 전사인쇄 / 화이트인쇄 ──→ 쿠마샵(외주)
```

### 13.2 신규 인프라 요구사항

- **신규 구입 필요:** 파일서버 + NAS (접수파일 취합/조판 전용)
- **프리프레스 설정:** 후지필름·모션원 프리프레스 커스텀 통신 필요
- **WINDOW SERVER:** 서버호스팅 (WEB + DB + STORAGE)
- **AMAZON:** 원본파일 UP + 접수파일 UP

🟡 **DECISION D-PM-29:** To-Be에서 파일 인프라 구성 결정. (현 As-Is: AWS S3 + 자체 NAS + 호스팅 서버 분산. 권장: AWS S3 단일 통합 + CloudFront CDN — Big-Bang 컷오버 핵심 결정. → ADR-002 후속과 연계)
