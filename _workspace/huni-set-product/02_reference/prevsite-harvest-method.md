## 0. 한 줄 결론

이전사이트 가격계산 = **서버 AJAX 조회 방식 (b)**. 옵션 선택 → `setActionCheck()`가 `/upload/product/a_<FORMCODE>.asp?...`를 **GET**으로 호출 → 응답(`::`·`|` 구분 단가 배열)을 hidden 필드에 채우고 클라이언트 `setPrice()`가 합산. **단가표는 클라이언트 JS에 없음**(서버 ASP가 단가 계산). → **권고 경로 = 가격AJAX 배치 호출**(폼 select로 옵션 enum 수집 + AJAX로 옵션→단가 매트릭스 획득). 클릭 없이 전 상품 가능.

★사용자 지적 실증: **트윈링 표지 단면 vs 양면 → 표지인쇄비 6,000 → 10,000** (AJAX 응답 `req_data[14]` 변동). 정적 `<select>` 목록으로는 "어떤 옵션이 가격을 움직이나"를 못 잡고, **동적 AJAX 응답으로만** 잡힌다. 설계가 표지를 "단면만"으로 둔 것은 **표지 양면 가격축 누락**(검증 트랙 인계).

## 1. 상품 목록 체계 (정찰 1)

- 상품 상세 URL = `https://www.huniprinting.com/product/goods.asp?pcode=N` (손님 주문 폼).
- **pcode = 일련번호**(상품별·1~150+ 범위·prd_cd와 다름). 책자세트=36 무선·37 PUR·38 트윈링·39 중철·40 하드커버무선·41 하드커버트윈링·44 싸바리.
- 카테고리 = breadcrumb `홈 > <카테고리> > <상품명>`(상품 상세에서만 신뢰·메뉴는 gif 스프라이트).
- **전 pcode 열거**: 정적 fetch가 그리드 미제공 → `seq 1..N` 네비게이트 스캔으로 breadcrumb 수집(기존 헬퍼 `_huni_live_pcode-index.csv`에 다수 인덱싱 완료). 한 번에 전 상품 URL 열거 = **pcode 순회로 가능**(인덱스 재사용 우선).

## 2. 폼 옵션 구조 일관성 (정찰 2)

- 옵션 = `<select name=tmp_pNN[...]>`. 패턴이 **상품군 간 일관**(tmp_p01 사이즈·tmp_p07_* 용지·tmp_p09 인쇄·tmp_p10 코팅·tmp_p18 제본·tmp_p20/tmp_p21_* 표지·tmp_p16/17 박형압·tmp_p05/06 링·투명커버·tmp_p22 포장).
- option value = **코드**(`단면;1`·`62`)이지 단가 아님. 단가는 AJAX가 채움.
- **폼코드(FORMCODE)** = 상품군별: BH01(책자)·PR01(실사)·GD01(굿즈)… `j_<FORMCODE>.js`(폼 로직)+`a_<FORMCODE>.asp`(가격AJAX) 짝. `setActionCheck`는 **전 상품 공통 함수명**.
- → 같은 파서로 select/option enum을 전 상품 긁을 수 있음(동일 패턴). 단 옵션 코드 의미는 폼코드별 다를 수 있어 매핑 필요.

## 3. ★가격계산 방식 — (b) 서버 AJAX (정찰 3·핵심)

### 3.1 메커니즘 (실측 입증)
```
옵션 onchange → calPrice() → check_pXX() (검증·hidden 코드 채움) → setActionCheck()
  → url = "/upload/product/a_BH01.asp?pcode=38&p01=..&p07=..&p09=..&p18=..&qty=..&page=..&p21_a=..&..."
  → req.open("GET", url); req.send()                                    ← 서버 가격조회 (b)
  → 응답: "0000::154|216|0.5|3|20000|0|1|35000|800|100|A|0|2|20000|6000|200|0|0|100|A|0|10"
            ↑상태  ↑req_data[0..]= 각 구성요소 단가 (| 구분)
  → setActionCheck가 req_data[7]=p08_d,[8]=p07_d,[11]=p10_d,[13]=p18_d,[14]=p21_e,[15]=p21_f… hidden에 채움
  → setPrice()가 클라이언트에서 단가 합산 → price_01 표시
```
- **(a) 클라이언트 JS 계산 = 아님**: `_amt`=number(0)·단가표 전역변수 없음. `j_BH01.js`=폼 검증/UI 로직만(단가 없음).
- **(b) 서버 AJAX = 맞음**: 단가 정답은 **`a_<FORMCODE>.asp` 서버**가 계산. 응답은 ASCII text(`::`로 status·`|`로 단가배열).

### 3.2 AJAX 파라미터 shape (책자 a_BH01.asp·실측 verbatim)
`pcode·p01(사이즈)·p_width·p_height·p07(용지)·p08(용지등급)·p09(인쇄)·p10(코팅)·p18(제본)·qty(수량)·page(페이지)·page_01·page_02·p20(표지유형)·p21_a(표지인쇄)·p21_b(표지용지)·p21_c(표지코팅)·p21_d·p22(포장)·dummy`. GET·인증 불요(손님 조회).

### 3.3 ★응답 필드 사전 (req_data 인덱스 → 구성요소·실측)
| idx | 필드 | 의미 | 단면 표지 | 양면 표지 |
|---|---|---|---|---|
| 0/1 | p_width/height | 작업 폭/높이 | 154/216 | 154/216 |
| 7 | p08_d | (용지등급가) | 35000 | 35000 |
| 8 | p07_d | 내지용지비 | 800 | 800 |
| 11 | p10_d | 내지코팅비 | 0 | 0 |
| 13 | p18_d | 제본비 | 20000 | 20000 |
| **14** | **p21_e** | **표지인쇄비** | **6000** | **10000** ← ★단/양면 변동 |
| 15 | p21_f | 표지(추가) | 200 | 200 |
| 18 | p21_gram | 표지평량 | 100 | 100 |

→ ★**표지인쇄비(p21_e)가 표지 단/양면에 따라 6000↔10000** = 사용자 지적 실증. 표지 인쇄면 = 가격 구성요소(설계 "표지 단면만" 누락).

## 4. ★효율 수집 방법 권고 (정찰 4)

### 권고 경로 = **가격AJAX 배치 + 폼 enum 하이브리드** (클릭 0·결정적)

| 단계 | 방법 | 산출 |
|---|---|---|
| **S1 pcode 로스터** | `_huni_live_pcode-index.csv` 재사용 + 빈 pcode만 네비 스캔 보강 | 전 상품 pcode·카테고리·상품명 |
| **S2 옵션 enum** | pcode당 1회 gstack goto → `<select>` 전수 파싱(§정적·이미 검증된 파서) | 상품×옵션축×옵션값(코드) 목록 |
| **S3 폼코드·파라미터 shape** | pcode당 `setActionCheck.toString()`에서 url 템플릿 1회 추출(FORMCODE·파라미터 순서) | a_<FORMCODE>.asp URL 스킴 |
| **S4 가격AJAX 배치** | S2 옵션값 조합을 S3 URL 템플릿에 대입 → **GET 직접 호출**(fetch·브라우저 1탭 내·클릭 0) → `::`·`|` 파싱 | 옵션조합→구성요소별 단가 매트릭스 |
| **S5 동적 가격축 발굴** | S4에서 **한 축만 바꿔 응답 단가 변동 관찰**(표지 단/양면처럼) → "가격을 움직이는 구성요소" 식별 | 가격영향 옵션축 사전(설계 누락 검출) |
| **S6 엑셀 매칭 검증** | S5 가격영향축 ↔ 권위 엑셀 구성요소 ↔ 라이브 t_prc_* 대조 | 누락/오매칭 결함보드(돈 직결) |

### 왜 이 경로인가 (근거)
- **클릭 0**: gstack 옵션 클릭 대신 AJAX URL을 fetch로 직접 호출(같은 origin·브라우저 1탭). 한 상품의 N 옵션조합을 순차 GET → 분 단위 수집.
- **결정적·정확**: 단가가 **서버 정답**(클라이언트 추정 아님). 응답 req_data 인덱스가 구성요소별 단가를 분리 제공 → "어느 구성요소가 얼마"까지 분해.
- **동적 가격축 포착**: §5 한 축 변경 관찰로 "가격을 움직이는 옵션"을 정적 목록 한계 없이 포착(트윈링 표지 양면처럼).

### 대안 경로 비교
| 경로 | 범위 | 정확도 | 리스크 |
|---|---|---|---|
| (A) **권고: 가격AJAX 배치+enum** | 전 상품·전 가격축 | 高(서버 단가 분해) | 조합폭발(옵션 다축 상품)·서버 부하(rate-limit 필요) |
| (B) 정적 select만 | 전 상품 옵션 enum | 中(가격 거동 못봄) | ★가격영향축 누락(사용자 지적 함정) |
| (C) gstack 클릭 시뮬 | 소수 상품 | 高 | 느림(클릭당 sleep)·대량 비현실 |

→ **(A) 권고**. (B)는 사용자 지적대로 부족(가격거동 누락). (C)는 실증·소수에만.

## 5. 전 상품 적용 절차 (방법만·실행은 별도 승인)

1. **응답 필드 사전 폼코드별 1회 추출**: req_data 인덱스→구성요소 매핑이 폼코드(BH01/PR01/GD01…)별로 다를 수 있음 → 폼코드당 setActionCheck 응답 파싱 1회로 사전 고정.
2. **조합 샘플링**: 다축 상품은 전수 조합 대신 **한 축씩 변경(OFAT)** + 권위 엑셀 가격축 교차로 핵심 조합만(조합폭발 방지).
3. **rate-limit·읽기전용 가드**: GET만·간격 sleep·동시성 1·주문/장바구니 URL 금지. 봇차단(403) 회피=gstack 브라우저 1탭 내 fetch(같은 세션).
4. **엑셀 권위 매칭**: 수집 단가매트릭스를 **검증 입력**으로만(가격 값 정답=엑셀·이전사이트=구조/거동 신호). 차이=조사신호(자동 교정·자동 주입 금지).

## 6. 리스크·한계

- **가격 값 ≠ 정답**: 이전사이트는 구 시스템(stale 가능)·권위 엑셀이 절대. AJAX 단가는 "구성요소가 가격을 움직이는가/어느 축인가"의 **구조 신호**로만 사용(값 대입 금지).
- **조합폭발**: 표지(인쇄×용지×코팅)×내지(용지×인쇄)×제본×사이즈×페이지×박 = 수천 조합/상품. OFAT+권위 교차로 축소 필수.
- **폼코드별 응답 사전 상이**: req_data 인덱스 의미가 BH01과 GD01이 다를 수 있음(폼코드당 1회 매핑 선결).
- **옵션 코드 의미**: option value 코드(`62`=백모120)는 사이트 내부 코드 → 후니 권위 코드/명으로 번역 필요(naming 유입 금지).
