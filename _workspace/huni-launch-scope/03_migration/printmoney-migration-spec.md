# 프린팅머니 이관 설계·명세 (printmoney-migration-spec)

> 산출: hls-migration-designer · 2026-06-30 · Huni-Launch-Scope(§28)
> ★위상: **설계·명세 중심**(원천 DB 미접근). 라이브 화면(`00_live/migration-screen-clues.md` §2) + Shopby 적립금 모델(`docs/shopby`)을 양 끝점으로. **실 이관은 인간 승인 후.**
> ★[HARD] 잔액=돈. 1원도 틀리면 안 됨 → **합계 대사(reconciliation)를 필수 GO 게이트로** 둔다.

---

## 0. 요약

- **이관 대상:** 회원별 **프린팅머니 잔액(balance)** + (보존 결정 시) **거래 원장(적립/사용/만료 내역)** + 만료정책.
- **모델 격차(OQ-G4):** 후니 프린팅머니=**충전식 선불 잔액**(PG로 미리 충전) / Shopby 적립금(accumulation)=**구매적립형**. **단, Shopby 적립금은 "현금성 포인트로 결제금액에 사용"**(accumulation-setting.mdx 근거) → **사용(usage) 의미는 동일**, 차이는 **적립(earning) 메커니즘**뿐.
- **핵심 발견:** Shopby는 **수동 지급/차감 + 외부 server API**로 임의 잔액을 회원에 시드 가능 →
  - `POST /profile/accumulations`(systemKey·회원별 즉시 지급·externalKey 추적)
  - 어드민 **적립금 일괄 지급(엑셀 등록)** + 사유 + 유효기간(1~71개월/제한없음)
  - `DELETE /profile/accumulations`(차감·보유액 초과 시 실패)
  - 웹훅 `ACCUMULATION_ADDED/SUBTRACTED`(감사 추적)
  → **legacy 잔액 마이그레이션은 적립금 수동지급으로 무손실 시드 가능.** GAP은 잔액 이관이 아니라 **향후 PG 선불충전(머니충전)** — 이는 이관 스코프 밖(운영 BFF).

## 1. 이관 대상 데이터셋 (관찰 기반)

| 데이터 | 출처(관찰) | 상태 |
|--------|-----------|------|
| 회원별 잔액(현재 사용가능 금액) | money.asp / 대시보드 "프린팅머니 N원" | ★확정 대상(단일 값) |
| 거래 원장(적립/사용/만료, 일시·금액·사유) | money.asp "내역 목록" 영역(테스트계정 0건→헤더 미관측) | ★스키마 원천 확인 필요 |
| 만료정책(유효기간·소멸) | 화면 미노출 | ★원천 확인 필요 |
| 머니충전(PG 선불충전) | 화면 미노출 | 이관 범위 외(운영 BFF) |

→ 마이그레이션 데이터셋(최소): **① 회원별 잔액(기준시각 스냅샷)**. 선택: **② 거래 트랜잭션 원장**.

## 2. 모델 매핑 — 두 방안 트레이드오프 (OQ-G4)

### 방안 A. Shopby 적립금 재해석 (★1차 권고)
구 프린팅머니 잔액을 **Shopby 적립금으로 이행**. 시드 = 회원별 수동지급 1건(`POST /profile/accumulations` 또는 어드민 엑셀 일괄).

| 항목 | 내용 |
|------|------|
| 잔액 시드 | 회원당 1 grant = 구 잔액 전액·사유="구사이트 프린팅머니 이관"·externalKey=이관키 |
| 사용 흐름 | 주문 calculate `accumulationUseAmt`로 결제 차감(기존 capability-map 정합) |
| 만료 | grant 유효기간 설정(잔여 유효기간 원천 있으면 반영, 없으면 정책값) |
| 장점 | native 운용(어드민 적립금관리·웹훅·마이페이지 노출 그대로)·추가 개발 최소·정산 정합 |
| 제약 | "적립금" 명칭 노출(→ 적립금 명칭을 "프린팅머니"로 설정 가능: accumulation-setting ①)·**향후 PG 선불충전은 별도 BFF 필요**·구매적립형 자동적립 규칙과 충전식이 한 잔액통에서 혼재(원장 구분은 사유/externalKey로) |

### 방안 B. 별도 충전식 원장 (BFF)
후니 BFF에 **충전식 잔액·트랜잭션 원장**을 별도 보유하고 결제 시 연동.

| 장점 | 충전식 모델 100% 보존·충전 트랜잭션 1급·복잡 만료/소멸 규칙 자유 |
|------|------|
| 단점 | 돈 무결성 직접 책임(이중원장 정합·Shopby 정산과 동기화 리스크 OQ-G3)·개발 XL·마이페이지/어드민 커스텀 |

### 권고
**1차 = 방안 A(적립금 재해석) + 명칭을 "프린팅머니"로 설정.** 잔액 마이그레이션·결제사용·만료가 native로 충족되고 돈 무결성 책임을 Shopby에 위임. **머니충전(향후 신규 충전)만 PG→`POST /profile/accumulations` BFF로 보강**(fit-gap No15 CUSTOM과 정합·이관 스코프 밖). 방안 B는 충전식 원장이 사업상 1급 요건으로 확정될 때만.

## 3. 잔액 vs 원장 — 보존 수준

| 수준 | 내용 | 트레이드오프 |
|------|------|-------------|
| L1 잔액만(★권고) | 회원별 현재 잔액 = 1 grant 시드 | 단순·대사 명확. 과거 거래이력은 구사이트 보관(열람 BFF 선택) |
| L2 원장 1:1 | 적립/사용/만료 각 건 재현 | Shopby는 과거 일자/사유 임의 backdate·임의 만료일 정밀재현 제약 → 완전 1:1 어려움·고비용. 합계 불일치 위험 |

→ **L1 권고**: 돈 정확성의 자는 **잔액 합계 대사**이지 거래 건수 재현이 아님. 과거 내역 열람 필요 시 구사이트 read-only 아카이브 또는 BFF 조회로 분리.

## 4. 이관 절차 (추출 → 변환 → 적재 → 검증 → 롤백)

```
[1 추출]  구 DB → 회원별 잔액 스냅샷(기준시각 동결·이후 거래 동결/델타관리)  ← 원천 제공 후(인간 승인)
   │        · 음수/0/비정상 잔액 분류
[2 변환]  잔액→grant 레코드(memberId·금액·사유·externalKey·유효기간)        ← 멱등·합계 보존
   │        · 합계 검증값 산출(Σ구잔액)
[3 적재]  Shopby 적립금 지급                                                  ← 회원당 1건·재시도
   │        · POST /profile/accumulations(externalKey 멱등) 또는 어드민 엑셀 일괄
[4 검증]  ★합계 대사 게이트(§6)                                              ← 통과 못하면 머니 컷오버 중단
[5 롤백]  DELETE /profile/accumulations(externalKey 기준)로 시드 회수·재실행  ← 멱등 회수 경로
```

- **동결(freeze):** 잔액 스냅샷 시각 이후 구사이트 머니 변동을 **동결**하거나 **델타 재적용** 규칙 필수(이중계상·누락 방지). 컷오버 전 충전/사용 중단 창(window) 권고.
- **멱등성:** `externalKey`(최대 60자)에 이관키 부여 → 재실행 시 중복 지급 방지·롤백 추적.
- **재시도:** 회원 단위 격리 재처리. 부분 실패가 합계 대사를 깨지 않도록 성공/실패 원장 유지.

## 5. 예외 처리 규칙

| 예외 | 규칙 |
|------|------|
| 0원 잔액 | 시드 생략(불필요 grant 0). 회원 이관과는 독립 |
| 음수 잔액 | ★시드 금지(적립금은 음수 불가)→보류 리스트·수기 결정(미수금은 §원장관리 백오피스 영역) |
| 만료 임박/만료분 | 구사이트에서 이미 소멸된 분은 잔액에 미포함 가정·정책 확인 |
| 회원 미존재(머니만 있고 회원 보류) | 회원 시드 선행 의존(머니는 회원 이관 GO 후 적재) |
| 중복 지급 | externalKey 멱등으로 차단 |
| 차감(롤백) 실패 | 회원이 이미 사용해 잔액 부족 시 DELETE 실패(A0002)→부분 회수 정책 |

## 6. ★합계 대사 검증 게이트 (머니 컷오버 GO 기준·HARD)

| 게이트 | 기준 | 방법 |
|--------|------|------|
| **P-G1 잔액 합계 대사** | **Σ(구 회원잔액) = Σ(Shopby 적립금 시드액)**, 차이 = **0원** | 추출 합계 ↔ 적립금 지급 합계 1원 단위 대조 |
| P-G2 건수 대사 | 시드 대상 회원수(잔액>0) = grant 성공건 + 보류(음수 등), 누락 0 | 카운트 reconcile |
| P-G3 샘플 1:1 | 대표 샘플 회원 잔액 = Shopby 가용 적립금(POST /accumulations/members/available 조회) | 1:1 조회 대조 |
| P-G4 멱등 재실행 | 동일 배치 2회 실행 시 추가 지급 0(합계 불변) | externalKey 검증 |
| P-G5 사용 가능성 | 시드 적립금이 주문 calculate accumulationUseAmt로 실제 차감되는지 종단 1건 | 종단 테스트 |
| P-G6 만료 정합 | grant 유효기간이 정책/원천대로 설정·소멸시점 검증 | 설정값 대조 |
| P-G7 동결 정합 | 스냅샷 이후 구사이트 변동분이 누락/이중계상 없이 처리 | 델타 대조 |

★**P-G1(합계 대사) 불일치 시 머니 컷오버 즉시 NO-GO.** 돈은 표면 통과 금지 — 1원 차이도 조사 신호.
