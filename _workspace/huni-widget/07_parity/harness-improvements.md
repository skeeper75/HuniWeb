# Huni-Widget 하네스 개선점 (2026-06-03 세션 도출)

이번 세션(동등성 게이트 → 코드 레벨 구조 정합 S0~S3)에서 실증된 개선점. A2에서 오케스트레이터·에이전트·CLAUDE.md에 반영, C에서 실행 지침화, B에서 팀으로 교차 검증.

## 1. 검증 깊이 — 캡처 표본 < 역공학 코드 권위 (단편 금지)
- **관찰**: 캡처 4모델 표본 게이트는 GO였으나, 표본이 단순 케이스만 써서 구조 결함(ATTB·복합축·color-chip)을 못 봄.
- **개선**: 역공학 소스코드(deob 4모듈)를 권위 명세로 삼는 **코드 레벨 구조 정합 검증축(S0 지도→S1 갭지도→S2 전상품→S3 보정)**을 정식 스테이지로.
- **반영처**: 오케스트레이터 — 동등성 검증 시 "캡처 표본 검증"과 "코드 정합 검증" 2층 게이트.

## 2. 독립 재검증 게이트 (자기보고 불신)
- **관찰**: 매 보정 후 hw-qa가 직접 재실행·재대조 → builder 자기테스트가 못 본 것을 반복 적발(F-2 직렬화 shape, C-A disable 미스매치, L-4 오판).
- **개선**: "보정(builder) → **독립 재검증(hw-qa, 직접 실행·캡처 대조) → GO**" 게이트를 모든 보정 라운드에 의무화. builder 자기보고는 GO 근거로 불충분.
- **반영처**: 오케스트레이터 보정 루프 + hw-qa 정의.

## 3. fixture가 직렬화 shape 결함 은폐
- **관찰**: fixture가 HTTP를 우회(reqBody 무시) → 어댑터 직렬화 shape 결함이 N개 테스트 통과에도 침묵(F-2).
- **개선**: 외부 API 호출 어댑터/직렬화는 **라이브 캡처 reqBody와 field-for-field 대조 테스트** 의무. fixture 미보유 시 **침묵 폴백 금지 → 명시 throw**(F-1).
- **반영처**: build 스킬 + hw-builder 정의. (메모리 [[fixture-masks-serialization-shape]])

## 4. INV(코어 불변) 완화 조건 명문화
- **관찰**: "위젯 코어 0줄"은 상품 확대엔 맞지만 구조결함 보정엔 코어(cascade/store)·계약 수정 불가피.
- **개선**: INV-3에 "**버그·구조결함 보정은 정당 예외** — 단 코어 수정 최소화 + 계약은 additive optional + `git diff --stat`로 변경 전수 명시·1줄 정당화" 조건 명문화.
- **반영처**: 오케스트레이터 불변식 + hw-builder.

## 5. 신규 leaf 사전정당 게이트
- **관찰**: 신규 컨트롤은 D4가 사전 정당화한 2개(MultiCheckGroup·AccPanel)만 → scope-creep 0.
- **개선**: 신규 leaf 컨트롤은 **사전 정당화(어느 갭이, 왜 기존 14종으로 불가)** 없이는 추가 금지. 가능하면 플래그 분기(group.multiple)로 신규 dispatcher case 회피.
- **반영처**: hw-builder + hw-architect.

## 6. 캡처 한계 진단 — 번들 상수 이식 경로
- **관찰**: 일부 데이터(반경 roundingConfigMap, ACC accFilterConfigMap)는 product_info 캡처로 해소 불가 — Vue 번들 상수가 본질.
- **개선**: 캡처 전 "이 데이터가 product_info에 있나 vs 번들 상수인가" 판별. 번들이면 **캡처가 아니라 deob 소스에서 상수 추출·이식**(날조 금지, 미등록은 명시 fallback).
- **반영처**: live-capture 스킬 + hw-reverse-engineer/architect.

## 7. 웨이브 분할 + 파일충돌 회피
- **관찰**: 리스크 오름차순(위험한 P4 맨끝) 순차 + 같은 파일 건드리는 보정은 단일 builder 순차(병렬 충돌 회피).
- **개선**: 보정 다항목은 **리스크 오름차순 웨이브**로 분할, 웨이브별 단일 builder + 웨이브마다 독립 재검증 게이트.
- **반영처**: 오케스트레이터 보정 패턴.

## 8. 커밋 완결성 검증
- **관찰**: isReadyToOrder BFF 배선 3파일이 커밋 누락 → 클린 체크아웃 빌드 깨짐 위험(타입만 있고 구현 없음).
- **개선**: 커밋 후 **HEAD 상태에서 게이트(tsc/vitest/build) 재확인** 의무 — 워킹트리 통과 ≠ 커밋 트리 통과. 보정 변경 파일 전수를 커밋 범위와 대조.
- **반영처**: manager-git 커밋 지침 + 오케스트레이터.

## 9. 팀 vs 서브 선택 기준 (B의 근거)
- **관찰**: 이번 세션은 단일 테스트베드 공유·순차 검증이라 서브에이전트로 진행. 독립 재검증은 회의적 단일이 본질이라 서브가 적합했음.
- **개선**: **독립 검증·단일 회의적 판정 = 서브** / **발견 공유·상충 토론·누락 보완이 품질을 높이는 분석·설계 = 팀(TeamCreate)**. 특히 "여러 전문가가 같은 산출물을 다른 렌즈로 보며 서로의 사각을 메우는" 작업은 팀.
- **B 핵심**: 코드 정합 결과를 **팀으로 교차 재검증** — 서브 순차에서 놓쳤을 미흡(예: 모듈 경계 간 상호작용, 한 에이전트만 본 가정)을 팀 상호토론으로 발견.
- **반영처**: 오케스트레이터 실행 모드 — 분석/설계/교차검증 Phase는 팀, 단일 독립검증은 서브.

## 10. 모델 정책
- **개선**: 추론집약 하네스(역공학 정독·정합검증)는 전 역할 opus. (workflow.yaml 적용 완료)

## 11. 진행 가시성 (운영 관찰)

- **개선**: 오케스트레이터에 단계 보드 + 백그라운드 위임 패턴 명문화.
