# 02. Aurora React Skin — 확장 가능 영역 vs 고정 영역

생성일: 2026-05-27
선행 문서: `01_overview.md`

---

## 핵심 질문

> 상품 상세 페이지의 옵션 영역을 통째로 자체 컴포넌트로 갈아끼울 수 있는가?

**잠정 답**: 기술적으로는 가능 (소스 공개, React 컴포넌트 단위 교체 가능), 그러나 **공식적인 slot/plug-in 메커니즘은 가이드에 명시되지 않음**. 즉 "직접 fork 후 컴포넌트 트리를 수정"하는 것이 유일한 길이며, 업스트림 변경분과의 머지 충돌은 사용자 책임.

---

## 1. 확장 메커니즘 — 가이드에서 확인된 사실

### 1.1 명시된 확장 패턴

| 패턴 | 가이드 출처 | 의미 |
|------|------------|------|
| `git remote add upstream` 후 fork 워크플로 | `[활용]PC 버전 커스텀 가이드.mdx` | **유일하게 명시된 커스터마이징 방식 = 소스 직접 수정 + 머지** |
| 셀러어드민 "단순복사" / "단순복사+디자인 수정" 옵션 | `5_설치.mdx` | 상품 등록 단위에서 수정 여부 분기. "디자인 수정"은 수정 작업 완료 후 정보를 입력해야 설치 진행 |
| `@shopby/react-components` Provider + useStateContext/useActionContext | `@shopby_react-components/index.mdx` | Context 기반 상태 주입 — 커스텀 컴포넌트가 동일 Context 구독 가능 |
| `ShopbyQuery.executeQuery/executeMutation` | `@shopby_shared.mdx` | API 호출이 추상화돼 있어 외부 API 호출도 동일 패턴으로 가능(예제 코드에 `jsonplaceholder.typicode.com` 호출 예시 존재) |
| `외부_스크립트_호출.mdx` | 별도 가이드 | 외부 JS(예: Edicus iframe 같은 3rd party)를 스킨에 주입할 수 있음 |
| `assets_경로_변경_가이드.mdx` | 별도 가이드 | 정적 자산 경로 커스터마이징 가능 |

### 1.2 명시되지 **않은** 확장 패턴 (= 위험 신호)

- **컴포넌트 slot 시스템 없음**. "이 prop을 통해 OptionPanel을 교체할 수 있다" 같은 공식 API 부재.
- **theming/디자인 토큰 시스템 명시 0건**. 색상/타이포/spacing의 변수화 가이드 없음. (Tailwind/CSS variables/디자인 토큰 라이브러리 채택 여부 미확인)
- **라우터 추가에 대한 가이드 없음**. 새 페이지(`/quote/business-card` 같은 견적 마법사 라우트)를 추가하는 공식 절차 부재. React Router 추정이므로 코드 레벨에서는 가능하나, 백엔드(Shopby) 어드민과의 메뉴 연동 흐름이 불명.
- **i18n 시스템 명시 0건**. 한국어 단일 운영 가정으로 보임.
- **Plugin/Extension API 부재**. WordPress 플러그인이나 Shopify App 같은 "외부 모듈 주입" 메커니즘 없음.
- **Skin 자체의 컨피그 스키마 부재**. "이 JSON 한 줄로 옵션 패널 비활성화" 같은 토글 없음.

→ **결론**: Aurora는 "참조용 React 템플릿"이지 "확장 가능한 framework"가 아니다. 커스터마이징 = fork + 코드 수정.

## 2. 상품 상세 옵션 영역 — 갈아끼우기 분석

### 2.1 현재 구조 (가이드 기반 추정)

상품 상세 화면(`상품_상세/index.mdx` + `상품_기본정보.mdx` + `추가상품.mdx`)에서 옵션 영역의 흐름:

1. `GET /products/{productNo}` → 상품 기본정보 (`baseInfo`)
2. `GET /products/{productNo}/options` → 옵션 정의 (`flatOptions` or `multiLevelOptions` or `input`)
3. 사용자가 옵션 선택 → 클라이언트에서 옵션 단위 옵션-가격 계산 → `POST /cart` 또는 `POST /order-sheets`

이 흐름의 모든 단계가 React 컴포넌트로 구현돼 있을 것이며, 옵션 패널은 별도 컴포넌트(가령 `OptionSelector.tsx` 또는 `ProductOptionPanel.tsx`)로 추정.

### 2.2 갈아끼우기 시나리오 — 3가지 옵션

**시나리오 A: OptionPanel을 통째로 교체 (1:1 swap)**
- 자체 `<PrintOptionPanel>` 컴포넌트를 작성, 부모 컴포넌트에서 import만 교체.
- 이 컴포넌트는 자체 옵션 상태(용지/사이즈/후가공/수량 20+필드)를 보유.
- 최종 `POST /cart` 호출 시 Shopby의 `productOption` 형식으로 매핑.
- **장애물 1**: Shopby의 `cartItem`은 `optionNo`(어드민에 등록된 옵션의 식별자) 단위로만 동작. 자체 옵션 폼이 만든 자유 조합을 Shopby가 모르는 옵션으로 보낼 수 없음.
- **장애물 2**: `optionNo`를 어드민에 미리 등록해두는 방식은 옵션 조합 수가 폭증(용지 5 × 사이즈 8 × 수량 10 × 후가공 토글 5 = 2000+ 조합)하면 비현실적.
- **장애물 3**: 가격이 옵션의 가산 단가(`+₩5,000`) 모델이라 Shopby 옵션 단가 구조와 호환되지만, **구간 할인(Tiered Pricing)**과 옵션 간 의존 규칙(예: A4지 + 200매 + 코팅 = 별도 단가)은 Shopby가 모름. → "임의 옵션 + 가격 계산은 클라이언트 자체"로 가더라도, 결국 서버가 받은 가격을 검증 못함 → **결제 무결성 위험**.

**시나리오 B: 옵션 영역만 자체, 가격·결제는 Shopby 사용**
- 자체 옵션 폼을 화면에 그리되, 옵션 확정 시 "사전 정의된 단일 SKU 1개"를 cart에 담음.
- 견적 정보는 별도 메타필드(`orderProductOptions[].optionInputs` 같은 텍스트 옵션, 또는 별도 자체 DB)에 저장.
- **장점**: Shopby의 주문/결제/배송/회원/적립금/쿠폰 전부 활용 가능.
- **단점**: 가격이 한 SKU의 고정가격에 묶이거나, 가격 계산 결과를 customField로 텍스트 송신해야 함. 어드민 주문관리 UI에서 견적 사양이 표시되지 않을 가능성. CMS와 견적 룰의 dual source of truth.

**시나리오 C: Aurora 위에 견적 wizard 라우트만 추가 + Shopby에 가짜 SKU**
- `/quote/business-card` 같은 신규 라우트 추가(Aurora 위에 얹기).
- 견적 결과를 "프로젝트(견적서)"로 자체 DB에 저장(Edicus처럼).
- 마지막 단계에 가짜 SKU 1개(`productNo: 99999`, 가격 `₩0`)를 Shopby cart에 담고, 견적 정보를 텍스트 옵션으로 첨부.
- 결제 시점에 동적 가격을 Shopby에 강제 주입할 수 있는지가 관건. (Shopby는 어드민 상품 가격을 신뢰하므로, **임의 가격 주입 API가 있는지 별도 확인 필요**.)

### 2.3 결정적 한계 — 가격 무결성

Shopby의 `POST /order-sheets/{no}/calculate`는 어드민 상품 가격 + 쿠폰/적립금 정보를 받아 서버 측 최종 가격을 산출. **클라이언트가 임의 가격을 보내 결제할 수 없음(서버 재계산)**. 인쇄 견적 사이트는 옵션 조합에 따른 동적 가격이 본질이므로, 이 구조와 정면충돌.

→ **결정**: 옵션 폼만 갈아끼우는 시나리오 A/B는 작동하지 않거나 결제 무결성을 깨뜨림. 시나리오 C는 가능하지만 "Shopby의 상품·가격 모델을 우회"하는 셈이므로 Shopby 채택의 의미가 반감.

## 3. 라우팅·페이지 추가 자유도

- **자유도**: 코드 수정 가능 → 신규 라우트 추가 가능 (React Router 가정).
- **제약**: 신규 라우트에서 사용할 데이터가 Shopby API에 없다면 별도 백엔드 필요. 즉 "Aurora 위의 견적 라우트"는 가능하지만 데이터 백엔드는 별도 구축.
- **제약 2**: 어드민에서 메뉴/내비게이션 설정은 Shopby 어드민이 담당. 신규 라우트가 메인 메뉴에 자동 노출되지 않을 가능성 → 하드코딩 필요.

## 4. 의존 강결합 식별 (Aurora 코어 가정)

가이드를 통해 추정되는 "갈아끼우기 어려운" 강결합:

| 강결합 가정 | 영향 |
|------------|------|
| 가격은 옵션 단위 단일값 (`price.salePrice` + `additionalPrice`) | 구간 할인 + 다차원 옵션 가산 모델 구현 불가 |
| 옵션 = 어드민에 사전 등록된 SKU | 동적 옵션 조합(견적)과 충돌 |
| 장바구니 = 옵션 단위(`cartNo` per `optionNo`) | 1견적 = 다항 옵션을 1행으로 표현하려면 별도 매핑 레이어 필요 |
| 결제 = Shopby 서버 측 재계산 | 클라이언트 측 가격 산식 신뢰 불가 |
| 약관 = Shopby 어드민 enum 기반 | 인쇄 견적 특수 약관(파일 검수 동의, 사양 확인 동의) 추가 시 어드민 확장 필요 |
| 추가상품(extraProducts) = 본상품 종속 | "옵션 부가가 아니라 별도 SKU의 추가 결제"라는 모델. 후가공이 별도 SKU여야 함 → 어드민에 후가공 SKU 잔뜩 등록해야 함 |
| 비회원 cart = localStorage 자체 관리 | 견적의 "임시저장 후 메일 전송" UX 자체 구현 필요 |
| URL 구조 | 가이드에 명시 없음. buysangsang의 `/shop/{id}/{name}/` 호환 여부 미상 |
| SEO/SSR | 가이드에 명시 없음. SPA 단일 페이지 모델 추정 → SEO 추가 작업 필요 |

## 5. 종합 — 확장성 점수 (0~3)

| 차원 | 점수 | 사유 |
|------|------|------|
| 컴포넌트 교체 | 1 | 공식 slot 없음. fork+코드 수정 가능. |
| 라우팅 추가 | 2 | 가능. 단 어드민 메뉴 연동 별도. |
| 디자인 토큰/theming | 1 | 명시 시스템 없음. CSS 직접 수정. |
| i18n | 0 | 명시 시스템 없음. |
| Plugin/Extension | 0 | 메커니즘 부재. |
| 신규 백엔드 도메인 | 0 | Shopby 백엔드에 없는 도메인은 별도 백엔드 필요. |
| **확장성 합산** | **4 / 18** | 약 22%. "참조 템플릿"에 가까움. |
