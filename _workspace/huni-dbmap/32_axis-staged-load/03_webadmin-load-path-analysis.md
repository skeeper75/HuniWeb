### 2-1. ⓐ/ⓑ 핵심 판정 (사용자 지정 2축)

- **④ 자재 오염(색/형상/사이즈가 자재행) = ⓐ(입력).** `load_materials:236-242`는 셀값을 그대로 INSERT한다. "오렌지/핑크 (3개1팩)"·"원형 90mm"가 자재행인 것은 v03 `05_자재정보` 시트에 그렇게 적힌 것을 무변환 전파한 결과. **변환 코드가 색을 자재로 바꾸는 로직이 없다.**
- **⑥ 카테고리 고아 = ⓐ(입력).** `load_categories`는 v03 `01_카테고리`의 상위코드 컬럼을 충실히 따른다(비면 NULL). 고아 노드는 **v03 시트가 부모를 비워 만든 것**. 코드 결함 아님.

### 2-2. 단 하나의 ⓑ 후보 = 코드 내장 교정 사전의 미완성

- `ENUM_ALIAS`(`:109-113`)·`MAT_TYP_OVERRIDE`(`:116-121`)는 **v03 라벨 오류를 코드가 부분 교정**하는 화이트리스트. 레더 4건(`:116-121`)은 코드로 .06(가죽)化. **즉 일부 v03 오류는 코드가 이미 고치고 있다.** 이 사전이 **불완전**(색/형상/카테고리 고아는 미커버)한 것을 "ⓑ 결함"으로 볼 수도 있으나, **본질은 ⓐ(입력)**다 — 사전은 입력 오류를 덮는 패치이지 변환 로직 자체의 버그가 아니다.

> **종합 진원 판정: 6축 전부 ⓐ(입력 v03)가 진원.** load_master는 무변환 전파기이며 도메인 의미를 변환하지 않는다. 순수 ⓑ(변환 코드 결함)는 **없다**. ⓒ도 없다. 코드의 유일한 도메인 개입(`ENUM_ALIAS`/`MAT_TYP_OVERRIDE`)은 v03 입력 오류를 코드로 덮는 **부분 패치**이며, 그 미커버 영역(색/형상/고아)이 곧 ⓐ로 남은 6축 결함이다.

## 3. 두 교정 경로 비교

### 3-1. 경로 X (라이브 직접 SQL) vs 경로 Y (webadmin 엑셀 재적재)

| 항목 | 경로 X (라이브 직접 SQL) | 경로 Y (교정 엑셀 재적재) |
|------|--------------------------|---------------------------|
| 방식 | UPDATE/INSERT로 라이브 t_* 직접 교정 | v03을 권위(상품마스터/가격표)로 바로잡은 **교정 엑셀**을 load_master에 먹여 재적재 |
| webadmin 코드 수정 | 0 | 0 (코드 불변·입력 파일만 교체) |
| **P-TRUNCATE 회귀** | **🔴 위험** — 개발자 `load_master --all` 재실행 시 `TRUNCATE … CASCADE`(`:168/185/198/214/231/253`)로 교정 전부 소멸 | **🟢 안전** — 재적재가 곧 교정본을 다시 INSERT(교정이 입력 파일에 박혀 영속) |
| 근본성 | 증상 교정(라이브 행만 고침·진원 v03 그대로) | **근본 교정**(진원 v03 입력 자체를 바로잡음) |
| 멱등성 | NOT EXISTS/ON CONFLICT 가드 필요(별도 설계) | load_master가 TRUNCATE+재INSERT라 자연 멱등(전량 재적재) |
| 가격사슬 영향 | UPDATE는 component_prices siz/mat 차원 키 보존 주의(기계 삭제 금지) | 재적재가 surrogate PK를 행순으로 재발급(`issue:152-157`) → **PK 재배치 위험**(아래 3-3) |
| 적용 주체 | 우리(인간 승인 후) | 개발자(load_master 실행 시점) |
| 통제 가능성 | 즉시·세밀 | 개발자 일정·입력 파일 배치에 의존 |

### 3-2. 축별 경로 권장

| 축 | 진원 | 권장 경로 | 이유 |
|:--:|:--:|:--:|------|
| ① 기초코드 | ⓐ | **Y**(코드그룹 신설 시 SEED 반영) + X(임시) | 코드그룹 부족=시드/입력. 근본은 SEED 또는 교정 엑셀. |
| ② 사이즈 | ⓐ | **Y**(경계 결정 후) | 평면화·option화 진원=v03 시트. 단 가격사슬 의존→재키 위험(3-3). |

> **전 축 경로 Y가 근본적**(진원이 ⓐ이므로). 경로 X는 **P-TRUNCATE 재적재 시 전부 소멸**하는 임시 교정에 불과 — 개발자가 v03을 재적재하는 한 회귀한다.

### 3-3. 경로 Y의 잔존 위험 (조건부)

1. **surrogate PK 재배치 위험.** `issue`(`:152-157`)는 엑셀 **행순**으로 `CAT/SIZ/MAT/PROC` surrogate를 재발급한다. 교정 엑셀에서 행을 추가/삭제/재정렬하면 **기존 라이브 PK(SIZ_000xxx 등)와 어긋날 수 있다** → 가격사슬(component_prices의 siz_cd·mat_cd 차원 키)·다른 관계행이 깨진다. **교정 엑셀은 기존 행순·코드를 보존**하고, 신규는 말미 append, 삭제는 use_yn='N'(행 제거 아님)로 해야 안전.
2. **상품은 안전.** PRD는 `identity=True`(`:252`)로 엑셀의 PRD코드를 그대로 쓴다 → 재적재해도 PK 불변.
3. **MES_ITEM_CD·qty_unit·constraint_json은 코드가 강제 NULL/폐기**(`:261/269/270`) — 교정 엑셀로 못 채운다(코드 수정 필요·ⓑ 백로그).

## 4. 경로 Y 실현 가능 여부 + 절차

### 판정: **조건부 가능 (CONDITIONAL-YES)**

**가능 근거(코드):** load_master는 입력 엑셀을 `XLSX` 상수(`:39`)로 읽고 시트명으로 `read_sheet`(`:55`)한다. **입력 엑셀의 시트 구조(시트명·헤더 컬럼명·코드 컬럼)만 v03과 동일하게 유지하면**, 내용이 교정된 엑셀을 같은 경로에 두는 것으로 코드 수정 0으로 정확 적재가 일어난다.

**불가/조건 영역:**
- 입력 엑셀 경로가 `XLSX` **상수로 하드코딩**(`:39`)·`main`이 경로 인자 미수신(`:517`) → **우리가 임의 경로를 주입할 수 없다.** 교정 엑셀은 **개발자가 `data/raw/prdmaster_full_migration_v03_20260518.xlsx`를 교정본으로 교체**(또는 개발자가 line 39를 새 파일명으로 변경=코드 수정·개발자 영역)해야 발효.
- 시트명/헤더가 v03과 1바이트라도 다르면 `read_sheet`의 `wb[sheet]`·`dict(zip(hdr,r))`가 KeyError로 실패.
- surrogate 재키 위험(3-3) → 행순·코드 보존 필수.

### 실현 절차

1. **교정 입력 엑셀 빌드** — v03 시트 구조(시트명·헤더·코드 컬럼) 그대로, 내용만 상품마스터/가격표(권위)로 교정:
   - ④ `05_자재정보`: 색/형상/용량/구수/인쇄면 행 제거(use_yn='N' 컬럼) — **행 삭제 아님**(PK 보존). mat_typ 정정(레더 .06 등은 `MAT_TYP_OVERRIDE` 이미 커버).
   - ⑥ `01_카테고리`: 고아 노드의 `상위카테고리코드` 컬럼 채움. `11_상품별카테고리`: prd↔cat 페어를 정상 잎노드로 재연결.
   - ② `04_사이즈정보`·`13_상품별사이즈`: 평면화 정정·경계 결정분 반영.
   - ⑤ `06_공정정보`·`14_상품별자재`: 누락 공정/연결 행 **말미 append**.
   - **기존 행순·코드 보존**(surrogate 재배치 방지). 신규는 말미.
2. **검증** — 교정 엑셀을 우리 검증 환경(롤백전용 DRY-RUN)으로 적재 시 surrogate가 라이브와 동일 매핑되는지(SIZ/MAT/CAT 코드 일치) 대조.
3. **개발자 전달** — 교정 엑셀 + "이 파일을 `data/raw/`의 v03 자리에 두고 `load_master --all`" 지시. **코드 수정 0.**
4. **선검증** — 개발자 재적재 후 라이브 6축 재실측으로 교정 발효 확인.

> 우리 산출 = **교정 입력 엑셀**(v03 시트 미러·내용 권위 교정). 개발자 액션 = 파일 교체 + 재적재. 양측 모두 코드 불변.

## 5. 개발자 협업 필요 항목 (ⓑ 코드 수정 백로그)

순수 ⓑ는 없으나, **교정 엑셀로 닫을 수 없는 코드 강제 영역**은 개발자 코드 수정이 필요하다.

| # | 항목 | 코드 라인 | 필요 수정 | 사유 |
|:--:|------|----------|-----------|------|
| C-1 | **MES_ITEM_CD 전량 NULL 강제** | `:261` | 중복 해소 후 NULL 강제 해제 | 코드가 None 하드코딩 → 교정 엑셀로 못 채움 |
| C-2 | **qty_unit_typ_cd NULL** | `:269` | 시트10에 컬럼 추가 + 코드 매핑 | 원천 컬럼 부재 |

| C-4 | **입력 엑셀 경로 하드코딩** | `:39`·`:517` | `main`에 경로 인자 추가(권장) | 우리가 임의 교정본 주입 불가 |
| C-5 | **P-TRUNCATE 재실행 가드** | `:168/185/198/214/231/253` | 재실행 정책 또는 교정 보존 | 경로 X 채택 시 회귀 방지(B-1) |

## 6. 최종 판정 요약

- **load_master 입력 = v03 엑셀 하드코딩**(`:39`)·무변환 전파기. 도메인 의미 변환 없음. 유일한 코드 교정=`ENUM_ALIAS`/`MAT_TYP_OVERRIDE`(`:109-121`)뿐.
- **6축 오류 진원 = 전부 ⓐ(입력 v03).** ④자재 오염·⑥카테고리 고아 모두 v03 시트 인코딩을 무변환 전파한 결과. 순수 ⓑ 없음.
- **진원이 ⓐ이므로 경로 Y(교정 엑셀 재적재)가 전 축 근본적·P-TRUNCATE에 안전.** 경로 X(직접 SQL)는 개발자 재적재 시 소멸하는 임시 교정.
- **경로 Y 실현 = 조건부 가능(CONDITIONAL-YES).** 코드 수정 0으로 가능하되 ① 시트 구조 v03 동일 ② 행순·surrogate 코드 보존 ③ 개발자가 입력 파일 교체+재적재 — 3조건 충족 시.
- **ⓑ 백로그(개발자 협업) = 6건**(C-1~C-6). 교정 엑셀로 못 닫는 코드 강제 영역(MES/qty_unit/constraint_json/경로인자/TRUNCATE가드).
