# dbm-category-audit — MAP 시트 검증 & 출시상태 3분류 방법론 (round-24)

상품마스터의 **MAP 시트(고객 카테고리 IA)** 가 실제 운영 가능한 상품과 제대로 연결되는지를
한 판에서 검증한다. 핵심 산출은 ① MAP↔상품 매칭 매트릭스 ② 미매칭 보드 ③ **출시상태 3분류표**.

## 1. 세 축의 정의 (혼동 금지)

| 축 | 실체 | 권위 역할 |
|----|------|----------|
| **MAP 시트** | 상품마스터 2번 시트. 11개 1차 카테고리(`01 엽서`~`11 에코백`) 컬럼 × `▶︎`하위분류 헤더 × 상품명 행 × `→`별칭(타 카테고리 교차참조) | 고객사이트가 **보여줄 카테고리 IA** = "있어야 할 상품 목록"의 권위 |
| **데이터 시트** | 상품마스터 3~13번 시트 11종(디지털인쇄·스티커·책자·포토북·캘린더·디자인캘린더·실사·아크릴·문구·굿즈파우치·상품악세사리) | 실제 **등록 대상 상품 데이터**(옵션·자재·공정·가격 컬럼 보유) = "준비된 상품"의 권위 |
| **라이브 DB** | Railway `t_prd_products`·`t_cat_categories`·CPQ 옵션·`t_prc_*` 가격사슬 | **현재 적재 사실** = 출시상태 판정의 1차 근거 |

[HARD] MAP의 카테고리(01엽서/03인쇄홍보물/04포스터/05사인…)는 데이터시트(디지털인쇄/실사…)와
**1:1이 아니다**. MAP은 고객 노출 분류, 데이터시트는 생산/등록 분류. 둘을 같은 것으로 보면 전부 오판한다.

## 2. MAP 시트 파싱 규칙

- 각 컬럼(1~11) = 1차 카테고리. R1 = 카테고리명(`01 엽서`), 이후 행 = 그 카테고리의 콘텐츠.
- `▶︎` 접두 = **하위분류(섹션) 헤더** (상품 아님). 예: `▶︎엽서`, `▶︎접지카드`.
- `→` 접두 = **별칭/교차참조** (다른 카테고리에 본체가 있는 상품의 재노출). 예: `→아크릴명찰`. 별칭은 중복 상품으로 카운트하지 말고 "교차참조"로 별도 표기.
- 일반 텍스트 = **상품 엔트리**(매칭 대상). 공백/장식 셀은 무시.
- 병합셀·주석 경고는 무시(openpyxl data_only=True 권장).

## 3. 상품명 정규화 & 매칭

상품명은 표기가 시트마다 다르다. 매칭은 **정규화 후 비교**:
- 공백·괄호·특수문자(`/`,`-`) 제거, 소문자화 불가(한글) → 음절 단위 비교.
- 동의어/축약 사전을 `35_category-map/_meta/alias-dict.csv`에 축적(예: `프리미엄명함`↔데이터시트 명함 행).
- 1차 = 완전일치, 2차 = 부분일치(후보 제시), 3차 = 도메인 추론(반드시 컨펌 플래그).
- 매칭 근거에 **MAP 좌표(컬럼/행) + 데이터시트 행 + 라이브 prd_cd**를 모두 기재. 추측 금지.

## 4. 출시상태 3분류 (권위 = 상품마스터 + 인쇄상품 가격표 분석 산출물)

[HARD] 출시상태는 **라이브 DB 적재 여부로 판정하지 않는다.** 라이브는 적재가 미완(round-7 옵션
전역미적재·round-16/18 가격사슬 단절)이라, 라이브로 판정하면 *아직 안 옮긴* 정상 상품이 전부
옵션부족으로 오분류된다(실제 round-24 1차 오류: opt_items=0을 ✅ 필수조건으로 강제 → priced=Y
정상상품 40건 오판). 권위 = **원천 데이터**(상품마스터 260610 + 인쇄상품 가격표 260527)와 그 분석
산출물(가격공식·옵션·BOM: round-2/6/11/16/18/26). 핵심 질문 = **"이 상품이 가격/견적 계산에 필요한
필수 옵션·요건을 원천 데이터상 갖추고 있는가."**

| 상태 | 정의 | 판정 조건 (원천 데이터 권위) |
|------|------|----------|
| **❌ 미출시(부재)** | 상품 자체가 원천에 없음 | 상품마스터 데이터시트에 상품 행 0 |
| **🟡 옵션부족** | 가격 계산에 필수적인 옵션·가격요건이 원천상에서도 빠져 견적 불가 | 상품마스터+가격표상 가격 계산 필수 입력(필수 옵션 또는 가격결정 차원: 사이즈/수량/용지/도수 등)이 결손 → 견적 산출 불가 |
| **✅ 정상등록가능** | 가격 계산 필수옵션·요건이 원천상 완비 → 견적 산출 가능 | 상품마스터+가격표상 그 상품의 가격공식·필수 입력이 모두 정의되어 견적 계산 성립 |

★**옵션 개수 > 0은 ✅ 조건이 아니다.** 고정가형·사이즈×수량 면적매트릭스형은 옵션 없이 가격표만으로
견적되는 정상 상품이다([[dbmap-price-formula-types-authority]]·[[dbmap-compute-in-app-db-stores-lookup]]).
"옵션부족"은 옵션 개수가 적다가 아니라 **가격 계산에 필수적인 입력이 빠져 계산 자체가 불가능**한 경우다.

★**라이브 적재 현황은 출시상태 판정 근거가 아니다** — '적재 진척' 참고용일 뿐. 가격 계산 가능성 판정은
가격 도메인 전문성이 필요하므로 `dbm-price-arbiter`/`dbm-price-engine-verifier`와 협업해 상품군별
가격 계산 방식(고정가/면적매트릭스/옵션조합)과 그 필수 입력 충족 여부를 원천 기준으로 도출한다.

추가 보조 라벨(필요 시): `◆교차참조`(→별칭), `➖N/A`(섹션 헤더 등 비상품).

라이브 적재 결손은 huni-dbmap 기존 라운드(round-6 CPQ·round-16/18 가격)로 **적재 라우팅만** 표기
— 이는 출시상태(✅/🟡)와 별개의 '적재 진척' 축이다.

## 5. 미매칭 보드 (양방향)

- **MAP-only**: MAP엔 있으나 데이터시트·라이브에 없음 → 대부분 ❌미출시. 카테고리·하위분류별 집계.
- **Sheet-only**: 데이터시트/라이브엔 있으나 MAP엔 없음 → 고객 노출 누락 후보(카테고리 미귀속). 별도 표기.
- 두 보드 모두 **카운트 + 목록 + 원인 가설**을 낸다. 침묵 누락 금지.

## 6. 산출물 (`_workspace/huni-dbmap/35_category-map/`)

- `01_map-parse.md` + `map-entries.csv` — MAP 전 엔트리 정규화 추출(좌표·유형[섹션/상품/별칭]).
- `02_matching-matrix.md` + `matching.csv` — MAP 엔트리 × {데이터시트 행, 라이브 prd_cd, 매칭등급}.
- `03_release-status.md` + `release-status.csv` — 상품별 출시상태 3분류 + 근거 + 라우팅.

- `_meta/alias-dict.csv` — 동의어 사전(누적).
- 재현 스크립트는 `scripts/`(엑셀 파싱·라이브 카운트 집계).

## 7. 라이브 실측 (읽기전용)

`dbm-schema-extract` 스킬의 psql 읽기전용 툴킷 재사용(`.env.local` RAILWAY_DB_*). SELECT만.
- `t_cat_categories` 트리(현재 카테고리 노드·고아 여부 — round-22 ⑥ 진단 정합).
- `t_prd_products` × `cat_cd`/`prd_nm` (라이브 상품 존재·카테고리 귀속).
- 상품별 `option_items`·`component_prices` 행수(옵션/가격 완비 판정).
- 쿼리는 셀 단위 배치로 최소화(세션 비용).

## 8. 원칙

1. **추측 0** — 매칭·상태 모든 판정에 좌표/행/prd_cd 근거. 미지는 가설+컨펌 플래그([[dbmap-print-method-not-absolute-axis]]).
2. **별칭≠상품** — `→` 교차참조를 실상품으로 이중카운트 금지.
3. **MAP≠데이터시트 분류** — 두 축의 카테고리 체계가 다름을 전제(§1).
4. **DB 미적재** — 검증·분류·매핑 명세까지만. 실 적재/교정은 인간 승인 후 기존 라운드 위임.
5. **정직한 보드** — 미매칭 양방향 전수. over/under-report 금지.
