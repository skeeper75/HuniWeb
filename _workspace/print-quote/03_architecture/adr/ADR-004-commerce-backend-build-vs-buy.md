- 상태: **Proposed** — 사용자 결정 대기
- 작성일: 2026-05-28
- 작성자: pq-architect
- 산출 경로: `_workspace/print-quote/03_architecture/adr/ADR-004-commerce-backend-build-vs-buy.md`
- 관련: ADR-001 (옵션 C Accepted — 자체 빌더 100%), ADR-003 (데이터 레이어 Open), ADR-004-shopby-delegation-scope.md (C-minimal 잠정 1순위)
- 범위 한정: **커머스 백엔드(회원/인증 · 결제 · 카트/주문 · CS) 레이어만**. Aurora UI 셸(배제 확정), Edicus SDK(유지 확정), 견적·카탈로그·가격엔진·관리자·인쇄검수(자체 구축 확정)는 본 ADR 대상 아님.

## 1. Context — 재평가의 트리거

ADR-001은 옵션 C(자체 빌더 100% + Shopby Server API 부분 활용)를 Accepted, ADR-004-shopby-delegation-scope는 C-minimal(PG만 Shopby, 나머지 자체)을 잠정 1순위로 두었다.

사용자가 새 전제를 제기: **"Agentic Coding으로 숙련 개발자가 커머스 기능을 직접 구현하는 것이 이제 현실적이다. Shopby Server API 위임이 V1에 정말 필요한가?"**

본 ADR은 이 질문에 **냉정한 build-vs-buy**로 답한다. 핵심 방법론은 각 capability를 **코드로 풀리는 문제**와 **코드가 아닌 문제(계약·인증기관·심사·법적 의무)**로 분리하는 것이다.

### 1.1 결정적 구분 — "개발자가 짤 수 있다" ≠ "런칭에 필요한 모든 것이 코드다"

Agentic Coding이 레버리지를 주는 것은 **애플리케이션 로직(코드)**이다. 다음은 코드가 아니며, 아무리 빨리 짜도 외부 일정·계약에 종속된다:

| 비코드 차단요소 | 성격 | 통제 가능성 |
|---|---|---|
| PG사 가맹 계약 (KG이니시스 등) | 사업자 심사·계약 | 외부 (PG사 심사 기간) |
| 카드사 심사 (PG 경유 간접) | 카드사 가맹점 심사 | 외부 |
| 에스크로/구매안전 서비스 의무 (전자상거래법 §13) | 법적 의무 + 에스크로 사업자 계약 | 외부 |
| 현금영수증 발급 의무 (소득세법/법인세법) | 법적 의무 + 국세청 연동 | 외부 (ecount 보유) |
| 본인확인기관(CI/DI) 유료 계약 (NICE/KCB) | 본인확인기관 계약 | 외부 — **단, 본 사업은 불필요(§2.1)** |
| 휴대폰 인증 SaaS 계약 (PASS 등) | 인증사 계약 | 외부 (경량) |
| 소셜로그인 심사 (네이버/카카오 검수) | 플랫폼 앱 검수 | 외부 (경량) |

→ 즉 **회원/카트/주문/CS는 코드 영역이라 Agentic Coding 레버리지가 크지만, 결제는 코드 비율이 작고 비코드(계약·심사) 비율이 크다.** 이 비대칭이 본 ADR의 결론을 결정한다.

### 1.2 용어 정정 — "Neon PG"는 결제 게이트웨이가 아니다

선행 ADR 다이어그램의 `Neon PG`는 **Neon(서버리스 Postgres) 자체 DB**를 가리키는 표기이며 결제 게이트웨이가 아니다. As-Is 실제 PG는 **KG이니시스(Inicis)** + 에스크로(이니시스 보유)다 (`order-flow.md` §1.2, `glossary.md` L166-168). 따라서 "자체 PG 연동 경험치"는 **이니시스 직연동** 기준으로 평가한다 — 후니프린팅은 As-Is(WordPress)에서 이미 이니시스를 운영 중이므로 PG 자체에 대한 경험·계약은 보유.

## 2. Capability Bucket별 평가

각 버킷: (a) Shopby 위임 가능 여부, (b) 자체 구축 난이도 — **코드 / 비코드 분리**, (c) 인쇄 도메인 fit, (d) 권고.

### Bucket 1 — 회원/인증

| 하위 요소 | Shopby 위임 | 자체 구축 (코드) | 자체 구축 (비코드) | 인쇄 fit |
|---|---|---|---|---|
| 가입/로그인/세션 | 가능 | NextAuth v5 — **낮음** (표준) | 없음 | 무관 |
| 소셜로그인 (네이버/카카오) | 가능 | OAuth 연동 — **낮음** | 플랫폼 앱 검수 (경량, 외부) | 무관 |
| 본인인증 CI/DI | 가능 | 본인확인기관 SDK 연동 — 中 | **본인확인기관 유료 계약 (외부)** | 무관 |
| 휴면계정 분리보관 (정보통신망법) | 가능 | cron + 분리 테이블 — **낮음** | 없음 (법규 준수는 로직) | 무관 |
| 약관/개인정보 동의 이력 | 가능 | `agreement_logs` 테이블 — **낮음** | 없음 | 무관 |

**핵심 사실 (ADR-004-shopby §1.3 사용자 확인):** 본인인증 **CI/DI 불필요** — 휴대폰 인증 수준이면 충분. 이로써 한국 커머스에서 Shopby 위임의 가장 큰 가치(본인확인기관 계약 대행)가 **본 사업에서 소멸**한다.

- **인쇄 도메인 fit:** 회원 모델은 도메인 중립. Shopby 위임이든 자체든 인쇄 견적에 영향 없음. 단, B2B 거래처별 청구·견적서 발송(`order-flow.md` §회계)은 Shopby 표준 회원 모델보다 자체 모델이 fit 높음.
- **권고:** **자체 구축.** 코드 난이도 낮음(NextAuth v5 — edicus.man Firebase Auth → NextAuth 마이그레이션 흐름 보유), 비코드 차단요소는 휴대폰 인증 SaaS 계약(경량)뿐. CI/DI 불필요로 Shopby 위임 가치 소멸.

### Bucket 2 — 결제(PG)

| 하위 요소 | Shopby 위임 | 자체 구축 (코드) | 자체 구축 (비코드) | 인쇄 fit |
|---|---|---|---|---|
| 카드/가상계좌/계좌이체 승인 | Shopby PG channel | 이니시스 SDK 연동 — 中 (이미 As-Is 보유) | **이니시스 가맹 계약·심사 (외부)** — 단 As-Is 계약 승계 가능 | 무관 |
| 에스크로/구매안전 | Shopby 위탁 | 이니시스 에스크로 모듈 — 中 | **에스크로 의무(전자상거래법) + 에스크로 사업자 계약** | 무관 |
| 간편결제 (네이버/카카오/토스) | Shopby 통합 채널 (1계약) | 각 페이 직계약·SDK — 中×N | **각 페이사 가맹 심사 N건 (외부, ~4주)** | 무관 |
| 현금영수증 | Shopby/PG | ecount 연동 — 낮음 | 국세청 연동 (ecount 보유) | 무관 |

**가장 냉정해야 할 버킷.** 결제는 **코드 비율이 작고 비코드(계약·심사) 비율이 크다.** Agentic Coding으로 이니시스 연동 코드는 수일 내 짤 수 있으나:
- **간편결제 다중 직계약**은 페이사별 가맹 심사가 각각 필요 — 코드가 아니라 심사 일정에 종속.
- 에스크로/구매안전은 **전자상거래법 의무**이고 에스크로 사업자(이니시스 등) 계약 종속.
- Shopby PG channel의 유일한 실질 가치 = **다중 PG·간편결제를 1계약·1통합으로 묶어주는 것** → 비코드 부담을 줄여줌.

- **인쇄 도메인 fit:** 결제 자체는 도메인 중립. 단 **동적 견적가 결제**(어드민 사전등록 SKU 없이 클라이언트 산출 단가로 결제)는 Shopby 위임 시 deal-breaker(ADR-001 §7, ADR-003 Q1) — Shopby가 어드민 가격을 단일 진실로 재계산하면 견적가 주입 불가. **이니시스 직연동은 결제 금액을 가맹점이 지정**하므로 동적 견적가 결제에 오히려 fit 높음.
- **권고:** **이니시스 직연동(자체) + 간편결제는 단계적.** As-Is 이니시스 계약 승계로 비코드 차단요소 대폭 축소. 동적 견적가 결제 무결성이 이니시스 직연동에서 자연스럽게 해결됨. 간편결제는 V1에 이니시스 통합형 간편결제(이니시스가 네이버/카카오/토스 중계) 우선 검토 → 직계약 N건의 비코드 부담 회피. **단 Open Question OQ-2 검증 필요.**

### Bucket 3 — 카트/주문/주문내역

| 하위 요소 | Shopby 위임 | 자체 구축 (코드) | 자체 구축 (비코드) | 인쇄 fit |
|---|---|---|---|---|
| 카트 (다건 묶음) | 가능 | 자체 cart 도메인 — **낮음** | 없음 | **Shopby 모델 fit 낮음** |
| 주문 생성·상태머신 | 가능 (표준 상태) | 자체 7-상태 머신 — 中 | 없음 | **Shopby 표준과 충돌** |
| 주문내역/조회 | 가능 | 자체 조회 — **낮음** | 없음 | 자체 fit 높음 |
| 파일 부속 상태머신 | **불가** (인쇄 특수) | 자체 — 中 | 없음 | **Shopby 표현 불가** |

**Agentic Coding 레버리지가 가장 큰 영역이자, 인쇄 도메인 특수성 때문에 Shopby fit이 가장 떨어지는 영역.**

후니의 주문 모델은 (`order-flow.md` §1):
- 7개 정식 주문상태 + **파일번호 단위 부속 상태머신 3종**(배송상품/파일업로드/Edicus 편집기) — 이중 상태머신.
- 신용결제(Case 5)는 미입금 우회 등 인쇄업 특수 결제 흐름.
- 수량구간가·옵션조합·작업파일·교정 재업로드 — Shopby 표준 주문 모델에 없는 1급 개념.

→ Shopby 표준 카트/주문에 이를 우겨넣으면 ADR-001 옵션 A가 빠진 함정(가격 무결성·옵션 모델 충돌)을 반복.

- **인쇄 도메인 fit:** Shopby 표준 = **낮음**. 자체 = **최상** (베이스라인 Order 7테이블 + 파일 상태머신이 화면에 1:1 매핑).
- **권고:** **자체 구축.** 코드 난이도 中, 비코드 차단요소 0. Agentic Coding 레버리지 최대. 인쇄 도메인 fit이 자체 구축을 강제.

### Bucket 4 — CS(문의/리뷰/쿠폰/적립금)

| 하위 요소 | Shopby 위임 | 자체 구축 (코드) | 자체 구축 (비코드) | 인쇄 fit |
|---|---|---|---|---|
| 상품문의/1:1문의 | 가능 | 자체 — **낮음** | 없음 | 자체 fit 높음 |
| 리뷰 | 가능 | 자체 — **낮음** | 없음 | 중립 |
| 쿠폰/할인 | 가능 | 자체 — 中 (할인 룰 엔진) | 없음 | **자체 가격엔진과 통합 필요** |
| 적립금 | 가능 | 자체 — 中 | 없음 | 포인트 전액결제(IA No.30) 특수 |

- **인쇄 도메인 fit:** 쿠폰/적립금은 **자체 가격엔진**(이미 자체 구축 확정)과 통합돼야 하므로 Shopby에 분리 위임 시 가격 산출 경계가 갈라짐. 포인트 전액결제(`huni-ia-master.md` No.30)는 표준 커머스와 다른 흐름.
- **권고:** **자체 구축.** 코드 난이도 中, 비코드 0. 쿠폰/적립금이 자체 가격엔진에 강결합되므로 위임 시 오히려 동기화 부담 증가.

## 3. 시나리오 비교 (A / B / C)

| | A. Shopby Server API 유지 | B. 하이브리드 (결제·PG만 위임) | C. 100% 자체구축 |
|---|---|---|---|
| 회원/인증 | Shopby | 자체 | 자체 |
| 결제 PG | Shopby PG channel | **Shopby PG channel** (다중 통합) | 이니시스 직연동 + 페이 직계약 N건 |
| 카트/주문 | Shopby | 자체 | 자체 |
| CS | Shopby | 자체 | 자체 |
| 코드 부담 | 동기화 코드 (중복) | 중 (도메인 코드, PG는 위임) | 高 (PG 통합 어댑터 포함) |
| 비코드 차단요소 | Shopby 계약 | Shopby PG 계약 1건 | **PG 다중 직계약 N건 + 에스크로** |

### 시나리오 A — Shopby Server API 유지 (커머스 백엔드 전면 위임)

- **정성 리스크:** 이중 어드민(Shopby 콘솔 + 자체), members 캐시·SSO·webhook polling 동기화 코드, IP 화이트리스트 CI/CD 종속, Shopby 5xx 장애 시 로그인·결제 정지, Shopby Enterprise 월정액. **동적 견적가 결제 deal-breaker**(ADR-001 §7)가 미해결이면 주문/결제 위임 자체가 불가.
- **런칭 차단요소:** Shopby Enterprise 계약, IP 화이트리스트 등록, 동적 가격 결제 지원 검증(미검증 시 차단).
- **평가:** CI/DI 불필요 + ecount 보유로 Shopby 핵심 위탁 가치 2개(본인확인·세금계산서) 소멸. **부담 > 가치.** 비추.

### 시나리오 B — 하이브리드 (결제·PG channel만 위임, 회원/카트/주문/CS 자체)

- **정성 리스크:** PG channel만 Shopby에 묶임(완전 독립 아님). 게스트 결제 시 Shopby memberNo 강제 여부 검증 필요(OQ-1). 동적 견적가를 PG channel이 수용하는지 검증 필요.
- **런칭 차단요소:** Shopby PG channel 계약(또는 이니시스 직연동으로 대체 가능), OQ-1/OQ-2 검증.
- **평가:** 코드 영역(회원/카트/주문/CS)은 Agentic Coding으로 자체화 → 인쇄 도메인 fit 확보. 비코드 부담이 큰 결제만 위임 → **다중 PG·간편결제 통합의 비코드 부담을 1계약으로 흡수**. 부담/가치 비율 우수.

### 시나리오 C — 100% 자체구축 (이니시스 직연동 + 페이 직계약)

- **정성 리스크:** 간편결제 페이사별 직계약 **N건 심사**(비코드, 외부 일정 종속, ~4주), 페이별 webhook 형식 상이 → 통합 어댑터 자체 구축, 단일 PG 장애 fallback 자체 구현, PG 수수료 협상력 분산.
- **런칭 차단요소:** **간편결제 직계약 N건의 심사 일정** — Agentic Coding으로 해결 불가한 비코드 차단요소. 에스크로 사업자 계약. V1 런칭 일정의 최대 리스크.
- **평가:** 코드는 모두 가능. 그러나 **결제 비코드 차단요소가 V1 런칭을 직접 지연**시킴. As-Is 이니시스 계약 승계 시 카드/가상계좌는 즉시 가능하나, 간편결제 다중 직계약이 병목. Shopby 완전 독립 의지가 명확할 때만.

## 4. 권고

### 권고: **시나리오 B — 하이브리드 (회원/카트/주문/CS 자체구축, 결제는 위임 또는 이니시스 직연동)**

단, 결제 위임처는 **Shopby PG channel과 이니시스 직연동 2안 병기**하여 OQ-2 검증 후 확정한다.

| 결제 분기 | 채택 조건 |
|---|---|
| B-1: Shopby PG channel | 동적 견적가 결제 지원(OQ-2-a) + 게스트 결제 가능(OQ-1) 확인 시 — 다중 간편결제 1계약 흡수 |
| B-2: 이니시스 직연동 | Shopby PG channel이 동적 견적가/게스트를 막을 때 — As-Is 계약 승계, 간편결제는 이니시스 통합형 우선 |

### 핵심 근거 3개

1. **코드/비코드 비대칭이 결론을 결정한다.** 회원·카트·주문·CS는 코드 영역이라 Agentic Coding 레버리지가 크고 비코드 차단요소가 0에 가깝다 → 자체구축이 합리적이고 인쇄 도메인 fit이 자체구축을 강제(이중 상태머신·동적 견적가). 반면 결제는 코드 비율이 작고 비코드(다중 PG 계약·심사·에스크로 의무) 비율이 커서, 100% 자체구축(시나리오 C)은 간편결제 직계약 N건의 심사가 V1 런칭을 직접 차단한다.

2. **Shopby 전면 위임(A)의 핵심 가치가 본 사업에서 소멸했다.** CI/DI 불필요(휴대폰 인증 충분) + ecount 보유(전자세금계산서·국세청 연동)로 한국 커머스에서 Shopby 위임의 2대 가치가 사라졌다. 남는 가치는 다중 PG 통합 1건뿐 → 그것만 위임하는 B가 최적.

3. **인쇄 도메인 fit이 카트/주문 자체구축을 강제한다.** 7-상태 + 파일 부속 상태머신 3종 이중 모델, 수량구간가·옵션조합·교정 재업로드는 Shopby 표준 주문 모델에 1급 표현이 없다 → 위임 시 ADR-001 옵션 A가 빠진 가격 무결성·옵션 충돌 함정 반복. 자체 베이스라인 Order 7테이블이 1:1 매핑된다.

## 5. 자체구축 시 코드가 아닌 차단요소 (런칭 블로커)

Agentic Coding으로 해결 불가, 외부 계약·심사·법적 일정에 종속:

1. **간편결제 직계약 심사 N건** (네이버페이/카카오페이/토스 각각 가맹 심사) — 시나리오 C의 최대 병목. B에서는 Shopby channel 또는 이니시스 통합형으로 회피.
2. **에스크로/구매안전 서비스 의무** (전자상거래법 §13, 선결제 통신판매) — 에스크로 사업자(이니시스 등) 계약 필요. As-Is에 에스크로 운영 중이므로 승계 검토.
3. **이니시스 가맹 계약·심사** — As-Is 계약 승계 시 회피 가능(검증 필요).
4. **휴대폰 인증 SaaS 계약** (PASS/NICE pin 등) — 경량이나 외부 계약 필요.
5. **소셜로그인 앱 검수** (네이버/카카오 개발자 검수) — 경량, 외부 일정.
6. **현금영수증/세금계산서 국세청 연동** — ecount 보유로 해결됨(비코드 차단요소 아님).

### 가장 큰 미해결 (BLOCKER)

**OQ-2 (동적 견적가 결제)** — 이 답이 결제 위임처(B-1 Shopby vs B-2 이니시스)를 가른다. Shopby가 동적 견적가를 막으면 결제도 자체(이니시스 직연동)로 가야 하고, 그 경우 시나리오 B는 사실상 "결제만 이니시스 직연동, 나머지 자체"가 되어 **ADR-004-shopby-delegation-scope의 C-extreme에 수렴**한다. 본 ADR과 ADR-004-shopby는 동일 결론(자체구축 우위 + 결제만 별도 취급)을 코드/비코드 렌즈로 재확인한다.

## 7. ADR-004-shopby-delegation-scope와의 관계

본 ADR(004b)은 기존 ADR-004-shopby-delegation-scope를 **대체하지 않고 보강**한다. 매핑:

| 본 ADR 시나리오 | shopby-delegation-scope 옵션 | 결론 일치 |
|---|---|---|
| A (전면 위임) | C-current | 둘 다 비추 (부담 > 가치) |
| B (결제만 위임) | C-minimal | 둘 다 1순위 |
| C (100% 자체) | C-extreme | 둘 다 2순위 (결제 비코드 차단요소) |

차이: 본 ADR은 **코드 vs 비코드(계약·심사) 분리**를 명시하여 "Agentic Coding이 결제 자체구축을 가능케 하는가?"라는 사용자 질문에 직접 답한다. 답: **코드는 가능하나 결제의 비코드 차단요소(다중 PG 직계약 심사)가 V1 런칭을 차단하므로 결제만 위임하는 B가 합리적.**
