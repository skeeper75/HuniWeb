## 1. 요약

| 지표 | 값 |
|------|-----|
| 총 상품 수 | 8 |
| 총 disable 규칙 (소재→후가공 비활성) | **209** |
| 총 후가공 옵션 | 80 (필수 63, 숨김 0) |
| 도수-제본 매핑 | 22 |
| 수량/페이지 제약 | 8 |
| 숨김 소재 | 0 (현재 캡처 범위 내) |
| 숨김 규격 | 0 |

### 핵심 발견

1. **제약조건은 `req_*/rst_*` 필드가 아니라 `pdt_disable_pcs_info` 테이블로 구현됨**
   - 기존 스크립트가 찾던 `sizeinfo.sizelist[].req_paper` 같은 구조는 구형 API(`product.js`)에서만 존재
   - 새 위젯(`book2025_item`, `vDigital_item`)은 `product_data` 기반 완전히 다른 구조 사용

2. **제약조건의 핵심 메커니즘은 "소재 선택 시 특정 후가공 비활성화"**
   - `pdt_disable_pcs_info`: MTRL_CD + PCS_CD 조합으로 비활성 규칙 정의
   - 209개 규칙 중 206개가 책자(book) 상품군에 집중

3. **수량/페이지 제약은 상품별 1건씩 정적으로 정의**
   - 서버 사이드 캐스케이드 없이 클라이언트에서 직접 검증

## 5. 상품군별 제약조건 패턴 차이

### 5.1 책자 (Book) -- 6개 상품

- **제약 메커니즘:** `pdt_disable_pcs_info` 중심 (총 206개 규칙)
- **핵심 패턴:** 내지 소재(RXOMO, RXWMO, RXNPW 등) 선택 시 코팅/박/미싱/접지/오시 등 후가공 비활성
- **도수-제본 매핑:** 3~5개 (양면/단면 도수에 따라 사용 가능한 제본 방식 결정)
- **수량 제약:** 
  - 윤전: minPage 2~10, maxPage 130~300 (대량 인쇄)
  - 토너: minPage 2~10, maxPage 20~200 (소량)
  - 스테플러/실제본: step=2 (양면이므로 짝수 페이지만)
- **규격:** 3~8종 (A4/B5 세로/가로, A5 등)

### 5.2 굿즈 (Goods) -- 1개 상품 (마이크 네임택)

- **제약 메커니즘:** disable 규칙 0개 (후가공 비활성 없음)
- **후가공 옵션:** 11개로 가장 많음 (다양한 마감 옵션)
- **수량 제약:** 1페이지 고정 (minPage=1, maxPage=1)
- **규격:** 4종
- **특징:** 제약이 거의 없는 단순 구조

### 5.3 아크릴 (Acrylic) -- 1개 상품 (명찰)

- **제약 메커니즘:** disable 3개 (RXIGC075 소재에서 COT/MIS/SCO 비활성)
- **후가공 옵션:** 4개 (모두 필수)
- **수량 제약:** 1페이지 고정
- **규격:** 2종
- **특징:** 소재 1종에 대해서만 제약 존재 (아크릴 특성상)

## 6. WowPress 12,892개 규칙과의 비교

| 비교 항목 | RedPrinting (신규 위젯) | WowPress |
|-----------|----------------------|----------|
| 제약조건 구현 방식 | `pdt_disable_pcs_info` 테이블 | `req_*/rst_*` 필드 (옵션 아이템별 인라인) |
| 제약 방향 | 소재 → 후가공 (단방향) | 양방향 (size↔paper, paper↔color 등) |
| 캡처 범위 | 8개 신규 위젯 상품 | 479개 전체 상품 |
| 총 규칙 수 | 209 | 12,892 |
| 상품당 평균 | 26.1 | 26.9 |
| 규칙 유형 | disable (비활성) | require + restrict (필요 + 제한) |
| 데이터 위치 | `product_data` (별도 테이블) | `product_option.option.*info.*list[]` (인라인) |
| 캐스케이드 트리거 | 클라이언트 Pinia 스토어 | 서버 API 재호출 |

### 핵심 차이점

1. **구조적 단순화:** RedPrinting 새 위젯은 "소재→후가공" 단일 방향 제약만 사용. WowPress의 양방향 `req_*/rst_*` 패턴(size↔paper↔color↔prsjob↔awkjob)보다 크게 단순화됨.

2. **캐스케이드 없음:** RedPrinting 신규 위젯은 옵션 변경 시 서버 재호출 없이 클라이언트에서 `pdt_disable_pcs_info` 테이블을 참조하여 UI를 갱신. WowPress는 옵션 변경마다 서버에 재요청하여 새로운 `req_*/rst_*` 데이터를 받아옴.

3. **상품당 규칙 밀도:** 상품당 평균 규칙 수는 비슷하나(~26), RedPrinting은 규칙 유형이 단순(MTRL→PCS 비활성)한 반면 WowPress는 5~6가지 제약 유형이 교차함.

## 7. 제약조건 데이터 구조 상세

### 7.1 `pdt_disable_pcs_info` (핵심 캐스케이드 제약)

```
{
  "MTRL_CD": "RXOMO080",   // 소재 코드 (내지 미색모조 80g)
  "PCS_CD": "COT_DFT",     // 후가공 코드 (코팅)
  "PCS_DTL_CD": null,      // 후가공 세부 코드 (없으면 전체 비활성)
  "NOTE": null
```

의미: "RXOMO080 소재가 선택된 경우 COT_DFT 후가공을 비활성화"

```
  "MAX_INN_PAGE": 130,     // 최대 내지 페이지
  "STEP_INN_PAGE": 1,      // 페이지 스텝 (2=짝수만)
  "MAX_THCK": 1000.00,     // 최대 두께 (mm)
  "INN_MAX_WGT": null,     // 내지 최대 평량
  "COV_MIN_WGT": null      // 표지 최소 평량
```

## 8. 레거시 상품 참고 사항

다음 상품군은 **새 위젯(Vue3+Shadow DOM)이 아닌 기존 `product.js`** 를 사용:
- 명함 (BC*)
- 전단지/리플렛 (LF*)
- 스티커 (ST*)
- 봉투 (BN*)

이들은 구형 `req_*/rst_*` 인라인 제약조건 구조를 사용하며, `cascade-capture.cjs`로는 캡처 불가. 기존 캡처 파일(`BCSPDFT_capture.json`, `BNSTDFT_capture.json` 등)은 별도 분석 필요.

## 9. 후니프린팅 설계 시사점

1. **제약 엔진 단순화 가능:** RedPrinting 신규 위젯의 제약조건은 "소재→후가공 비활성" 단일 테이블로 충분. WowPress처럼 양방향 복잡 그래프가 필요 없음.

2. **클라이언트 사이드 처리:** 제약조건 데이터가 초기 로드 시 한 번에 전달되므로, 옵션 변경마다 서버 재호출 불필요. Pinia/Zustand 스토어에서 로컬 필터링.

3. **수량 제약은 정적:** `pdt_prn_cnt_info`는 상품 정의 시 고정. 동적 수량 제약(옵션에 따라 수량 범위 변경)은 없음.

4. **데이터 모델링:** 후니프린팅 DB에 `product_disable_pcs` 테이블(material_code, pcs_code, pcs_detail_code) 하나면 핵심 캐스케이드 구현 가능.
