# vessel-mat-type-relabel (MAT_TYPE 오라벨 교정) — vessel-level 분류축 정비

> rpm-vessel-designer. `MAT_TYPE.09 "파우치"·.10 "악세사리"`가 *자재유형* 코드 그릇을 *상품군명* 버킷으로 만든 vessel-level 오라벨. 자재 분류축을 자재유형으로 회복하는 그릇 정비.
> 권위 = 라이브 read-only 실측(2026-06-17·본 세션 직접 측정). design ≠ apply.
> **버전:** v2.0 (GS 신축). ★ open decision 중대 — 행 영향 큼·신중.

## 0. 한 줄 평결
**이것은 "분류축 의미"의 vessel-level 결함이다 — 그릇(코드 그룹)은 있으나 *코드값 정의*가 축을 오염**(자재유형 축에 상품군명이 섞임). 그러나 **교정 방식이 미묘**: .09/.10 코드를 단순 재명명하면 안 된다 — 그 코드에 매달린 69+43=112행이 *애초에 자재가 아닌 것*(형상/색/구수/인쇄면)이라, 코드 교정보다 **행을 올바른 축으로 축이동(B-3 data)하고 남는 진짜 자재만 정상 MAT_TYPE으로 재분류**하는 것이 정답. 즉 vessel 측 산출 = **(a) .09/.10이 자재유형이 아님을 *판정*하고 (b) 진짜 자재를 받을 정상 코드(.05 특수/.06 가죽 등)가 충분한지 확인 + 부족 시 신소재 코드행 추가 후보 제시**까지. 코드 *재명명/폐기* 자체는 행 의존(112행 + 80/82 상품 BOM) 때문에 **인간 승인·dbmap B-3 선후 조율 필수 = open decision**.

## 1. search-before-mint (라이브 실측 — 결정적)
`t_cod_base_codes` MAT_TYPE 라이브 14종(실측):
```
.01 디지털인쇄용지  .02 제본부자재   .03 아크릴      .04 금속
.05 특수소재        .06 가죽         .07 부자재      .08 실사소재
.09 파우치 ★        .10 악세사리 ★   .11 스티커용지  .12 사입자재
.13 합판 스티커용지  .14 합판 봉투용지
```
| 코드 | 축 정합성 | 판정 |
|---|---|---|
| .01~.08·.11~.14 | 소재/용지/금속/아크릴 = **자재유형** | ✅ 정상 분류축 |
| **.09 파우치** | 상품군명(파우치는 상품이지 자재 아님) — 69행이 형상/색/인쇄면/구수 | 🔴 오라벨 |
| **.10 악세사리** | 상품군명(악세사리는 상품군) — 43행 동형 | 🔴 오라벨 |

**라이브 .09 행 실증(GS 재측):** `MAT_000263 원형 90mm`·`MAT_000274 사각 57mm`(형상/사이즈)·`MAT_000271 단면`/`MAT_000272 양면`(인쇄면)·`MAT_000277 2구`(구수) — **자재가 하나도 없다**. .09는 "파우치 상품의 옵션값"을 자재로 적재한 버킷. → 코드 자체가 자재유형 축을 깨뜨림(분류축에 상품군명 + 그 안에 비자재 값).

**search-before-mint 결론:** 정상 자재유형 코드(.05 특수소재·.06 가죽 등)는 이미 풍부 → .09/.10에 들어있어야 할 *진짜* 자재(파우치 본체 = 레더/캔버스/타이벡/메쉬/린넨)는 round-22가 이미 .06/.05 등으로 41행 적재함(GS 실측). 즉 **진짜 자재를 받을 정상 그릇은 충분** — .09/.10은 *비울 수 있는* 오라벨 버킷이다. 신규 코드 mint는 신소재(우드/코르크/규조토·data-gap)에만 필요(GPM-4·ddl-proposer).

## 2. 그릇 설계 (vessel 측 — 분류축 회복 판정 + 정상 코드 충분성)
### 2.1 .09/.10 = 자재유형 아님 판정 (vessel-level 결론)
- MAT_TYPE 축은 *자재 물성/소재* 분류여야 한다(인쇄용지/아크릴/금속/특수/가죽). 상품군명(파우치/악세사리)은 **카테고리 축**(`t_cat_categories`) 소관 — MAT_TYPE에 있으면 안 됨.
- → vessel 정비 목표: .09/.10을 **자재유형 축에서 제거**(use_yn='N') 하되, **선결로 그 112행을 올바른 축으로 축이동**(B-3 data): 형상→`t_siz_sizes`·색→CPQ option·구수→bundle·인쇄면→print_side. 행이 다 빠진 뒤 코드 비활성.

### 2.2 정상 자재 코드 충분성 + 신소재 후보 (사다리)
- 진짜 파우치/악세사리 *본체 자재*(레더/캔버스/메쉬/린넨)는 이미 .05/.06에 적재(round-22 41행) → 재분류 그릇 충분.
- **신소재**(우드/코르크/규조토/벨벳/세라믹·data-gap)는 .05 특수소재 흡수 vs 신규 MAT_TYPE 코드행 — **search-before-mint: .05 특수소재로 흡수 가능하면 신규 코드 mint 0** 권장. 가격/옵션이 소재별 분기 필요하면 신소재 코드행(`MAT_TYPE.15 목재` 등) 후보 = ddl-proposer. **단 이는 data-gap 트랙(GPM-4)이지 본 vessel 산출 아님** — 여기선 "정상 코드 충분·신소재는 .05 흡수 우선" 판정까지.

### 2.3 사다리 평결
- 신규 테이블/컬럼 **0**. 코드행 변경 = `.09/.10 use_yn='N'`(폐기 아닌 비활성) — **단 행 선이동 후**. 신소재는 .05 흡수(코드 mint 0) 우선.

## 3. 정규화 / 영향 (★중대 — 신중)
- **무손실:** 112행이 *비자재*라 자재 축에서 빼는 것이 무손실(오히려 손실 복구). 형상/색/구수는 목적지 축(siz/option/bundle)이 무손실 수용.
- **무중복:** .09/.10 제거 시 자재유형 축이 순수 자재만 → 분류축 함수종속 정상화(MAT_TYPE → 자재물성).
- **★영향(큼):**
  - **112 자재행 + 80/82 상품 BOM 의존**(round-22): .09/.10 행을 `t_prd_product_materials`가 링크 중 → 코드 비활성 *전에* 반드시 (i) 행을 목적지 축으로 이동 (ii) 상품 BOM 재배선 (iii) 자재행 use_yn='N' = **마지막**. 선폐기 시 80상품 본체 BOM 전손(round-22 [HARD] FK 위상).
  - 정상 .05/.06 적재분(41행)은 무영향.
- **롤백:** 코드 use_yn 'N'→'Y' 복구 가능. 단 행 축이동(B-3)은 별 롤백(dbmap undo.sh).
- **적용 순서(HARD):** ① 목적지 그릇 선행(siz/option/bundle = V-3·기존) → ② B-3 축이동(data·dbmap) → ③ 상품 BOM 재배선(data) → ④ .09/.10 행 잔여 0 확인 → ⑤ .09/.10 use_yn='N'(vessel). **본 vessel(⑤)은 ①~④ 완료가 전제.**

## 4. 판정
- MAT_TYPE 오라벨: **vessel-level 분류축 결함 확정**(코드값이 축 의미 오염) → 교정 그릇 = `.09/.10 use_yn='N'` (신규 0) + 신소재 .05 흡수. **단 행/BOM 의존으로 dbmap B-3와 강결합 = 단독 적용 불가·open decision.**

## 5. DDL 참조
- 코드 비활성(use_yn UPDATE) = ddl-proposer 코드 패턴(경량). 신소재 코드행(채택 시) = ddl-proposer. 행 축이동 SQL = dbmap round-22 B-3(vessel 트랙 밖).

## 6. open decision (★중대 — 행 영향 큼·신중)
1. **.09/.10 재명명 vs 비활성 vs 보존:** 재명명("파우치"→자재유형)은 의미 부재(파우치는 자재유형 아님) → **비활성(use_yn='N') 권장**, 단 112행 전이동 후. 폐기(DELETE)는 금지(감사/롤백·FK).
2. **신소재 코드:** .05 특수소재 흡수 우선(mint 0). 소재별 가격/옵션 분기 필요 입증 시만 신규 MAT_TYPE 코드행(ddl-proposer·data-gap 트랙).
3. **B-3 선후 조율(HARD):** 본 vessel은 dbmap B-3(축이동)·GPM-4(본체자재 BOM)·상품 BOM 재배선이 *선행*해야 안전. RP-Meta 단독 적용 불가 — dbmap 트랙과 인간이 순서 조율.
4. 실 적용 = 인간 승인(특히 80상품 BOM 영향으로 가장 신중).
