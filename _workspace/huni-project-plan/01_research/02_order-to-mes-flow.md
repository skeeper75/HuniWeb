# 02. 주문 → 생산(MES) → 포장 → 배송 전체 흐름

> 후니프린팅 리뉴얼 일정관리 IA 문서용. **주문형 생산(made-to-order)** 구조.
> 전 흐름을 **사이트 내부 기능** vs **외부 연동**으로 분리 정리.
>
> **원천:**
> - `docs/huni/후니프린팅_주문프로세스_20251001.pdf` (12p) — 주문 프로세스·자동알림·인쇄타입·파일명 Rename·발주·파일 FLOW
> - `docs/huni/후니프린팅_공정관리_시행초안_20260210.pdf` (2p) — 공정플로우(MES 측) 17 케이스
> - `docs/huni/table-spec_260610.html` — 라이브 t_* 35테이블 설계도
> - 라이브 DB 읽기전용 SELECT 실측 (Railway `railway` DB)
>
> **표기 규칙:** `실측:` = 라이브 DB SELECT 결과 / `추정:` = 문서·도메인 기반 추정.

## 0. 핵심 결론 (한눈에)

**실측: 라이브 DB(35 t_* 테이블)는 "상품·가격 마스터" 만 존재한다. 주문·결제·파일·장바구니·배송·MES 런타임 테이블은 0개다.**

- 주문/결제/파일/배송/장바구니 관련 테이블 검색 결과: **부재** (`information_schema.tables` 전수 조회)
- MES 연동 컬럼은 **딱 하나**: `t_prd_products.MES_ITEM_CD` (상품→MES 품목코드 브릿지)
- 따라서 "주문 → 생산정보 전달 → MES" 의 **주문·전달 레이어 전체가 to-be(신규 구축)** 다. 마스터(상품/옵션/가격)만 적재돼 있다.
- 원천 PDF는 현재 **수동 운영(웹하드+EXCEL+생산관리 프로그램 분리)** 의 Pain point를 그리며, 리뉴얼은 "Front 페이지에서 가상계좌/자동알림/장바구니/파일업로드/편집기/추가배송비로 해결" + "MES의 상품과 사용자 상품을 매칭하여 파일네임 일관화" 를 목표로 한다 (주문프로세스 PDF p1 하단 Pain point 종합).

## 1. 전체 흐름도 (텍스트 단계)

각 단계에 **[내부]** = 후니 리뉴얼 사이트 내부 기능 / **[외부]** = 외부 서비스 연동 라벨.

```
 ① 상품선택                         [내부]   카탈로그/상품 상세 (t_prd_products)
   │
 ② 옵션선택 (위젯)                  [내부]   CPQ 옵션 위젯 (option_groups/options/items)
   │
 ③ 가격확인                         [내부]   가격엔진 evaluate_price (t_prc_*)
   │
 ④ 파일업로드 / 에디터 디자인        [내부+외부]
   │   ├─ 파일업로드 상품            [내부 UI] + [외부 S3] 원본파일 업로드
   │   └─ 편집기(에디터) 상품        [외부 에디쿠스/Edicus] 디자인→랜더링 + [외부 S3] 결과 저장
   │
 ⑤ 장바구니                         [내부]   (현재 부재 → 리뉴얼 신규. PDF: "장바구니가 없음")
   │
 ⑥ 결제                             [외부 PG] 이니시스 (카드/가상계좌/에스크로/포인트)
   │                                          → 승인 콜백으로 "결제완료" 상태 전이
   │  ─────────── 결제완료 트리거 ───────────
   │
 ⑦ 생산파일 생성                    [내부+외부]  주문번호·파일번호 채번, 접수파일 등록,
   │                                          파일명 Rename, 썸네일 자동생성
   │                                          (편집기 상품은 [외부 Edicus Prepress] 랜더링 경유)
   │
 ⑧ 생산정보 전달                    [내부→외부]  선택옵션 → 생산 BOM/공정/사이즈/수량 환원,
   │                                          MES_ITEM_CD 매칭하여 MES로 전달
   │
 ⑨ MES 생산관리                     [외부 MES]   제작대기→제작중→제작완료 상태머신,
   │                                          의뢰서 출력, 발주(외주처별), 생산다운 여부 Y
   │
 ⑩ 공정 (생산)                      [외부 MES]   디지털출력→코팅→재단→후가공/제본/가공 (17 케이스)
   │                                          + 자체제작 / 외주제작 분기
   │
 ⑪ 포장                             [외부 MES]   1차포장(바코드)→박스포장 / 제작완료 바코드
   │
 ⑫ 배송                             [내부+외부]  송장출력, 택배/퀵/직접방문 분기
                                                + [외부 택배사] 송장발급 + [외부 알림톡/SMS]
```

**자체제작 vs 외주제작 분기 (주문프로세스 PDF p3 "03. 인쇄타입 및 제작옵션"):**
- 자체제작: 디지털출력·실사출력·특수출력·라텍스(화이트)·반칼/완칼커팅·시트커팅·레이저커팅·UV평판출력·전사인쇄·제본·도장
- 외주제작: 책자·현수막·합판인쇄·고주파·폰케이스 / 선택적 외주: 동판제작·박작업·톰슨/도무송·중철제본

## 2. 사이트 내부 기능 표

> 담당 구분: **인쇄개발** = 견적위젯·가격엔진·파일/생산정보(인쇄 도메인) / **쇼핑개발** = 카탈로그·장바구니·결제·주문·배송(커머스)
> 입력 IA 매핑: F1~F5 = 주문전환 단계, D1~D2 = 파일 단계 (IA 문서 입력 단계 라벨)

| 단계 | 기능 (쉬운말) | 담당 | 관련 t_* 엔티티 | 입력 IA 매핑 | 난이도 |
|------|--------------|------|----------------|-------------|--------|
| ① 상품선택 | 상품 목록·상세 보여주기. 어떤 상품인지 고르기 | 쇼핑개발 | `t_prd_products`, `t_cat_categories`, `t_prd_product_categories` | F1 (상품 진입) | 하 |
| ② 옵션선택 (위젯) | 사이즈·소재·도수·수량 등 옵션 고르는 견적 위젯 | 인쇄개발 | `t_prd_product_option_groups`, `_options`, `_option_items`, `_constraints`, `t_siz_sizes`, `t_clr_color_counts` | F1 (옵션) | 상 |
| ③ 가격확인 | 고른 옵션으로 실시간 가격 계산 | 인쇄개발 | `t_prc_price_formulas`, `_formula_components`, `_price_components`, `_component_prices`, `t_prd_product_prices`, `t_dsc_*` | F2 (가격/견적) | 상 |
| ④-1 파일업로드 | 인쇄용 원본파일(AI/PDF/JPG) 올리기 | 인쇄개발 | `t_prd_products.file_upload_yn` (실측: Y=219) | D1 (파일업로드) | 중 |
| ④-2 에디터 디자인 | 웹에서 직접 디자인(편집기) 후 저장 | 인쇄개발 | `t_prd_products.editor_yn` (실측: Y=121) | D2 (에디터) | 상 |
| ⑤ 장바구니 | 여러 상품 담아 한 번에 주문 (현재 부재) | 쇼핑개발 | (런타임 테이블 부재 → 신규) | F3 (장바구니) | 중 |
| ⑥ 결제 진입/완료 처리 | 결제수단 선택→PG 호출→승인콜백 받아 주문확정 | 쇼핑개발 | (주문/결제 테이블 부재 → 신규) | F4 (결제) | 상 |
| ⑦ 생산파일 생성 | 주문번호·파일번호 채번, 파일명 Rename, 썸네일 자동생성 | 인쇄개발 | `t_prd_products` (file/editor 플래그·MES_ITEM_CD) | D1/D2 후속 | 상 |
| ⑦ 파일명 일관화 | `품목_출력사이즈_양단면_소재명_거래처명_고객명_파일고유번호_수량` 규칙으로 Rename | 인쇄개발 | `t_prd_products`, `t_siz_sizes`, `t_mat_materials` | D1/D2 | 상 |
| ⑧ 생산정보 환원 | 선택옵션 → 생산 BOM(자재/공정)·사이즈·수량으로 환원 | 인쇄개발 | `t_prd_product_materials`, `_processes`, `_sizes`, `t_proc_processes`, `t_mat_materials` | F5 (주문확정 후) | 상 |
| ⑫ 주문 상태조회 | 고객이 미입금→결제완료→제작중→출고완료 추적 | 쇼핑개발 | (주문 상태 테이블 부재 → 신규) | F5 (마이페이지) | 중 |
| ⑫ 배송정보 입력 | 배송지·배송방법(택배/퀵/방문)·추가배송비(제주 등) | 쇼핑개발 | (배송 테이블 부재 → 신규) | F4/F5 | 중 |

| **택배사 / 퀵 / 배송** | 송장발급·배송추적, 합배송, 추가배송비 | API 또는 화면 (송장출력·송장EXCEL) | 택배사 계약(송장 API), 퀵 연동, 제주/추가배송비 정책 | 하나의 주문에 여러 송장 부여(PDF Pain point) 지원 결정 |

> **참고:** 외주 생산처(후지필름 등)는 엄밀히는 "MES 하위 외주 연동" 이나, Front 사이트 관점에서는 MES 내부 업무로 캡슐화됨. IA 일정관리상으로는 MES 트랙 안에 포함.

## 4. 생산정보 전달 상세

**트리거:** 결제완료(`결제완료` 상태 전이) → ① 생산을 위한 파일 생성 + ② 생산에 필요한 정보 전달.
근거: 주문프로세스 PDF p1 (주문상태머신: 미입금→결제완료→제작대기→제작중→제작완료→출고완료) + p9 "09. 발주업무 프로세스" + p10 "10. 주문상태별 업무 프로세스".

### 4-2. ② 생산정보 전달 — 선택옵션 → 생산 환원

고객이 위젯에서 고른 **옵션(차원)** 이 생산용 **BOM/공정/사이즈/수량** 으로 환원되어야 함:

| 선택옵션 (위젯) | 생산정보로 환원 | 환원 t_* 엔티티 | 라이브 상태 (실측) |
|----------------|----------------|----------------|-------------------|
| 사이즈 | 작업사이즈·출력사이즈·출력용지규격(판형) | `t_siz_sizes`, `t_prd_product_sizes`, `t_prd_product_plate_sizes` | 적재됨 |
| 소재 선택 | 자재 BOM (parent + usage) | `t_mat_materials`, `t_prd_product_materials` | 적재됨 |
| 후가공/공정 옵션 | 공정 BOM (코팅·재단·박·제본 등) | `t_proc_processes`, `t_prd_product_processes` | 적재됨 |
| 도수(인쇄색상) | 도수/별색 | `t_clr_color_counts`, `t_prd_product_print_options` | 적재됨 |
| 수량 | 제작수량 (제작수량≠출력수량 이슈, PDF p6) | `t_prd_product_bundle_qtys`, `min/max/dflt_qty` | 적재됨 |
| (상품 자체) | **MES 품목 매칭** | `t_prd_products.MES_ITEM_CD` | **실측: 16/275만 채워짐** |

> 즉 **생산정보의 "재료"(자재/공정/사이즈/도수/수량)는 이미 마스터에 적재**돼 있고, 부족한 것은 **① 주문↔이 마스터를 잇는 주문/주문상세 런타임 테이블, ② MES로 보내는 매칭 키(MES_ITEM_CD)** 다.

### 4-3. MES 연동 컬럼 라이브 실측

**실측 (Railway `railway` DB, 읽기전용 SELECT):**

- **MES 연동 테이블/컬럼은 단 하나**: `t_prd_products.MES_ITEM_CD` (전수 검색 결과 다른 `%mes%` 컬럼 없음, `change_message` 는 Django admin 로그로 무관).
- **적재 상태:** 전체 상품 **275행 중 MES_ITEM_CD 채워진 행 = 16행 (5.8%), NULL = 259행 (94.2%).**
- **포맷:** `NNN-NNNN` (카테고리코드-순번). 예: `OPP접착봉투 = 012-0004`, `탁상형캘린더 = 007-0001`, `엽서캘린더 = 007-0003`, `볼체인 = 012-0015`.
- **`t_cus_customers` (거래처):** 테이블은 존재하나 **0행 (빈 테이블)**. 컬럼 = `cus_cd, cus_nm, grade_cd(등급), reg_ymd, use_yn` — 등급/이름만 있고 주문 연결 없음.
- **주문/결제/파일/배송/장바구니 테이블:** `information_schema` 전수 조회 결과 **0개** (`%ord%`, `%pay%`, `%file%`, `%ship%`, `%deliv%`, `%cart%`, `%mes%` 패턴 매칭 부재).

**해석:**
- MES 매칭 키(MES_ITEM_CD)가 **94% NULL → 생산정보 전달의 핵심 브릿지가 사실상 미구축**. 리뉴얼 선행 과제로 **전상품 MES_ITEM_CD 채번** 이 필수 (PDF Pain point "MES의 상품과 사용자의 상품을 매칭하여 파일네임을 일관화").
- 주문 런타임 전체(주문·주문상세·결제·파일·배송)가 to-be → 쇼핑개발 신규 구축 영역.

## 5. 일정관리 IA 시사점 (요약)

- **인쇄개발 영역(적재 완료된 마스터 활용):** 위젯·가격엔진·생산정보 환원은 마스터(자재/공정/사이즈/도수)가 이미 있어 "조립"에 가까움. 단 MES_ITEM_CD 채번·파일명 일관화·썸네일 자동생성은 신규.
- **쇼핑개발 영역(전부 신규):** 장바구니·결제(이니시스)·주문상태머신·배송/송장 — 라이브 런타임 테이블 부재로 **그릇부터 신규 구축**. Shopby 채택 여부가 분기점.
- **외부 연동 선행조건(PM):** 에디쿠스 SDK 계약, S3 presigned 정책, 이니시스 가맹, 알림톡 템플릿 승인, **MES↔Front 통신 규격 + MES_ITEM_CD 전수 채번**, 택배사 송장 API.
- **최우선 리스크:** MES_ITEM_CD 94% NULL = 생산정보 전달 브릿지 미완. 주문 런타임 테이블 전무 = 커머스 백엔드 미설계.
