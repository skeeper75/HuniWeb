BATCH-12를 데이터가 흘러가는 경로부터 차근히 설명드리겠습니다.

데이터가 라이브 DB에 들어오는 경로

후니 상품 데이터는 이런 파이프라인을 거쳐 라이브 DB에 적재됩니다:

① 상품마스터 원본 엑셀   ← 진짜 정답 (우리가 L1로 보유)
        │
        ▼  [누군가가 "정규화" 작업 — 이 단계에서 오류 발생]
② v03 정규화 엑셀        ← 결함의 진짜 진원 (오류 多·우리 레포엔 없음)
   (prdmaster_full_migration_v03)
        │
        ▼  [load_master.py = 그대로 옮기기만 함]
③ 라이브 DB (t_*)        ← 우리가 본 "잘못된 결과" (피고)

핵심: 라이브 오류는 적재 코드 잘못이 아니다

round-13 내내 확인한 사실은 — 라이브 DB의 오류는 load_master.py(적재 코드)의 잘못이 아니라는 것입니다. 코드를 읽어보니 load_master는 v03 엑셀의 값을 **변환 없이 그대로 옮기는 "순수 전파기"**였습니다(코드 Read로 입증). 즉 ②에서 이미 틀린 채로 들어왔고, ③은 그걸 충실히 복사했을 뿐입니다.

★ Insight ────────────────────────────────────
번역 비유: 원문(①)→중간 번역본(②, 오역 포함)→최종 출판본(③). 최종본의 오류는 식자공(load_master)이 아니라 중간 번역(v03)의 오역 탓. 식자공은 번역본 그대로 조판했을 뿐.

구체 예: "코팅"이 라이브에서 재료로 잘못 등록된 것(BATCH-3) — v03 엑셀이 코팅을 자재 칸에 적어뒀고, load_master가 그대로 옮긴 결과. load_master를 아무리 봐도 "코팅=공정으로 바꾸라"는 로직이 없음(전파기니까).
──────────────────────────────────────────────

그래서 BATCH-12의 선택: 어디서 고칠까

문제는 — ②(v03 엑셀)는 webadmin 팀 영역이고 우리 레포에 없습니다. 그래서 교정을 두 군데서 할 수 있습니다:

(a) DB에서 직접 교정 (하류) — 권장

③ 라이브 DB의 잘못된 값을 우리가 직접 고침. 우리는 정답(① 상품마스터 원본)을 알고 있으니 즉시 가능.
- 장점: 빠름, webadmin 팀 일정에 안 묶임, 우리가 통제
- ⚠️ 단점: 누군가 나중에 load_master를 다시 돌리면, ②의 틀린 값이 우리 교정을 다시 덮어씀(회귀). → 그래서 "멱등 UPSERT + 재실행 가드"가 필요(이미 캘린더에서 plate .01→.03 퇴행 위험으로 실증됨)

(b) webadmin 팀이 v03를 수정 (상류)

②의 정규화 엑셀 자체를 고침. 근본 원인 제거 — 다시 돌려도 안전.
- 장점: 재실행해도 안 틀림(원본이 옳아지니까)
- ⚠️ 단점: v03 파일 접근·webadmin 팀 일정·소유권에 의존. 우리가 통제 못 함

(c) 둘 다 병행

DB 즉시 교정(a)으로 운영 정합을 먼저 확보 + v03 상류 수정(b)도 점진 진행해 재발 방지.

왜 이게 "전 트랙 방향"인가

이 결정이 10 family 전부에 영향을 줍니다. round-13이 찾은 거의 모든 오적재가 v03 진원이기 때문입니다. 그래서:
- (a)를 고르면 → round-5/6/10 교정 작업이 전부 "DB 델타 + 재실행 가드" 설계로 진행
- (b)를 고르면 → 우리는 교정 명세만 webadmin 팀에 넘기고, 실 적재는 그쪽에서
- (c)면 → 둘 다 트랙 가동

[참고] 이 프로젝트의 확립된 directive는 이미 "v03 = 미분석 오류 多 → 정답 참조 금지, 정답은 상품마스터 L1"이고, webadmin은 사용자(지니)님 소유의 별도 레포(HuniProductPrice2)입니다. 즉 (a)와 (b) 모두 결국 사용자 통제 범위 안이라, 핵심은 "지금 라이브를 빠르게 바로잡을지(a)" vs "근본을 고쳐 재발을 막을지(b)"의 우선순위 선택입니다.

시트별 매니페스트 = 2단 구조
├─ ① 실무진 정리표 (한글 도메인
│    상품명 | 무엇이 틀렸나 | 어떻게 고치나 | 영향 | 근거
├─ ② 기술 적재본 (개발/실행용)
│    - 기초데이터: size/자재/공정/도수/판형/묶음수 (UPSERT)
│    - CPQ: option_groups→optionst
│    - 정규화 가드: table-spec 도메인 + sql/ 제약 화이트리스트
│    - 타임스탬프: 신규 reg_dt=no변경이력
└─ ③ 안전장치: 백업 → 롤백전용 DRY-RUN → (인간 승인) COMMIT → 재검증

검증은 dbm-validator(생성자≠검증자)가 맡고, 4 변경시트는 도메인·경쟁사 근거를 재대조합니다.
