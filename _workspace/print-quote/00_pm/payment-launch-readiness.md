# 결제 온보딩 & 런칭 준비 체크리스트

**작성:** 2026-05-30
**목적:** KG이니시스 결제 적용 절차 + 후니프린팅 리뉴얼 런칭에 필요한 전체 준비사항을 개발/실무진 역할로 분리 정리
**관련 BLOCKER:** OQ-3(이니시스 계약 승계), OQ-EsCrow(에스크로 승계), 누락 R1(결제 데이터 마이그레이션), 누락 R2(연동 버전 호환성) — `commerce-backend-scenario.md` §5 참조

---

## 0. 핵심 — "라이브 사이트가 없는데 심사를 어떻게?"

가장 큰 오해를 먼저 풉니다. **결제 적용은 "개발"과 "심사(상용 계약)"가 분리된 2개 트랙**입니다.

| 트랙 | 사이트 필요? | 무엇을 하나 | 누가 |
|---|:-:|---|---|
| **A. 개발·연동** | ❌ 불필요 | 이니시스가 공개한 **테스트 MID(`INIpayTest`)**로 라이브 사이트 없이 즉시 결제창 연동·테스트. 심사 불필요 | 개발 |
| **B. 상용 전환(심사)** | ✅ 필요 | 오픈 직전 스테이징이 "심사 가능 상태"가 되면 가맹점관리자에서 **MID 추가 + 새 도메인 URL 등록** 신청 → 원천사 심사 → 상용 MID 수령 | 실무진 + 개발 |

즉 **"사이트가 없어서 심사를 못 한다"가 아니라**, 개발은 테스트 MID로 지금 바로 시작하고, 심사는 컷오버 직전에 신청하면 됩니다. 그리고 후니는 **이미 이니시스 가맹점**이므로, 신규 계약(상담 2~5영업일)이 아니라 **기존 가맹점의 "온라인 서비스 추가"**(접수 자체는 수십 초, 원천사 심사는 별도) 경로를 탑니다.

## 1. 이니시스 결제 적용 5단계 (As-Is 가맹점 기준)

```
[1단계] 현 가맹 상태 파악 ──> [2단계] 테스트 연동 개발 ──> [3단계] 심사 가능 상태 구축
                                                                      │
        [5단계] 상용 MID 교체·검수 <── [4단계] MID/URL 추가 신청·심사 ┘
```

### 1단계 — 현 가맹 상태 파악 (실무진, OQ-3/R2 해소)
- 가맹점관리자([gw.inicis.com](https://gw.inicis.com)) 로그인 계정·현재 MID 확인
- 현재 어떤 지불수단(카드/가상계좌/계좌이체/간편결제)을 쓰는지 목록화
- **현 연동 모듈 버전 확인** (R2 함정): As-Is가 구판 표준결제창이면 신규 Next.js 스택에서 신규 키·API로 재연동 필요할 수 있음
  - ✅ **확인됨 (2026-05-30 라이브 크롤)**: As-Is huniprinting.com은 **INIStdPay 표준결제창**을 PC(`/INIStdPay/input.asp`)·모바일(`/INIStdPayMobile/input.asp`)로 사용 중. Classic ASP 연동이라 신규 Next.js에서는 동일 표준결제창을 **신규 구현**(서버 승인·signature 검증을 JS/TS로 재작성)해야 함 — 계약(MID/키)은 승계하되 코드는 재작성
- 신규계약/서비스변경 문의: **1588-4954** (고객센터), 신규계약팀 02-3430-5858

### 2단계 — 테스트 연동 개발 (개발, 사이트 불필요)
- 테스트 MID 적용: 일반결제 `INIpayTest` / 에스크로 `iniescrow0` / 정기결제 `INIBillTst`
- 표준결제창 JS 연동: `https://stdpay.inicis.com/stdjs/INIStdPay.js`
- 필수 파라미터: `mid`, `signature`, `mKey`, `timestamp`, `version`, `oid`(주문번호), `price`
- **서버 승인·검증 로직**: 결제창 결과 수신 → 서버에서 승인 API 호출 → 위변조 검증(signature 재계산) → 주문 상태 확정 → DB 저장
- 매뉴얼: [manual.inicis.com](https://manual.inicis.com/)

### 3단계 — 심사 가능 상태 구축 (실무진 + 개발)
원천사(카드사) 심사를 통과하려면 사이트가 아래 상태여야 함:
- 실제 판매 가능한 **상품 3~5개 이상** 등록
- 상품 상세페이지에 **배송기간·교환·환불 정책** 표기
- 사이트 **하단(footer)에 사업자정보**: 상호명·사업자등록번호·연락처·사업장 주소·대표자명·통신판매업 신고번호
- (스테이징 도메인으로 신청 가능 여부는 고객센터 확인 — 보통 오픈 예정 도메인 등록)

### 4단계 — MID/URL 추가 신청·심사 (실무진)
- 가맹점관리자 → **[변경/추가 > 서비스 추가 및 변경] → [1.3 MID 추가 > 1.3.1 MID 추가 및 변경]**
- 새 도메인 URL 등록 (URL·호스팅정보 앞에 `*` 기호 필수)
- 접수 후 **카드사 심사 통상 7~10영업일** (지불수단별 원천사 심사 별도, 보증보험 추가 요구 가능)
- 판매상품 종류·최고단가·판매방식에 따라 추가 심사·보증보험 발생 가능

### 5단계 — 상용 MID 교체·최종 검수 (개발)
- 심사 통과 후 발급된 **상용 MID·키**로 코드의 테스트 MID 교체 (환경변수 분리: `.env.local`)
- 실거래 1건 소액 테스트 → 승인·취소·환불·정산 흐름 검증
- 정산 대사 로직 확인 (PG 정산 내역 ↔ 자체 주문 DB)

## 2. 에스크로(구매안전서비스) — OQ-EsCrow (실무진 + 개발)

- **법적 의무**: 통신판매업 신고 사업자는 구매안전서비스 의무. **현금성 거래(가상계좌·계좌이체)는 에스크로 의무**, 카드결제는 선택.
- **처리 방법**: 대부분 PG 가입 시 부가서비스로 함께 신청 → 이니시스에서 **에스크로 확인증** 발급 (테스트 MID `iniescrow0` 존재 = 이니시스가 에스크로 결제창 제공)
- 실무진: 이니시스 서비스 추가 시 에스크로 함께 신청 + 확인증 수령 → 통신판매업 신고에 첨부
- 개발: 에스크로 결제창([escrow_pc 매뉴얼](https://manual.inicis.com/pay/escrow_pc.html)) 연동, 배송완료 → 구매확정 → 대금지급 흐름 구현

## 3. 전체 런칭 준비사항 — 개발 vs 실무진 역할 분담

> 🔴 = V1 런칭 차단(필수) · 🟡 = 중요 · 🟢 = 권장

### A. 실무진(오너/지니·운영) 담당 — 계약·법무·정책·콘텐츠

| # | 준비사항 | 우선 | 메모 |
|---|---|:-:|---|
| R-1 | 이니시스 가맹점관리자 계정·현 MID·지불수단 현황 확인 | 🔴 | 1단계, OQ-3 |
| R-2 | 이니시스 MID 추가 + 새 도메인 URL 등록 신청 | 🔴 | 4단계, 카드사 심사 7~10일 |
| R-3 | 에스크로(구매안전) 서비스 신청 + 확인증 수령 | 🔴 | OQ-EsCrow, 법적 의무 |
| R-4 | 통신판매업 신고 (도메인/상호 변경 시 갱신) | 🔴 | 에스크로 확인증 첨부 |
| R-5 | 사업자정보 확정 (footer 표기용 5종) | 🔴 | 심사 요건 |
| R-6 | 환불·교환·배송 정책 문구 확정 | 🔴 | 심사 요건 + 법적 고지 |
| R-7 | 이용약관·개인정보처리방침 문안 | 🟡 | 법무 검토 권장 |
| R-8 | 심사용 상품 3~5개 콘텐츠(상세·이미지·가격) | 🔴 | 3단계 |
| R-9 | 배송비 정책 6종 확정 (D-PM-31) | 🔴 | 가격엔진 Step6·결제화면 차단 |
| R-10 | 도메인 소유권·이전 결정 (신규 vs 기존 huniprinting) | 🟡 | URL 등록과 직결 |
| R-11 | 간편결제(네이버/카카오) 직계약 여부·시점 결정 | 🟢 | OQ-4, V1.5/V2 이연 가능 |

### B. 개발 담당 — 연동·검증·인프라·마이그레이션

| # | 준비사항 | 우선 | 메모 |
|---|---|:-:|---|
| D-1 | 테스트 MID 결제창 연동(일반) | 🔴 | 2단계, 사이트 불필요 |
| D-2 | 서버 결제 승인·위변조 검증 로직 | 🔴 | signature 재계산·무결성 |
| D-3 | 에스크로 결제창 + 구매확정 흐름 | 🔴 | 가상계좌·계좌이체 대상 |
| D-4 | 상용 MID 환경변수 분리·교체 체계 | 🔴 | 5단계, `.env.local` |
| D-5 | 정산 대사 로직 (PG내역 ↔ 주문DB) | 🟡 | 가맹점 직접 책임 |
| D-6 | **Inquiry/InquiryQuote 엔티티** 추가 | 🔴 | B2B V1 후속(별건) |
| D-7 | **결제 데이터 마이그레이션**(As-Is 진행중 주문/결제/환불 이관) | 🔴 | 누락 R1, 컷오버 단절 구간 |
| D-8 | As-Is 연동 모듈 버전 ↔ 신규 스택 호환성 점검 | 🟡 | 누락 R2 |
| D-9 | V-001 게스트결제 회원강제 PoC | 🟡 | 검증 필요 |
| D-10 | OQ-2 동적 견적가 결제 지원 PoC | 🟡 | 이니시스 직연동은 가맹점지정금액이라 자연지원 예상 |
| D-11 | 도메인·SSL·호스팅 인프라 셋업 | 🔴 | URL 등록 전제 |
| D-12 | footer 사업자정보·정책 페이지 구현 | 🔴 | 심사 요건(R-5/R-6 콘텐츠 수령 후) |

## 4. 권장 진행 순서 (의존관계 기준)

1. **지금 즉시 (병렬)**
   - 실무진: R-1(가맹 현황 파악) → 이니시스 1588-4954 상담으로 "기존 가맹점 신규 사이트 추가" 경로·심사조건·스테이징 도메인 심사 가능 여부 확인
   - 개발: D-1·D-2(테스트 MID 연동) — 사이트 없이 착수 가능
2. **컷오버 1~2개월 전** (심사 리드타임 선소진)
   - 실무진: R-5·R-6·R-8 콘텐츠 확정 → R-2(MID/URL 신청) → R-3(에스크로) → R-4(통신판매업)
   - 개발: D-11·D-12로 스테이징을 "심사 가능 상태"로
3. **심사 통과 후**
   - 개발: D-4(상용 MID 교체) → D-5(정산대사) → D-7(데이터 마이그레이션) → 실거래 검수
