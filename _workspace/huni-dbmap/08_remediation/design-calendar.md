# design-calendar(디자인캘린더) 시트 결함 처리 — 도메인 렌즈 라이브 규명 (round-3 remediation)

> **작업 목적(내재화):** 누군가 엑셀을 인쇄 도메인 이해 부족으로 DB에 적재해 **누락·오매칭·모호**가 생겼다.
> 본 문서는 그 결함을 *올바로 처리(수정)*하기 위해, **라이브 Railway DB 직접 read-only SELECT를 권위**로
> design-calendar(디자인캘린더) 상품군을 종단 규명한다. 도메인 지식(07_domain KB)은 결함을 옳게 분류하는 렌즈다.
> 정제된 DB가 인쇄 자동견적 위젯의 기반이 된다.
>
> **권위·안전 원칙(HARD):** 라이브 SELECT = 진실(추출본·v2 2026-06-04은 stale 가능).
> 추정·임의판단 0 — 라이브·엑셀 L1·도메인 KB·PDF 실재 근거만. 미지/추정은 가설+컨펌 플래그.
> INSERT/UPDATE/DDL 0. DB 미적재 유지(본 문서는 처리 *방향* 설계). `db railway`·비표준포트·JOIN KEY=`prd_nm`.
>
> **작성** 2026-06-05 · 식별자/컬럼/코드/SQL 영어, 설명 한국어. 라이브 SELECT 3시트 공통 9회(전부 read-only).

## ⚠️ 머리말 — 본 검증이 v2 대비 뒤집은 가장 큰 사실 (권위 반전)

**디자인캘린더 시트는 calendar 시트와 *동일한 5개 prd_cd(PRD_000108~112)*를 가리킨다. DB에 별도의 "디자인캘린더" 상품은 존재하지 않는다.**

라이브 직접 확인 결과(가장 큰 반전):

- **JOIN KEY=prd_nm으로 디자인캘린더 시트의 5상품명(탁상형/미니탁상형/엽서/벽걸이/와이드벽걸이)을 라이브 조회하면
  calendar 시트와 *완전히 같은* PRD_000108~112로 떨어진다.** `디자인캘린더`/`디자인`을 포함한 캘린더 상품은 DB에 없다
  (라이브 SELECT #9: `디자인` 검색 결과 = 포토북[디자인명] 계열뿐, 캘린더는 108~112 단일 세트).
- 두 시트 차이: design-calendar는 **가격 포함(`가격` 컬럼)·`디자인보유(●)`·`페이지사양` 단일값(30P/26P/12P/13P)·종이 직접명(`몽블랑 190g`)**.
  calendar는 `*별도설정`·`장수(필수)` 다중값·가격 없음. → **디자인캘린더 = 동일 상품의 "디자인 제공(완성 디자인) variant"** 성격.
- **결함의 본질:** 디자인캘린더가 별도 상품/옵션 축으로 DB에 인코딩되지 않음. 동일 prd_cd에 디자인 제공 여부를
  구분하는 축이 없다 → 위젯에서 "디자인캘린더 vs 일반캘린더"를 구분 못함. **모호(CONFIRM)의 뿌리.**

**교훈(HARD 재확인):** prd_nm only JOIN에서 "두 시트가 같은 상품을 가리키는지"는 라이브 매칭으로만 드러난다.
디자인캘린더를 calendar와 별개 상품으로 가정하면 결함을 오판한다.

## ① design-calendar 상품군 — 도메인 이해 (라이브 권위)

**라이브 SELECT #1,9:** 디자인캘린더 시트 5 prd_nm 매칭 + `디자인` 키워드 전수.

디자인캘린더 시트 = calendar와 동일 **탁상형·미니탁상형·엽서·벽걸이·와이드벽걸이** 5상품(`prd_nm` 동일).
라이브 매칭 = **PRD_000108~112**(별도 prd_cd 없음).

|------------------------------|:------------------:|:--------------------:|---------------------------|
| 탁상형캘린더 | PRD_000108 | **동일** | 종이=몽블랑190g·페이지=30P·가격=10400·디자인보유● |
| 미니탁상형캘린더 | PRD_000109 | **동일** | 몽블랑190g·26P·6500·● |
| 엽서캘린더 | PRD_000110 | **동일** | 몽블랑190g·12P·4000·우드거치대 |
| 벽걸이캘린더 | PRD_000111 | **동일** | 몽블랑190g·13P·9900·고리형트윈링·● |
| 와이드벽걸이캘린더 | PRD_000112 | **동일** | 몽블랑190g(3절)·13P·24000·고리형트윈링·● |

- **디자인캘린더 = "후니 제공 디자인(템플릿) 포함" 판매 형태.** `디자인보유(●)`·`가격 포함`·페이지 고정(30P 등)이
  핵심 차이. 일반 캘린더(고객 디자인 업로드, `*별도설정` 종이·장수 가변)와 **같은 물리 상품·다른 판매축**.
- 라이브 확인: 별도 "디자인캘린더" 상품 미존재(0건). use_yn=N 없음(108~112 전부 Y).
- **qty_unit_typ_cd = NULL:** 272 전상품 NULL — 고유 결함 아님(§4 G-DC-5).

> **삼각대/고리형트윈링(사용자 지적):** design-calendar=삼각대/고리형트윈링 가공 — 엑셀 `캘린더사양_캘린더가공`에
> 삼각대(그레이/블랙)·고리형트윈링제본이 명시. calendar와 동일 캘린더가공 축(택일그룹 결함 공유).

| `페이지사양` = **`30P`·`26P`·`12P`·`13P`(단일 고정)** | 디자인 페이지 고정 | page_rule(?) / variant |

## ④ 결함 표 — 누락(MISSING)·오매칭(MISMATCH)·모호(CONFIRM) (도메인 렌즈, 라이브 재확인)

### G-DC-1 [CONFIRM · 구조적 모호] 디자인캘린더 = 별도 상품/축 미인코딩

- **라이브 재확인(SELECT #1,9):** 디자인캘린더 시트 5상품이 calendar와 **동일 prd_cd(108~112)**. DB에 디자인캘린더 전용 상품/옵션축 **0건.**
- **도메인 의미:** 디자인캘린더 = "완성 디자인 포함" 판매 형태(고정 페이지·고정 종이·고정가). 일반캘린더 = "고객 디자인 업로드"(가변).
  같은 물리 상품의 **2개 판매 모드**인데 DB엔 이를 구분하는 축 없음.
- **판정:** **CONFIRM(구조적).** 디자인 제공 여부를 ① 별도 prd_cd 신설 ② variant/옵션축 ③ `편집기/디자인보유` 플래그 중 무엇으로 인코딩할지 미정.
  현재는 위젯에서 "디자인캘린더"를 별개 상품으로 노출 불가.

### G-DC-2 [MISMATCH/CONFIRM] 종이 — 디자인캘린더 고정(몽블랑190g) ↔ 일반 IMPORT 가변 혼재

- **도메인 근거:** 디자인캘린더 종이 = `몽블랑 190g` **단일 고정**(R-MAT-1 직접). 일반캘린더 = `*별도설정` IMPORT 가변.
- **라이브:** 동일 prd_cd(108~112)에 calendar의 IMPORT 자재(다행)가 적재. 디자인캘린더 고정 종이와 **자재 집합이 충돌**.
- **판정:** 디자인캘린더와 일반캘린더가 같은 prd_cd를 쓰는 한, **종이 옵션 집합을 분리할 수 없음**(G-DC-1 의존).
  G-DC-1을 별도축으로 풀면 디자인캘린더는 몽블랑190g 단일 자재로 제약. **CONFIRM.**

### G-DC-3 [MISSING] 캘린더가공 택일그룹 멤버 연결 (calendar G-CL-1 공유)

- **라이브:** 동일 prd_cd → calendar와 동일하게 **택일그룹 멤버 연결 끊김(excl_grp_cd NULL)·삼각대 거치 공정 108·109 미적재.**
- **판정:** calendar.md G-CL-1·G-CL-2와 **동일 결함**(공유 상품). 디자인캘린더 고유 추가 없음 — calendar 처리 시 동시 해소.

### G-DC-4 [MISSING/CONFIRM] page_rule 고정 페이지 미적재

- **도메인 근거:** 디자인캘린더 `페이지사양 = 30P/26P/12P/13P` **단일 고정** → page_rule(min=max=고정) 적재가 자연스러움.
- **라이브:** 108~112 page_rule 0행. 고정 페이지 미적재.
- **판정:** **CONFIRM** — 디자인캘린더 페이지를 page_rule(고정)로 적재할지, variant로 둘지(G-DC-1 인코딩에 의존).

### G-DC-5 [글로벌 갭] qty_unit_typ_cd 전 NULL

- 272 전상품 NULL — 고유 결함 아님(글로벌 정책).

## ⑥ 실무 컨펌 질문 목록 (top — 디자인캘린더 구조 중심)

1. **[R1·최우선·구조]** 디자인캘린더는 calendar와 **동일 prd_cd(108~112)**를 쓰고 있음(DB에 별도 상품 없음).
   디자인 제공 형태를 ① 별도 prd_cd 신설 ② 동일 상품의 variant/옵션축 ③ `디자인보유/편집기` 플래그 중 무엇으로 인코딩할 것인가?
2. **[R2]** 디자인캘린더는 종이가 `몽블랑190g` 단일 고정, 일반캘린더는 `*별도설정` IMPORT 가변 — R1을 별도축으로 풀면
   자재 집합을 분리하는데, 디자인캘린더는 몽블랑190g 1종으로 제약하는 게 맞는가?
3. **[R4]** 디자인캘린더 페이지(30P/26P/12P/13P 고정)를 page_rule(min=max)로 적재할지, 선택 불가 고정 variant로 둘지?
4. **[R3·캘린더 택일그룹]** 디자인캘린더도 calendar와 동일 캘린더가공 택일그룹 결함(멤버 연결 끊김)을 공유 —
   calendar 처리(GRP-CAL-가공 멤버 복원)로 동시 해소되는 것이 맞는가?
5. **[가격]** 디자인캘린더는 `가격` 컬럼(완제품 고정가) 포함 — 이는 round-2 가격엔진(t_prc_*) 영역이나,
   "디자인 포함가 vs 일반 견적가"를 어느 축에서 구분하는지 가격 설계와 연계 확인 필요.

## ⑦ 큰 그림 적용 (도메인 → 엑셀 → DB결함 → 처리 → 위젯)

- **도메인 KB:** "디자인 제공(완성형) vs 고객 업로드"는 후니 주문프로세스(PDF p9 case1 디자인파일여부 Y/N)의 핵심 분기 — 디자인캘린더가 그 Y측.
- **엑셀 L1(무손실):** design-calendar 시트의 `디자인보유 ●`·`가격`·고정 페이지·직접 종이명이 일반 calendar와의 차이를 정확 보존 → 두 시트가 같은 상품의 다른 판매축임을 규명.
- **DB 라이브 권위:** prd_nm only JOIN으로 디자인캘린더 5상품이 calendar와 동일 108~112임을 확정 → "디자인 축 미인코딩"이라는 구조 결함이 드러남(가정으론 오판).
- **처리방향:** R1(디자인 제공 축 결정)이 선결. 그 위에 자재·페이지 분리, 택일그룹은 calendar와 공유 해소.
- **위젯 기반:** 위젯에서 "디자인캘린더"를 일반캘린더와 구분 노출하려면 R1 인코딩이 선결. 미해소 시 위젯이 두 판매모드를 분리 못함 → 디자인캘린더 상품 자체가 위젯에 표현 불가.
