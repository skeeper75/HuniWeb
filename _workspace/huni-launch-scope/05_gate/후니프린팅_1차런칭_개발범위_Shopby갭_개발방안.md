# 후니프린팅 신규 사이트 — 1차 런칭 개발범위 · Shopby 갭 · 개발방안

> 작성: 2026-06-30 · 후니프린팅 리뉴얼(Classic ASP → 신규 Shopby 기반)
> 이 문서는 **무엇을 그대로 쓸 수 있고(해결), 손봐야 하고(부분), 새로 만들어야 하는지(미해결)** 를 162개 기능 전수로 가른 뒤, 1차 런칭에 들어갈 것을 골라 "어떻게 만들지"까지 정리한 **개발범위 설계서**입니다.
> ★ 이 문서는 **설계·명세까지** 입니다. 실제 프로그램 구현·고객 데이터 옮기기·DB 저장은 **사람이 최종 승인한 뒤** 별도 팀(위젯 구현·Shopby 연동·DB 작업)에 넘깁니다.
> ★ 어려운 영어 용어는 풀어서 설명합니다.

---

## 용어 미리보기 (쉽게)

- **Shopby(샵바이)** — NHN이 만든 쇼핑몰 솔루션(상품·회원·장바구니·주문·결제·정산을 기본 제공하는 "쇼핑몰 뼈대"). 후니의 새 사이트는 이 위에 올립니다.
- **위젯(widget)** — 상품 페이지 안에서 손님이 사이즈·종이·수량 등을 고르고 **실시간 가격**을 보는 옵션 선택 화면. 후니가 직접 만듭니다(인쇄 옵션이 너무 다양해 Shopby 기본 옵션으로는 표현이 안 됨).
- **BFF(중계 서버)** — 위젯·Shopby·후니 가격엔진 사이에서 데이터를 **번역·중계**하는 후니 서버. "Backend For Frontend"의 약자로, 화면을 위한 전용 뒷단입니다.
- **가격엔진(evaluate_price)** — 후니가 이미 가진 가격 계산 프로그램. 옵션을 넣으면 정확한 가격이 나옵니다. **가격의 최종 권위는 항상 이 서버**입니다(화면은 보여주기만).
- **webadmin** — 후니가 지금 쓰는 상품·가격 등록 관리자(파이썬 Django). 신규에서도 **그대로 유지**합니다.
- **마이그레이션(migration)** — 옛 사이트의 회원·머니 같은 데이터를 새 사이트로 **옮기는 작업**.

### 1-3. ★런칭 전에 반드시 결정해야 하는 GATE (관문)

아래 4개는 **돈·주문이 걸린 핵심 관문**입니다. 이게 막히면 1차 런칭 품질이 깨집니다.

| GATE | 무엇 | 왜 중요 |
|------|------|---------|
| **G1. 가격 전달 방식** | 위젯이 계산한 정확한 가격을 **Shopby 장바구니·주문에 손실 없이 싣는 길** 확정 | Shopby 장바구니에는 "임의 가격을 직접 넣는 칸"이 **아예 없음**(원문 확인). 특히 가로×세로 자유입력·면적/구간 단가 같은 **견적형 상품**은 안전한 길이 아직 미확정 → **1차 최대 위험** |
| **G2. 정산 기준 확정** | 세금계산서·정산이 "주문라인 등록가"와 "실제 결제액" 중 **무엇을 진짜로 보는지** | 가격을 임시값으로 넣으면 **정산이 틀어질 수 있음** |
| **G3. 머니 이관 안전장치** | 옛 프린팅머니 잔액을 1원도 안 틀리게 옮기는 검증 | 고객 돈. 만료 기본값=**제한없음**·전 회원 잔액 전수 대조 필요 |
| **G4. 외부 계약** | 이니시스 결제·알림톡/문자·휴대폰 인증·MES(생산설비) 통신규격·파일 검수기준 | 계약/규격이 늦으면 해당 기능 전체가 멈춤 |

## 2. 1차 런칭(Phase 1) 세부 개발범위

1차는 **위젯 + Shopby + 후니 BFF + webadmin** 4층 구조로 만듭니다.

| 층 | 역할 | 1차에 담는 것 |
|----|------|---------------|
| **위젯**(옵션 선택 화면) | 옵션 고르기·제약·실시간 가격 표시·파일 올리기 | 상품옵션(33)·사이즈/소재/도수(34~38)·제본(42·43)·제약(48)·가격표시(49)·파일업로드(52·53)·주문검증·플로팅 박스·장바구니(61~64) |
| **BFF**(중계 서버) | 가격엔진 호출·옵션↔Shopby 번역·생산정보 환원·파일 식별 | 가격(49)·주문변환(62)·장바구니(64)·생산BOM(157)·파일명(159) |
| **Shopby**(쇼핑몰 뼈대) | 회원·장바구니·주문·결제·배송·정산 | 로그인/가입/회원(1·2·4·6·8·22·24)·배송지(78·79)·결제(80)·주문그릇(162) |
| **webadmin**(현 관리자 유지) | 상품·가격·기초데이터 등록 | 상품/가격관리(100~108)·MES 채번(158) |

### 2-1. ★동적 가격 — 가장 중요한 돈 관문 (49·62·64)

인쇄는 옵션을 바꿀 때마다 가격이 달라집니다. 이걸 정확히 보여주고, 정확히 주문에 싣는 게 핵심입니다.

- **49 옵션 변경 시 실시간 재계산** [새로 만듦·초대형]
  - 만드는 곳: 위젯(가격 요청) + BFF(가격 계산).
  - 흐름: 위젯이 옵션을 바꿀 때마다 → BFF `POST /price/evaluate` 호출 → 후니 가격엔진(evaluate_price)이 정확한 가격을 돌려줌 → 위젯이 표시만.
  - 원칙: **가격의 진짜 주인은 서버**. 위젯이 자체적으로 가격을 다시 계산하지 않습니다. 가격이 0원으로 나오면 그건 **항상 오류 신호**입니다.
  - 주의: 옵션 바꿀 때마다 서버 왕복이라 약간의 지연 보정(디바운스)·캐싱 필요.

- **62 선택옵션 → 주문 데이터로 변환** [새로 만듦·대형] ★돈 관문
  - 문제: 위에서 말한 대로 Shopby 장바구니에는 **임의 가격을 직접 넣는 칸이 없습니다**(원문에서 customPrice·orderPrice 등 전수 0건 확인). 그래서 계산된 가격을 그대로 못 싣습니다. 아래 3가지 우회로 중 골라야 합니다.
    - **방법 A (가격별 옵션 미리 등록)** — 가능한 가격 조합마다 옵션을 미리 만들어 둠. **고정가·조합이 적은 상품**에 안전(정산도 안전). 단, 조합이 많으면 폭발.
    - **방법 B (주문 직전 즉석 옵션)** — 주문 순간에 계산가를 옵션으로 즉석 등록. 손실 없으나 **Shopby 상품심사("수정하면 승인대기")에 막힐 위험**이 있어, 자동승인 예외인지 **확인이 관문**(G1).
    - **방법 C (수량 1 + 총액을 옵션금액으로)** — IA 비고 방식. 단 정산이 어긋날 수 있어 **정산 기준(G2) 확정 후** 검증 필요.
  - 권고: **고정가·소조합 상품부터 방법 A로 시작**. 가로×세로·면적/구간 같은 견적형은 G1·G2 해소 전까지 단계 확대.

- **64 장바구니 담기** [부분·중형] ★돈 관문
  - Shopby 표준 `post-cart`로 담되, 위젯 선택값을 상품/옵션으로 번역해서 넣음. 담기 직전 Shopby `get-cart-validate`로 살 수 있는 상태인지 점검.

- ★ **명세 보강(검증 중 발견한 공백)**: 장바구니 다음의 **주문서 단계**(`주문서 작성 → 최종금액 계산 → 결제예약`)까지 한 줄로 명세해야 합니다. Shopby에 해당 기능(post-order-sheet·calculate·reserve)이 **실재함을 원문에서 확인**했으니, 위젯→장바구니→**주문서→최종계산**→결제까지 전 경로를 빠짐없이 설계에 넣습니다.

### 2-2. ★옵션 위젯 — 인쇄 옵션이 Shopby에 안 맞음 (33·34~38·42·43·48)

- **왜 새로 만드나**: Shopby 선택옵션은 **최대 5종까지**(원문 option.mdx:44 확인)인데, 인쇄는 사이즈·종이·도수·코팅·후가공·제본 등 5종을 훌쩍 넘고, 가로×세로 자유입력처럼 **연속값**도 있습니다. → Shopby 옵션으로는 못 담으므로 **위젯이 옵션 화면을 직접 소유**합니다.
- **33 상품별 옵션 목록** [새로 만듦·대형]: 상품마다 옵션 구성이 달라서, BFF가 후니 옵션 데이터를 위젯에 내려주면 위젯이 옵션 트리를 그립니다.
- **34~38 사이즈/비규격/수량/소재/도수** [전부 새로 만듦·중형]: 각 선택이 위젯의 선택값(siz_cd·가로/세로·수량·mat_cd·도수)으로 들어가고, 가격은 가격엔진이 계산.
- **42·43 제본·내지/표지** [새로 만듦·대형/중형]: 책자는 여러 부품을 합치는 셋트 상품이라, 위젯이 구성하고 BFF가 셋트 가격을 계산.
- **48 옵션 제약(같이 못 고르는 조합)** [새로 만듦·대형]: "이 종이엔 이 코팅 안 됨" 같은 규칙을 위젯이 평가. 규칙 데이터를 먼저 정리해야 함.

### 2-3. 파일/편집 (52·53·75)

- **52 PDF 파일 업로드** [새로 만듦·대형]: 위젯에서 파일을 외부 저장소(S3)에 올리고, 파일 식별값만 주문라인에 텍스트로 붙임(파일 원본은 후니가 보관).
- **53 편집 미리보기·썸네일** [새로 만듦·중형]: 올린/편집한 결과의 작은 미리보기 생성.
- 온라인 편집기(Edicus) 연동(54)은 3차.

### 2-4. 회원 영역 (5·21·22·23·24·29)

- **5 정보입력·21 회원정보수정** [부분·중형]: Shopby 표준 프로필에 더해, 후니 회원은 **개인+사업자정보+회사주소 2개**까지 있어서(30개 넘는 항목) 사업자 항목은 추가항목(extraInfo)/별도 저장으로 확장.
- **22 비번입력·23 비번변경·24 탈퇴** [해결]: Shopby 표준. 단 **변경/탈퇴 전 본인확인 방법은 결정 필요**.
- **29 마이페이지 메인** [부분·소형]: 회원할인율·프린팅머니 잔액·옵션보관 요약을 대시보드로 조립. 할인 적용순서(상품금액→회원할인→쿠폰→머니)를 결제에 맞춤.

### 2-5. 상품 리스트 (31·153·154)

- **153 전시 카테고리 운영툴** [부분·중형]: Shopby 전시 카테고리로 다단계 진열. 옛 사이트는 정렬/필터가 없는 단순 나열이라 신규에서 개선.
- **31 리스트(검색결과)·154 검색/필터** [부분]: Shopby 검색에 인쇄 필터축(소재/사이즈/도수)을 더함.

### 2-6. 주문/배송/결제 그릇 (76·77·78·79·80·81·150·161·162)

- **162 주문 런타임 그릇** [부분·초대형] — ★중요: 옛 사이트엔 **주문 테이블 자체가 없습니다**(실측 확인). 그래서 **Shopby 주문 기능을 그릇으로 채택**하고, 후니는 인쇄 사양·파일·생산정보만 라인에 연결.
- **80 결제(이니시스)** [재분류: 부분·가격브리지 종속]: Shopby 결제 표준 + 이니시스 연동. 단 결제 검증이 "라인가=후니가" 일치를 전제하므로 **G1 가격브리지 해소가 선행**.
- **77 배송·78·79 배송지·150 도서산간** [해결/부분]: Shopby 표준 + 지역 추가배송비 데이터.
- **161 주문상태 추적** [부분·중형]: Shopby 주문조회 + 인쇄 생산 상태(미입금→제작중→출고완료).

### 2-7. ★생산 브릿지 — 주문을 공장으로 (137·157·158·159)

- **157 생산정보 전달(주문→생산BOM 환원)** [새로 만듦·대형] ★주문 관문: 결제완료 주문의 선택값을 **생산용 자재명세(BOM)** 로 되돌려 MES(생산설비)에 전달. MES 통신규격 확정 필요.
- **158 전상품 MES 코드 채번** [새로 만듦·중형]: 지금 라이브는 275개 중 16개만 코드가 있음(94% 미채번) → 전상품 채번 체계.

### 2-8. 상품/가격 관리 (100~108) — webadmin 그대로

## 3. 회원/프린팅머니 마이그레이션 설계

옛 사이트의 회원과 프린팅머니를 새 사이트로 옮기는 작업입니다. **실제 옮기기는 사람 승인 후** 별도 트랙이며, 여기서는 설계만 합니다. 옛 DB 원본 미접근이라 **화면 관찰 + Shopby 모델**로 그린 청사진입니다.

### 3-1. 회원 이관

- **그대로 옮김(손실 없음)**: 아이디·이름·이메일·휴대폰·전화·주소·상호·사업자번호·SMS/마케팅 동의값·등급(매핑)·휴면/탈퇴 상태.
- **추가항목/주소록으로 보존**: 대표자명·업태·업종·직업·회사전화·팩스·회사주소1/2.
- **★못 옮김 → 대체**: 비밀번호. 옛 비번은 암호화(단방향 해시)라 새 시스템과 호환 불가 → **옮기지 않고 재설정**합니다.
  - 기본: 첫 로그인 시 이메일로 비번 재설정(방법 A).
  - 보조: 휴대폰만 있는 회원은 임시비번 문자(방법 B), 소셜로그인 회원은 네이버/카카오 전환(방법 C).
  - **전 회원이 1회 재인증** 하는 것은 불가피 → 이관 공지문에 포함.
- **약관/개인정보 재동의**: 사이트 이전은 처리방식 변경을 수반할 수 있어 **재동의 또는 사전 고지** 필요.
- **★교정 반영(검증 적발)**: 마케팅 수신동의는 동의일시 원본이 없으면 **"이관일로 기록"하지 말 것**(날짜 날조·법적 위험). 원본 없으면 **수신 보류 + 재동의 캠페인**으로 통일합니다.

### 3-2. 프린팅머니 이관 (★돈 — 1원도 안 틀리게)

- **모델 차이**: 후니 프린팅머니는 **미리 충전하는 선불 잔액**, Shopby 적립금은 **구매 시 쌓이는 포인트**. 단 **사용(결제 차감) 방식은 같습니다**.
- **권고 방안 A (적립금으로 재해석)**: 회원별 잔액을 Shopby 적립금으로 **1건씩 수동 지급(시드)** 해서 옮김. 적립금 명칭을 "프린팅머니"로 바꿔 표시 가능(원문 확인). 향후 새 충전(PG 결제)만 별도 중계 서버로 보강.
- **보존 수준**: **현재 잔액만** 옮기는 것을 권고(L1). 과거 거래내역 1:1 재현은 비용 크고 합계 어긋날 위험 → 필요 시 옛 사이트 조회로 분리.
- **★안전장치(검증 교정 반영)**:
  - **합계 대사**: Σ(옛 회원 잔액) = Σ(옮긴 적립금), 차이 **0원** 이어야 통과. 1원이라도 차이나면 **즉시 중단**.
  - **전 회원 전수 대조**(샘플만으로 부족): 회원 한 명씩 옛 잔액 ↔ 새 잔액을 전부 비교(회원 간 잔액이 뒤바뀌는 사고 방지).
  - **만료 기본값 = 제한없음**: 옛 잔여 유효기간을 모르면 **소멸시키지 말 것**(고객 돈 소멸 방지).
  - **멱등 키**: 같은 작업을 두 번 돌려도 중복 지급 0(externalKey 활용).
  - 음수 잔액(미수금)은 적립금으로 못 옮김 → 별도 보류·수기 결정.
