# 1차 런칭 범위 세부 흐름 (상품리스트 · 회원 · 프린팅머니)

라이브 As-Is 실측. 모든 화면 근거 URL 첨부. 읽기 탐색만(주문/결제/제출 0).

---

## A. 상품리스트 → 주문 흐름

### A-1. 진입 (GNB 6 대분류)
디지털인쇄 / 책자·캘린더 / 실사출력 / 굿즈제작(파우치&백, NEW) / 패키지·포장재 / 디자인의뢰.

### A-2. 카테고리 리스트
- `/product/list.asp?pcode=N` — 상품 썸네일 그리드. 카테고리당 다수 상품(→ 상세).
- pcode 약 70종 관측(1,3~19,22~28,31~41,44,47,50,52,53,55,57,59~66,68~70,74,80~92 등).
- 정렬/필터 UI: 리스트 페이지에서 별도 정렬·필터 컨트롤 미관측(그리드 나열형). 라이브 ASP 특성.

### A-3. 상세/주문 — 4 패턴(중요)
| 패턴 | 용도 | 옵션 메커니즘(실측) |
|------|------|----------------------|
| **`/product/goods.asp?pcode=`** | 견적형 주문(디지털·실사·책자) | 폼3·select 다수+수량 input. 예 pcode=72 → `tmp_width, tmp_height, tmp_p02, tmp_p05, tmp_p06` (가변사이즈 + 옵션 p02/p05/p06) + 수량 → 가격계산. **현 사이트의 자동견적 핵심 화면.** |
| **`/goods/view.asp?pcode=`** | 카탈로그형(굿즈/완제품) | 단순 `qty` 선택. 옵션 고정형 완제품. |
| `/package/view.asp?pcode=` | 패키지/셋트 | 패키지 구성(세부 미조사). |
| `/design/sangsang.asp?pcode=` | 디자인 의뢰(상상) | 의뢰형(세부 미조사). |

- `HUNI_LIVE_GOODS_URL`(.env)의 `product/goods.asp?pcode=`가 곧 견적형 주문 진입점.
- 견적→옵션보관함(`/mypage/save.asp`) 저장 동선 존재(대시보드 "저장된 내역" 표기로 확인).
- 장바구니/주문 버튼은 미클릭(읽기 탐색 한정). cart 버튼 셀렉터는 product/goods.asp에서 미검출(주문 동선은 별도 확인 필요).

### A-4. 1차 범위 시사
- As-Is 주문은 **견적형(product/goods.asp)** 과 **카탈로그형(goods/view.asp)** 2갈래로 분기 → 신규 위젯/Shopby 브리지 설계 시 두 주문 모델을 모두 흡수해야 함.
- 옵션은 사이즈(가변 width/height) + 코드형 옵션(tmp_pNN) 조합 = 현 가격엔진(t_prc_*)의 차원 입력과 대응.

## B. 회원 영역

### B-1. 인증
- 로그인: `/login/login.asp` → POST `/login/login_ok.asp` (필드 `user_id`, `user_pwd`, hidden `where`, 이미지 submit).
- 로그아웃: `/login/logout.asp`.
- 회원가입: `/join/intro.asp` (★로그인 상태에선 main 리다이렉트 → 가입 폼 전체 미접근. 로그아웃 후 재탐색 필요).

### B-2. 마이페이지 메뉴 (13)
주문조회 · 옵션보관함 · 쿠폰/이용권 · 프린팅머니 · 나의 상품Q&A · 1:1문의 · 나의 이용후기 · 체험단 활동내역 · 회원정보수정 · 비밀번호변경 · 증빙서류발급 · 회원탈퇴 · (대시보드 main).

### B-3. 회원정보(modify.asp) 필드 = 회원 데이터 스키마(마이그레이션 핵심)
**개인:** user_pwd, user_email, check_email(뉴스레터 수신), user_zip/user_zone/user_addr1/user_addr2/jibun_addr(주소: 우편번호·도로명·상세·지번), user_tel1/2/3(전화), user_hp1/2/3(휴대폰) + check_sms·check_bunho·temp_sms·sms_number(SMS 인증/수신).
**사업자(세금계산서용):** user_job(직업), user_company(상호), user_number1/2/3(사업자번호), user_owner(대표자), user_uptae(업태), user_upjong(업종), user_c1zip/zone/addr1/addr2(회사주소1), user_ctel1/2/3(회사전화), user_cfax1/2/3(팩스), user_c2zip/zone/addr1/addr2(회사주소2/추가배송지 추정).
→ **회원 = 개인정보 + 사업자정보 + 다중주소** 구조. 신규 회원 모델/마이그레이션 매핑의 1차 소스.

### B-4. 회원 등급/할인
- 대시보드에 **회원할인율(%)** 표기 = 등급 기반 할인 필드 존재. (쿠폰 안내: "주문금액에서 회원할인 제외 후 추가 할인" → 회원할인이 1차, 쿠폰이 2차 적용 순서.)

### B-5. 증빙(tax01.asp)
결제수단별 발행: 신용카드=불가(매출전표로 세액공제), 계좌이체/무통장/에스크로=현금영수증·세금계산서 택일 발행.

### B-6. 탈퇴(out.asp)
아이디·비밀번호·이름·이메일·휴대전화·탈퇴사유. (제출 안 함.)

## C. 프린팅머니 (= 적립금/예치금)

### C-1. 조회 화면 (`/mypage/money.asp`)
- "현재 사용가능 금액" = 잔액(balance) 표기 1건.
- 적립/사용 **내역 테이블**(거래 history) 영역 존재(현재 "내역이 없습니다").
- 대시보드(main.asp)에도 "프린팅머니: N원" 요약 표기.

### C-2. 충전/적립/사용 동선
- 별도 "머니충전" 메뉴는 마이페이지 좌측에서 미관측(money.asp 단일). 충전은 주문/결제 흐름 내 또는 별도 페이지일 가능성 — **미발견(추가 탐색 필요)**.
- 쿠폰/이용권은 프린팅머니와 별개 화면(`/mypage/coupon.asp`).

### C-3. 1차 범위 시사
- 프린팅머니 = 잔액 + 거래내역(적립/사용/만료) 2요소. 마이그레이션 시 **회원별 잔액 + 트랜잭션 원장** 두 데이터셋 필요.
- 화면=잔액만 노출, 충전·적립 규칙은 화면 비노출 → 백오피스/결제 연동에서 확인해야 함(화면≠DB).
