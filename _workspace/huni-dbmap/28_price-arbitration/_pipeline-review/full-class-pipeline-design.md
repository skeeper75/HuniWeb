## 결론 먼저 (5줄)

1. **PRF_DGP_A 완주는 "1회성 영웅 작업"이 아니라 재사용 가능한 게이트 시퀀스를 낳았다** — G-DATA(GATE-S 구조 + GATE-A 의미 2단) → G-CHAIN → G-CALC → 벤치마크 → 정립 → 재검증. 이 시퀀스를 §1 템플릿으로 고정한다.
2. **횡단 결함 3종(D-2 멤버십·D-1 출력판형·D-1b prc_typ)은 클래스 고유가 아니라 스키마 전체 결함이다 — 라이브로 실증됨.** D-2: `opt_cd` 3,481행 전부 NULL + OPT_REF_DIM에 가격comp 참조 부재. D-1b: `prc_typ_cd` 전 comp가 `PRICE_TYPE.01` 단일값이고 단조성 시그니처로 총액형 의심 comp **22개**(후가공·명함박·완칼도무송) 적발. → §2 진입 체크리스트로 **먼저 스캔**해 클래스마다 재발견하는 낭비를 없앤다.
3. **중단없는 진행의 핵심 = "검증 차단"과 "승인큐로 미룸"의 분리** — 매핑 결함 *발견*은 멈추지 않고 진행하되, 그 결함의 *해소*(실 DB 적용·엔진 의미결정)는 승인큐로 빼서 후속 클래스 검증을 막지 않는다(§3).
4. **다음 착수 권고 = PRF_BIND_SUM(제본 합산형 4상품).** 레드 캡처(PRBKYPR 8조합) 보유로 경쟁사 벤치마크가 즉시 되는 유일한 클래스 + 합산형이라 PRF_DGP_A 패턴 직접 재사용(§4).
5. **신규 에이전트 0.** 기존 자산(verifier·arbiter·benchmark·validator·option-mapper·ddl-proposer·round-13)의 호출 순서·산출 경로만 템플릿화. 골든 케이스·RTM은 산출물(파일)이지 에이전트가 아니다(§5·§7).

## 2. 횡단 결함 카탈로그 → 진입 체크리스트 (S0)

PRF_DGP_A에서 발견한 D-2·D-1·D-1b를 **알려진 결함 패턴**으로 정식화. 각 클래스 S0에서 **먼저 이 4종을 라이브 스캔**해 "이 클래스에 해당 패턴이 있나"를 즉시 판정 → 클래스마다 처음부터 파헤치는 비용 제거.

### 2-1. 라이브 실증된 결함 패턴 (시그니처 검증 완료 2026-06-14)

| ID | 패턴 | 라이브 탐지 시그니처 | 검증 결과 | 영향 범위 |
|---|---|---|---|---|
| **D-2** | 옵션 선택→comp 멤버십 매핑 부재 (합산형에서 어느 comp를 합산할지 결정 불가 → 과대합산) | ① `SELECT count(*),count(opt_cd) FROM t_prc_component_prices` → opt_cd 전부 NULL이면 멤버십 미표현 ② `OPT_REF_DIM` base_code에 가격comp 참조타입(.08) 부재 | 🔴 **전역 confirmed** — opt_cd **3,481행 전부 NULL**, OPT_REF_DIM .01~.07만(가격comp 참조 없음) | 합산형 전 클래스(DGP_A~F·BIND·FOLD) |
| **D-1** | 단가행 siz_cd=출력판형인데 입력은 작업사이즈 → 직접 매칭 0행 | 상품 `t_prd_product_sizes.siz_cd` ∩ 그 comp `component_prices.siz_cd` = ∅ 이고, `t_siz_sizes.note`에 전지치수 존재 | ✅ PRF_DGP_A=전지 룩업으로 해소(SIZ_000499). 클래스마다 변환표 필요 여부 다름 | 디지털·면적형 등 출력판형 쓰는 클래스 |
| **D-1b** | `prc_typ_cd`가 단가형(.01)로 오적재됐으나 실제는 수량구간 고정총액형(가격표 "합가") → 단가×주문수량 과대합산 | `prc_typ_cd='PRICE_TYPE.01'`인 comp 중 min_qty↑ 시 unit_price **비감소**(총액형 신호; 진짜 단가형은 수량↑시 장당가 하락) | 🔴 **22개 comp 의심 confirmed** — prc_typ_cd가 **PRICE_TYPE.01 단일값**(전 comp 단가형). 단조성 위반 comp 22개: 후가공(CREASE/PERF/VARIMG/VARTEXT/CORNER_ROUND)·명함박(NAMECARD_FOIL_S1/S2 STD/HOLO)·완칼도무송(CUT_FULL_DIECUT/PERF) | **클래스 횡단** — PRF_DGP_A·NAMECARD·STK·GANGPAN·완칼 보유 클래스 |
| **D-3** | t_dsc 수량구간 할인 미연결(공식이 할인 참조하나 t_dsc 0행) | 상품 할인테이블 바인딩 vs `t_dsc_*` 행 존재 대조 | (PRF_DGP_A는 0행 — 별도 분리) | round-1 구간할인 적재 미반영 클래스 |

> **D-1b 횡단성의 의미(돈-크리티컬):** prc_typ_cd가 라이브 전체에서 PRICE_TYPE.01 단일값이라는 것은 **적재 시 prc_typ를 일괄 .01로 넣었다**는 뜻 = 총액형이어야 할 22개 comp가 전부 단가형으로 오적재. 클래스별로 재발견하면 22번 같은 진단을 반복하게 된다. S0 스캔 1회로 "이 클래스의 comp 중 D-1b 의심은 X·Y"를 즉시 리스트업 → S2 GATE-A가 그 comp만 가격표 권위로 ⓐ/ⓑ/ⓒ 판별.

### 2-2. S0 진입 스캔 표준 쿼리 묶음 (읽기전용·재현용)

각 클래스 S0에서 클래스 comp 집합(`formula_components` 경유)을 구한 뒤 아래를 돌려 `entry-scan.md`에 결과만 기록. **검증은 멈추지 않음** — 패턴 적발돼도 S1으로 진행하고, 해소는 승인큐(§3).

```
-- (A) 공식유형 판정: 클래스 공식의 frm_typ / comp 개수 → 합산형 vs 고정가형
-- (B) D-2 멤버십: 클래스 comp 중 차원-멤버십 일치(자동) vs 옵션-멤버십(부재) 분류
--     option_items에서 클래스 상품의 옵션값 → ref_dim_cd 분포(.08 없으면 멤버십 자리 없음 확정)
-- (C) D-1 출력판형: 상품 작업사이즈 siz_cd ∩ comp 단가행 siz_cd = ∅ 여부 + note 전지치수 유무
-- (D) D-1b prc_typ: 클래스 comp 중 prc_typ='01' & 수량구간 비감소(총액형 의심) 리스트
-- (E) D-3 할인: 상품 할인 바인딩 vs t_dsc 행 존재
```

> **S0 산출 = 결함 사전탐지표.** "이 클래스는 D-2 해당(합산형, 옵션 N그룹 멤버십 부재), D-1b 의심 comp 3개(오시1~3줄), D-1 변환 필요(전지 룩업), D-3 무관" 식 한 줄 요약. 이게 GATE-A의 검사 범위를 미리 좁혀 클래스당 분석량을 줄인다.

> **트레이드오프(스캔 비용 vs 재발견 비용):** S0 스캔은 클래스마다 라이브 쿼리 5묶음 추가 = 소량 비용. 그러나 이를 안 하면 GATE-A에서 매번 "왜 과대합산되나"를 바닥부터 추적(PRF_DGP_A가 겪은 22M원 진단). **재발견 비용 >> 스캔 비용** → S0 채택. 단, 시그니처는 "의심 신호"지 확정이 아님 — 확정은 항상 가격표 권위 대조(S2). S0는 범위 좁히기 도구.

## 3. 중단없는 진행 + 승인큐 분리

검증 파이프라인은 멈추지 않되, 사람 승인이 필요한 지점을 **승인큐(approval queue)**로 빼서 후속 클래스 검증을 막지 않는다.

### 3-1. 분류 기준 — "검증 차단" vs "큐로 미룸"

| 구분 | 정의 | 행동 | 예 |
|---|---|---|---|
| **검증 차단 (BLOCKING)** | 그게 없으면 *이 클래스의 검증 자체가 불가능* | 해당 클래스만 S7까지 못 감 → "차단" 마킹 후 **다음 클래스로 이동**(전체는 안 멈춤) | 참조 마스터가 round-13 MIS-LOADED(틀린 행 조회) → GATE-A A-2 차단 → round-13 해당 시트 선행 필요 |
| **승인큐 (DEFERRED)** | 결함은 *진단·정립 완료*, 남은 건 *실 DB 적용·엔진 의미결정*뿐 | `_approval-queue.md`에 항목 적재 + **클래스는 "정립 완료(적용 대기)"로 진행** | D-2 가설 A(option_items+.08)·D-1b ⓒ 정정·엔진 evaluate_price 구현 — 전부 인간/webadmin Phase11 |

**판별 규칙(한 줄):** *"이 결함을 풀지 않으면 다음 케이스/클래스의 검증 판정을 내릴 수 있나?"* → 내릴 수 있으면 DEFERRED(큐), 못 내리면 BLOCKING(차단·다음 클래스로).

### 3-2. 왜 D-2·D-1b가 BLOCKING이 아니라 DEFERRED인가 (중요)

PRF_DGP_A에서 D-2·D-1b는 *발견·진단·정립*이 끝났다(가설 A 채택·ⓒ 확정·재계산 44,330/104,330 재현). 남은 건 **실 DB 적용**(base_code 1행·option_items 12행·prc_typ 정정)과 **엔진 구현**(webadmin Phase11) — 둘 다 인간 승인 영역. → **DEFERRED.** 이 덕에 PRF_DGP_A를 "정립 완료(적용 대기)"로 닫고 **PRF_BIND_SUM으로 즉시 이동** 가능. 만약 모든 결함을 BLOCKING으로 잡으면 1클래스에서 DB 적용 승인을 기다리며 전체가 정지한다.

> **반례(BLOCKING 정당):** 어느 클래스의 참조 size/자재가 round-13에서 *아직 감사 안 됨 또는 MIS-LOADED 잔존*이면, 그 클래스 가격은 *틀린 마스터 행*을 조회하므로 재계산 판정 자체가 무의미 → BLOCKING. 이건 큐로 미루면 안 됨(틀린 값으로 GO 낼 위험).

### 3-3. 승인큐 구조 (`28_price-arbitration/_approval-queue.md` — 신규 산출)

| 항목 | 클래스 | 결함 | 정립안(GO) | 트랙 | 재검증 기준 | 컨펌 | 상태 |
|---|---|---|---|---|---|---|---|
| AQ-1 | PRF_DGP_A | D-2 멤버십 | 가설A: OPT_REF_DIM.08 + option_items 12행 | ddl-proposer→option-mapper(round-6) | 보정 하드코딩 없이 활성 comp 한정 재현 | C-2 확정 | 적용 대기 |

> **트레이드오프(큐 적재 vs 즉시 적용 요구):** 큐로 미루면 결함이 라이브에 *남아있는 채로* 후속 검증이 진행된다(가격뷰어가 아직 비활성이라 실청구 위험 0이지만, 적재된 데이터는 여전히 틀림). 대안 = 결함마다 즉시 적용 승인 요구 → 전체 정지. **권고: 큐 적재**(엔진 미구현 = 실청구 0이라 적용 시점 유예 안전) + 큐 우선순위는 "여러 클래스 공유 결함 우선"(D-1b 22comp가 1순위 — 한 번 고치면 다클래스 해소).

- **경쟁사 벤치마크가 즉시 되는 유일 클래스**(레드 PRBKYPR 8조합 캡처 보유) → S4 B-gate를 실제로 한 번 돌려보는 첫 사례. PRF_DGP_A는 경쟁사 🟡라 벤치마크 미검증이었음.

- 4상품 소규모 → 템플릿 일반화의 첫 회귀 테스트로 적합.

**2순위 = DGP_B/C/D/E (디지털 합산형 잔여 9상품).** PRF_DGP_A와 **동일 공유 comp·동형 결함(D-2·D-1·D-1b)** → 진단 재유도 0, 멤버십 표만 상품별로 확장. 골든값은 라이브 보강 필요(🔴).

**3순위 = PRF_POSTER_FIXED (28·최대·영향 최대).** 고정가형/면적매트릭스 대표. D-2 N/A(완제품 단가행 직접)라 GATE-A 검사축이 다름(차원조합 커버리지) → 템플릿의 "고정가형 분기"를 처음 작동시키는 사례. 면적 오모델 이력 있어 신중.

**4순위 = D-1b 횡단 정정 트랙(별도).** STK·NAMECARD·완칼의 D-1b 22comp를 클래스별로 만나기 전에 **한 번에 가격표 권위로 재판별**(승인큐 AQ-3). 클래스 검증과 병행 가능(독립 트랙).

> **의존 재배치 vs class-map §3 델타:** class-map은 PRF_DGP_A→BIND→POSTER 순이었는데, 본 재검토도 BIND 1순위로 동일(경쟁사 가용성이 결정적). 단 **DGP_B~E를 POSTER 앞으로** 올린다 — 동형 결함 상속으로 비용 최저·진도 빠름. POSTER는 검사축이 달라(고정가형) 별도 준비 필요하므로 합산형 잔여를 먼저 소진.

## 9. 미해소 컨펌 (사용자/실무진 판정)

- **C-RTM1 (BLOCKING 강도):** GATE-A A-2를 round-13 GO *필수 선행*으로 강제할지 vs 미GO 클래스는 "마스터 미정합 경고 + 조건부 계산"으로 둘지(arbiter-review C-1 승계). 보수=필수 선행이나 184 다수 미감사 시 진도 정체. **권고: 골든 케이스로 검증하는 클래스는 필수 선행(돈-크리티컬), 184 증분 흡수분은 경고 후 진행.**
- **C-D1b-SCOPE (신규·중대):** D-1b가 22 comp 횡단으로 confirmed됨. 클래스별로 만날 때마다 정정할지 vs **AQ-3 횡단 트랙으로 22 comp를 한 번에 가격표 권위 재판별**할지. **권고: 횡단 트랙**(클래스 검증과 병행·재유도 0). 단 prc_typ 정정 방식(ⓒ-1 .02+엔진 vs ⓒ-2 신규 .03)은 webadmin Phase11 결정 대기(C-D1b 승계).
- **C-NEXT (다음 착수):** 본 설계 권고 = PRF_BIND_SUM 1순위(경쟁사 즉시·합산형 재사용). 사용자가 영향 최대(POSTER 28)를 우선할지 vs 패턴 검증(BIND 4)을 우선할지 — **권고: BIND**(템플릿 일반화 회귀 + 경쟁사 첫 실행). POSTER는 고정가형 분기 준비 후.
- **C-GOLDEN-EXT (외부 골든):** 합산형은 단일 가격표 셀이 없어 외부 견적가 1:1 대조가 엔진 구현 후로 유예됨. 이 유예를 수용할지, 아니면 사이트 견적/레드 캡처로 외부 골든을 선확보할지 — **권고: BIND는 레드 캡처(🟢)로 외부 골든 선확보 가능 → BIND를 외부골든 첫 사례로**.
