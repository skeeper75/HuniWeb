## A. "추가(옵션)" 분류 적합성 — 핵심 쟁점 [판정: 부합]

### A.1 쟁점
mapper는 엑셀 "추가(옵션)" 컬럼(큐방/끈/각목+끈)을 `option_group`(OG-CHUGA)으로 처리. 설계문서 §1①은 "추가상품=템플릿(SKU)". 이 둘이 다른 개념이므로, mapper가 설계문서를 부합했는지 개념을 혼동했는지 판정.

### A.2 직접 출처 대조 (추측 배제)

**1차 출처(silsa-l1.csv line 87~92, 엑셀 명시):** "추가(옵션)" 컬럼의 세로 캐스케이드 값 =
`추가없음 / 큐방(4개)추가 / 끈(4개)추가 / 각목(900이하)+끈(4개) 추가 / 각목(900초과)+끈(4개) 추가`. 컬럼 헤더 원문 = `추가(옵션)_추가` + `추가(옵션)_추가옵션가격`.

**설계문서 §1① addon=template의 정의(line 17~20):** "단순 상품 링크 ❌ → *상품 + 옵션(예: **OPP봉투+사이즈+수량**)* 을 박제한 구성 템플릿... MES 전달 시 **별도 주문상세번호**". 즉 addon=template은 **독립적으로 판매·생산되는 완제상품**(OPP봉투처럼 그 자체가 상품)을 부모상품에 끼워 파는 것.

**설계문서 §1②/§3② option의 정의(line 24~26, 50~51):** "옵션 = 상품에 등록된 차원행(자재/공정/사이즈)을 참조하는 묶음", "옵션 재료는 **그 상품에 등록된 차원 내에서만**".

### A.3 판정 근거 (3축 분류)

| 옵션값 | 물리 정체 | 환원 차원 | addon(완제상품)인가 option(부속/공정)인가 |
|---|---|---|---|
| 큐방(4개) | 현수막에 박는 부착 부속 | 부착 공정 PROC_000081 | option — 독립판매 완제품 아님, 부착 공정으로 환원 |
| 끈(4개) | 현수막에 다는 끈(부착) | 부착 공정 PROC_000081 | option — 부착 공정으로 환원 |
| 각목+끈 | 현수막 거치용 나무막대+끈 | 끈=공정 / 각목=셋트(set, 물리부속) | option(복합) — 각목은 부속이지 OPP봉투 같은 독립상품 아님 |

**부합 근거 3건:**
1. **엑셀 컬럼명 자체가 "추가(옵션)"** — 작성자(후니 실무진)가 이를 "옵션"으로 명명. 엑셀 명시값=권위(메모리 [dbmap-domain-knowledge]) → option_group 처리는 엑셀 권위 정합. addon=template은 설계문서가 "OPP봉투+사이즈+수량"으로 예시한 **독립 완제상품**이고, 큐방/끈/각목은 독립 판매 SKU가 아니다.
2. **"그 상품에 등록된 차원 내에서만"(§3②) 충족** — 큐방·끈=부착 공정(PROC_000081, 라이브 EXISTS), 각목=셋트 차원으로 환원. 전부 차원으로 resolve 가능 → §0 제1원칙("모든 선택은 자재/공정으로 환원") 충족.
3. **설계문서 §1② 택일그룹 일반화** — "옵션그룹 = 옵션 모음 + 택1/택N → 공정 택일그룹의 일반화". 추가(옵션)이 택1 캐스케이드(한 셀=한 값)이므로 SEL_TYPE.01 택일그룹 = 정확히 이 일반화의 사례.

→ **개념 혼동 아님.** mapper가 §10 D-4에서 "각목 귀속: set(부속) vs addon(template)"을 명시적으로 자문하고 "각목=부속이므로 set 1차 권고"로 라우팅한 것은 두 개념을 구분하고 있다는 증거다. 거치대(독립 완제상품 성격)는 R6에서 명시적으로 제외(워크스루의 가상 차용 폐기) — 진짜 addon 후보를 오히려 보수적으로 배제했다.

### A.4 반대 가능성 점검 (적대적)
각목은 "나무막대 완제품"이라 addon(template)일 여지가 있는가? — 점검 결과 **아니다**: 각목은 단독 판매 카탈로그 상품(ref-products 0행)이 아니라 현수막 거치 부속이고, 엑셀이 "추가(옵션)" 한 셀에 "각목+끈"으로 묶어 표현했다(독립 주문상세번호 단위가 아님). mapper의 set 귀속은 정당. **단 D-4는 사용자 확인 후보로 정직하게 열어둠** — 이는 침묵 결정이 아니라 에스컬레이션이므로 결함 아님.

**A 판정: ✅ 부합. 개념 혼동 없음. 추측 없음(엑셀 컬럼명 권위).**

## B. 비치수 범위·조건부 수치를 constraint로 본 것 [판정: 부분 결함 — 설계 정정 완료]

### B.1 쟁점
R-SIZE-NONSPEC(가로 500~1750)·R-GAKMOK-HEIGHT·R-BONGJE-PARAM이 설계문서 §3③ constraint 의도("차원값 간 호환/금지/필수동반")에 맞는가, 아니면 사이즈 차원 속성(범위)이라 constraint 모델 오용인가.

### B.2 직접 대조
- **설계문서 §3③(line 69, 73):** "임의 옵션/차원 조합 + 불리언 로직(호환·금지·필수)", **"한계: JSONLogic은 *검증*(이 조합 OK?)용"**. 즉 constraint = **차원값 간 관계** 검증용.
- **라이브 admin 실측(live-admin-groundtruth §2.2 LV-1/2):** 폼빌더 표준 var = 차원 코드 7종(siz_cd·proc_cd 등)뿐. **연속 수치(width/height)·모드 플래그 var 부재.**
- **1차 출처(silsa-l1.csv row109):** 비규격 가로 500~1750·세로 500~5000 = **사이즈 차원의 입력 범위 속성**.

### B.3 판정
비치수 가로/세로 범위는 **차원값 간 관계가 아니라 사이즈 차원 자체의 수치 범위 속성**이다 → 당초 constraint로 모델한 것은 **모델 오용**이 맞다. **그러나 결정적으로, mapper가 이 오용을 스스로 적발·정정했다:** `silsa-live-reconciliation.md` LV-2에서 3 constraint를 전부 **GAP(표현 불가)**으로 분류하고 **라이브 적재 0건**(`_GAP.csv` 격리, `status=GAP`)으로 처리. constraint_json은 NULL 유지. 검증처를 products 범위 컬럼(A)/비표준 var JSONLogic(B)/앱 런타임(C)으로 ddl-proposer·도메인에 라우팅(침묵 선택 안 함).

→ **모델 오용은 사실이나, 그 오용분이 라이브로 적재되지 않고 GAP으로 격리됨.** 적대 감사 관점에서 "잘못 만든 규칙이 적재될 위험"은 **0**이다. mapper의 자기정정(2026-06-08 reconciliation)이 이 결함을 이미 닫았다.

### B.4 잔존 결함
**[B-1 · MINOR]** `t_prd_product_constraints_GAP.csv`가 `logic_target_spec` 컬럼에 비표준 var JSONLogic(`size_mode/width/height`)을 보존한다. 의도 보존은 정당하나, 이 파일이 실수로 적재 glob에 포함되면 라이브 부재 컬럼(`logic_target_spec`은 라이브 `logic`과 다른 이름)이라 즉시 실패한다. 파일명 `_GAP` 접미·`status=GAP` 컬럼으로 격리는 돼 있으나, **적재 로더가 `_GAP.csv`/`_BLOCKED.csv`를 명시적으로 제외하는 글롭 규칙이 적재 매니페스트에 문서화됐는지** 확인 필요. → 라우팅: **mapper/load-builder**(매니페스트에 제외 글롭 명시).

**B 판정: ⚠️ 모델 오용 사실이나 self-correct로 라이브 적재 0건 격리. 정합. 잔존 B-1(MINOR).**

## C. [CONFIRM] 추측의 INSERTABLE 혼입 [판정: 부합 — 추측 누출 0]

### C.1 쟁점
양면테입→{대상:테입}·큐방→{대상:큐방}·각목=set 등 `[CONFIRM]` 항목이 option_items INSERTABLE 9행에 포함. 추측이 "적재 가능"으로 분류돼 라이브에 새어들 위험이 있는가.

### C.2 직접 대조 — 무엇이 실제로 적재되는가
**load_silsa/t_prd_product_option_items.csv 실제 9행(직접 확인):**
```
컬럼 = prd_cd,opt_cd,item_seq,ref_dim_cd,ref_key1,ref_key2,qty,use_yn
양면테입  → OPT_REF_DIM.04, ref_key1=PROC_000081, ref_key2=(빈칸)
큐방(4개) → OPT_REF_DIM.04, ref_key1=PROC_000081, ref_key2=(빈칸)
```
**결정적 사실:** 적재 CSV에는 `ref_param_json` 컬럼이 **없다**(라이브 부재, R3·LV-5 확증). `{대상:테입}`·`{대상:큐방}` 같은 **`[CONFIRM]` 추측은 적재 CSV 어디에도 들어가지 않는다.** 실제 적재되는 값은 `ref_key1=PROC_000081`(부착 공정)뿐이고, 이건 추측이 아니라 **확실**이다 — 큐방·양면테입·끈 전부 물리적으로 "부착" 행위이고 부착 공정은 라이브 EXISTS이며 트리거가 검사하는 것은 ref_key1=proc_cd의 차원행 존재뿐.

### C.3 판정
- 트리거 `fn_chk_opt_item_ref`가 검사하는 것 = `(ref_dim_cd, ref_key1)`가 차원행으로 EXISTS. → PROC_000081은 EXISTS → INSERTABLE은 **확실**(추론 아님).
- `[CONFIRM]`이 붙은 것은 **param 의미**(테입/큐방 enum 정합)뿐인데, **param은 적재되지 않는다**(ref_param_json 부재). → **추측은 적재 경계를 넘지 못한다.** option/note 컬럼의 `[CONFIRM]` 텍스트는 프로비넌스(설계 메모)일 뿐 라이브 데이터 아님.

→ **추측의 라이브 누출 0.** "INSERTABLE = 추측 적재"가 아니라 "INSERTABLE = ref_key1 확실, param은 애초에 미적재". mapper가 §4.2에서 "큐방은 enum 미존재나 ref_key1=081은 INSERTABLE — item은 proc_cd EXISTS만 검사, param은 트리거 미검사"로 정확히 구분.

### C.4 잔존 결함
**[C-1 · MINOR]** 양면테입을 부착 공정(PROC_000081)으로 환원한 것은 **도메인 추론**이다(L1 "양면테입" ≠ enum "테입", 또한 "양면테입"은 부착 부속이 아니라 접착 자재일 수도 있음). ref_key1=081 자체는 트리거 통과하나, **양면테입이 정말 "부착" 공정인지 vs 별도 자재/공정인지**는 엑셀 명시 매핑이 아니다. mapper가 `[CONFIRM]`(§10 #3)으로 정직 표기했으므로 침묵 추측은 아니나, INSERTABLE로 분류돼 적재 후보에 든 점은 사용자 확인 전 적재 시 잘못된 공정 귀속이 라이브에 들어갈 여지. → 라우팅: **사용자/도메인 확인**(양면테입=부착 공정 환원 정당성). 단 이는 ref_key1 레벨이라 적재돼도 트리거 무결성은 유지(MINOR).

**C 판정: ✅ 부합. 추측(param)은 적재 경계 밖. 잔존 C-1(MINOR, 양면테입 공정 귀속 확인).**
