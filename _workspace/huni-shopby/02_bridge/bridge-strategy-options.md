# bridge-strategy-options.md — 브리지 전략 후보 A~D × 5축 트레이드오프

> 전략 강요 금지 — 하네스가 후보+트레이드오프를 제시, 권고·확정은 architect/gate가 수렴.
> 5축: 스펙 가능성 · 무손실성 · 운영부담 · 정산/세금/클레임 정합 · 위젯 적합.
> 근거: `shopby-product-model.md`·`product-price-bridge-spec.md`(스펙 인용 포함). 작성: 2026-06-25.

---

## 0. 공통 전제 (모든 전략에 적용되는 스펙 사실)

1. Shopby 카트/주문 라인에 **가격 입력 필드 없음**. 라인가 = 서버가 `salePrice+addPrice(−할인)` 재산출
   (`product-price-bridge-spec §3.1`, M-PRICE-1).
2. 가격 변동 표현 = **옵션행(addPrice) 추가** 또는 **추가상품(productNo) 추가**뿐 (M-OPT-1, M-EXTRA-1).
3. calculate가 라인가를 무조건 재산출 → 후니 계산가 생존 = "후니가=Shopby 등록가" 동기화 시에만 (§3.3).
4. 사양 텍스트(`optionInputs.inputValue`)·식별·수량은 어느 전략에서나 무손실 전달 가능.

## 1. 전략 정의 (재확인)

- **전략 A — 카탈로그 정식 동기화**: 후니 상품/CPQ 옵션을 Shopby 상품/옵션으로 등록·동기화. 가격분기 조합마다
  옵션행(addPrice). Shopby 가격을 그대로 활용. (= `product-price-bridge-spec` 경로 P-A)
- **전략 B — 컨테이너 상품**: Shopby에 상품군당 대표상품 소수, 구성·계산가는 옵션/추가상품/customProperty로.
  가격은 후니 권위 유지 시도. (가격 주입 한계는 P-C/P-D 의존)
- **전략 C — 커스텀가 직접 주입**: 후니 계산가를 라인 가격으로 직접. (스펙상 라인 가격 필드 없음 → **순수형 C는 불가**,
  실현형 = 주문 직전 옵션/상품 동적 생성으로 가격을 등록가화 = P-B)
- **전략 D — 혼합**: 노출/검색=동기화(A), 가격=주문 직전 동적 등록(B/P-B), 사양=텍스트(F5)+메타(F7) 조합.

## 2. 5축 트레이드오프 표

| 축 \ 전략 | A 카탈로그 동기화 | B 컨테이너 상품 | C 커스텀가 직접주입 | D 혼합 |
|-----------|------------------|-----------------|---------------------|--------|
| **① 스펙 가능성** | ✅ 가능. `put-product-options addPrice`(`product-server-public.yml:1804`)로 옵션행 등록. 단 조합 유한·열거 가능해야 | ⚠️ 상품/옵션 등록은 가능하나 **라인 임의가 주입 필드 없음**(§3.1) → 가격은 결국 P-C/P-D 동기화 의존 | ❌ **순수형 불가**(라인 가격 필드 0건). 실현형=주문 직전 옵션 동적 생성(P-B)만 | ✅ A의 노출 + P-B의 가격. 스펙 조합으로 성립 |
| **② 무손실성** | ⚠️ 조합 열거 가능 범위만 무손실. CPQ 연속/대규모 조합은 옵션 폭발로 사실상 손실 | ❌ 가격 무손실 보장 못함(컨테이너 1가격 동시성 충돌, P-C). 사양은 텍스트로 무손실 | ✅ 가격 무손실(라인=정확히 후니 계산가, P-B 성립 시) | ✅ 가격 무손실(P-B) + 노출 무손실(A) |
| **③ 운영부담** | 🔴 높음. 전 상품×전 조합 옵션 동기화·재고/가격 변경 전파·심사(judgement) | 🟡 중. 카탈로그 단순(대표상품), 그러나 가격 정합 로직·동시성 처리 추가 | 🔴 높음. 주문마다 옵션 동적 생성+잔존 옵션 청소+동시성+심사(OQ-3) | 🔴 가장 높음. A 동기화 + B 동적 가격 둘 다 운영 |
| **④ 정산/세금/클레임 정합** | ✅ 가장 안전. Shopby 등록가=실청구가=정산 기준 일치(추정, OQ-4). surtaxType·commission 정상 | 🔴 위험. Shopby 라인가 ≠ 후니 권위가면 정산이 어느 가격을 쓰는지 미상(OQ-4). 클레임 환불액 불일치 위험 | 🟡 P-B로 라인가=후니가면 정합. 단 동적 옵션의 정산 추적성·취소시 옵션 상태 미검증 | 🟡 P-B 라인가 일치 시 정합. 복잡도가 정합 검증 부담↑ |
| **⑤ 위젯 적합** | 🔴 낮음. 위젯 CPQ 자유선택 ↔ 사전열거 옵션 불일치(위젯이 못 만든 조합=주문 불가) | 🟡 중. 위젯이 사양·계산가 산출 후 컨테이너+텍스트로 전달. 가격 권위는 위젯/후니 | ✅ 높음. 위젯 계산가를 그대로 주문가로(P-B). evaluate_price 단일 권위 보존 | ✅ 높음. 위젯=가격권위(P-B), Shopby=노출/주문관리 |

범례: ✅ 유리 / 🟡 보통·조건부 / ⚠️ 제약 있음 / 🔴 부담·위험 / ❌ 불가.

## 3. 전략별 핵심 리스크 한 줄

- **A**: CPQ 조합 폭발(후니 가격은 사이즈/수량/자재/후가공 연속·매트릭스 — `engine-contract §9` 면적매트릭스/구간형 →
  유한 옵션으로 열거 시 수천~수만 행, 위젯이 만드는 조합을 사전에 다 등록 불가).
- **B**: 가격 주입 필드 부재로 "컨테이너+자유금액"이 스펙상 성립 안 함 → 결국 P-B 동기화 필요 → B 단독 의미 약함.
- **C**: 순수 직접주입은 스펙상 불가. 실현은 P-B(주문 직전 동적 옵션)로 귀결.
- **D**: 가장 강한 무손실+위젯 적합이나 운영/정합 복잡도 최고 → 동적 옵션 생성(P-B)의 정합·동시성·심사 검증이 게이트.

## 4. enterprise 보강 — 전략별 영향 갱신 (option.mdx·judgement.mdx 정독)

> 1차 산출 후 enterprise 운영 매뉴얼(`product/add-list/add/sale-info/option.mdx`·`product/judgement.mdx`)을
> 정독해 스펙 추론을 운영 권위로 확증/갱신. 새 사실 5건.

| ID | enterprise 사실 | 근거 | 영향받는 전략·축 |
|----|----------------|------|-----------------|
| E1 | 선택형 옵션 **최대 5개**·텍스트 옵션 최대 5개·옵션가=고정값 | option.mdx:44-45,60-62 | **A ①스펙가능성 하향** — 후니 CPQ 축이 5 초과(자재·사이즈·도수·후가공·코팅·묶음…)면 일체형 단일 상품에 못 담음. 상품 분할 필요 |
| E2 | **선택옵션 = "추가상품 판매 등으로 활용"**(native) | option.mdx:88 | **B/D ②무손실 상향** — 후가공 가산을 추가상품 API 대신 native 선택옵션(addPrice)으로 표현 가능. 운영부담 ↓ |
| E3 | **상품 정보 수정 시 "수정 후 승인대기"**(상품심사) | judgement.mdx:9,35 | **C/D(P-B) ①스펙가능성·③운영부담 악화** — 주문 직전 옵션 동적 변경이 판매 차단 유발 위험. 자동승인 예외 없으면 **P-B 사실상 불가** = C/D 핵심 관문(M-JUDGE-1) |
| E4 | native 수량할인 = **필수옵션 수량·금액 기준** 정률/정액 | option.mdx:162 | **전 전략 ④정합** — 후니 "구간단가"(구간별 다른 단가)와 모델 상이. 후니 final_price에 이미 반영하고 Shopby 할인=0 두는 설계가 무손실(OQ-5) |
| E5 | **세트 상품 세트옵션은 분리형(필수/선택) 불가** | option.mdx:157 | 후니 셋트(`evaluate_set_price`) 매핑 시 옵션 구성 제약 — 셋트 상품군 별도 취급 |
| E6 | 정산/세금계산서/거래명세서 **전부 NATIVE**(Shopby) | recommendations.md:61-63 | **B/D ④정합 위험 재확인** — 정산이 Shopby 라인가 기준일 개연성↑. 후니 권위가 분리(placeholder) 시 정산 왜곡 위험(OQ-4 미해소) |

> **갱신 함의:** ① 전략 A는 "축≤5·유한조합" 상품군에만 무손실(E1). ② 후가공 가산은 native 선택옵션이 1차
> 경로(E2)라 추가상품 API 의존이 줄어 D의 운영부담 일부 완화. ③ **그러나 P-B(동적 옵션 가격)는 상품심사(E3)가
> 가로막을 수 있어 D의 가격 메커니즘 자체가 위태** — D 채택은 "addPrice 변경이 자동승인되는가"(OQ-3) 확인이 선결.

## 5. 미수렴 결정 사항 (architect/gate로)

1. CPQ 조합이 유한·열거 가능한 상품군(예 고정가형·소조합·축≤5)과 연속·매트릭스형(면적·구간) 상품군을 **전략 분리**할지
   (전자=A, 후자=D/P-B). → 상품군별 하이브리드. ★토대 §2 재실측: 카트가능 GREEN ≈ 60상품(스티커·포스터·엽서/카드·
   책자 등)부터 — 이 중 고정가형/소조합을 전략 A 파일럿 후보로.
2. P-B(주문 직전 동적 옵션 생성)가 상품심사 "수정 후 승인대기"를 유발하는지(E3·M-JUDGE-1) = **D 채택의 관문**(OQ-3).
   자동승인 예외 없으면 D는 성립 불가 → 전략 A(사전 옵션 동기화)로 회귀.
3. 정산이 라인 salePrice/addPrice를 권위로 쓰는지(E6·OQ-4) 확인 후 B/D의 가격 분리 허용 여부 = **정합 관문**.
4. evaluate_price 단일 권위 계약 보존(후니 directive) ⇒ Shopby 가격을 권위로 쓰지 않는다면 D/P-B 외 선택지 사실상 없음.
   단 P-B가 E3로 막히면, **남는 무손실 길 = "후니 final_price를 사전 옵션 addPrice로 일괄 동기화(A)"** 뿐 →
   조합 폭발(E1)과 동기화 부담(③)을 감수하거나, 가격 권위 분리(B/D)의 정산 정합(E6)을 architect가 재정의.
5. 후가공/추가옵션 가산을 native 선택옵션(E2)으로 표현할지 vs 추가상품 API(M-EXTRA-1)로 할지 — E2가 운영 단순(권장 검토).

> ★권고 아님(트레이드오프만). architect가 상품군별 적합 전략을 수렴, gate(SB2)가 무손실·계산가 생존을 재검증.
