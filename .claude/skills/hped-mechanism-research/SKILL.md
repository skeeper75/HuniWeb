# hped-mechanism-research — 가격 장치 원리 정의·지식격차 방법론

5개 가격 장치의 역할을 원리로 정의하고, 아는 것과 모르는 것을 분리한다. **결론이 아니라 이해**가 목적이다.

## 왜 이 방법론인가

장치의 역할을 정확히 정의하지 않으면 잘못 적재된다. "검증"은 정의가 선 뒤에야 가능하다. 따라서 정의 단계에서 확신도를 명시하고, 모르는 것을 정직하게 드러내야 후속 검증·적재가 오판하지 않는다.

## 장치 역할 정의 — 6요소 템플릿

각 장치(가격공식·가격구성요소·할인테이블·가격뷰어·가격시뮬레이터)를 다음 6요소로 정의한다:

| 요소 | 질문 |
|------|------|
| **목적** | 이 장치는 가격계산에서 무엇을 책임지는가 (한 문장) |
| **입력** | 무엇을 받는가 (어느 테이블/컬럼/사용자 선택) |
| **출력** | 무엇을 내놓는가 (값·구조) |
| **연결** | 다른 4개 장치와 어떻게 이어지는가 (데이터 흐름) |
| **경계** | 이 장치가 책임지지 **않는** 것 (오해 방지) |
| **확신도** | 아래 4단계 중 (출처 명기) |

## 확신도 4단계 [HARD]

모든 정의 항목·메커니즘 진술에 반드시 부착한다:

- `[확정·코드근거]` — pricing.py/price_views.py에 실제로 그렇게 구현됨 (file:line).
- `[확정·설계산출물]` — docs/prcx01·pricing-erd에 명시 (코드 미확인 시 코드 확인 필요 표기).
- `[추정·도메인]` — 인쇄 가격 도메인 원리상 그럴 것 (라이브 미확인).
- `[미지·리서치필요]` — 모름. 컨펌질문으로 큐잉.

> 미지를 확정처럼 쓰는 것이 이 하네스가 막으려는 실패다. 애매하면 낮은 확신도로.

## 조합 메커니즘 서술 원칙

후니 관리자가 상품요소를 조합해 가격을 만드는 흐름을 서술하되:
- **추상 일반론 금지** — 분석된 파일럿 상품군(엽서 합산형·현수막/실사 면적매트릭스·아크릴 면적+두께)이 실제로 각 장치를 어떻게 타는지 구체 예로.
- **어느 장치가 어느 차원을 책임지는지** 명시 (예: 면적매트릭스의 가로×세로는 component_prices의 siz_width/siz_height 티어, 두께는 mat_cd 비수량 차원).
- **합쳐지는 메커니즘** (합산형=원자 룩업 후 합, 면적=매트릭스 셀 단가, 할인=순차곱) 서술.

## 지식맵 산출 (knowledge-map.md)

2분할:
- **아는 것** — `[확정]` 항목 종합. 검증·적재가 신뢰할 토대.
- **모르는 것** — `[추정]`·`[미지]` 항목 + 각각에 컨펌질문(누가 답할 수 있는가: 코드실측/도메인/사용자).

## 권위 순서 [HARD]
① 라이브 코드(실제 동작) ② 설계 산출물(의도) ③ 라이브 스키마/데이터 ④ 인쇄 도메인(보강·갭만). 코드와 설계가 어긋나면 발견으로 기록(code-schema-auditor와 조율).
