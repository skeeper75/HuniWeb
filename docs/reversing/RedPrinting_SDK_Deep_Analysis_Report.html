1. 핵심 결론: 왜 productRedWidgetSDK.js는 난독화하지 않았는가?

결론: productRedWidgetSDK.js는 의도적으로 가볍게 유지된 "연결 다리(Bridge)" 역할이며, 보호할 핵심 로직이 포함되어 있지 않습니다. 진짜 비즈니스 로직은 CDN에 호스팅된 별도 파일(widget.js)에 미니파이되어 있습니다.

난독화하지 않은 이유 3가지

① 보호할 로직이 없음 — 이 파일에는 자체 fetch 호출이 0건이고, 가격 계산/API 통신/데이터 처리 로직이 전혀 없습니다. 단순히 RedWidgetSDK 인스턴스를 생성하고 콜백을 연결하는 글루(glue) 코드입니다.

② 핵심 로직은 CDN 별도 파일에 분리됨 — widget.js(438KB)가 Vue 3 앱으로서 모든 UI 렌더링, 상태 관리, API 통신, 가격 계산을 담당합니다. 이 파일은 이미 미니파이되어 있어 역공학이 어렵습니다.

③ 서버 측 보안 의존 — 가격 조회는 공개 API이고, 주문/결제는 서버 세션(쿠키)으로 보호됩니다. 클라이언트 코드 난독화가 보안에 실질적 기여를 하지 않습니다.

8.2 암호화가 없는 이유 분석

① 가격 조회 API는 공개 정보

가격 계산 API(get_ajax_price_vTmpl)가 반환하는 데이터는 공개 가격표 정보입니다. 누구나 웹사이트에서 옵션을 변경해 볼 수 있으므로, 이 데이터를 암호화할 필요가 없습니다.

② 민감한 작업은 서버 세션으로 보호

실제 주문/결제는 로그인 후 서버 세션 쿠키로 인증됩니다. initProductData.member에서 비로그인 상태는 GUEST@redprinting으로 식별되며, 주문 시 로그인이 요구됩니다.

③ HTTPS가 전송 중 암호화를 담당

TLS/HTTPS가 모든 네트워크 통신의 전송 구간 암호화를 담당합니다. 추가적인 애플리케이션 레벨 암호화는 이중 보호(defense-in-depth)를 제공하지만, 이 유형의 서비스에서는 일반적으로 불필요합니다.

④ 클라이언트 코드 난독화의 한계

productRedWidgetSDK.js를 난독화해도 브라우저 DevTools로 widget.js의 Shadow DOM 내부와 Pinia 스토어, 네트워크 요청을 모두 확인할 수 있으므로, 난독화의 실질적 보안 효과가 없습니다.
